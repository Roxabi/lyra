## Source

GitHub issue #320 — "feat(core): enable token-level streaming from Claude CLI to adapters". User testing confirmed `--include-partial-messages` works in interactive stream-json mode, emitting `stream_event/content_block_delta` NDJSON lines with incremental text chunks.

## Problem

Lyra buffers complete LLM responses before sending them to users. The Claude CLI already supports token-level streaming, and the downstream plumbing (hub dispatch, adapter edit-in-place, outbound dispatcher) is fully implemented. The gap is in the CLI protocol layer → driver → agent path: no code parses `stream_event` deltas, and no method returns an `AsyncIterator[str]` from the CLI process.

### Current data flow (non-streaming)

```
user message
  → pool_processor._process_one()
    → agent.process(msg, pool, on_intermediate=cb)
      → provider.complete(pool_id, text, model_cfg, system_prompt)
        → CliPool.send()
          → cli_protocol.send_and_read()
            → read_until_result()  # blocks until "result" event
              ← CliResult(result=full_text)
            ← CliResult
          ← CliResult
        ← LlmResult
      ← Response(content=full_text)
    → dispatch_response(msg, response)
```

### With streaming (what we need)

```
user message
  → pool_processor._process_one()
    → agent.process(msg, pool)
      → provider.stream(pool_id, text, model_cfg, system_prompt)
        → CliPool.send_streaming()
          → cli_protocol.send_and_read_stream()
            ← AsyncIterator[str]  (yields content_block_delta texts)
          ← AsyncIterator[str]
        ← AsyncIterator[str]
      ← AsyncIterator[str]
    → dispatch_streaming(msg, iterator, outbound)  # already implemented
```

### Downstream plumbing (READY — no changes needed)

| Layer | File | Method | Status |
|-------|------|--------|--------|
| Pool processor | `pool_processor.py:220` | `isinstance(result, AsyncIterator)` check → `dispatch_streaming()` | Ready |
| Hub dispatch | `hub_outbound.py:142` | `dispatch_streaming(msg, chunks, outbound)` | Ready |
| Outbound dispatcher | `outbound_dispatcher.py:88` | `enqueue_streaming(msg, chunks, outbound)` | Ready |

## Outcome

When streaming is enabled for an agent, users see response text appear progressively (word-by-word) in Telegram and Discord via edit-in-place, instead of waiting for the full response — particularly impactful for long-form responses (code, summaries, reasoning chains) where wait time is most noticeable. Non-streaming agents continue to work exactly as today.

## Appetite

1-week cycle. The downstream plumbing is ready — work is concentrated in 6 files across CLI protocol, driver, agent, and config layers.

## Shapes

### Shape 1: Parallel streaming path (separate methods)

Add dedicated streaming methods alongside existing non-streaming ones:

- `cli_protocol.py`: new `send_and_read_stream()` → async generator that yields `content_block_delta` texts, handles `system/init` and `result` for session management
- `cli_pool.py`: new `send_streaming()` → returns `AsyncIterator[str]`
- `cli.py`: new `ClaudeCliDriver.stream()` → returns `AsyncIterator[str]`
- `simple_agent.py`: `process()` returns `AsyncIterator[str]` when `streaming=True` in agent config
- `cli_pool_worker.py`: add `--include-partial-messages` to `_build_cmd()` when streaming enabled
- `agent_config.py`: add `streaming: bool = False` to `ModelConfig`

**Trade-offs:**
- Pro: Clean separation — non-streaming path is untouched, zero risk of regression
- Pro: Natural fit — downstream already expects `AsyncIterator[str]`, `pool_processor.py:220` already routes it
- Pro: Easy to test each layer independently
- Con: Some duplication between `send_and_read()` and `send_and_read_stream()` (session_id extraction, timeout handling)

**Rough scope:** M

### Shape 2: Unified path with dual return type

Make existing methods accept a `streaming: bool` kwarg and return `CliResult | AsyncIterator[str]`:

- `read_until_result(streaming=True)` → returns `AsyncIterator[str]` instead of `CliResult`
- `CliPool.send(streaming=True)` → returns `CliResult | AsyncIterator[str]`
- `ClaudeCliDriver.complete(streaming=True)` → same

**Trade-offs:**
- Pro: DRY — single code path, less duplication
- Con: Complex union return types break the `LlmProvider` protocol (currently returns `LlmResult`)
- Con: Interleaved logic — streaming and non-streaming branches in the same function makes `read_until_result` even more complex (already 150 lines)
- Con: Locking: `CliPool.send()` holds `entry._lock` and awaits the full result — streaming must release the lock after writing stdin but before the iterator is consumed, fundamentally different lifecycle

**Rough scope:** M (but higher risk)

### Shape 3: Callback promotion (extend on_intermediate)

Repurpose the existing `on_intermediate` callback to fire per-token instead of per-turn:

- `read_until_result()` calls `on_intermediate(delta_text)` for each `content_block_delta`
- No new methods needed — just handle `stream_event` in the existing event loop

**Trade-offs:**
- Pro: Minimal API surface change — no new methods, no new return types
- Con: Semantic mismatch — `on_intermediate` means "intermediate turn" (multi-turn), not "token chunk". Conflating these breaks the abstraction
- Con: Downstream plumbing expects `AsyncIterator[str]` — would need a callback→iterator bridge (`asyncio.Queue` as adapter), adding complexity
- Con: Doesn't compose — can't have both intermediate turn callbacks AND token streaming simultaneously (multi-turn streaming scenario)

**Rough scope:** S (but architecturally wrong)

## Fit Check

```mermaid
flowchart TD
    subgraph "Shape 1: Parallel Path"
        S1A[send_and_read_stream] -->|AsyncIterator| S1B[CliPool.send_streaming]
        S1B -->|AsyncIterator| S1C[ClaudeCliDriver.stream]
        S1C -->|AsyncIterator| S1D[SimpleAgent.process]
        S1D -->|AsyncIterator| S1E[pool_processor dispatch_streaming]
    end
    subgraph "Shape 2: Unified"
        S2A["read_until_result(streaming=True)"] -->|Union type| S2B["CliPool.send(streaming=True)"]
        S2B -->|Union type| S2C["complete(streaming=True)"]
        S2C -.->|breaks LlmProvider protocol| S2D[Protocol change needed]
    end
    subgraph "Shape 3: Callback"
        S3A[on_intermediate per token] -->|callback| S3B[Queue bridge]
        S3B -->|AsyncIterator| S3C[dispatch_streaming]
        S3A -.->|semantic conflict| S3D[intermediate turns]
    end
```

**Shape 1 is the clear winner:**

1. **Architectural fit**: The downstream plumbing already expects `AsyncIterator[str]` — Shape 1 produces exactly that. `pool_processor.py:220` already has `isinstance(result, AsyncIterator)` routing.

2. **Locking lifecycle**: `CliPool.send()` holds `entry._lock` for the full request/response cycle. Streaming requires a different lifecycle: acquire lock → write stdin → release lock (iterator consumed externally). A separate `send_streaming()` method handles this naturally.

3. **Risk**: Shape 1 doesn't touch the existing non-streaming path at all. Shape 2 risks regression via union return types breaking the `LlmProvider` protocol and the `entry._lock` lifecycle mismatch. Shape 3 breaks the `on_intermediate` abstraction.

4. **Constraints**: Backward compat (opt-in per agent) is naturally satisfied — `ModelConfig.streaming` gates which method is called. In `SimpleAgent.process()`, the flag determines whether `provider.stream()` or `provider.complete()` is called; `pool_processor.py:220` then routes the return type accordingly.

*Dashed arrows (-.->)  in the diagram above indicate incompatibility / architectural breakage.*

**Eliminated:**
- Shape 2: Union return types, protocol breakage, locking lifecycle mismatch
- Shape 3: Semantic conflict with `on_intermediate`, requires callback→iterator bridge, can't compose with multi-turn

## Edge Cases (from architect review)

Four contracts the spec must define:

1. **Cancel-in-flight with live streaming iterator.** `_process_with_cancel()` cancels `agent_task` when a new message arrives. If `dispatch_streaming` is mid-iteration, `CancelledError` fires into the generator, leaving subprocess stdout unread. The OS pipe buffer fills (`limit=1024*1024`), the subprocess blocks on write, and the next `send()` hits the idle timeout. **Required:** the generator's `aclose()` must call `CliPool.reset(pool_id)` to kill the subprocess cleanly.

2. **Session ID propagation.** `pool_processor.py:257` reads `getattr(result, "session_id", None)` on the exhausted generator. The generator object must expose a `session_id` attribute, set when the `result` event is consumed. Remains `None` on cancellation.

3. **Tool-use block delta filtering.** The CLI emits `content_block_delta` for both `text_delta` and `input_json_delta` (tool-use input). Only `text_delta` should be yielded. The existing `read_until_result()` filters by `b.get("type") == "text"` — the streaming path needs the equivalent filter or raw JSON fragments reach users.

4. **`--include-partial-messages` is spawn-time.** If `ModelConfig.streaming` is toggled while a process is live, the process never emits stream events. `streaming` must participate in `ModelConfig.__eq__` (unlike `cwd` which uses `compare=False`) so the existing `model_config != entry.model_config` mismatch check in `CliPool.send()` forces a respawn.

See also: ADR-028 (`docs/architecture/adr/028-token-level-streaming-path-shape.mdx`).
