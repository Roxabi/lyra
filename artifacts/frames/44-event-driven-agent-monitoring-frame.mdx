## Problem

The current agent monitoring system relies on polling/cron ticks to check agent state. Every tick
generates token-heavy context regardless of whether anything has changed — agents may be idle and
stable, yet the system still processes monitoring context on every cycle. This is the dominant source
of wasteful token consumption in steady-state operation.

A reference implementation (knowledge `813ffe17`) demonstrates that replacing this pattern with an
event-driven architecture reduces monitoring token consumption by approximately 95%.

## Who

- **Primary:** Lyra system internals — the asyncio event loop and agent lifecycle management
- **Secondary:** End users (lower latency/cost from reduced token waste), operators (easier to observe
  meaningful state changes vs. noisy polling logs)

## Constraints

- Must integrate with the existing `asyncio.Queue`-based bus architecture (stable since #112 merged)
- Layer 1 event emission must be lightweight — no LLM calls in the hot path
- Event types must be typed (`AgentEvent` dataclass) for static analysis compatibility
- Related issue #111 (bash pre-check monitoring) covers a deterministic pre-LLM check and is
  complementary, not superseded by this work
- The event system must be LLM-provider-agnostic — works regardless of whether agents use Claude
  direct, OpenRouter, Claude Code CLI, or any other backend

## Out of Scope

- Measuring the actual production token reduction (95% claim) — measurement is a follow-up
- Integration with #104 (LLM circuit breaker) — separate issue
- Changes to inbound/outbound queue split (handled by #112, already merged)

## Complexity
