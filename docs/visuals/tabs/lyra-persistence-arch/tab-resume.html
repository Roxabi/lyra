Session Resume Paths

Three ordered paths to resume a Claude session — priority, triggers, guards, failure modes, and the ResumeStatus returned

1
reply-to-resume
Highest priority — user explicitly replies to an assistant message
MessageIndex
Trigger
msg.reply_to_id is not None AND pool is not busy AND not group chat
Store read
MessageIndex.resolve(pool_id, reply_to_id) → session_id in O(1) PK lookup
Returns
RESUMED — or falls through if message_id not indexed
✅ Cleanest path. No file check needed. Works even when the subprocess is still alive. Correctly resumes the exact session the user was replying to.
2
thread-session-resume
Discord only — threads carry their session in the DB
ThreadStore
Trigger
platform_meta["thread_session_id"] present — injected by Discord adapter on thread messages
Store reads
1. ThreadStore.get_session(thread_id, bot_id) → (session_id, pool_id)
2. pool.resume_session(session_id) → resume_and_reset() → _session_file_exists()
Returns
RESUMED if file exists, else falls through to Path 3 with path2_attempted=True
⚠️ Timing bug: _session_file_exists() globs ~/.claude/projects/*/{session_id}.jsonl. In --input-format stream-json mode, the .jsonl is written when the subprocess exits — not during streaming. So if the subprocess is still alive (normal case), this check returns False and Path 2 silently fails even when context is intact. Fix: bypass file check if subprocess for that session is alive. (→ finding #5)
3
last-active-session
Fallback — last assistant turn's session for this pool
TurnStore
Trigger
pool.is_idle AND turn_store is not None AND NOT group chat
Store read
TurnStore.get_last_session(pool_id) → most recent assistant turn's session_id (index scan)
Guards
Guard 1: last_sid is None → SKIPPED (no prior turn)
Guard 2: last_sid == pool.session_id → SKIPPED ✅ FIXED
Otherwise: pool.resume_session(last_sid) + file check
✅ Bug #415 fixed: Guard 2 (last_sid == pool.session_id) now explicitly returns SKIPPED instead of falling through to FRESH. This eliminates the spurious ⚠️ warning sent to users when the pool is already on the correct session.
ResumeStatus enum
RESUMED — session successfully set in _resume_session_ids[pool_id] for next spawn. No user notification.
SKIPPED — nothing to do (pool already on session, group chat safety skip, no prior turns). No user notification.
FRESH — no session could be resumed. _notify_session_fallthrough() → sends ⚠️ "Couldn't resume your previous session — starting fresh" to the user.
Root cause of bug #415 — false-positive ⚠️ warnings

Scenario: pool has session a73c10d6. User sends a new message. Path 2 tries to resume a73c10d6 → _session_file_exists() returns False (subprocess still alive, .jsonl not flushed) → falls to Path 3 → get_last_session() returns a73c10d6 → guard fires: last_sid == pool.session_id.

Before fix: Guard logged at DEBUG, fell through to return FRESH if path2_attempted else SKIPPED. Since path2_attempted=True, returned FRESH → sent ⚠️ to user, even though context was fine (the subprocess was alive and _resume_session_ids still had the right value).

After fix: Guard now return ResumeStatus.SKIPPED explicitly. No spurious warning. The actual spawn still uses the correct session from _resume_session_ids.
