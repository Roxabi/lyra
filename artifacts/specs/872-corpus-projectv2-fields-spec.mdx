## Context

`corpus.db` stores issue metadata from GraphQL but does not capture GitHub Project V2
single-select fields (Lane, Size, Priority, Status). The dep-graph v5 reader
(`dep_graph/v5/data/corpus.py:_project_lane_size`) compensates by inferring lane and
size from label prefixes (`graph:lane/`, `size:`). That fallback was always temporary —
it was explicitly marked as a SINGLE SWAP POINT pending this issue. With
`roxabi-plugins#119` (enrollment + backfill) closed, the hub project now has full
coverage; the label-projection path is dead weight and a maintenance trap.

Priority and Status are unavailable entirely in `corpus.db`, blocking downstream
consumers (v6 API, dashboards) from reading them without a second round-trip.

All touched files live in `roxabi-live`. The PR body uses `closes Roxabi/lyra#872` so
issue closure fires automatically on merge.

## Goal

Persist all four Project V2 single-select fields (Lane, Size, Priority, Status) in
`corpus.db` via GraphQL, and replace the label-inference path in the v5 dep-graph reader
with direct column reads — eliminating the label fallback entirely.

## Non-goals

- Dropping `layout.json` from v5 (separate decision, not part of #872).
- Deleting `graph:*/size:*` labels from GitHub issues (label cleanup, separate issue).
- Migrating the v6 API or any corpus consumer beyond the v5 dep-graph reader.
- Changing the `edges` table or `sync_state` schema.
- Adding SQL indexes on `priority` or `status` (no SQL filter path exists yet).

## API / Schema Changes

### schema.py — four new TEXT columns

`SCHEMA_VERSION` bumped 2 → 3. `_migrate()` gains four `ALTER TABLE issues ADD COLUMN`
calls, each in its own `try/except sqlite3.OperationalError` block guarded by the
existing `"duplicate column"` pattern:

```
lane     TEXT   -- Project V2 Lane option name (e.g. "a1", "standalone")
priority TEXT   -- Project V2 Priority option name (e.g. "P1", "P2 - Medium")
size     TEXT   -- Project V2 Size option name (e.g. "M", "L")
status   TEXT   -- Project V2 Status option name (e.g. "In Progress", "Done")
```

All four default to `NULL`. Existing rows remain valid; columns are hydrated on the next
`make corpus-sync` pass (lazy migration — corpus is a local cache, one sync cycle with
NULLs is acceptable).

### graphql.py — ISSUES_QUERY extended

`ISSUES_QUERY` gains a `projectItems(first: 5)` fragment on each issue node:

```graphql
projectItems(first: 5) {
  nodes {
    project {
      id
    }
    fieldValues(first: 20) {
      nodes {
        ... on ProjectV2ItemFieldSingleSelectValue {
          name
          field {
            ... on ProjectV2FieldCommon {
              name
            }
          }
        }
      }
    }
  }
}
```

`first: 5` on `projectItems` covers issues enrolled in multiple boards without the
fragility of `first: 1`. `first: 20` on `fieldValues` covers all custom fields in
typical V2 boards.

### sync.py — _parse_project_fields + upsert_issue + closed_hop_pass

New helper `_parse_project_fields(node: dict, canonical_project_id: str | None) -> dict`
filters by canonical project ID and returns `{"lane": ..., "priority": ..., "size": ...,
"status": ...}` (all four `None` when unmatched or `canonical_project_id` is `None`).

`upsert_issue` extended from 11 to 15 positional keys in docstring, INSERT tuple, and
ON CONFLICT SET clause.

`closed_hop_pass` stub dict (currently lines 291–303) gains four `None` keys:
`"lane": None, "priority": None, "size": None, "status": None`. Omitting these would
cause a `KeyError` on every closed-hop path — this is the highest-probability silent
breakage risk in the PR.

### dep_graph/v5/data/corpus.py — _project_lane_size renamed to _project_fields

`_project_lane_size(labels: list[str])` replaced by
`_project_fields(lane, priority, size, status)` returning
`{"lane_label": lane, "priority": priority, "size": size, "status": status}`.

The key `lane_label` is preserved (v5 consumers already reference `issue["lane_label"]`
per `test_corpus.py` line 122 — renaming would break all v5 consumers).

`_fetch_issues` SELECT extended from 11 to 15 columns:
```
..., is_stub, lane, priority, size, status FROM issues
```

`LANE_LABEL_PREFIX`, `SIZE_LABEL_PREFIX` constants and the full body of
`_project_lane_size` deleted. No feature flag — `roxabi-plugins#119` confirmed complete
hub enrollment.

### CORPUS_PROJECT_IDS env var

Single env var loaded at `sync.py` module level into `_CORPUS_PROJECT_IDS: dict[str, str]`.

Format: `owner/repo:project_node_id` pairs, comma-separated:

```
CORPUS_PROJECT_IDS=Roxabi/lyra:PVT_kwDOB8J6DM4BQrDj,Roxabi/llmCLI:PVT_kwDOB8J6DM4BU-5r,Roxabi/voiceCLI:PVT_kwDOB8J6DM4BQrEv,Roxabi/imageCLI:PVT_kwDOB8J6DM4BQzJC
```

| Repo | Project ID |
|------|-----------|
| Roxabi/lyra | PVT_kwDOB8J6DM4BQrDj |
| Roxabi/llmCLI | PVT_kwDOB8J6DM4BU-5r |
| Roxabi/voiceCLI | PVT_kwDOB8J6DM4BQrEv |
| Roxabi/imageCLI | PVT_kwDOB8J6DM4BQzJC |

When the env var is absent or a repo is not listed, `_parse_project_fields` returns all
four `None` (safe — no crash, no fallback to deleted label path). When `projectItems`
nodes exist but none match the configured ID, log a `stderr` warning (signals
misconfiguration vs. genuine non-enrollment).

## Rollout

1. Merge PR in `roxabi-live` — body contains `closes Roxabi/lyra#872`.
2. `_migrate()` runs automatically on next `bootstrap()` call (triggered by `make corpus-sync`).
3. Set `CORPUS_PROJECT_IDS` in `.env` with all four repo:project_id pairs before running sync.
4. Run `make corpus-sync` — new columns hydrate lazily from live GraphQL.
5. No fallback retention: label-projection path is deleted in V3; if `CORPUS_PROJECT_IDS`
   is unset, all four columns store `NULL` for all repos (safe, no crash).
6. No data migration script required — corpus is a local cache; one sync cycle with NULLs
   is the accepted downtime window.
