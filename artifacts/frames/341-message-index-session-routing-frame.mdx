## Problem

Session resumption via reply-to is broken across all platforms (Telegram, Discord) and contexts (DMs, threads). When a user replies to a previous bot message, Lyra should resume the Claude session from that message ‚Äî but it doesn't. Three root causes compound:

1. **`ContextResolver` was never wired** in bootstrap ‚Äî `hub._context_resolver` was always `None`, making reply-to-resume dead code. *(Fixed in staging.)*
2. **`conversation_turns.reply_message_id`** is a fragile, dual-purpose column ‚Äî it only indexes assistant messages, has no DB index (O(n) scan), no platform/scope scoping (Telegram cross-chat collisions), and mixes turn auditing with session routing.
3. **Streaming session_id race** ‚Äî `dispatch_streaming()` returns immediately, so `pool.session_id` is never updated from the iterator. Assistant turns get logged with a random UUID. *(Fixed in staging.)*

Even with the staging fixes, the underlying data model is wrong: the ContextResolver now runs but returns stale or incorrect sessions from `conversation_turns`.

## Who

- **Primary:** Mickael ‚Äî the sole user of Lyra across Telegram and Discord. Reply-to is the natural UX for resuming a conversation thread, and it silently fails today.
- **Secondary:** Any future Lyra user on any platform.

## Constraints

- Single domain: `core/` stores + pipeline + pool. No adapter changes needed (adapters already populate `reply_to_id`).
- Must coexist with existing `conversation_turns` (audit trail) ‚Äî `message_index` is additive, not a migration.
- SQLite-only (no new DB engine). `aiosqlite` single-writer is fine for Lyra's throughput (~50 exchanges/day).
- Cold start acceptable: empty table on first deploy, Path 3 (`get_last_session`) covers until traffic populates the index. Not a regression.
- Must handle both streaming and non-streaming paths with correct `session_id` at population time.
- `INSERT OR IGNORE` conflict policy ‚Äî preserve original session mapping on message edits.

## Out of Scope

- Group chat per-user filtering (requires `user_id` in schema ‚Äî Phase 2).
- Claude CLI session file lifetime management (external system).
- SDK backend resume (no session mechanism in `AnthropicSdkDriver`).
- Backfilling from `conversation_turns` (data quality too poor).
- Thread‚Üísession mapping (existing `discord_threads` table handles that).
