## Problem

Lyra's inbound path currently forks at the adapter boundary into **two separate
envelope types on two separate buses**:

- `InboundMessage` → `LocalBus[InboundMessage]` (alias `InboundBus`) → `MessagePipeline` → pool → agent.
- `InboundAudio` → `LocalBus[InboundAudio]` (alias `InboundAudioBus`) → standalone `AudioPipeline` consumer loop → STT → re-inject as `InboundMessage` onto the text bus.

This split produces several concrete costs:

- **Duplicate serialization / sanitization paths.** #525 had to sanitize `platform_meta` at the NATS trust boundary twice — once for the text bus, once for the audio bus. Any future trust-boundary invariant must be applied in both places.
- **Duplicate bootstrap wiring.** `inbound_bus` and `inbound_audio_bus` are created, passed, and subscribed independently in hub, both adapters (telegram, discord), and all three supervisor entry points (`hub_standalone`, `adapter_standalone`, `unified`).
- **Two NATS subject trees.** `lyra.inbound.<platform>.<bot_id>` and `lyra.inbound.audio.<platform>.<bot_id>` are wired in parallel via `NatsBus` + `NatsChannelProxy`, doubling migration and deprecation surface.
- **AudioPipeline is a bespoke consumer loop, not a stage.** `core/audio_pipeline.py` (~440 LOC) drains its own queue, calls STT, then re-enqueues onto `InboundBus`. It bypasses `MessagePipeline`'s guards and fail-fast semantics for the pre-STT portion of its flow.
- **Semantic mismatch.** `InboundMessage` already carries `modality: Literal["text", "voice"] | None`. Audio already *wants* to be a message with a modality tag — the parallel class is historical drift.

## Who

- **Primary:** Hub/core maintainers and adapter developers. Every new adapter (future: CLI channel, web channel) currently has to wire and normalize to **two** envelope types. Every pipeline invariant has to be enforced in **two** places.
- **Secondary:**
  - Operators — simpler bootstrap and fewer supervisor program logs to correlate during incidents.
  - Future modality work (image, video, multimodal agents) — the modality field is the right extension point; a new species per modality does not scale.

## Constraints

- **Three supervisor processes on Machine 1.** `lyra_hub`, `lyra_telegram`, `lyra_discord` run independently and can restart independently. The NATS topic migration must roll out without orphaning in-flight audio messages or requiring a strict deploy ordering.
- **NATS subject tree is a wire contract.** Collapsing `lyra.inbound.audio.*` into `lyra.inbound.*` is a breaking topic change. Needs a deprecation window where both trees are accepted during transition, then the old tree is removed.
- **STT-free deployments must still work.** When `hub._stt is None`, audio messages currently receive an `stt_unsupported` reply. This behaviour must survive the refactor — no silent drops, no crashes.
- **Related to #525.** The dual-sanitization concern exposed by #525 should be **eliminated** by this refactor, not reintroduced. The single post-NATS sanitization path must cover both modalities.
- **InboundAudio consumers exist beyond the bus drain loop.** `agent.py`, `audio_pipeline.py`, `hub_outbound.py`, `simple_agent.py`, `anthropic_agent.py`, `pool_processor.py` all import or reference the type. Each site needs an individual migration decision (rename to `InboundMessage`, or remove).
- **Frozen dataclasses.** Both types are `@dataclass(frozen=True)` — field additions to `InboundMessage` for audio payloads must preserve frozen semantics and schema versioning.
- **Schema version bump.** `InboundMessage.schema_version: int = 1` — adding audio fields is a compatible additive change (new optional fields), but NATS deserialization on older subscribers must handle unknown fields gracefully. Needs verification against `nats_bus.py` deserialization.
- **Test surface is large.** Any test currently asserting `isinstance(msg, InboundAudio)` or constructing one will need migration. Grep-and-replace alone will not suffice — behavioural tests around STT failure modes must be preserved.

## Out of Scope

- **New modalities (image, video, file attachments as first-class).** This refactor enables them but does not add them. Attachments remain `list[Attachment]` on `InboundMessage`.
- **TTS refactor.** `OutboundAudio` and the TTS dispatch path stay as they are. Only the inbound audio flow is touched. Moving TTS to a pipeline stage is a follow-up.
- **Schema v2 / major message format redesign.** This is an additive v1 change, not a version bump.
- **Pipeline stage ordering / plugin system rework.** We reuse the existing `MessagePipeline` stage interface — no new stage infrastructure.
- **STT provider or whisper-model changes.** `self._hub._stt` semantics and the `is_whisper_noise` heuristic are preserved as-is.
- **Retroactive migration of historical audio messages.** Any in-flight messages on the old NATS subject during deploy are accepted via the deprecation window; there is no backfill.

## Complexity

**Tier: F-full** — multi-domain (hub, adapters, NATS, pipeline, bootstrap), non-trivial migration (dual NATS subject tree with deprecation window), and several architectural unknowns that warrant an `/analyze` phase before committing a spec.

Signals observed:
- Multiple domains touched: `core/message.py`, `core/inbound_bus.py`, `core/inbound_audio_bus.py`, `core/audio_pipeline.py`, `core/hub/*`, `adapters/*` (both platforms), `bootstrap/*` (3 entry points), `nats/*`.
- New architectural pattern: converting a bespoke consumer loop (`AudioPipeline.run()`) into a pipeline stage — needs a design decision on where STT sits in the stage order relative to guards/trust resolution/command routing.
- Backward compatibility: dual NATS subject acceptance during rollout, schema evolution on frozen dataclasses.
- Multi-consumer blast radius: ~7 modules currently import `InboundAudio`; each needs an individual decision.
