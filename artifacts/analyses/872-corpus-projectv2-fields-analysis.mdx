## 3. GraphQL Extension

### Fragment design

GitHub's `projectItems` returns a `ProjectV2ItemConnection`. Each node's `fieldValues` is a polymorphic union. The correct inline fragment for single-select fields is `ProjectV2ItemFieldSingleSelectValue`.

```graphql
projectItems(first: 5) {
  nodes {
    project {
      id
    }
    fieldValues(first: 20) {
      nodes {
        ... on ProjectV2ItemFieldSingleSelectValue {
          name
          field {
            ... on ProjectV2FieldCommon {
              name
            }
          }
        }
      }
    }
  }
}
```

This fragment resolves `field.name` (e.g. `"Lane"`, `"Size"`, `"Priority"`, `"Status"`) and `name` (the selected option, e.g. `"a1"`, `"M"`, `"P1"`, `"In Progress"`).

`first: 5` on `projectItems` is sufficient — issues enrolled in the hub are in at most 2 projects (hub board + production board). `first: 20` on `fieldValues` covers all custom fields in typical V2 boards.

### Single-issue / multi-project disambiguation

An issue may appear in multiple projects (e.g. `Roxabi/lyra` board PVT_kwDOB8J6DM4BQrDj and `roxabi-production` board). The sync code must read only the **canonical project** and ignore others.

**Best long-term answer: environment variable `CORPUS_PROJECT_ID` per repo, resolved at sync time.**

Rationale: the four hub project IDs are already known (frame section "Risks" lists lyra=PVT_kwDOB8J6DM4BQrDj, llmCLI=PVT_kwDOB8J6DM4BU-5r, etc.) but they differ per repo. Hard-coding a single global ID would silently miss issues from non-lyra repos. A per-repo env var table adds config surface. The cleanest approach: store a mapping in `.env` / `pyproject.toml` `[tool.roxabi-live]` under key `corpus_project_ids` (a dict `owner/repo → project_node_id`), loaded once at sync startup. Default to `None` → skip project-field extraction (returns NULL) for repos not in the map. This is zero-config for the common case and explicit for enrolled repos.

Implementation in `run_repo_sync`:

```python
canonical_project_id = _CORPUS_PROJECT_IDS.get(f"{owner}/{name}")

def _parse_project_fields(node: dict, canonical_project_id: str | None) -> dict:
    if not canonical_project_id:
        return {"lane": None, "priority": None, "size": None, "status": None}
    for item in node.get("projectItems", {}).get("nodes", []):
        if item.get("project", {}).get("id") == canonical_project_id:
            fields: dict[str, str | None] = {"lane": None, "priority": None, "size": None, "status": None}
            for fv in item.get("fieldValues", {}).get("nodes", []):
                fname = (fv.get("field") or {}).get("name", "").lower()
                if fname in fields:
                    fields[fname] = fv.get("name")
            return fields
    return {"lane": None, "priority": None, "size": None, "status": None}
```

The field name comparison uses `.lower()` to guard against GitHub changing casing (`"Lane"` vs `"lane"`). The option value `fv.get("name")` is the display string, not the internal ID — this matches what label-projection currently stores and what v5 consumers expect.

### GraphQL cost impact

Current `ISSUES_QUERY` cost per page ≈ 1–2 points (nodes fetched × field complexity). Adding `projectItems(first:5) { fieldValues(first:20) }` per issue node increases cost. The `rateLimit { cost remaining resetAt }` line in `ISSUES_QUERY` (line 41 of graphql.py) is already surfaced by `log_rate_limit` in `sync.py` (lines 135–139). No throttle logic change is needed; see section 8 for risk assessment.

## 4. Migration

### Pattern

`_migrate()` currently uses a single `try/except sqlite3.OperationalError` block for one column (schema.py lines 51–58). For four new columns, extend with one block per column — do not use a loop, because a loop would silently swallow the first successful `ALTER TABLE` if any later one raises an unrelated error.

```python
def _migrate(conn: sqlite3.Connection) -> None:
    """Apply incremental migrations to existing DBs."""
    # Migration 1 — edges.kind (SCHEMA_VERSION 1 → 2)
    try:
        conn.execute("ALTER TABLE edges ADD COLUMN kind TEXT NOT NULL DEFAULT 'parent'")
        conn.commit()
    except sqlite3.OperationalError as exc:
        if "duplicate column" not in str(exc).lower():
            raise

    # Migration 2 — projectV2 fields on issues (SCHEMA_VERSION 2 → 3)
    for col in ("lane TEXT", "priority TEXT", "size TEXT", "status TEXT"):
        try:
            conn.execute(f"ALTER TABLE issues ADD COLUMN {col}")
            conn.commit()
        except sqlite3.OperationalError as exc:
            if "duplicate column" not in str(exc).lower():
                raise
```

### Idempotency check — PRAGMA table_info

The `"duplicate column"` string match on the OperationalError message is the established pattern in this codebase (schema.py line 57). It is sufficient: SQLite raises `"table X already has column Y"` which contains `"duplicate column"` when the column exists. An alternative using `PRAGMA table_info(issues)` before attempting ALTER is more explicit but adds a round-trip per column; the current catch-and-ignore pattern is consistent with Phase 1 precedent and acceptable for a local cache DB.

### SCHEMA_VERSION bump

Increment `SCHEMA_VERSION = 2` → `SCHEMA_VERSION = 3` (schema.py line 8). The `bootstrap()` function writes this value via `PRAGMA user_version` (line 67). There is no reader that branches on `user_version` today, so the bump is informational only — but it is required by the frame's AC-1.

## 8. Risks

### GraphQL response size growth

Adding `projectItems(first:5) { fieldValues(first:20) }` per issue node increases bytes per page. At 100 issues/page, this adds ~100 × (5 items × 20 field values × ~40 bytes/value) ≈ 400 KB uncompressed. `gh api graphql` streams over HTTPS (compressed); practical increase is ~40–80 KB/page. Not a bandwidth concern on localhost, but it does increase GraphQL **complexity cost** (not just bytes). The `rateLimit.cost` field in the existing response already surfaces this — the V2 gate test should assert `cost` remains below a threshold (e.g. `< 20` per page) to catch runaway complexity.

### Rate limits

GitHub GraphQL rate limit is 5000 points/hour. Current incremental syncs fetch only recently-updated issues (since cursor), typically 0–20 issues per repo. Full re-syncs (no cursor) are the risky case: at 100 issues/page × N pages × increased cost-per-page, a full re-sync of 1000 issues at cost=10/page uses 100 points — well within budget. No throttle change is needed, but add a post-sync assertion on `remaining > 100` to the V2 gate test.

### Project ID config drift across repos

The frame lists four hub project IDs (lyra=PVT_kwDOB8J6DM4BQrDj, llmCLI=PVT_kwDOB8J6DM4BU-5r, etc.). Issues from different repos route to different projects. The disambiguation design (section 3: env-var-backed per-repo map) handles this cleanly. Drift risk: if a repo is enrolled in a new project and the map is not updated, project fields silently store NULL — not a crash, and the label-fallback will already be deleted by V3. Mitigation: the `_parse_project_fields` helper logs a stderr warning when `projectItems` nodes exist but none match the configured project ID, signalling a misconfiguration rather than enrollment gap.

### upsert_issue signature change breaking closed_hop_pass

`closed_hop_pass` (sync.py lines 252–307) builds a stub dict at lines 291–303. This dict must gain the four new keys as `None`:
```python
stub = {

    "lane": None,
    "priority": None,
    "size": None,
    "status": None,
}
```
If omitted, the upsert will fail with a `KeyError` on the new positional args. This is a **high-probability silent failure** if the stub dict is not updated in the same PR.

### sync.py file-length gate

`sync.py` is currently 339 lines and already exempt at "320 lines" in `tools/file_exemptions.txt` (line 8). The actual count is 339 (verified by read). Adding `_parse_project_fields()` (~20 lines), updating `upsert_issue` (+8 lines), and updating `run_repo_sync` (+5 lines) brings the file to approximately 372 lines. The exemption entry must be updated to reflect the new line count and reference issue #872.

## 9. Open Questions — Resolved

### Q1: How to identify the canonical project for each repo?

**Answer: per-repo mapping in `.env` / config, loaded as `_CORPUS_PROJECT_IDS: dict[str, str]`.**

Hard-coding a single project ID would miss lyra-adjacent repos (llmCLI, voiceCLI, etc.). A per-repo env var per key (e.g. `CORPUS_PROJECT_ID_ROXABI_LYRA`) creates explosion. The cleanest approach is a single env var `CORPUS_PROJECT_IDS=Roxabi/lyra:PVT_...,Roxabi/llmCLI:PVT_...` parsed at module load into a dict. If the env var is absent, project field extraction is skipped for all repos (all columns store NULL) — this is safe and avoids import-time errors on machines without the env set.

### Q2: Should `_project_lane_size` be renamed to `_project_fields`?

**Answer: yes, rename.**

The frame says "rename or extend to `_project_fields` optional" (frame line 118). The existing name is a misnomer after the swap (it no longer reads labels, and it now surfaces four fields not two). Rename to `_project_fields` with updated docstring. All references are within `corpus.py` and `test_corpus.py` — no other module imports this private function.

### Q3: What GraphQL `first:` value for `projectItems`?

**Answer: `first: 5`.**

The frame constraint says `projectItems(first:1)` (frame line 38), but this is fragile: an issue in two projects returns two nodes, and `first:1` may return the non-canonical project depending on API ordering. `first: 5` is a safe ceiling (no issue realistically belongs to more than 5 projects) and the disambiguation logic in `_parse_project_fields` filters by project ID regardless of order.

### Q4: Should the fallback label-parsing be kept as a fallback during transition?

**Answer: no.**

The frame decision table (frame line 138) is explicit: "Remove immediately in V3 (no feature flag)". `roxabi-plugins#119` confirmed full hub coverage. The frame's AC-3 grep check enforces this. A conditional fallback would add dead-code complexity with no safety net value — if project fields are NULL (unenrolled issue), the caller receives `None`, which is the correct representation.

### Q5: Should `priority` and `status` be indexed?

**Answer: not in this PR.**

No current consumer filters by `priority` or `status` in SQL. The v5 reader loads all issues into memory and filters in Python. Adding indexes preemptively adds schema complexity without a measured benefit. This is a follow-on if a v6 SQL query path emerges.
