## Overview

`lyra_harness` is a standalone NATS service (Milestone M2 of the distributed Lyra epic, #633). It receives a `HarnessRequest` from the hub, runs the full agentic loop (LLM calls â†’ tool execution â†’ inject results â†’ repeat), streams `RenderEvent` chunks back to the hub's reply-to inbox during execution, and returns a final message containing `updated_history`. The process is fully stateless â€” any worker can serve any `pool_id`.

This service allows the hub to delegate compute-heavy agentic turns to a dedicated worker process (local or remote) without coupling the hub's lifecycle to LLM execution. It mirrors the `TtsAdapterStandalone` / `NatsTtsClient` pattern already established in the codebase.

## Wire Format

### HarnessRequest (hub â†’ harness)

Envelope follows the schema-version pattern used by all NATS messages in the codebase (`_version_check`, `adapter_base._validate_envelope`).

```json
{
  "schema_version": 1,
  "request_id":   "<uuid4>",
  "pool_id":      "<str>",
  "session_id":   "<str>",            // tracing only â€” see Â§Stateless Guarantee
  "message":      "<str>",
  "history":      [
    {"role": "user",      "content": "<str>"},
    {"role": "assistant", "content": "<str>"}
  ],
  "agent_config": {
    "name":          "<str>",
    "system_prompt": "<str>",
    "model":         "<str>",
    "base_url":      "<str | null>",
    "cwd":           "<str | null>",
    "tools":         ["Read", "Write", "Edit", "Bash", "Grep"],
    "max_turns":     "<int | null>",
    "streaming":     true,
    "skip_permissions": false
  }
}
```

#### Field decisions

| Field | Decision | Rationale |
|---|---|---|
| `history` | Inline array | Object store ref deferred to #521. Inline is sufficient for current turn lengths (<100 KB); guarded by NATS `max_payload`. |
| `agent_config` | Full blob per request | Stateless guarantee: harness resolves no state â€” hub is the single source of truth. No synchronisation protocol needed. |
| `tools` | Names only | Harness resolves tool implementations from its own registry keyed by name. Names are the stable API surface. |
| `max_turns` | Field inside `agent_config` | Matches `ModelConfig.max_turns`; `null` means unlimited. |
| `streaming` | Field inside `agent_config` | Harness always runs `LiteLLMDriver` in streaming mode (required for `RenderEvent` chunks). Wire field present for forward-compat; non-streaming mode is not yet supported. |
| `model` / `base_url` | Inside `agent_config` | Harness calls `LiteLLMDriver` (#665) directly; driver is configured per-request from these fields. |
| `cwd` | `agent_config.cwd` (string) | Filesystem path used by tool executor; serialised as string (not `Path`) for JSON compat. |

#### Credentials

`api_key` is never transmitted over NATS. The harness resolves it from its own environment or secrets store at request time, matching `ModelConfig.api_key`'s docstring: "Resolved via secrets store at runtime; None = use env var". No credential field appears in `HarnessRequest`.

#### History inline threshold

No per-field byte threshold is enforced in code. The NATS server's `max_payload` (default 1 MB) acts as the hard ceiling. If a conversation grows beyond that ceiling, the hub must truncate history before sending (oldest turns dropped first, system prompt preserved). Object-store ref support is deferred to #521.

### RenderEvent chunks (harness â†’ hub, during execution)

Format reuses `NatsRenderEventCodec` exactly as used by `NatsChannelProxy`. The harness publishes to `msg.reply` (the hub's ephemeral inbox) as each event is emitted.

```json
{
  "stream_id":  "<request_id>",
  "seq":        0,
  "event_type": "text" | "tool_summary" | "stream_end" | "stream_error",
  "payload":    { ... },
  "done":       false
}
```

Terminal conditions follow `NatsRenderEventCodec.is_terminal()`:
- `stream_end` â€” always terminal (explicit sentinel).
- `stream_error` â€” always terminal (error path).
- `tool_summary` â€” never terminal.
- All other types â€” terminal when `done=true`.

### Final message (harness â†’ hub, after loop completes)

Published to `msg.reply` after all `RenderEvent` chunks and after the `stream_end` sentinel.

```json
{
  "schema_version":  1,
  "request_id":      "<uuid4>",
  "ok":              true,
  "updated_history": [
    {"role": "user",      "content": "<str>"},
    {"role": "assistant", "content": "<str>"}
  ],
  "error":           null
}
```

Error path:

```json
{
  "schema_version":  1,
  "request_id":      "<uuid4>",
  "ok":              false,
  "error":           "tool_execution_failed" | "llm_error" | "turn_limit_exceeded" | "internal_error",
  "updated_history": null
}
```

The response envelope inherits `schema_version` from the request; future protocol revisions must bump it.

The hub distinguishes chunk messages from the final message by the presence of `updated_history` (or `ok`) at the top level. Chunk messages carry `event_type` at the top level; final messages carry `ok`.

## Tool Execution Model

### Tools running in-harness

The following tools execute inside the harness process. They require local filesystem access and are resolved by name from the harness tool registry.

| Tool name | Implementation | Requirement |
|---|---|---|
| `Read` | `lyra.tools.read_tool` | Local FS |
| `Write` | `lyra.tools.write_tool` | Local FS |
| `Edit` | `lyra.tools.edit_tool` | Local FS |
| `Bash` | `lyra.tools.bash_tool` | Local shell |
| `Grep` | `lyra.tools.grep_tool` | Local FS |

`cwd` from `agent_config.cwd` is applied as the working directory for all tool executions. When `cwd` is `null`, the harness defaults to its own process working directory.

### Deferred tools

| Category | Deferral |
|---|---|
| Remote/MCP tools | Deferred â€” no remote protocol defined yet. |
| WebFetch, WebSearch | Deferred â€” network egress policy TBD. |

### Tool name resolution

The harness maintains a static registry: `dict[str, ToolExecutor]`. On startup, it registers the in-process tools listed above. Unknown tool names in `agent_config.tools` are logged as warnings and skipped â€” the agentic loop proceeds with the subset that resolved. This prevents a bad config from hard-failing an entire session.

### Agentic loop structure

```
while turns < max_turns (or unlimited):
    response = LiteLLMDriver.stream(history, system_prompt, tools)
    for event in response:
        if TextLlmEvent:
            emit RenderEvent chunk â†’ msg.reply
        if ToolUseLlmEvent:
            result = ToolExecutor.run(event.tool_name, event.input, cwd)
            emit ToolSummaryRenderEvent chunk â†’ msg.reply
            inject tool result into history
    if ResultLlmEvent.is_error:
        publish stream_error sentinel + final {ok: false}
        return
    if no tool calls this turn:
        break  # model satisfied, no more tool calls
    turns += 1

publish stream_end sentinel
publish final {ok: true, updated_history: history}
```

## NATS Subjects

| Subject | Direction | Purpose |
|---|---|---|
| `lyra.harness.request` | hub â†’ harness | Inbound request subject; queue group `harness-workers` |
| `lyra.harness.health.{worker_id}` | harness â†’ hub | Worker heartbeat, published every 5 s |
| `_INBOX.harness.<uid>` | harness â†’ hub | Ephemeral reply-to; created by hub via `nc.new_inbox()` |

### Queue group

`harness-workers` â€” constant defined in `src/lyra/nats/queue_groups.py` alongside `TTS_WORKERS` and `STT_WORKERS`. Ensures exactly-one delivery when multiple harness workers are running.

### Heartbeat subject pattern

`lyra.harness.health.{worker_id}` where `worker_id` is `harness-workers-{hostname}-{pid}` (same derivation as `NatsAdapterBase._worker_id`). The hub subscribes with wildcard `lyra.harness.health.*` and tracks freshness with a TTL of 15 s (TTL matches TTS). This differs from the flat-subject pattern used by TTS/STT â€” which publish to a single subject with `worker_id` in the payload â€” but is functionally equivalent.

Heartbeat payload extends `NatsAdapterBase.heartbeat_payload()`:

```json
{
  "worker_id":       "harness-workers-hostname-1234",
  "service":         "harness-workers",
  "subject":         "lyra.harness.health.harness-workers-hostname-1234",
  "queue_group":     "harness-workers",
  "host":            "hostname",
  "uptime_s":        42.3,
  "ts":              1234567890.0,
  "active_requests": 2,
  "llm_driver":      "litellm"
}
```

`subject` and `queue_group` are base fields injected by `NatsAdapterBase.heartbeat_payload()`; the harness merges its own fields (`active_requests`, `llm_driver`) on top.

## Stateless Guarantee

The harness owns zero durable state. Per-request data flow:

```
HarnessRequest contains:
  pool_id      â†’ logged for tracing only, not stored
  session_id   â†’ logged for tracing only, not stored
  history      â†’ consumed for this turn, returned as updated_history
  agent_config â†’ used for this turn, discarded after

Harness never writes to:
  - Any SQLite store
  - Any NATS JetStream KV
  - Any file outside tool execution scope (tool writes are user-directed)
```

Any worker in the `harness-workers` queue group can serve any `pool_id` at any time. There is no affinity, no sticky routing, and no warm-up requirement. Workers are interchangeable.

## Observability

The harness is the critical instrumentation point for the agentic loop. Without harness-level observability, LLM `Generation` records (from the LiteLLM OTel callback) appear isolated with no parent span, no tool-call context, and no end-to-end latency. Harness must expose the agentic structure as spans and events so backends (Langfuse, Phoenix, Jaeger) can reconstruct the turn.

### Observability Model

| Concept | Maps to | Emitted by |
|---|---|---|
| User turn (one `HarnessRequest`) | `Trace` | Hub (root); `trace_id` propagated on `HarnessRequest` |
| Harness agentic-loop iteration | `Span` (child of Trace) | Harness (`HarnessLoop.run`) |
| LLM inference call | `Generation` (child of Span) | LiteLLM OTel callback (automatic once enabled) |
| Tool execution (Read/Write/Edit/Bash/Grep) | `Event` (child of Span) | Harness (`ToolExecutor.run`) |
| Pool identity | `session_id` | `pool_id` from `HarnessRequest` copied onto Trace and Span |

### Wire-Format Extension

One optional field is added to the top-level `HarnessRequest` envelope:

```json
{
  "schema_version": 1,
  "request_id":   "<uuid4>",
  "trace_id":     "<str | null>",
  "pool_id":      "...",
  "session_id":   "...",
  "message":      "...",
  "history":      [...],
  "agent_config": {...}
}
```

#### Field decision

| Field | Decision | Rationale |
|---|---|---|
| `trace_id` | Optional string | Hub-provided root trace id. Harness attaches all spans/generations/events as children. `null` means no root -- harness still emits spans but they have no parent (useful for synthetic/CLI-triggered requests). |

### Provider Wiring

- `HarnessAdapterStandalone.__init__(obs: ObservabilityProvider = NoOpObsProvider())` -- dependency-injected; default NoOp so tests and dev require no backend.
- `HarnessLoop` receives `obs` from the adapter and calls `obs.start_span(trace_id, name="harness.iteration", attributes={"turn": n, "pool_id": pool_id, "session_id": session_id})` at the top of each iteration; `obs.end_span(span)` on completion or error.
- `ToolExecutor.run()` records `obs.record_event(span, name="tool.{name}", input=..., output=..., error=...)` per tool call.
- LiteLLM OTel callback is enabled at driver init: `litellm.callbacks = ["otel"]`. Each LLM call inside a span automatically emits a `Generation` under the current span; harness does not call `record_generation` explicitly.

### Span Attributes (Minimum Set)

| Attribute | Source | Notes |
|---|---|---|
| `pool_id` | `HarnessRequest.pool_id` | Tags for session grouping |
| `session_id` | `HarnessRequest.session_id` | Duplicated for backend compat (Langfuse uses `session_id`) |
| `agent.name` | `agent_config.name` | Filter traces by agent |
| `agent.model` | `agent_config.model` | Filter by provider/model |
| `turn` | loop counter | Iteration index inside the turn |
| `request_id` | `HarnessRequest.request_id` | Correlate with hub logs |

### Failure Mode

Observability-provider errors are swallowed by the harness -- a failing obs call must never abort the agentic loop. `NoOpObsProvider` is used if none is configured. Errors inside the real provider are logged at `warning` level and dropped.

## Hub Fallback

The hub-side `NatsHarnessClient` follows the same circuit-breaker + heartbeat-liveness pattern as `NatsTtsClient`:

1. `NatsHarnessClient.start()` subscribes to `lyra.harness.health.*` to track worker freshness.
2. Before each call, `_any_worker_alive()` checks that at least one heartbeat was seen within the last 15 s.
3. If no worker is alive, or if the circuit breaker is open, the client raises `HarnessUnavailableError`.
4. The hub catches `HarnessUnavailableError` and falls back to `agent.process()` inline (existing behaviour pre-M2).

The fallback is transparent to the user. A `log.warning` is emitted at each fallback occurrence for observability.

## Failure Modes

| Failure | Detection | Hub Behavior |
|---|---|---|
| No worker alive | `_any_worker_alive()` false before send | Raises `HarnessUnavailableError`; hub falls back to `agent.process()` inline. Final message is produced normally. |
| Circuit breaker open | Checked before send | Same as above â€” `HarnessUnavailableError`. |
| Per-request stream timeout | No final message received within 30 s (configurable) | Hub emits a `stream_error` `RenderEvent` to the pool with error code `harness_stream_timeout`; treats the turn as failed. No `updated_history` is returned. |
| Stream abort / worker crash mid-turn | NATS connection drop or no data within timeout | Treated as stream timeout; same `stream_error` path above. |

`HarnessUnavailableError` fires before the request is sent. Stream timeout fires after the request is sent but before the final message arrives â€” these are distinct failure modes with different recovery paths. The 30 s default is sized for a multi-tool agentic turn; it should be surfaced as a configurable parameter on `NatsHarnessClient`.

## Acceptance Criteria

1. `HarnessAdapterStandalone` subscribes to `lyra.harness.request` with queue group `harness-workers` and extends `NatsAdapterBase`.
2. A `HarnessRequest` with valid `schema_version=1` is accepted; any other version is dropped with a log warning (existing `_validate_envelope` behaviour).
3. `RenderEvent` chunks are published to `msg.reply` during the agentic loop using `NatsRenderEventCodec.encode()`.
4. The final message `{ok, updated_history, error}` is published to `msg.reply` after the `stream_end` sentinel.
5. All five in-harness tools (Read, Write, Edit, Bash, Grep) are registered and resolved by name.
6. Unknown tool names are warned-and-skipped; the loop continues with remaining tools.
7. `cwd` from `agent_config` is applied to all tool executions; `null` defaults to the process cwd.
8. `max_turns` from `agent_config` caps the loop; `null` means unlimited.
9. Heartbeat published to `lyra.harness.health.{worker_id}` every 5 s.
10. `NatsHarnessClient` raises `HarnessUnavailableError` when no live worker is detected (heartbeat stale > 15 s) or circuit breaker is open.
11. Hub falls back to `agent.process()` inline on `HarnessUnavailableError` with a `log.warning`.
12. A unit test covers the fallback path (mock `NatsHarnessClient` raises `HarnessUnavailableError`).
13. A unit test covers the agentic loop with a mock LLM driver emitting one tool call followed by a final text response.
14. No SQLite store, JetStream KV, or persistent state is written by the harness process.
15. `HarnessAdapterStandalone` accepts an optional `ObservabilityProvider` in its constructor; default is `NoOpObsProvider`.
16. A `trace_id` present on `HarnessRequest` is propagated as the parent of every span/event emitted during that request; `trace_id=null` causes spans to be emitted without a parent.
17. A unit test covers loop execution with a mock `ObservabilityProvider` that records start/end/event calls, asserting the expected sequence for a one-tool one-response turn.
