Component reference

InboundBus

One async queue per platform. Adapters push InboundMessage objects here. Hub consumes. Provides backpressure and decouples adapter I/O from agent execution.

src/lyra/core/inbound_bus.py

Hub

Central router. Receives all messages from InboundBus, resolves scope_id (chat ID), looks up or creates a Pool, and dispatches the message to it.

src/lyra/core/hub.py

Pool

A running conversation context: holds message history, circuit breaker state, and a reference to the current AgentBase instance. One Pool per scope_id. Pools are created lazily and cleaned up on timeout.

src/lyra/core/pool.py

AgentBase

Stateless singleton. Holds Agent config dataclass. On every message: hot-reload check (O(1)), then calls LLM API with full history. AnthropicAgent and SimpleAgent extend this.

src/lyra/agents/base.py

AgentStore

Write-through in-memory cache of all AgentRow DB records. get(name) is O(1). upsert() writes to both DB and cache atomically, bumping updated_at. Used by hot-reload.

src/lyra/db/agent_store.py

OutboundDispatcher

Per-bot async queues with exponential-backoff retry. Circuit breaker opens after sustained failures (configurable threshold), closes on probe success. Prevents cascading failures from flaky bot APIs.

src/lyra/core/outbound.py

Key design constraints

AgentBase is a stateless singleton. All per-conversation state lives in Pool. This means one agent config serves all concurrent conversations without locking.
Hot-reload is lazy, not proactive. Config changes are picked up on the next inbound message. There is no background polling task.
Pool → Hub back-reference is a known issue. Some Pool code holds a reference back to Hub, creating a circular dependency. Tracked as a high-priority audit finding.
Unbounded Pool growth. Pools are created per scope_id with no eviction policy. Long-running instances accumulate idle pools. A TTL-based cleanup is a pending improvement.
