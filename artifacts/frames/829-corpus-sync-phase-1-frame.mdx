## Problem

The dep-graph tool is built on a per-project cache (`lyra-v2-dependency-graph.gh.json`) and a visibility rule scoped to a single "primary repo" (`Roxabi/lyra`). Cross-repo dep chains that aren't 1-hop from a lyra issue stay invisible — concretely, the voiceCLI chain `#69 → #53/55/80/81/85 → #83` is labelled `graph:lane/a2` but never renders, even after the transitive-backward-walk fix landed.

The current model also can't support the agreed long-term UX: "select a project (repo), see its tree plus subtrees from other projects that share a node." That requires a workspace-wide data layer — one corpus, many readers. This issue builds that corpus.

Phase 1 is strictly the data plane. The v5 reader, the project-select dropdown, and the overlap renderer are deliberately deferred to Phases 2–3 so this change stays reviewable.

## Who

- **Primary:** Mickael — needs the full Roxabi-org issue graph locally to drive the redesigned dep-graph and, later, roxabi-dashboard.
- **Secondary:** the future `roxabi-dashboard` package — will treat this corpus as its input, so the schema and CLI must survive extraction from `scripts/corpus/` to a dedicated repo (Phase 5) without a rewrite.

## Constraints

- **Language / stack:** Python + `uv` (project standard). SQLite via stdlib `sqlite3` — no ORM, keep the schema portable.
- **Storage location:** `~/.roxabi/corpus.db` (outside the repo, like `~/.roxabi/forge/lyra/visuals/*.gh.json` today).
- **GitHub access:** `gh auth token` already exists; reuse, don't require a new PAT.
- **Rate limits:** must respect GraphQL 5000 points/hr; log `cost`/`remaining`/`resetAt` each run.
- **Incremental:** re-sync must re-fetch only issues with `updatedAt > last_synced_at` per repo. Full refresh should stay cheap enough for nightly cron (< 2s of quota on current org size).
- **Edge canonicalisation:** store `blocks` in one direction only; derive `blocked_by` via a reverse index. Halves the table and removes drift risk.
- **Cross-repo refs:** stored as `owner/repo#N` strings — never bare numbers.
- **+1 closed hop rule:** after the open-set fetch, pull stub meta for any closed blocker referenced in `blocked_by` but not already in the corpus (mirrors the existing `fetch.py` second-pass behaviour).

## Out of Scope

- v5 reader refactor (`compute_visible` → `tree(repo) ∪ shared_subtree(Q, P)`) — **Phase 2**.
- Toolbar project-select dropdown and overlap renderer — **Phase 3**.
- Cron / timer wiring for scheduled sync — **Phase 4**.
- Extracting `scripts/corpus/` into the `roxabi-dashboard` repo — **Phase 5**.
- Any change to `dep_graph/` (legacy path) or `v5/` readers. Those keep reading `gh.json` until Phase 2 flips them.
- Body text storage — keep only `body_hash` for drift detection to keep the DB small.
- Webhooks / real-time sync — polling is sufficient.
