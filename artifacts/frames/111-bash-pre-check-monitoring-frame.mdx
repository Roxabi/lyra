## Problem

Any monitoring or watchdog logic that calls an LLM to assess hub/agent health burns tokens even when everything is fine. A cron-style health check calling the LLM every N minutes wastes tokens on idle "all good" responses.

The Elvis Sun pattern (two-layer monitoring) achieves ~95% token reduction by using a deterministic bash/Python pre-check (Layer 1) that only escalates to an LLM call (Layer 2) when an anomaly is detected.

Without this, ongoing hub monitoring on Machine 1 will accumulate unnecessary LLM costs proportional to check frequency.

## Who

- **Primary:** Hub operator (Mickael) — runs Lyra 24/7 on Machine 1, needs low-cost health monitoring
- **Secondary:** Future multi-machine deployments — same pattern applies to any monitored node

## Constraints

- Must run on Machine 1 (`roxabituwer`, Ubuntu Server 24.04) as systemd timer or cron job
- Layer 1 checks must be zero-token (bash/Python, no LLM calls)
- Layer 2 LLM call only on anomaly — reuses existing Lyra LLM infrastructure (including #104 circuit breaker, now merged)
- Telegram notification on failure (existing bot integration)
- Thresholds configurable via TOML (aligns with Lyra config pattern)
- Part of Epic #101 (Phase 0 — Bot core parity)

## Out of Scope

- Memory/session lifecycle idle detection (handled by #83)
- Multi-machine orchestration (future — single-machine first)
- Dashboard or web UI for health status
- Auto-remediation (Layer 2 suggests remediation, does not execute it — tracked in #120)
