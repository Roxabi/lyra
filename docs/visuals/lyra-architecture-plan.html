Hub

Central Dispatcher

Single consumer loop pops messages from the bounded bus. Resolves bindings (exact key or wildcard fallback), creates/retrieves pools, dispatches to agents or command router.

Adapters (Spokes)

Channel Producers

Each adapter runs as independent asyncio task. Normalizes platform-specific messages into unified Message format. Verifies identity before constructing Message (security contract).

Pools

Per-User State

Mutable state lives in per-user Pool: history, message deque (50 turns), asyncio.Lock. Sequential processing per user, parallel across users. Zero race conditions.

System 02

The Bus

Bounded asyncio.Queue with backpressure handling. Never drops messages — queues, acknowledges, and waits.

Design

Bounded Queue (100 slots)

  • Multiple producers — each adapter pushes normalized messages
  • Single consumer — Hub.run() loop pops and processes
  • Backpressure — queue full → adapter sends "message received, ~Xs wait" ack, then blocks on await bus.put()
  • No message drops — queues instead of discarding
Rate Limiting

Per-User Sliding Window

  • 20 messages / 60 seconds per user
  • Enforced via Hub._is_rate_limited() with deque sliding window
  • Inactive users auto-cleaned when deque empties
  • Prevents abuse without penalizing normal usage
Design Rationale
At personal scale (1–5 users), a simple asyncio.Queue is sufficient. No external message broker needed. The bounded size (100) provides backpressure. Phase 2 will add NATS for inter-machine coordination when Machine 2 comes online.

System 05

Memory System

Five-level memory architecture. Phase 1 implements levels 0 (working) and 3 (semantic via roxabi_memory). Full cascade planned for Phase 2.

LevelNameImplementationStatus
0 Working Memory Active context window in LLM Done
1 Session Memory Multi-turn state per pool (deque, 50 turns) Phase 2
2 Episodic Memory Dated Markdown files Phase 2
3 Semantic Memory SQLite + BM25 + embeddings (roxabi_memory) Integrating
4 Procedural Memory Learned skills, preferences Phase 3
Pool History (L0)

Working Memory

list[Message] + deque[dict] (SDK format, capped at 50 turns). Consolidation planned: time-decay on old turns, auto-compaction when context window limit approaches, L0→L3 cascade.

roxabi_memory (L3)

Semantic Memory

Separate package imported via uv.sources. Vault at $ROXABI_VAULT_DIR (default ~/.roxabi-vault). Personas loaded from personas/ dir. URL dedup via indexed normalized_url columns.

Command Router

Detects /command messages via regex. Routes builtins (/help, /circuit) and plugin commands. Admin-only commands gated by _admin_user_ids set.

/help — list commands /circuit — admin circuit status /echo — plugin example
System 07

Configuration

Three-tier configuration: env vars for secrets, TOML for agent config, separate TOML for circuit breaker. All hot-reloadable.

Tier 1 — Secrets

.env File

TELEGRAM_TOKEN, DISCORD_TOKEN, ANTHROPIC_API_KEY, ROXABI_VAULT_DIR. Never in TOML.

Tier 2 — Agent

Agent TOML

[agent] name, persona. [model] backend, model, tools. [prompt] system. [plugins] enabled list. Hot-reload on mtime change.

Tier 3 — System

lyra.toml

[circuit_breaker.*] failure_threshold, recovery_timeout. [admin] user_ids. System-wide settings.

System 08

Resilience Patterns

Circuit breaker, hot-reload, rate limiting, backpressure handling. Production-grade reliability without external infrastructure.

stateDiagram-v2
  [*] --> CLOSED
  CLOSED --> OPEN : failures >= threshold
  OPEN --> HALF_OPEN : recovery_timeout elapsed
  HALF_OPEN --> CLOSED : probe succeeds
  HALF_OPEN --> OPEN : probe fails
Circuit breaker state machine: fast-fail when providers are down, probe for recovery
Circuit Breaker
  • Per-provider circuits (anthropic, telegram)
  • Configurable thresholds (default: 5 failures)
  • Recovery timeout (default: 60 seconds)
  • Half-open probes allow single request through
  • Admin visibility via /circuit command
Hot-Reload
  • Agent TOML — reloaded on file mtime change
  • Persona files — reloaded if changed
  • Plugin handlers.py — reloaded without restart
  • CliPool respawn — on system_prompt change

Competitive Position
Compared to the 5 Claw Family projects analyzed, Lyra's ~300-line hub is the most minimal core by far (vs OpenFang 137K, IronClaw 99K, OpenClaw 39MB). The design trades scale for auditability and readability — all code readable in an afternoon. The two-machine architecture provides a unique advantage: dedicated hub + on-demand GPU without VRAM contention. Key patterns to adopt from peers: hybrid RRF search (OpenClaw), decorator LLM chains (IronClaw), credential proxy (NanoClaw), three-tier memory (OpenFang), and provider registry (Nanobot).
