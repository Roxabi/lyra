## Source

Issue #76: "feat: direct Anthropic SDK agent (replace CLI subprocess)" — Part of Phase 1b (Agent core, epic #73).
Frame: `artifacts/frames/76-direct-anthropic-sdk-agent-frame.mdx` (approved).

User instruction: "Check first that Anthropic SDK agent allow subscriptions." Validated during framing: OAuth subscription auth is viable for personal use, but requires dual-auth strategy.

## Problem

Lyra's agent execution path goes through Claude CLI subprocess (`CliPool` → `claude` binary → NDJSON protocol). This blocks streaming, prevents native tool use, sandboxes the environment, and ties auth to the CLI binary's OAuth flow. The current architecture has no streaming support at any layer — `Response` is a plain string, adapters send once, hub dispatches once.

## Outcome

An `AnthropicAgent` class that calls the Anthropic API directly, delivering:

1. **True token-by-token streaming** from SDK to Telegram/Discord (edit-in-place pattern)
2. **No sandbox** — agent runs in Lyra's process, direct access to vault, memory, calendar, and all services
3. **No cold start** — direct HTTP calls replace ~12s Node.js subprocess spawn
4. **Custom tool definitions** — Python functions the agent can call (vault search, calendar, memory)
5. **System prompt injection** for persona (#75) — first-class `system=` parameter
6. **Session persistence** — conversation history stored in Pool, survives process restarts
7. **Model switching** — per-request parameter, trivial (no `control_request` needed)
8. **Dual auth** — OAuth subscription primary, API key fallback, configurable per agent
9. **`SimpleAgent` remains default** — migration is opt-in via TOML config (`backend = "anthropic-sdk"`)

## Appetite

2-week cycle, scoped as:
- **Week 1**: AnthropicAgent core (SDK client, streaming pipeline, API key auth, basic tool loop)
- **Week 2**: Adapter streaming integration (Telegram/Discord edit-in-place, debounce), TOML config, tests
- **Deferred**: OAuth token refresh mechanism (follow-on issue — start with API key, add OAuth as enhancement)

## Shapes

### Shape 1: Raw Messages API (`anthropic` package)

Use the `anthropic` Python SDK directly with `AsyncAnthropic.messages.stream()`.

**Architecture:**
```
User msg → Hub → AnthropicAgent.process()
                    → AsyncAnthropic.messages.stream()
                    → yield text deltas
                    → Hub.dispatch_streaming()
                    → Adapter.send_chunk() (edit message in-place)
```

**Key implementation details:**
- `AsyncAnthropic` for native async (fits Lyra's asyncio architecture)
- `messages.stream()` context manager yields `text_stream` (token-by-token)
- Tool loop: check `stop_reason == "tool_use"`, execute Python function, append `tool_result`, continue
- Conversation history: pass full `messages` array each turn (managed in Pool as `sdk_history: list[dict]`)
- Token counting: `response.usage.input_tokens` / `output_tokens`
- Auth: `api_key` parameter accepts API keys. OAuth tokens require custom header (`Authorization: Bearer sk-ant-oat01-...`) — needs httpx `auth` hook with 401 retry for token refresh.
- `process()` returns `Response | AsyncIterator[str]` — hub branches on `isinstance`

**Trade-offs:**
- Pro: Full control over agent loop, tool system, conversation management
- Pro: Minimal dependencies (just `anthropic` package)
- Pro: System prompt is a first-class parameter — perfect for persona injection
- Con: Must implement tool loop, conversation history, token budget management manually
- Con: OAuth token support needs custom httpx middleware + 401 retry path for token refresh
- Con: No built-in session persistence — must build from scratch

**Rough scope:** L (new agent class + streaming pipeline + tool executor + adapter changes)

### Shape 2: Agent SDK (`claude-agent-sdk-python`)

Use Anthropic's Agent SDK, which wraps the Messages API with built-in tool execution, session management, and context compaction.

**Architecture:**
```
User msg → Hub → AnthropicAgent.process()
                    → claude_agent_sdk.query(prompt, options)
                    → async for message in stream
                    → Hub.dispatch_streaming()
                    → Adapter.send_chunk()
```

**Key implementation details:**
- Async-only (`async for message in query(...)`)
- Built-in tools: Read, Edit, Bash, Glob, Grep, WebSearch, etc.
- Session management: `resume=session_id` for multi-turn
- Context compaction: auto-summarizes old history when approaching context limit
- `CLAUDE_CODE_OAUTH_TOKEN` env var supported natively — no custom headers needed
- Permission system: fine-grained tool access control

**Trade-offs:**
- Pro: OAuth subscription auth works out of the box
- Pro: Session persistence, context compaction, tool execution all handled
- Pro: Less code to write and maintain
- Con: Designed for agentic coding tasks, not conversational hub-and-spoke — structurally misaligned with Lyra's architecture
- Con: Brings Claude Code's tool system (Read, Edit, Bash) which conflicts with Lyra's custom tools (vault, calendar, memory)
- Con: Session management duplicates Pool's role — two sources of conversation state
- Con: Custom tools harder to integrate alongside built-in tools
- Con: Streaming interface yields `Message` objects, not raw text deltas — adapter integration less flexible

**Rough scope:** M (agent class + adapter changes, less custom code but integration complexity)

### Shape 3: Hybrid — Messages API agent + CliPool streaming upgrade

Two parallel improvements:
1. **Short-term**: Add streaming callbacks to `CliPool` (like 2ndBrain's `on_assistant_message`) + session persistence + model switching. Immediate UX win, no auth change.
2. **Long-term**: Build `AnthropicAgent` with raw Messages API alongside, for opt-in migration.

**Architecture:**
```
Phase A (immediate):
  CliPool._read_until_result() → yield assistant events → callback → adapter.edit_message()

Phase B (parallel):
  AnthropicAgent → AsyncAnthropic.messages.stream() → same streaming pipeline
```

**Trade-offs:**
- Pro: Immediate streaming on CLI path (free with subscription)
- Pro: De-risks SDK migration — CLI gets streaming first, SDK agent inherits same pipeline
- Pro: Session persistence + model switching close 2ndBrain gaps immediately
- Pro: Streaming pipeline built once, used by both backends
- Con: Two phases = more total work (L+ scope)
- Con: CLI streaming is message-level (assistant events), not token-level
- Con: Delays the full SDK benefits (custom tools, system prompt)

**Rough scope:** L+ (CliPool streaming + session persistence + SDK agent + shared streaming pipeline)

## Codebase Impact

### Hub contract change (critical design decision)

`AgentBase.process()` return type becomes `Response | AsyncIterator[str]`:

```python
@abstractmethod
async def process(self, msg: Message, pool: Pool) -> Response | AsyncIterator[str]: ...
```

`Hub.run()` branches on `isinstance`:
```python
result = await agent.process(msg, pool)
if isinstance(result, Response):
    await self.dispatch_response(msg, result)
else:
    await self.dispatch_streaming(msg, result)
```

`ChannelAdapter` Protocol gains a streaming method:
```python
class ChannelAdapter(Protocol):
    async def send(self, original_msg: Message, response: Response) -> None: ...
    async def send_streaming(self, original_msg: Message, chunks: AsyncIterator[str]) -> None: ...
```

### Conversation history format (Pool change)

The SDK requires `list[dict]` (MessageParam format), not Lyra's `Message` objects. Add to `Pool`:

```python
@dataclass
class Pool:
    pool_id: str
    agent_name: str
    sdk_history: list[dict] = field(default_factory=list)  # SDK message format
    # ... existing fields
```

This avoids lossy conversion from `Message` objects on every call. Connects to existing eviction TODO in `pool.py`.

### Auth implementation details

| Method | Header | SDK support | Token refresh |
|--------|--------|-------------|---------------|
| API key | `x-api-key: sk-ant-api...` | Native (`api_key=`) | None needed |
| OAuth token | `Authorization: Bearer sk-ant-oat01-...` | Needs httpx `auth` hook | Every 8h — catch `AuthenticationError`, refresh, retry |

OAuth with raw SDK requires an httpx `auth` hook with 401 retry:
```python
import httpx
import anthropic

class OAuthAuth(httpx.Auth):
    def __init__(self, token: str, refresh_token: str):
        self.token = token
        self.refresh_token = refresh_token

    def auth_flow(self, request):
        request.headers["Authorization"] = f"Bearer {self.token}"
        response = yield request
        if response.status_code == 401:
            self.token = self._refresh()  # call refresh endpoint
            request.headers["Authorization"] = f"Bearer {self.token}"
            yield request

    def _refresh(self) -> str:

        ...

http_client = httpx.AsyncClient(auth=OAuthAuth(oauth_token, refresh_token))
client = anthropic.AsyncAnthropic(http_client=http_client)
```

**Note:** OAuth refresh is deferred from the 2-week appetite. Start with API key auth. OAuth hook added as enhancement.

### Streaming pipeline design (shared across shapes)

```
AgentBase.process() -> Response | AsyncIterator[str]
  ↓
Hub.run(): isinstance check
  ↓ (streaming path)
Hub.dispatch_streaming(msg, chunks: AsyncIterator[str])
  ↓
Adapter.send_streaming(msg, chunks):
  1. Send initial placeholder message → get msg_id
  2. async for chunk in chunks:
       accumulated_text += chunk
       if debounce_elapsed(~500ms):
           await edit_message(msg_id, accumulated_text)
  3. Final edit with complete text
```

Telegram: `bot.edit_message_text(msg_id, text)` — rate limit ~30 edits/min → debounce at ~500ms
Discord: `message.edit(content=text)` — rate limit ~5 edits/5s → debounce at ~1s

## Risks

Ordered by priority (P0 = design around, P3 = monitor):

| P | Risk | Impact | Mitigation |
|---|------|--------|------------|
| P0 | OAuth token expires (8h) | Agent fails mid-conversation | Defer OAuth to follow-on issue. When implemented: httpx `auth_flow` with 401 retry + refresh endpoint. |
| P0 | Streaming rate limits (Telegram ~30/min, Discord ~5/5s) | Edits dropped, API errors | Debounce logic required — send edit every ~500ms (Telegram) / ~1s (Discord), not every token. |
| P1 | Conversation history unbounded | Token budget exceeded, cost spikes | Cap at N turns in `Pool.sdk_history` (connect to existing `pool.py` eviction TODO). |
| P1 | Initial send fails → edit chain breaks | Streaming response lost silently | Catch send failure, fall back to non-streaming `send()` with accumulated text. |
| P1 | Pool lock held during entire stream | "Stop generating" commands queued until stream completes | Known trade-off — document as limitation. Pool lock ensures sequential per-user, which is correct. Future: cancel token via `asyncio.Task.cancel()`. |
| P2 | OAuth revoked by Anthropic | Agent stops working | Dual auth — API key fallback in config. Low probability for personal use. |
| P3 | SDK breaking changes | Agent breaks on update | Pin `anthropic` version in pyproject.toml, test in CI. |

## Fit Check

**Shape 1 (Raw Messages API)** is the best fit:
- Full control over tool system — Lyra needs custom tools (vault, memory, calendar), not Claude Code's built-in tools
- System prompt injection is a first-class parameter — needed for persona (#75)
- Agent runs in Lyra's process — no sandbox, direct access to all services
- OAuth requires a small httpx hook — manageable, and deferred from initial scope
- Aligns with Lyra's existing architecture (asyncio, hub-and-spoke)

**Shape 2 (Agent SDK)** is eliminated:
- Designed for agentic coding tasks, structurally misaligned with Lyra's conversational hub-and-spoke
- Brings Claude Code's tool system which conflicts with Lyra's custom tool design
- Session management duplicates Pool's role — two sources of conversation state

**Shape 3 (Hybrid)** is a valid alternative if CLI streaming is valued as a hedge:
- The frame strategy is "build both, CLI default, migrate when ready" — which Shape 3 directly serves
- However, Shape 1 already provides a CLI fallback (SimpleAgent unchanged) and API key auth eliminates the OAuth failure risk that would make CLI streaming a necessary hedge
- Shape 3 adds ~40% more scope (CliPool streaming + session persistence) for a marginal improvement on the CLI path that will be replaced by the SDK agent anyway
- If CLI streaming is desired independently, it should be a separate issue (#77 2ndBrain parity audit)

**Recommendation:** Go with **Shape 1** (Raw Messages API). Start with API key auth (week 1-2). Add OAuth hook as enhancement (follow-on issue). CLI streaming is out of scope — tracked separately via #77.
