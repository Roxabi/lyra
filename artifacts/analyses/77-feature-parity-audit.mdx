## New Issue Discovered

### Hub Command Sessions (created as sub-issue of #73)

During the audit, we identified that Lyra needs a **command session pattern** — the hub should route specific commands to dedicated lightweight agent sessions with appropriate model/tools/prompt, separate from the main conversation.

**Commands identified from 2ndBrain usage patterns**:

| Command | Session type | Model | Behavior |
|---------|-------------|-------|----------|
| `/add <url>` | Dedicated | Sonnet | Scrape URL + Claude metadata extraction + vault add |
| Bare URL | Dedicated | Sonnet | Same as `/add` — hub detects URL and routes |
| `/explain <text\|url>` | Dedicated | Sonnet | Scrape (if URL) + deep analysis/explanation |
| `/summarize <url>` | Dedicated | TBD | Scrape + concise TL;DR |
| `/search <query>` | Direct (no session) | — | Vault FTS query, return results immediately |

**Why this matters**: In 2ndBrain, the Telegram bot had dedicated plugin handlers for each command type. In Lyra, the agent processes all messages, but some commands benefit from isolated sessions with task-specific prompts rather than polluting the main conversation context.
