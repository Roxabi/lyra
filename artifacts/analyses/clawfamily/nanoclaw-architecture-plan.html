Key Insight
NanoClaw avoids an event bus entirely. All coordination happens through SQLite polling (messages) and filesystem polling (IPC). This keeps the system debuggable and crash-resilient — files persist across restarts.

1 Memory — SQLite + Filesystem Per-Group

SQLite Tables
  • chats + messages
  • scheduled_tasks + task_run_logs
  • router_state + sessions
  • registered_groups
Filesystem Layout
  • groups/{name}/CLAUDE.md — editable memory
  • groups/{name}/conversations/ — archived
  • groups/global/CLAUDE.md — shared read-only
  • data/sessions/{group}/.claude/ — SDK sessions
No Vector Search
NanoClaw uses file-based search (Glob/Grep inside containers) instead of embedding or vector databases. No automatic compaction — sessions are full transcripts managed by the SDK.

GroupQueue States
  • active — container processing messages
  • idleWaiting — 30min keep-alive window
  • pendingMessages — queued for processing
  • retryCount — up to 5 retries
Why Filesystem IPC?
  • Simplicity — just JSON files on disk
  • Crash resilience — files persist across restarts
  • Debuggable — inspect files directly
  • Container-friendly — mounted volumes

Credential Proxy
  • Runs on host port 3001
  • API Key mode: proxy injects x-api-key header
  • OAuth mode: exchanges placeholder for real token
  • Containers never see real credentials
Security Model
  • Project root mounted read-only
  • Writable paths mounted separately
  • Mount allowlist: mount-allowlist.json
  • Blocks .ssh, .aws, .gnupg automatically

7 Architectural Patterns

Pessimistic Timestamps
  • Advance lastTimestamp before processing
  • On crash: skip message (idempotency over completeness)
  • Prevents duplicate sends at the cost of rare missed messages
Idle Container Streaming
  • Containers stay alive for 30 minutes idle
  • Follow-up messages piped to stdin
  • No container restart for multi-turn conversations
Session Per Group
  • Complete isolation between groups
  • No cross-group data leakage possible
  • Each group has own CLAUDE.md memory file
Skills Over Configuration
  • Code transformation, not runtime plugins
  • Users fork and apply skills that modify source
  • ~7K LOC is auditable, modifiable, debuggable

8 Error Handling

Mechanism Config Behavior
Retries Max 5, exponential backoff 5s base delay, doubles each attempt
Container timeout 30 minutes Kill process + mark as failed
Cursor advancement Pessimistic Prevents duplicate sends on crash
Task failures SQLite logging Logged to DB, task marked as paused

9 Lyra Comparison — Adopt vs Study

Adopt for Lyra
  • Credential proxy pattern — never expose real API keys to agents
  • Pessimistic timestamp handling — idempotency over completeness, skip on crash
  • IPC via filesystem — simple, resilient, debuggable inter-process communication
  • Per-group session isolation — no cross-group data leakage
Study Further
  • Container-based agent isolation — Docker overhead vs security tradeoff
  • Skills as code transformation — fork-and-modify vs runtime plugins
  • Idle container streaming — 30min keep-alive for multi-turn efficiency
  • Mount allowlist security model — filesystem-level access control
Key Takeaway
NanoClaw's "small is defensible" philosophy (7K LOC) trades feature breadth for auditability and security. Lyra can adopt its credential proxy and pessimistic patterns while maintaining its own plugin-based extensibility model.
