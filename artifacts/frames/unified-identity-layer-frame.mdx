## Problem

Lyra has no concept of "a person." Every platform identity (`tg:user:7377831990`, `dc:user:389408866774810625`) is a standalone key with no link between them. Trust grants, preferences, memory, and conversation history are all keyed by platform-scoped IDs.

This was invisible when trust resolution happened adapter-side (adapters used bare platform IDs before normalization). Commit `96ac94b` (C3 — trust re-resolution on Hub side, #456) moved auth to the Hub, where `user_id` is already the normalized prefixed form (`tg:user:X`). But `seed_from_config()` still writes grants with bare IDs (`"7377831990"`). The lookup mismatches, the owner is treated as BLOCKED, and messages are silently dropped.

**This is not just a seeding bug.** The key mismatch is the first visible symptom of a missing abstraction. Without a unified identity:

1. **Trust is fragmented.** Granting OWNER on Telegram doesn't make you OWNER on Discord — you need separate grants, and they can drift.
2. **Prefs don't follow you.** TTS language preference set on Telegram is invisible on Discord (`prefs_store` keyed by platform ID).
3. **Memory is siloed.** Conversation context, L0 buffer, extracted facts — all platform-scoped. Lyra on Discord has no idea what you discussed on Telegram.
4. **NATS makes it worse.** When Hub runs as a standalone process (C4, #457), messages arrive over NATS with platform-prefixed IDs. Every consumer (prefs, memory, trust, pool routing) must independently resolve "who is this person." Without a shared identity layer, each will reinvent its own mapping — or not bother.
5. **Bus carries unauthenticated traffic.** C3 removed the adapter-side auth gate. Every message — including from blocked users — now sits on the Bus before TrustGuardMiddleware filters it. A flood from blocked users can exhaust queue slots (100/platform + 500 staging) and backpressure legitimate users.

**Why now:** C3 is deployed and broken in production. The NATS roadmap (C4 Hub extraction, #457) will deepen the dependency on platform-scoped IDs if identity isn't fixed first. Memory L1-L3 (#166/#128, Phase 3) needs a stable user key to associate long-term facts and preferences across sessions and platforms.

## Who

- **Primary:** Mickael — sole operator and end user. Blocked from his own bot right now. Uses both Telegram and Discord daily.
- **Secondary:** Future multi-user scenarios (Phase 5 — semi-public deployment). Any non-admin user on multiple platforms would hit the same fragmentation.

## Constraints

- **Solo dev.** Implementation must be sliceable — the critical auth fix cannot wait for the full cross-platform memory integration.
- **Backward-compatible migration.** Existing `auth.db` grants, `prefs_store` entries, and memory records keyed by platform IDs must be migrated or dual-readable. No data loss.
- **Config format stability.** `owner_users = [7377831990]` in `config.toml` is a clean, platform-native format. Changing it to prefixed IDs shifts complexity to the operator. Prefer keeping config simple and letting the seeding layer add context.
- **NATS opt-in constraint.** Identity must work identically in embedded mode (LocalBus) and distributed mode (NatsBus). No NATS dependency.
- **Bus flooding.** C3 removed the adapter-side auth gate. The identity layer should consider whether a lightweight pre-Bus filter is needed, or whether bounded queues + rate limiting are sufficient.

## Out of Scope

- **Cross-platform conversation merging.** Unifying the user key enables future cross-platform memory recall, but actually merging active Pool state across platforms is a separate design problem (pool routing, scope semantics). This frame covers the identity foundation, not pool federation.
- **OAuth / external identity providers.** No third-party identity federation. Mapping is config-seeded or self-service `/link` within Lyra.
- **Anonymous / public user identity.** Users without a linked identity remain platform-scoped. The identity layer is for known users with explicit grants.

## Complexity

**Tier: F-full** — New abstraction layer spanning 5+ domains (auth, prefs, memory, pool routing, config seeding), requires DB schema migration, introduces cross-platform semantics that don't exist today.

Signals observed:
- **Files touched:** auth_store, authenticator, prefs_store, memory, hub, multibot, config, message, middleware_stages, pool + new identity store + migration + tests = 15+
- **Domains:** auth (trust grants), prefs (user preferences), memory (L0-L3), pool routing (scope resolution), config seeding (bootstrap), adapters (normalization)
- **New patterns:** `users` + `platform_identities` tables, identity resolution protocol, `/link` command flow
- **Unknowns:** migration strategy for existing platform-keyed data, cross-platform pool routing semantics, `/link` challenge flow design, Bus pre-filter trade-offs
- **Arch impact:** Every consumer of `msg.user_id` downstream of the Hub is affected
