Voice & Persona Config #343
DB-first in V11. persona_json and voice_json are stored directly in the agents table. Edit them with lyra agent edit <name> — changes hot-reload without restart. Legacy persona, tts_json, stt_json columns are auto-migrated on AgentStore.connect().
persona_json

Controls the agent's identity, tone, and language behavior.

jsonpersona_json structure
{
  "name":         "Lyra",
  "role":         "Personal AI assistant",
  "language":     "en",           // primary response language
  "instructions": "Be concise and direct. ...",
  "style": {
    "tone":       "professional",   // casual | professional | terse
    "emoji":      false,
    "markdown":   true
  }
}
Persona vs. instructions. instructions in persona_json is prepended to every system prompt. Keep it short — it runs on every turn. Use role for the one-line identity sentence shown in /status.
voice_json

Merged TTS + STT configuration. Replaces the legacy tts_json / stt_json split.

jsonvoice_json structure
{
  "tts": {
    "engine":   "chatterbox",     // chatterbox | coqui | gtts
    "voice":    "en_female_1",
    "speed":    1.0,
    "enabled":  true
  },
  "stt": {
    "engine":   "whisper",        // whisper | vosk
    "model":    "base",
    "language": "en",
    "enabled":  true
  }
}
fallback_language
Column
fallback_language TEXT DEFAULT 'en'

Used when user locale detection fails and persona_json.language is not set. Always has a value — defaults to 'en'.

Priority order
Language resolution

/lang <code> session override → persona_json.language → i18n_language → fallback_language

patterns_json

Trigger patterns used by smart routing to dispatch messages to the right agent or backend.

jsonpatterns_json structure
{
  "trigger": [
    "^/code",          // regex — route to coding agent
    "\\bpython\\b",
    "\\brefactor\\b"
  ],
  "exclude": [
    "^/reset"          // never route these patterns here
  ]
}
Patterns are evaluated against the raw message text before the / prefix check. smart_routing_json on the hub agent defines which agent receives matches from each pattern group.
Hot-reload Behavior
1
Edit persona via CLI
lyra agent edit <name> — opens interactive DB editor, bumps updated_at.
2
Runtime polling detects change
AgentStore polls updated_at on each incoming request. Changed rows are reloaded into the in-memory cache.
3
Next message uses new config
No restart needed. Active sessions pick up the new persona on the next turn.
