## Problem

`Hub.run()` (hub.py:673–738) is a 65-line monolithic message loop with cyclomatic complexity ~15 (flagged `noqa: C901`). Every routing concern — platform validation, rate limiting, binding resolution, agent lookup, pairing gate, command dispatch, circuit breaker, and pool submission — lives in a single `while True` block. This makes the method hard to reason about, test in isolation, and extend.

Additionally, `_pairing_gate_drop` (hub.py:648-649) contains a bare `except Exception: pass` that silently swallows dispatch failures, violating the project's error-handling standards (ADR-017 V-01).

## Who

- **Primary:** Developers maintaining or extending Hub message routing
- **Secondary:** Operators who rely on logs to diagnose message-handling failures

## Constraints

- Must not change external behavior — pure refactor (message routing outcomes identical)
- #204 (Pool → Hub decoupling) has landed — PoolContext protocol is available
- Hub is the central runtime loop — changes must be thoroughly tested
- Audio loop (hub.py:564-585) is a separate concern and out of scope

## Out of Scope

- Refactoring the audio processing loop (`audio_loop`)
- Changing routing semantics or message formats
- Performance optimization (not the goal, though pipeline may enable it later)

## Complexity

**Tier: F-lite** — Single domain (hub core), clear scope (extract pipeline + fix 1 silent except), no new architectural patterns.
