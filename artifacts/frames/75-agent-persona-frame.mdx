## Problem

Lyra's identity is currently a raw system prompt string embedded in the agent TOML config. While it works for basic LLM interactions, this approach falls short in three ways:

1. **No structured persona definition.** Name, personality traits, tone, expertise areas, and behavioral rules are all mixed into a single freeform string. There's no way to programmatically access individual persona attributes (e.g., extracting the agent's name for a greeting, or tone for voice generation).

2. **Not reusable across surfaces.** The persona needs to inform multiple outputs beyond the LLM system prompt: CLAUDE.md project instructions, voice/TTS generation (tone, speaking style), and future channel-specific adaptations. A raw string can't serve these use cases.

3. **No composability.** Adding a new trait or instruction means editing a monolithic string. A structured config would let persona fields be composed, overridden, or extended per context.

## Who

- **Primary:** Mickael — defines Lyra's personality and uses it across chat, voice, and dev tooling.
- **Secondary:** Lyra itself — the persona config shapes how the agent presents across all channels.

## Constraints

- Must extend the existing TOML-based agent config (`lyra_default.toml`), not replace it.
- The `[prompt].system` field must remain functional (backwards-compatible).
- Persona fields must be usable by both the LLM prompt builder and external consumers (voice generation, CLAUDE.md templating).
- Keep the config human-readable and easy to edit — this is a "soul file" that Mickael will craft by hand.
- Existing agent tests must continue to pass.

## Out of Scope

- Per-channel tone adjustments (Discord formal, Telegram casual) — deferred to a future issue.
- Dynamic persona switching at runtime (multiple personalities).
- Memory/context integration (separate issue #73 scope).
- Voice generation implementation itself — this issue only provides the persona data that voice will consume.

## Complexity
