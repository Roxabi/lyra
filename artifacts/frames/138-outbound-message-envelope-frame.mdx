## Problem

Today, agents return a bare `str` (or `AsyncIterator[str]` for streaming) and each channel adapter independently formats it for its platform. Telegram-specific MarkdownV2 escaping, Discord's 2000-character chunking, streaming edit-in-place logic, and button construction all live directly in adapter code — with no typed contract between the hub and the adapter about *what is being asked to send*.

This creates implicit coupling: adapters must make assumptions about content type, formatting intent, and whether a message is a streaming chunk or a final response. Adding a new platform means re-discovering these conventions by reading existing adapter code.

The fix is a single `OutboundMessage` dataclass that the hub and agents produce. Adapters consume it through a uniform `render(msg, context)` method, owning all platform-specific translation internally.

## Who

- **Primary:** The message bus / hub — produces `OutboundMessage` instead of raw strings; no longer needs platform awareness.
- **Secondary:** Channel adapters (Telegram, Discord) — implement `render()` against a stable contract.
- **Downstream unblocked:** #99 (hub command sessions — buttons via `OutboundMessage.buttons`) and #79 (voice TTS — voice notes as `Attachment` on `OutboundMessage`).

## Constraints

- `OutboundMessage` and supporting types (`Button`, `CodeBlock`, `Attachment`) must be defined in `core/message.py` (sibling to `InboundMessage` normalization).
- Streaming path must be preserved: `edit_id` + `is_final=False` drives edit-in-place; `is_final=True` marks the final chunk.
- Telegram adapter: MarkdownV2 escaping, 4096-char chunking, streaming edit, InlineKeyboardMarkup.
- Discord adapter: standard Markdown, 2000-char chunking (split or embed), streaming edit, ActionRow components, thread routing via `thread_id`.
- Hub and agents must produce zero platform-specific fields — all platform logic stays in adapters.
- Should be implemented after or alongside InboundMessage normalization (sibling issue); shares `core/message.py`.

## Out of Scope

- InboundMessage normalization (separate sibling issue).
- Voice TTS integration (#79) — `OutboundMessage` unblocks it but does not implement it.
- Hub command sessions (#99) — unblocked by this, not implemented here.
- New channel adapters beyond Telegram and Discord.
- Rich embed construction beyond what's needed to demonstrate the adapter protocol.
