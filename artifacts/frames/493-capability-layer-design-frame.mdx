## Problem

Lyra has three surfaces that need to invoke capabilities — the LLM turn loop, adapter
automation rules (#475), and slash commands — but no shared way to do so. Without a
common protocol, each surface grows its own parallel implementation of the same underlying
operations (vault writes, web scraping, file reads, shell execution), leading to divergence
and a fragmented mental model.

Additionally, #477 (tool registry) and #478 (plugin SDK) share a hard dependency: #478
must produce objects that satisfy the same `ToolHandler` contract that #477 defines. If
that contract is not locked before either child starts, one issue can silently break the
other mid-flight.

This epic locks the shared design first. Neither child starts implementation until this
is approved.

## Who

- **Primary:** All three entry points — LLM turn loop, adapter rules, slash commands —
  consume the registry through a single dispatch interface.
- **Secondary:** Plugin authors (#478) — must produce `ToolHandler`-compliant objects
  from manifests; the registry has no knowledge of how they were created.

## Constraints

- The `ToolHandler` protocol is the only shared seam between #477 and #478. It must be
  stable before either child ships.
- Three entry points have different execution contexts (multi-turn async, event-driven,
  direct dispatch) but must call the same registered capability.
- `ToolRegistry` must not know about manifests — that is #478's concern.
- Naming is dot-namespaced (`vault.save`, `web.fetch`, `shell.run`) — namespace doubles
  as permission scope for future E-stop / approval flows.
- Shell sandboxing model is unresolved — deferred to #477 spec.

## Design (locked here — children implement against this)

### ToolHandler protocol

```python
class ToolHandler(Protocol):
    name: str           # dot-namespaced: "vault.save", "web.fetch", "shell.run"
    description: str    # shown to LLM in tool definitions
    input_schema: dict  # JSON Schema — validated by registry before execute()
    read_only: bool     # advisory hint for E-stop / approval layer
    async def execute(self, input: dict) -> str
```

### ToolRegistry interface

```python
class ToolRegistry:
    def register(self, handler: ToolHandler) -> None
    def get_definitions(self) -> list[dict]     # Anthropic tool_use format
    async def dispatch(self, name: str, input: dict) -> str
    def get(self, name: str) -> ToolHandler     # for adapter rules + slash commands
```

- `dispatch()` validates input against schema → calls `execute()` → wraps exceptions as
  error strings (never raises to caller)
- `get_definitions()` serializes all registered handlers into the Anthropic `tools` array
- Single module-level singleton; injected into agent at bootstrap

### Entry point contracts

**LLM tool calls** (`SimpleAgent`, implemented in #477):
```
stop_reason == "tool_use"
→ registry.dispatch(name, input)
→ append tool_result to messages
→ re-invoke LLM
→ repeat until stop_reason == "end_turn" (max 10 turns)
```

**Adapter rules** (`ChannelRuleEngine`, consumed by #475):
```
rule pattern matches
→ registry.get("vault.save").execute(input)
→ result injected into rule response template
```

**Slash commands** (new issue — not in #477 or #478):
```
/note:add <text>
→ registry.dispatch("vault.save", {"content": text})
→ confirm to user directly, no LLM
```

### How #478 plugs in

```
manifest file
→ Plugin SDK validates + parses (Pydantic)
→ generates ToolHandler-compliant instance
→ registry.register(handler)
```

Registry has no knowledge of manifests. The seam is exactly `ToolHandler`.
