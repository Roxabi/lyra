🚀 lyra-stack — recommended for production

✦ lyra-stack

A single supervisord instance manages all Roxabi daemons (lyra, voicecli_tts, voicecli_stt…). Each project repo owns its supervisor configs; lyra-stack/conf.d/ contains only symlinks. Recommended for Machine 1 (24/7 hub).
bash
# Clone alongside lyra
git clone https://github.com/Roxabi/lyra-stack ~/projects/lyra-stack

# One-time setup: register lyra symlinks + start supervisord
cd ~/projects/lyra
make register      # registers lyra conf symlinks into lyra-stack

cd ~/projects/lyra-stack
make start         # start supervisord (idempotent)
make ps            # status of all programs

# Lyra-specific shortcuts
make lyra          # start lyra
make lyra reload   # restart
make lyra logs     # tail stdout
make lyra errors   # tail stderr
make lyra status   # status

# Or from inside lyra repo:
cd ~/projects/lyra
make lyra          # delegates to lyra-stack
make deploy        # git pull → pytest → supervisorctl restart on Machine 1
  • Adapters run as independent processes — telegram can restart without touching discord
  • Machine 1 remote control: make remote status, make remote logs, make remote reload
  • make deploy runs git pull + pytest + restart on Machine 1 in one command

Under the hood — message flow
flow
1. Adapter receives message → normalizes to InboundMessage(platform, bot_id, scope_id, text, …)
2. Enqueued on per-platform asyncio Queue (bounded=100, immediate ACK on full)
3. InboundBus staging queue → Hub.run_loop()
4. Hub: resolve_binding(platform, bot_id, scope_id) → agent="lyra_default"
5. Pool created/reused for this scope — one Pool = one isolated conversation
6a. text starts with "/" → CommandRouter.dispatch() → handler → response  (zero tokens!)
6b. normal text → agent.process(msg, pool) → LLM (claude subprocess)
7. OutboundDispatcher: verify routing → deliver reply to originating channel only
Scope isolation
RoutingKey format
# (platform, bot_id, scope_id) → (agent, pool_id)
telegram · lyra · chat:555   → pool telegram:lyra:chat:555
discord  · lyra · thread:888 → pool discord:lyra:thread:888
Two different bots talking to the same user in the same channel have completely isolated history.

TG DM/groupchat:{chat_id}
TG forum topicchat:{id}:topic:{id}
Discord threadthread:{thread_id}
Discord channelchannel:{channel_id}
Agents
Create, edit, and manage agents. Agents live in SQLite (~/.lyra/auth.db) — TOML files are seed sources only.
💾
AgentStore (v4): Agent config is now stored in SQLite (~/.lyra/auth.db). TOML files are seed sources — edit them, then run lyra agent init to import into DB. Runtime reads from DB only.
Create / Update an agent
1
Scaffold a new agent TOML
lyra agent create my_agent   # interactive wizard → writes TOML
📁
User overrides: Place a file at ~/.lyra/agents/my_agent.toml to override the system default without modifying the repo. User-level files take precedence during seed.
2
Edit the TOML
tomlsrc/lyra/agents/my_agent.toml
[agent]
name              = "my_agent"
memory_namespace  = "my_agent"      # isolates memory from other agents
persona           = "lyra_default"  # system prompt from ~/.roxabi-vault
show_intermediate = false           # true → show ⏳ intermediate turns to user
permissions       = []

[model]
backend   = "claude-cli"            # "claude-cli" | "anthropic-sdk"
model     = "claude-sonnet-4-6"
max_turns = 10
tools     = ["Read", "Grep", "Glob", "WebFetch", "WebSearch"]
# Bash + Write omitted — prevents prompt injection → RCE

[agent.smart_routing]               # requires backend = "anthropic-sdk"
enabled = false

[agent.smart_routing.models]
trivial  = "claude-haiku-4-5-20251001"

[plugins]

[workspaces]                         # each key → / slash command

projects = "~/projects"

# Voice config (all optional — omit to inherit voicecli globals)
[tts]

[stt]

Command routing flow
🔀
Full routing architecture — prefix comparison, 6 dispatch layers, ! fallthrough, and decision table are all documented in the 🔀 Routing tab.
flow
User types "/echo hello"
    ↓
CommandRouter.is_command(msg) → True
    ↓
CommandRouter.dispatch(msg)
  ├── parse: command="/echo", args=["hello"]
  ├── plugin registry: "/echo" → cmd_echo handler
  └── calls: await cmd_echo(["hello"], msg)
    ↓
Response("hello") → OutboundDispatcher → User

LLM agent is NEVER called — zero tokens, <100ms

# "!" prefix: same dispatch, but unknown → LLM fallthrough (not error)
User types "!unknown foo"
    ↓ router finds no handler, returns None
    ↓ hub._submit_to_pool() — LLM sees "!unknown foo"

STT — Speech to Text (optional)
✅
STT is optional. Omit STT_MODEL_SIZE from .env → STT is disabled and Lyra starts normally. Set it to enable STT. When set, must have a non-empty value (no default — raises RuntimeError if empty string).
bash.env
# Omit entirely to disable STT
STT_MODEL_SIZE=large-v3-turbo  # no default
STT_DEVICE=auto               # cpu | cuda | auto
STT_COMPUTE_TYPE=auto         # float16 | int8 | auto
📚
Personal vocab: Drop voicecli.vocab at ~/.voicecli/voicecli.vocab (one word/phrase per line). Loaded automatically — shared with the voicecli dictate daemon.

Key architectural decisions
Python + asyncio
Go/Rust/Node eliminated. Python AI ecosystem is unbeatable; asyncio is sufficient for 1–5 I/O-bound users.
Agent = stateless singleton
Immutable config from AgentStore (SQLite) shared across all pools. All mutable state lives in Pool. No duplication, no race condition.
SQLite (not Postgres)
SQLite + WAL + aiosqlite covers personal use. aiosqlite is mandatory — sync SQLite blocks the event loop.
2-machine split
Machine 1 (24/7 hub): Telegram, Discord, STT, embeddings, SQLite. Machine 2 (on-demand): heavy LLM (Phase 2).
Command Routing
How messages dispatch from raw platform event through 6 ordered layers to handler, LLM, or error — and why ! behaves differently from /.
The Two Prefixes
/
Strict mode
Hard command — / prefix
Known command → dispatched to registered handler
Unknown command → returns "Unknown command: /foo" error immediately
LLM never sees an unrecognised / command
!
Soft mode
Fallthrough allowed — ! prefix
Known command → dispatched identically to / — same registry, same handler
Unknown command → router returns None → hub falls through to LLM pool
Only error behavior differs — same command set, same dispatch order

Where Does a New Command Go?
ScenarioLayerHow to add it
Affects session or bot state (clear history, change workspace) Hub Builtin Add to _DEFAULT_BUILTINS in command_router.py + implement handler method
Isolated LLM task that must not pollute chat history Session Add handler in session_commands.py, register in CommandRouter._session_commands
Feature togglable per-agent, hot-reloadable without restart Plugin Create plugins/myname/plugin.toml + handlers.py, enable in agent TOML
Needs LLM reasoning but user wants /-like entry syntax Passthrough Call router.register_passthrough("name") in agent.py, rewrite message before pool
Discord voice channel control (guild-only, stateful, before mention filter) Adapter Add to _handle_voice_command() in adapters/discord.py before mention filter
User wants command syntax but fallthrough to LLM is acceptable ! Prefix Nothing to add — user types !anything and router falls through automatically
Everything else LLM Pool No command needed — plain text routes straight to LLM agent
⚠️
No native platform command menus. Lyra does not call Telegram's setMyCommands() or Discord's app_commands tree. All routing is application-level text parsing — the same command set works on both platforms, plugins hot-reload without Discord tree resync. Adding Telegram autocomplete (cosmetic only): bot.set_my_commands([BotCommand(...)]) at startup.
