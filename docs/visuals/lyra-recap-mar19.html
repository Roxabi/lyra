04 Décisions clés

Per-scope asyncio.Queue — pas de NATS pour l'instant
Problème : plusieurs scopes envoyant en parallèle créaient des interleaves (messages Telegram/Discord mélangés). Solution retenue : une asyncio.Queue par (platform, scope_id) avec un Task drainer dédié — sérialisation garantie par scope, concurrence entre scopes conservée. NATS serait la bonne solution Phase 2 mais introduit une dépendance réseau lourde trop tôt. Compromis : solution in-process, pas distribuée — ok pour Phase 1.
4547421 · docs: 0242b45 40bbe6e 06e0ed7
ProcessorRegistry remplace SessionCommandHandler Protocol
Problème : SessionCommandHandler était un Protocol monolithique — chaque nouvelle session command devait modifier le router. Solution retenue : decorator-based registry @register("/cmd") avec BaseProcessor.pre() + .post(). Ajouter une commande = créer un fichier dans processors/, zéro modif du core. Bénéfice : les réponses arrivent dans l'historique de session (follow-up questions fonctionnels).
6ce0b39 · suppression: session_commands.py
lyra-send comme plugin Claude Code, pas commande intégrée
Pourquoi plugin : lyra-send n'est utile qu'en contexte Claude Code (pas en prod Lyra). En faire un skill permet de l'appeler depuis n'importe quel projet via le marketplace, sans alourdir le runtime Lyra. Architecture : le skill interroge l'API HTTP de Lyra pour envoyer — Lyra expose un endpoint, le skill orchestre depuis Claude Code.
29bee50 · plugins/lyra-send/skills/send/SKILL.md
Arc docs architecture séparés du CLAUDE.md (gap-analysis, current-state, target)
Contexte : Les docs docs/architecture/current-state.md, gap-analysis.md, target-architecture.md sont créés mais pas encore commités. Ils documentent le gap majeur identifié : LlmProvider.stream() absent (seul complete() existe), pas de StreamProcessor domain layer, type outbound AsyncIterator[str] pas encore AsyncIterator[RenderEvent]. Ces docs tracent la direction Phase 2.

06 Mental model essentials

1. InboundBus → Hub → Pool — c'est un pipeline, pas un monolithe

  • Chaque message entre par l'adapter → InboundBus → Hub détermine le pool_id → Pool.submit()
  • Chaque conversation = un Pool isolé avec sa session, son agent, son historique
  • pool_id = <platform>:<bot>:<scope>:<id> — jamais hardcoder un scope

2. OutboundDispatcher : per-scope, pas per-platform

  • Depuis #364 : serialisation garantie par scope (ex. thread Discord ou chat Telegram)
  • Deux scopes différents peuvent envoyer en parallèle — c'est voulu
  • Le circuit-breaker reste par platform — une platform HS coupe tout son scope

3. ProcessorRegistry : decorator pattern, pas router

  • @register("/vault-add") — auto-enregistrement à l'import
  • processors/__init__.py importe tout — ne jamais oublier d'y ajouter un nouveau processor
  • pre() enrichit le message avant LLM — post() fait les side-effects après réponse

4. Agents dans auth.db — TOMLs = seeds uniquement

  • Toujours lyra agent init après édition d'un TOML — sinon aucun effet
  • Le runtime lit uniquement la DB, jamais les TOMLs directement
  • lyra agent init --force pour écraser une row existante

5. Streaming : deux chemins dans simple_agent

  • streaming=True → cli_protocol stream tokens via on_intermediate callbacks
  • Fix du jour (#1a70315) : l'early-return bypassait ces callbacks — maintenant corrigé
  • Le debounce est init au 1er token, pas au démarrage — évite les edits inutiles

6. LlmProvider n'a que complete() — pas de stream()

  • Le streaming passe par on_intermediate callbacks dans complete(), pas un vrai stream protocol
  • Les tool_use events ne sont jamais exposés à l'appelant (gap documenté)
  • C'est un gap connu — LlmProvider.stream() est dans la roadmap pré-Phase 2

07 Hotspots de dette cognitive

Haute
src/lyra/core/cli_protocol.py — +163/-40 non commités, rôle ambigu
C'est le fichier le plus modifié dans le working tree actuel. cli_protocol.py orchestre le streaming bas-niveau entre le subprocess Claude CLI et le Pool. Avec le fix on_intermediate et les changements en cours, son contrat exact n'est pas évident. Plusieurs call sites le consomment différemment.
→ Ajouter un docstring de module expliquant : "Ce module gère X, pas Y. on_intermediate est appelé à chaque token, pas à chaque message. Le path streaming=True bypass N." Commiter les changes en cours avant de continuer.
Haute
33 fichiers modifiés non commités — risque de perte de contexte
Le working tree contient discord_inbound.py, multibot_wiring.py, agent_refiner.py, pool_observer.py, message_pipeline.py, plus une dizaine de fichiers de tests modifiés. Aucun ne correspond à une feature visible. Très probablement des reviews findings en cours (Mar 20 = 4 commits de fix post-review).
→ git stash ou commit atomique par domaine avant de switcher de sujet. Les docs architecture (current-state, gap-analysis, target-architecture) sont aussi non commités — ils devraient l'être.
Moyenne
processors/__init__.py — import manuel obligatoire
Chaque nouveau processor doit être importé dans processors/__init__.py pour s'auto-enregistrer. C'est un gotcha silencieux : oublier l'import = processor jamais chargé, aucune erreur.
→ Ajouter un commentaire explicite dans __init__.py : "Ajouter ici tout nouveau processor — sans cet import, le @register ne s'exécute pas." Ou utiliser pkg_resources/importlib pour l'autodiscovery.
Moyenne
outbound_dispatcher.py — scope reap threshold non documenté
La constante _SCOPE_REAP_THRESHOLD dans outbound_errors.py détermine quand un scope idle est supprimé. Sa valeur et les conséquences d'un mauvais réglage (memory leak vs GC agressif) ne sont pas documentées dans le module.
→ Ajouter un commentaire sur _SCOPE_REAP_THRESHOLD expliquant la formule (ex. "X scopes max en mémoire avant reap — calibré pour N platforms × M users concurrents").
Moyenne
Gap docs non commités : current-state, gap-analysis, target-architecture
Ces trois fichiers capturent l'état exact de l'architecture et les gaps vers Phase 2. Ils existent dans le working tree mais ne sont pas versionnés. Si le contexte est perdu, la direction Phase 2 sera à reconstituer.
→ Commiter les 3 docs + ADR 031 dans un commit docs(architecture): document current state, gaps, and target for Phase 2.
Faible
CVE-2025-69277 pynacl — override uv actif mais non propre
pynacl>=1.6.2 est forcé via [tool.uv] override-dependencies et le TODO #271 est dans pyproject.toml. discord.py 2.7.1 confirme la compatibilité API. C'est une solution fonctionnelle mais fragile si discord.py évolue.
→ Surveiller la prochaine release discord.py pour retirer l'override. Ajouter au changelog Phase 2 prep.
