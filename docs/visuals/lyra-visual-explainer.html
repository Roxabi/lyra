Message Flow — Deep Dive

Full inbound/outbound pipeline, audio path, backpressure, and message envelope.

InboundMessage Envelope
@dataclass(frozen=True)
class InboundMessage:
  id: str
  platform: str               # "telegram" | "discord" | ...
  bot_id: str                 # "main" (one bot per platform)
  scope_id: str               # canonical routing scope (computed by adapter)
  user_id: str                # canonical sender ID (rate-limiting, pairing)
  user_name: str
  is_mention: bool
  text: str                   # normalized plain text (markup stripped)
  text_raw: str               # original text with platform markup
  attachments: list[Attachment]
  reply_to_id: str | None
  thread_id: str | None
  timestamp: datetime
  locale: str | None
  trust: Literal["user", "system"]
  platform_meta: dict         # platform-specific routing data (chat_id, guild_id, ...)
Inbound Pipeline
1
Channel adapter receives event
Telegram on_message / Discord on_message — adapter normalizes to InboundMessage
2
Per-channel inbound queue
Isolated bounded queue per adapter. Provides per-channel backpressure.
3
InboundBus staging queue
Shared bounded queue (maxsize=100). Feeds the Hub consumer loop.
4
Hub.run() → resolve_binding()
Lookup (platform, bot_id, scope_id) → (agent, pool_id) in bindings table.
5
get_or_create_pool()
Pool created lazily. TTL eviction prevents memory leaks (#205).
6
pool._process_loop
asyncio.Task per pool. Sequential within scope, parallel across scopes.
7
agent.process(msg, pool)
Stateless agent processes message with pool's mutable state.
Outbound Pipeline
1
Agent returns OutboundMessage
Carries RoutingContext (channel + bot_id) for verification (#152).
2
OutboundDispatcher.enqueue()
Routes to the correct per-channel outbound queue by platform.
3
Adapter verifies routing
Verifies channel + bot_id matches before sending. Prevents misrouting.
4
Platform API call
Telegram sendMessage / Discord channel.send
Backpressure

When staging queue is full (100 messages): adapter sends immediate ack ("message received, ~Xs wait"), then performs await bus.put() — blocks until slot frees.

Architecture Decisions

Key choices made, why, and what was explicitly rejected. Living ADR log.

Language & Runtime
ADR-001

Python + asyncio over Go / Rust / Zig / Node

Adopted

Decision: Python 3.12 + asyncio everywhere for backends, CLIs, and agents.

Why: Python's AI/ML ecosystem is unbeatable (fastembed, faster-whisper, aiogram, aiosqlite). asyncio is sufficient for 1–5 I/O-bound users. Other stacks offer no compensating benefit at this scale.

❌ Go — no AI ecosystem ❌ Rust — too low-level for prototyping ❌ Zig — experimental ❌ Node — ecosystem mismatch for AI
ADR-002

Python-first Library Paradigm

Adopted 2026-03-08

Decision: Every project ships as a Python library (importable, __all__ contract) + thin CLI shell. No logic in cli.py.

Why: Lyra calls from voicecli import generate_async directly — no subprocess, no HTTP server. Uniform patterns across all projects. Type-checked, IDE-navigable APIs.

Rule: Library first → CLI exposes it. Models load lazily (no side effects on import). Cross-project deps via uv add --editable path/to/lib.

Infrastructure
ADR-003

2-Machine Split — Hub + AI Server

Adopted

Decision: Machine 1 (RTX 3080, 24/7) handles Hub + TTS + embeddings + STT. Machine 2 (RTX 5070 Ti, on-demand) handles heavy LLM inference.

Why: Eliminates VRAM contention. Machine 1 stays within 6 GB / 10 GB budget. Machine 2 can run Qwen 14B or Gemma 27B uncontested.

Constraint: Machine 1 must never shut down. Machine 2 is optional — cloud LLM is the fallback when it's off.

ADR-004

Cloud LLM by Default — Local LLM in Phase 2

Adopted

Decision: AnthropicSdkDriver + ClaudeCliDriver for Phase 1. OllamaDriver (Machine 2) deferred to Phase 2 via NATS worker.

Why: Cloud LLM unblocks Phase 1 immediately. Local LLM adds infra complexity (NATS, worker health, circuit breaker) that isn't justified until Phase 1 is stable.

Abstraction: LlmProvider protocol (#123 ✅) means swapping drivers requires zero core changes.

ADR-005

asyncio.Queue Now, NATS in Phase 2

Adopted

Decision: Phase 1 bus = asyncio.Queue (in-process, bounded, simple). Phase 2 replaces with NATS + JetStream for persistence and Machine 2 coordination.

Why: asyncio.Queue is sufficient for single-process personal use. Adding NATS before Phase 1 is stable would complicate debugging with no user-facing benefit.

Data & Storage
ADR-006

SQLite over PostgreSQL

Adopted

Decision: SQLite + WAL mode + aiosqlite for all persistence (memory, sessions, vault).

Why: Personal use. Single-writer workload. Zero ops overhead. WAL mode gives concurrent reads. FTS5 and sqlite-vec are built-in extensions — no separate search service needed.

❌ PostgreSQL — ops overhead, overkill ❌ Redis — extra process, no persistence guarantee
ADR-007

fastembed ONNX over sentence-transformers

Adopted

Decision: fastembed ONNX runtime for embeddings instead of sentence-transformers + PyTorch.

Why: ONNX runtime is non-blocking — no run_in_executor needed. sentence-transformers blocks the asyncio event loop. nomic-embed-text via ONNX gives the same quality at lower overhead.

ADR-008

aiosqlite is Mandatory — Not Optional

Constraint

Fact (not a choice): Synchronous SQLite in an asyncio event loop blocks all coroutines. aiosqlite wraps SQLite in a background thread, making it safe in async contexts. This is non-negotiable.

Agent & Routing Model
ADR-009

Stateless Agent Singleton

Adopted

Decision: Agent = immutable config (prompt, permissions, namespace). All mutable state lives in the Pool. Agent is a singleton shared across all pools.

Why: Eliminates duplication and race conditions. Multiple pools can share one agent with zero coordination overhead. Upgrade path: add agent_state: dict to Pool if atomic SLMs need sub-ms recall — zero refactoring.

ADR-010

scope_id Replaces user_id in RoutingKey

Adopted (#125)

Decision: RoutingKey = (platform, bot_id, scope_id). scope_id is the canonical conversation scope, not the sender.

Why: Group chats and threads have multiple senders but one pool. Routing by user_id would create one pool per user in a group — wrong. Scope captures the conversation unit correctly: chat:NNN, thread:NNN, channel:NNN.

ADR-011

External Tools — Install, Wrap, Declare

Adopted

Decision: External CLIs (VoiceCLI, ImageCLI, Google Workspace) follow a 3-layer pattern:

LayerWhatWhere
InstallCLI binary on PATH via setup.shHost machine
WrapThin roxabi-plugins skill (SKILL.md only)roxabi-plugins repo
DeclareAgent TOML declares tool accesslyra repo

Why: No forking or vendoring. Upstream maintains the CLI; we maintain the skill wrapper. Bash allowlist now, MCP later.

Inter-machine & Network
ADR-012

httpx HTTP/2 over gRPC

Adopted

Decision: httpx async + HTTP/2 for inter-machine communication (Machine 1 ↔ Machine 2). OpenAI-compatible API on Machine 2.

Why: gRPC is unnecessary at personal-use throughput. httpx is already a project dependency. OpenAI-compatible API means any LLM runtime (Ollama, llama.cpp) works without a custom protocol.

❌ gRPC — overkill, extra tooling ❌ WebSocket — not needed for req/resp LLM calls
Memory Scope Decisions
ADR-013

Reduced Memory Scope for Phase 1

Adopted

Decision: Phase 1 ships only L0 (working) + L3 (semantic). Levels 1, 2, 4 deferred until real need arises.

Why: L1 (session) and L2 (episodic) add complexity with marginal value at Phase 1 scope. L4 (procedural) requires L3 to be proven stable first. "Add when the real need arises" is the guiding principle.
