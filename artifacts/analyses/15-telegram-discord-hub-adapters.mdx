## Source

Issue #15 + frame doc: the hub skeleton (D3-D4) is live. D5 goal is real traffic. Scope expanded
to Telegram + Discord in parallel. Core constraint surfaced during framing: not a library wiring
task — it is a routing model design problem (channel × bot_id matrix, platform context
heterogeneity, attribution, security).

## Problem

The current `Message.channel` is a plain string (`"telegram"` or `"discord"`). The current
`BindingKey(channel, user_id)` has no `bot_id` dimension. This means:

- Only one Telegram bot and one Discord bot can be registered simultaneously
- No way to route by bot identity (e.g. LegalTech bot vs personal assistant bot)
- Platform-specific context (Discord thread ID, Telegram topic, forum post) lives in untyped
  `metadata` — invisible to the type checker and easy to drop
- Attribution is incomplete: a message knows its `user_id` but not *which bot received it*

The hub needs a **routing key** that encodes `(platform, bot_id, user_id)` and a
**platform context** type that captures Discord/Telegram specifics without losing them.

Reference benchmarks used: **OpenClaw** (Node.js, 50+ channels), **Vercel AI SDK**
(multi-provider message lifecycle), and **Chat SDK** (chat-sdk.dev — multi-platform bot SDK
with Telegram + Discord adapters).

## Outcome

Lyra can run N Telegram bots and M Discord bots simultaneously on the same hub, with correct
message routing and no loss of platform-specific context, while preserving all existing Telegram
behavior. Operators can wire a new bot by registering an adapter and a binding — no changes to
the hub core.

## Reference Benchmarks

### OpenClaw (Node.js)

**Session key pattern**: `agent:<id>:<channel>:<account>:dm:<peer>`
- `channel` = platform name
- `account` = bot token identity (multi-bot support)
- `peer` = the user or group being spoken to

**Binding precedence** (first match wins):
1. Exact peer match (DM/group ID)
2. Parent peer inheritance
3. Guild + roles
4. Guild ID
5. Account ID
6. Channel-wide fallback

**What we adopt**: the `(platform, bot_id, user_id)` routing key structure. Binding precedence
is a future enhancement — for Phase 1 we implement exact + wildcard only.

**What we skip**: static config-file bindings (Lyra uses code-registered bindings), workspace
directory isolation (Lyra uses pool objects).

### Vercel AI SDK

**Two-tier message model**:
- `UIMessage` — full application state (persisted, metadata-rich)
- `CoreMessage` — lossy format for LLM consumption (`convertToModelMessages()`)

**What we adopt**: separating the *hub message* (full attribution + platform context) from the
*LLM input* (minimal turn history). `Message` = hub message. When dispatching to the LLM,
adapters convert to a minimal context list. This avoids bloating the LLM context with routing
metadata it doesn't need.

**Message parts** (Text, Tool, File, Reasoning): directly applicable to `MessageContent` — the
existing union type is the right direction. We extend it but don't redesign it.

**What we skip**: React hooks, streaming to UI clients (not relevant for Python asyncio).

### Chat SDK (chat-sdk.dev)

TypeScript multi-platform bot SDK supporting Telegram, Discord, Slack, Teams, and others.
Directly implements both adapters we need — most applicable reference for input/output handler
patterns.

**Normalized message model** (platform-agnostic):
```
id          → platform message ID
threadId    → "adapter:channel:thread"   ← confirms our pool ID format
text        → plain text content
author      → { userId, userName, fullName, isBot, isMe }
isMention   → bool  ← required for group chat dispatch
attachments → Attachment[]
metadata    → { timestamps, edit status }
```

**What we adopt**:
- Thread ID format `"adapter:channel:thread"` = our pool ID `f"{platform}:{bot_id}:{user_id}"` —
  identical structure, independently derived
- `author.isBot` + `isMention` fields → add to `Message` for group chat skill dispatch
- `message_id` on `DiscordContext` → needed for reply targeting (already flagged by architect)
- Two-tier output: `thread.post()` for new messages vs `sent.edit()` for streaming updates

**Telegram adapter specifics (from Chat SDK):**
- **Webhook-only** — no polling. Requires a live HTTP endpoint receiving POST from Telegram.
  In Lyra: FastAPI route (`/webhooks/telegram/{bot_id}`) running as an asyncio task alongside
  the hub.
- Secret token validated via `X-Telegram-Bot-Api-Secret-Token` header on every request.
- `bot.getMe()` called at startup to auto-detect bot username for mention matching.
- Known limitation: Telegram does not expose full message history to bots — adapter only
  caches messages it has seen. Pool history is the source of truth for context.

**Discord adapter specifics (from Chat SDK):**
- **Two reception modes** — critical distinction:
  - HTTP Interactions endpoint → receives button clicks, slash commands, verification pings only.
    **Does NOT receive regular messages.**
  - Gateway WebSocket (persistent connection) → required for regular `on_message` events.
- For Lyra Phase 1 (text messages only): **Gateway mode is mandatory**.
- `discord.py` Gateway client must share the same asyncio event loop as the hub:
  `asyncio.gather(hub.run(), discord_client.start(token))` — they are co-routines in one loop.
- Requires `Message Content Intent` enabled in Discord Developer Portal — without it, message
  text is empty.
- Requires `Server Members Intent` for `author.userName` / `author.fullName` resolution.

**Adapter startup pattern** (derived from Chat SDK + discord.py + python-telegram-bot v20):
```python
# All three run as co-routines in the same event loop
async def main():
    await asyncio.gather(
        hub.run(),                          # hub bus consumer loop
        telegram_app.run_webhook(...),      # FastAPI + PTB webhook
        discord_client.start(token),        # discord.py gateway
    )
```

**What we skip**: TypeScript/JSX cards, Redis state layer (Lyra uses Pool objects + aiosqlite),
serverless deployment model (Lyra is a persistent process on Machine 1).

## Shapes

### Shape A — Compound channel string (minimal)

Keep `Message.channel` as a string but use a compound format: `"telegram:bot_main"`,
`"discord:bot_server1"`. `BindingKey` stays `(channel, user_id)` unchanged. Platform context
goes in `metadata`.

```python
# Example
msg = Message(
    id="...",
    channel="telegram:bot_legaltech",  # compound string
    user_id="user_123",
    ...
    metadata={"thread_id": "...", "topic_id": "..."},  # untyped
)
key = BindingKey("telegram:bot_legaltech", "user_123")
hub.register_adapter("telegram:bot_legaltech", adapter)
```

**Trade-offs:**
- Pro: Zero changes to `BindingKey`, `Hub`, `Pool`, existing tests
- Pro: Backward compatible — existing Telegram adapter just changes its channel string
- Con: Platform context in untyped `metadata` — type checker can't help, easy to drop fields
- Con: Routing logic must string-split `channel` to extract platform or bot_id
- Con: Brittle if naming conventions drift (`:` separator collisions possible)

**Rough scope:** XS — 1 day

---

### Shape B — Typed routing key + PlatformContext (recommended)

Replace `BindingKey(channel, user_id)` with `RoutingKey(platform, bot_id, user_id)`. Add a
typed `PlatformContext` union to `Message`. Register adapters by `(platform, bot_id)` pair.

```python
class Platform(str, Enum):
    TELEGRAM = "telegram"
    DISCORD = "discord"

@dataclass(frozen=True)
class TelegramContext:
    chat_id: int
    topic_id: int | None = None      # supergroup topics
    is_group: bool = False

@dataclass(frozen=True)
class DiscordContext:
    guild_id: int
    channel_id: int
    message_id: int                  # needed for reply targeting in dispatch_response
    thread_id: int | None = None     # threads, forum posts
    channel_type: Literal["text", "thread", "forum", "voice"] = "text"

PlatformContext = TelegramContext | DiscordContext

@dataclass
class Message:
    id: str
    platform: Platform               # NEW: coarse platform type
    bot_id: str                      # NEW: which bot received this
    channel: str                     # deprecated alias for platform.value — removed after migration
    user_id: str
    user_name: str                   # NEW: display name (from author.userName in Chat SDK)
    is_mention: bool                 # NEW: was the bot @-mentioned? (Chat SDK: isMention)
    is_from_bot: bool                # NEW: is sender a bot? (Chat SDK: author.isBot)
    content: MessageContent | str
    type: MessageType
    timestamp: datetime
    platform_context: PlatformContext  # NEW: typed, not metadata
    trust: Literal["user", "system"] = "user"
    metadata: dict = field(default_factory=dict)

class RoutingKey(NamedTuple):        # replaces BindingKey
    platform: Platform
    bot_id: str
    user_id: str                     # "*" wildcard supported

# Hub adapter registry keyed by (platform, bot_id)
hub.register_adapter("telegram", "bot_legaltech", telegram_adapter)
hub.resolve_binding(msg)  # uses msg.platform + msg.bot_id + msg.user_id
```

**Implementation requirements (non-negotiable):**
- `dispatch_response` must switch to `adapter_registry[(msg.platform, msg.bot_id)]` in the
  **same commit** as `RoutingKey` is introduced. There is no valid intermediate state where
  the registry is still `dict[str, ...]` while `RoutingKey` is live.
- Pool IDs must incorporate `bot_id` to avoid namespace collision: canonical format is
  `f"{platform}:{bot_id}:{user_id}"`. The `register_binding` collision guard must validate
  uniqueness across `(platform, bot_id, user_id)` triples, not just `(channel, user_id)`.

**Trade-offs:**
- Pro: Type-safe — `TelegramContext.topic_id` vs `DiscordContext.thread_id` are explicit
- Pro: Routing key encodes full attribution; no string parsing
- Pro: Aligns with OpenClaw's `(channel, account, peer)` session key model
- Pro: Adding a 3rd platform (Signal, Slack) is additive — new `PlatformContext` variant
- Pro: `RoutingKey` is the natural key for rate limiter and sender allowlist
- Con: `Message` changes break existing tests (2 files, ~60 lines — acceptable with no live traffic)
- Con: `Hub.adapter_registry` type changes from `dict[str, ChannelAdapter]` to
  `dict[tuple[Platform, str], ChannelAdapter]`
- Con: Slightly more upfront work (~200 lines changed vs XS Shape A)

**Rough scope:** M — 2–3 days

---

### Shape C — Adapter instance as identity (adapter-first)

Each bot instance is a uniquely named adapter. `Message.channel` = adapter instance name
(e.g. `"telegram_legaltech"`, `"discord_server1"`). Add `platform` field for coarse grouping.
`BindingKey` stays `(channel, user_id)` — `channel` is now the adapter instance name.

**Trade-offs:**
- Pro: Zero changes to `BindingKey`, Hub routing logic, `Pool` — only 1 new field on `Message`
- Pro: Simplest implementation path
- Con: **Breaking conceptual change**: `channel` was the stable public contract for routing
  (documented as `"telegram" | "discord"`); it silently becomes an opaque adapter instance name.
  Every piece of code that treats `channel` as a platform name is now wrong.
- Con: Platform context still untyped in `metadata`
- Con: Querying "all Telegram messages" requires filtering `platform` not `channel`

**Rough scope:** S — 1 day

## Fit Check

| Criterion | Shape A | Shape B | Shape C |
|-----------|---------|---------|---------|
| Multi-bot support | ✓ (string hack) | ✓ (typed) | ✓ (named adapter) |
| Type-safe platform context | ✗ | ✓ | ✗ |
| Zero Telegram regression | ✓ | ✓ | ✓ |
| Hub routing logic unchanged | ✓ | ✗ | ✓ |
| Attribution completeness | partial | full | partial |
| Extensible to 3rd platform | brittle | additive | additive |
| Implementation scope | XS | M | S |
| Aligns with OpenClaw model | ✗ | ✓ | partial |
| `channel` field contract preserved | ✓ | partial (alias) | ✗ (broken) |
| Input handler model understood | ✓ | ✓ | ✓ |
| Output handler model understood | ✓ | ✓ | ✓ |

**Recommended: Shape B.**

Rationale: the routing key and platform context are the *invariant core* of this system. Getting
them right now avoids a painful migration once real traffic flows. The cost is ~200 lines of
changes to `message.py`, `hub.py`, and tests — all of which have no live traffic yet. Shape A
and C both defer the typing problem into `metadata`, which will surface when the first Discord
thread message silently drops its `thread_id` or the first rate limiter tries to key on
`(bot_id, user_id)` and has to parse a string.

Shape C is the worst option despite the lowest scope: it silently breaks the documented meaning
of `Message.channel` (from platform name to adapter instance name) without any type-system
enforcement. Anyone reading the code later will be confused.

**Migration path:**
- Telegram adapter is implemented first and verified against existing tests (zero regression).
- `Message.channel` is kept as `platform.value` (deprecated alias) until both adapters pass.
- Discord has no existing adapter — no regression risk. Discord scope for D5-D7: **text channel
  messages only** (slash commands, forum threads, and DMs are explicitly deferred to a follow-up
  issue).
- If Telegram adapter work reveals deeper coupling, Discord is deferred to D6+. The routing
  model (Shape B) is the minimum viable delivery for the week; Discord adapter is bonus.

## Appetite

1-week cycle (D5–D7). This is a design-first issue. Implementation effort is ~300–400 lines
including tests.

**Minimum viable outcome for the week**: routing model (Shape B) + Telegram adapter live.
**Discord adapter**: target D6-D7, deferrable to the following cycle if Telegram takes longer.

## Open Questions for Spec

1. **Observability on unmatched routing key**: if `resolve_binding(msg)` returns `None`, what
   does the hub do? Log and drop? Dead-letter queue? Explicit error reply? Must be defined before
   the run loop is implemented.
2. **Bot token rotation**: what happens when a bot token is rotated or revoked mid-session?
   Which pools are invalidated? Deferred to operations docs, but the adapter interface should
   not make rotation impossible (e.g. no singleton token stored at import time).
3. **Discord event scope**: text channel messages only (D5-D7). Slash commands, threads, DMs,
   and forum posts are explicitly deferred to a follow-up issue.
4. **Discord Gateway intents**: `Message Content Intent` and `Server Members Intent` must be
   enabled in the Discord Developer Portal before the adapter can receive message text or resolve
   usernames. This is a manual setup step — must be documented in the spec's prerequisites.
5. **Telegram webhook registration**: the webhook URL must be registered with Telegram via
   `setWebhook` at adapter startup. The FastAPI route (`/webhooks/telegram/{bot_id}`) must be
   reachable from Telegram's servers — requires a public URL or ngrok tunnel for local dev.
   Machine 1 (24/7, public IP) is the production target.
6. **`is_mention` dispatch rule**: in group chats, should the hub only dispatch to skills when
   `is_mention=True`? Or always? This is a per-binding policy decision for the spec.

## Security Notes

From the frame, applied per Shape B:
- **Webhook origin verification**: handled at adapter level (Telegram: `X-Telegram-Bot-Api-Secret-Token`; Discord: gateway WebSocket, token never logged)
- **Sender allowlist**: keyed on `(bot_id, user_id)` — `RoutingKey` is the natural key
- **Bot token secrets**: env vars only, not logged — convention enforced in adapter base class
- **Rate limiting**: per `RoutingKey` token bucket (shape B makes this trivially type-safe)
- **Hub isolation**: internal bus unreachable from network (unchanged)
