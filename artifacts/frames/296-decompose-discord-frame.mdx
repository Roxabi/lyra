## Problem

`adapters/discord.py` is 1,452 lines — the largest file in the project. The `DiscordAdapter` class alone is ~1,150 LOC mixing five distinct concerns: gateway handling, text formatting, audio processing, voice mode, and thread lifecycle. This makes the file hard to navigate, test in isolation, and modify safely.

Parent issue #293 targets decomposing all oversized modules. This is the discord adapter leg of that effort, following the same extract-and-delegate pattern already applied to `hub.py` (#305) and `agent.py` (#295).

## Who

- **Primary:** Developers maintaining or extending the discord adapter — formatting changes, audio pipeline tweaks, and thread logic updates currently require navigating a 1,400-line file.
- **Secondary:** Reviewers — smaller, focused modules are easier to review and reason about.

## Constraints

- Must not change runtime behavior — pure structural refactor.
- `_shared.py` (273 LOC) already provides `chunk_text`, `push_to_hub_guarded`, `buffer_audio_chunks` — new modules must reuse these, not duplicate.
- All existing tests must pass (`uv run pytest`), linter must stay clean (`uv run ruff check .`).
- Follows the same decomposition pattern as #305 (hub.py) and #295 (agent.py).

## Out of Scope

- Refactoring `_shared.py` or moving shared utilities.
- Changing the `DiscordAdapter` public API or adapter protocol.
- Extracting helpers (`_reply_safe`, `_start_typing`, `_cancel_typing`, `_resolve_channel`) — these stay in the main adapter as they're tightly coupled to gateway state.
- Extracting `DiscordConfig` / `load_discord_config` — only ~20 LOC, not worth a separate module.

## Complexity

**Tier: F-lite** — Clear scope, single domain (discord adapter), well-defined extraction targets with no architectural decisions needed.

Signals observed:
- Single domain: all changes within `adapters/` directory
- Clear extraction targets: formatting (~150 LOC), audio (~150 LOC), threads (~100 LOC)
- Established pattern: follows #305 and #295 decomposition approach
- No new abstractions or interfaces needed — extract functions/methods, update imports
- Acceptance criteria are concrete and measurable (≤300 LOC target)
