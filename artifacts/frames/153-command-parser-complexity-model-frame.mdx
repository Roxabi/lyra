## Problem

The system already has two partial implementations:

1. **`CommandRouter`** (`src/lyra/core/command_router.py`) ŌĆö routes `/`-prefixed commands to plugin handlers, but:
   - Only supports `/` prefix; `!` prefix messages fall through to the LLM as plain text
   - No `CommandParser`/`CommandContext` abstraction ŌĆö parsing is embedded in `dispatch()`
   - No command ŌåÆ (agent_id, pool_id) routing table; commands route to *handlers*, not agents

2. **`ComplexityClassifier`** (`src/lyra/llm/smart_routing.py`) ŌĆö classifies text using keyword/word-count heuristics with 4 levels (TRIVIAL, SIMPLE, MODERATE, COMPLEX), but:
   - Operates on raw text only ŌĆö no attachment signals, turn count, or command context
   - Model mapping is per-agent TOML config (4 levels ŌåÆ model strings), not a centralized `LOW/MEDIUM/HIGH ŌåÆ Haiku/Sonnet/Opus` table

The gap: without a `CommandParser`, `/imagine` is treated as a chat message routed to the default LLM agent instead of the image agent. Without rich `ComplexityEstimator` signals, short messages with attachments or long sessions are misclassified as TRIVIAL and served by Haiku when they need Sonnet.

## Who

- **Primary:** Lyra hub pipeline ŌĆö `Hub.run()` + `CommandRouter.dispatch()` are the integration points
- **Secondary:** Agent owners who add new commands (they need the routing table entry, not code changes)

## Constraints

- Must not break existing `CommandRouter` plugin system (plugins register via `PluginLoader`)
- `SmartRoutingDecorator` already exists ŌĆö `ComplexityEstimator` must integrate with it, not replace it
- `!` prefix is reserved for skill-style commands (lighter-weight, no pool); `/` is agent commands
- `InboundMessage` type is the source of truth ŌĆö no new message fields without updating `Message` dataclass
- Dynamic model upgrade mid-generation is explicitly deferred to a follow-up issue

## Out of Scope

- Dynamic upgrade mid-generation (complex streaming concern, future issue)
- Memory isolation (#memory-isolation ŌĆö separate issue)
- New LLM providers or model configs beyond Haiku/Sonnet/Opus mapping
- Changing how `PluginLoader` registers commands

## Complexity

**Tier: F-lite** ŌĆö Clear scope across 2 files (`command_router.py`, `smart_routing.py`) + new `command_parser.py`, single domain (core routing pipeline), no unknowns beyond the `!`-prefix handler contract.

Signals observed:
- Files touched: ~5 (command_parser.py new, command_router.py, smart_routing.py, agent.py config, hub.py integration)
- Technical risk: known patterns, extends existing abstractions
- Architectural impact: shared pipeline types (CommandContext added to InboundMessage)
- Unknowns: 1 (how `!`-prefix handler contract differs from `/`-prefix in practice)
