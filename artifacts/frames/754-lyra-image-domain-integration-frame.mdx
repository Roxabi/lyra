## Problem

ADR-047 (revised 2026-04-16) codifies the connector ownership split: **satellite-owned workers, lyra-owned contracts/publishers/nkeys**. The satellite half already shipped — imageCLI#50 landed `imagecli nats-serve` on 2026-04-15, subscribing to `lyra.image.generate.request` via `NatsAdapterBase`. But the lyra side is hollow: there is no authoritative wire contract for `lyra.image.*`, no hub-side publisher to emit requests from agents or plugins, no `image-worker` nkey identity in `gen-nkeys.sh`, and no supervisor entry wiring the satellite worker into Machine 1.

Until all four pieces land, the subject is effectively undefined — the satellite worker cannot authenticate against NATS, agents have no typed client to publish through, and the ownership table in ADR-047 carries an "in progress" row. This issue closes the lyra half of the image domain connector, mirroring what #766 just did for voice and what #658 did originally for the voicecli contract.

## Who

- **Primary:** lyra hub maintainer wiring the image publisher into the agent/plugin layer — owns the contract ADR, the client shape, and the nkey/ACL surface.
- **Secondary:** imageCLI satellite maintainer (consumes the published contract and the provisioned nkey seed); reviewers auditing alignment with ADR-044 / ADR-046 / ADR-047; future domain connectors (memory, llm) that will use this as the template alongside voice.

## Constraints

- **Contract shape must follow ADR-044** — subject tree, envelope with `contract_version: "1"`, queue group `IMAGE_WORKERS`, explicit error codes, user-controlled field enumeration for sanitization.
- **Binary payload domain — max payload declaration required** per ADR-047 Rule 4 (images are binary like audio). Must pick an explicit byte ceiling and document it in the ADR.
- **Publisher shape follows `nats_tts_client.py` / `nats_stt_client.py`** — subject constants, serialization, heartbeat tracking, error mapping. No new patterns; lift the existing shape.
- **nkey provisioning must honor ADR-046 Invariant 2** — `image-worker` added to `IDENTITIES` in `deploy/nats/gen-nkeys.sh`, ACL subject-scoped (subscribe `lyra.image.generate.request`, publish `lyra.image.generate.heartbeat`), no cross-domain access.
- **Supervisor integration** — `imagecli_gen` entry in `deploy/supervisor/conf.d/` generated via `make gen-conf` from `deploy/agents.yml`, `startsecs` tuned for image model warmup on RTX 3080, env carries the nkey seed path.
- **No inline dataclasses or subject f-strings** on the lyra side — when roxabi-contracts gains a `voice`-style `image` submodule (future), the publisher must be able to swap to it cleanly. For this issue: tolerate inline defs (matches current `nats_tts_client.py` pre-#766) but shape them to be portable.
- **ADR-047 ownership table must be updated** — image row status → shipped, with links to this PR and the nkey/auth.conf regeneration.

## Out of Scope

- Extracting the image contract into `packages/roxabi-contracts/src/roxabi_contracts/image/` — the contracts domain port for image is tracked separately (parallel to #763 for voice).
- Any behavioral change to `imagecli nats-serve` on the satellite side — already landed via imageCLI#50.
- Image model selection, GPU tuning, or prompt/negative-prompt defaults on the satellite — driven by imageCLI config.
- Agent/plugin-layer UX (command parsing, progress reporting to the user) — consumed downstream of the publisher but out of scope here.
- Tagging / publishing `roxabi-contracts` with an image submodule — happens in a later epic.
- Auth.conf rollout to Machine 1 production — regenerating auth.conf is in scope, pushing/reloading NATS is a separate deploy task.

## Complexity

**Tier: F-lite** — Single domain (NATS transport + deploy wiring), clear scope enumerated in the issue acceptance checklist, no new architecture (contract shape follows ADR-044, publisher follows `nats_{tts,stt}_client.py`). Four concrete artifacts: one ADR, one Python module, two deploy file edits. No unknowns that require `/analyze`.

Signals observed:
- Single domain: `src/lyra/nats/` + `deploy/nats/` + `deploy/supervisor/` + `docs/architecture/adr/` — no cross-package Python runtime change beyond the new client module.
- Pattern-lift: contract shape is ADR-044 applied to image; publisher is `nats_{tts,stt}_client.py` with image field names.
- Enumerable acceptance: 6 acceptance criteria in the issue body are all mechanically verifiable.
- One judgment call: the max payload ceiling for image responses (binary domain — ADR-047 Rule 4) needs a concrete number backed by expected output resolutions, documented in the ADR.
- Dependencies resolved: imageCLI#50 (done), ADR-047 (codified 2026-04-16).
