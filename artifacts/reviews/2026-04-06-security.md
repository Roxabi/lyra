## nkey Authentication (343ff69)

**Where are keys stored?**
Seed files live at `/etc/nats/nkeys/{hub,llm-worker,monitor}.seed`, mode 0600, root-owned. Path is injected via `NATS_NKEY_SEED_PATH` env var; no key material appears in source or `.env.example`. The env var is correctly excluded from the example file (commented out, labelled production-only).

**Is auth enforced server-side or just client-side?**
Client-side wiring only in this commit. The NATS server config (`nats.conf`) already requires nkey auth per the frame doc, but production currently runs `nats-local.conf` (no auth) because the Python side was not providing credentials until now. This commit unblocks switching to `nats.conf` but does not make that switch — the server-side enforcement is a follow-up deployment action. Until `nats.conf` is activated in production, NATS remains unauthenticated on the wire.

**Rotation story?**
Not addressed. `gen-nkeys.sh` generates once; no rotation mechanism or secret-age check exists. Seeds are long-lived.

**Verdict: WARN**
The implementation is correct and well-tested. The gap is operational: dev connects without auth (by design), prod still runs unauthenticated NATS until `nats.conf` is deployed. No code bug — deployment gap only.

## platform_meta Sanitization (1f0c911)

**Which fields are sanitized? How?**
Allowlist in `src/lyra/nats/_sanitize.py:13-23` covering nine keys: `guild_id`, `channel_id`, `message_id`, `thread_id`, `channel_type`, `chat_id`, `topic_id`, `is_group`, `thread_session_id`. All other keys, including any `_`-prefixed internal keys, are stripped. Applied at the NATS receive boundary in `NatsBus._staging` handler and in `inbound_audio_legacy.py`.

**Is there a risk of prompt injection via platform_meta reaching LLM?**
Platform_meta values are used for routing logic only (pool_id construction, session lookup) — they do not appear in prompt templates or LLM context in the reviewed code paths. However, value types and lengths are not validated: an attacker who can publish to the NATS bus could inject an arbitrarily large string into `thread_session_id`, which is passed verbatim to `TurnStore.get_session_pool_id()` as a SQLite query parameter. The query is parameterized (`WHERE session_id = ?`) so SQLi is not possible, but there is no length cap.

**Is it validated at inbound NATS boundary or later?**
At the NATS receive boundary — correct placement. The `dataclasses.replace()` call in `nats_bus.py:268-273` replaces `platform_meta` before the item reaches the staging queue, so downstream code sees only sanitized keys.

**Verdict: WARN**
Key allowlisting is solid. Residual gap: values of allowlisted keys are untyped and unbounded. A crafted oversized `thread_session_id` string traverses the pipeline and reaches the DB. Impact is low (parameterized query prevents injection; memory cost is bounded by NATS message size), but it is worth noting.

## Cache Bounding + TTL Reaper (4f5c884)

**What caches were unbounded?**
`NatsOutboundListener._cache` (inbound message correlation cache) and `_stream_outbound` / `_stream_queues` (streaming chunk assembly). Streams had no timeout: a hung stream goroutine would hold queue objects indefinitely.

**New limits and TTLs — are they appropriate?**
- `_MAX_CACHE_SIZE` (inbound cache): hard eviction of oldest entry when full, with warning log. Size was already present but the eviction logging was improved.
- Stream timeout: `asyncio.wait_for(q.get(), timeout=120.0)` added in `_events()` coroutine. 120s is generous but finite — appropriate for a chatbot response window.
- TTL reaper: `_REAPER_INTERVAL_SECONDS = 30` with a separate background task evicting stale `_cache_ts` entries. No max TTL value visible in this diff for the reaper threshold — would need the full `nats_outbound_listener.py` to confirm, but the architecture is correct.

**Memory exhaustion / DoS risk assessment**
Before this commit: a flood of unique stream_ids from the NATS bus would cause unbounded growth of `_stream_queues` and `_stream_outbound`. After: streams time out at 120s and are garbage-collected. The inbound cache evicts on overflow. Risk is materially reduced. Residual: `_stream_queues` growth between reaper cycles at high message rates — no per-connection rate limit exists.

**Verdict: PASS**
Both the inbound cache and stream assembly are now bounded. TTL reaper architecture is sound. No new vulnerabilities introduced.

## Trust Model Assessment

The NATS bus is a semi-trusted internal boundary. The trust model after these commits is:

- **Adapter → NATS → Hub**: adapters publish `InboundMessage` with `platform_meta`. After 1f0c911, keys are allowlisted at the hub-side NATS receive boundary before entering the pipeline. This is correct placement.
- **Hub → NATS → Adapter**: hub publishes `OutboundMessage`. Adapter deserializes via `deserialize_dict`. No sanitization on this path, which is acceptable since hub is the trusted originator.
- **NATS authentication**: currently optional (dev mode). Server-side enforcement pending activation of `nats.conf`. Until then, any process that can reach `localhost:4222` can publish to any NATS subject without credentials.
- **Session scope isolation**: added in 1f0c911. `thread_session_id` from untrusted `platform_meta` is now validated against `pool_id` ownership before a session resume is accepted. This closes the cross-pool session hijack vector.

The trust boundary is clear and improving. The main residual gap is that NATS server-side auth enforcement is not yet activated in production.
