## Problem

The codebase has no automated guards on code complexity or file/function size. Large, complex functions can slip through review unnoticed, accumulating technical debt as the codebase grows. With P0 priority, this should be addressed before the codebase grows further.

## Who

- **Primary:** Developers contributing to Lyra — enforced guardrails prevent complexity creep during normal development
- **Secondary:** Reviewers — automated checks reduce cognitive load during code review

## Constraints

- Must use **ruff** (already the project linter) — no new tools
- Thresholds must be **standard Pylint/McCabe defaults** (complexity 10, statements 50, args 5, ~500 lines/file)
- All existing code must pass or have justified `noqa` annotations
- File length enforcement needs a mechanism (CI script or pre-commit hook) since ruff doesn't natively support it

## Out of Scope

- Stricter-than-default thresholds (can tighten later)
- New linting tools beyond ruff
- Refactoring existing code to reduce complexity (only fix violations or add justified `noqa`)

## Complexity

**Tier: F-lite** — Single domain (tooling config), clear scope, touches `pyproject.toml` + potentially a CI/pre-commit script + fixing any existing violations.

Signals:
- Single domain: developer tooling / linter config
- Clear deliverable: ruff rule config + file length check
- No architectural decisions needed
- May require fixing existing violations across multiple files
