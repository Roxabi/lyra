## Problem

V1 (Telegram adapter) and V2 (STTService) are merged — audio messages reach agents as `AUDIO`-typed `Message` objects carrying an `AudioContent` payload with a temp file path. But agents don't act on them yet: `AnthropicAgent` and `SimpleAgent` have no `AUDIO` branch, so voice messages silently fall through without transcription or reply.

This slice wires the STT layer into both agents: call `STTService.transcribe()`, prepend a grounding prefix for the LLM call (while keeping raw transcript in history), handle empty/noise results with a retry prompt, and guarantee temp file deletion via `try/finally` in all paths (success, empty, exception). It also wires `STTService` instantiation in `__main__.py` with startup config validation.

## Who

- **Primary:** Lyra users sending voice messages via Telegram — they expect a text reply without invoking `/transcribe`.
- **Secondary:** Developers extending agents — the `AgentBase.stt` slot establishes the injection pattern for future agent types.

## Constraints

- ADR-013: temp file cleanup is owned by the agent, not `STTService`. The `try/finally` block must live in `process()`.
- Grounding prefix (`🎤 ...`) must be included in the LLM call text but stripped from `pool.sdk_history` to avoid polluting conversation context.
- `STTService` is optional (`stt: STTService | None`) — agents must remain functional when STT is not configured.
- Integration tests must cover all 4 paths: success, empty transcript → retry, transcription exception → error reply, temp file always deleted.

## Out of Scope

- V4 (observability / metrics) — separate issue.
- Telegram adapter changes — already done in V1.
- STTService internals — already done in V2.
- Discord adapter STT wiring — not part of this slice.
