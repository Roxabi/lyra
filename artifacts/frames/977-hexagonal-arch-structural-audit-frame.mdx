## Problem

The Lyra codebase has grown to 50+ modules across `src/lyra/` and two packages, but the structural enforcement hasn't kept pace. A full audit surfaced five concrete violations of the declared hexagonal architecture ‚Äî ranging from a kernel inversion (domain importing from application layer) to a circular dependency between the two shared packages. The declared layer model in `docs/ARCHITECTURE.md` and `.importlinter` covers the main stack but leaves six cross-cutting modules unowned and the entire `packages/` directory outside quality gate enforcement.

The violations are not theoretical: the kernel inversion means `core/` is coupled to `lyra.commands` (a plugin namespace), the circular package dep means `roxabi-contracts` and `roxabi-nats` cannot be independently versioned or tested, and the floating modules (`obs/`, `stt/`, `tts/`, `errors.py`, `config/`, `integrations/`) have no import rules ‚Äî anyone can depend on them from any layer with no enforcement.

## Who

- **Primary:** all contributors to the Lyra codebase ‚Äî violations will compound as the codebase grows
- **Secondary:** downstream consumers of `roxabi-nats` and `roxabi-contracts` packages ‚Äî circular dep blocks clean independent versioning

## Constraints

- `core/` must remain the hexagonal kernel: zero upward imports after fix
- `.importlinter` is the enforcement mechanism ‚Äî all layer rules must be expressible there
- `packages/roxabi-nats` and `packages/roxabi-contracts` are published packages ‚Äî breaking changes require semver bump
- Quality gate scripts in `tools/` are synced from `roxabi-plugins` ‚Äî changes to the gate logic itself must go through that repo; only config (`qg.conf`, exemption files) is project-local
- 23 `core‚Üíinfrastructure` exemptions are tracked per-ADR ‚Äî removal must be coordinated with the ongoing stores migration (#935)

## Out of Scope

- Renaming or reorganizing `src/lyra/adapters/` internals (telegram, discord, clipool) ‚Äî layout is stable
- Changes to `bootstrap/` wiring logic ‚Äî composition root is intentionally import-everything
- Adding new agents or commands
- Any change to `packages/roxabi-contracts` schemas (contract versioning is a separate concern)

## Complexity

**Tier: F-full** ‚Äî multi-domain refactor touching core domain layer, two published packages, quality gate config, and `.importlinter` rules simultaneously.

Signals observed:
- Multiple independent domains: `core/`, `packages/roxabi-nats`, `packages/roxabi-contracts`, `tools/`
- New abstraction required: `shared/` layer (new namespace, new import rules)
- Unknowns: best strategy for breaking the circular package dep (extras vs. separate test-helpers package vs. moving helpers into roxabi-nats)
- Risk: touching `core/agent/agent.py` (kernel inversion fix) requires verifying all callers of `PLUGINS_DIR` resolve correctly after the change
- 23 exemptions interact with the active stores migration (#935) ‚Äî sequencing matters
