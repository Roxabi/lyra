## Problem

Audio outbound has no Hub-level dispatch entry point. `render_audio()` and `render_audio_stream()` exist on both adapters, and `OutboundDispatcher.enqueue_audio()` was added in #175, but there is no `Hub.dispatch_audio()` or `dispatch_audio_stream()` to complete the pipeline.

The streaming audio path is worse: adapter `render_audio_stream()` methods call `self.render_audio()` directly, completely bypassing the dispatcher — circuit breaker ownership, queue ordering, and observability are lost for streamed audio.

Without this work, agents and the TTS pipeline have no typed, channel-agnostic way to send audio through the bus. Text has `dispatch_response`, attachments have `dispatch_attachment`, but audio has nothing at the Hub level.

## Who

- **Primary:** Lyra engine — unified outbound path for all message kinds (text, streaming, audio, attachments)
- **Secondary:** Operators — consistent circuit breaker behavior and failure observability across all outbound kinds

## Constraints

- Must follow existing dispatcher pattern (`dispatch_*` on Hub → `enqueue_*` on OutboundDispatcher → worker loop → adapter call → CB record)
- `ChannelAdapter` protocol already defines `render_audio()` — no signature changes needed
- Streaming audio must drain cleanly when CB is open (same pattern as streaming iterator drain)
- Cherry-picks audio-related items from #186: `_AUDIO_EXTS` whitelist move to `_shared.py`, streaming helper dedup

## Out of Scope

- TTS provider selection or routing changes
- New observability/metrics infrastructure (use existing CB recording)
- Changes to `render_audio()` / `render_audio_stream()` adapter implementations (beyond wiring)
- Non-audio items from #186 (chunk_text, _msg, platform guards, test cleanup)
- Live audio streaming (#185 — separate F-full concern)

## What's Missing

| Component | Gap |
|-----------|-----|
| `Hub.dispatch_audio()` | No hub-level entry point for single audio |
| `Hub.dispatch_audio_stream()` | No hub-level entry point for streaming audio |
| `OutboundDispatcher.enqueue_audio_stream()` | No dispatcher support for streaming chunks |
| `OutboundAudioBus` | No dedicated bus class for outbound audio routing |
| Streaming audio CB ownership | `render_audio_stream()` calls adapter directly, bypassing dispatcher |
| `_AUDIO_EXTS` in `_shared.py` | Whitelist only in Discord adapter (from #186) |

**Tier: F-lite** — single domain (outbound audio path), clear patterns to mirror from existing text/attachment dispatch, well-defined gaps.

- Files: `hub.py`, `outbound_dispatcher.py`, `_shared.py`, adapters (wiring only), tests
- Patterns: mirrors `dispatch_attachment()` + `dispatch_streaming()` implementations
- New class: `OutboundAudioBus` follows `InboundAudioBus` structure
