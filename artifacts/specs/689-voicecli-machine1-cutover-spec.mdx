## Context

Promoted from [689-voicecli-machine1-cutover-frame.mdx](../frames/689-voicecli-machine1-cutover-frame.mdx). Executes Slice S3 of parent spec [658-voicecli-nats-decouple-spec.mdx](658-voicecli-nats-decouple-spec.mdx).

Machine 1 currently runs four voice-related supervisor programs:

| Program | Command (today) | Role |
|---|---|---|
| `voicecli_stt` | `voicecli stt-serve` (Unix socket) | native STT worker |
| `voicecli_tts` | `voicecli tts-serve` (Unix socket) | native TTS worker |
| `lyra_stt` | `lyra adapter stt` → `stt_adapter_standalone.py` | NATS ⇄ voicecli bridge |
| `lyra_tts` | `lyra adapter tts` → `tts_adapter_standalone.py` | NATS ⇄ voicecli bridge |

After S2 (voicecli sibling repo merged), `voicecli nats-serve {stt,tts}` exists. This slice swaps the `voicecli_{stt,tts}` supervisor commands over to the NATS mode, leaving `lyra_stt` / `lyra_tts` **untouched** (S4 deletes them). The NATS subject contract is frozen by ADR-044; hub code is already contract-stable.

## Goal

`voicecli_{stt,tts}` on Machine 1 serve voice requests over NATS (`lyra.voice.{stt,tts}.request`) with heartbeats visible from hub, verified by `make voice-smoke` round-trip, with a one-command rollback.

## Users

- **Primary:** Mickael (ops) — runs the cutover, owns rollback
- **Secondary:** Lyra voice-note users (Telegram + Discord) — cannot tolerate voice-path outage > ~30 s during swap

## Expected Behavior

1. Pre-flight: ADR-044 merged (#658 S1 done); voicecli sibling repo released with `nats-serve` (S2 done); NATS server running on Machine 1 with auth.conf enforcing nkey ACLs for voice subjects.
2. Ops provisions the nkey seed file under `~/.lyra/nkeys/` readable by the voicecli supervisor user.
3. Ops updates `deploy/supervisor/conf.d/voicecli_{stt,tts}.conf` on Machine 1: `command=` → new `run_voicecli.sh {stt,tts}` wrapper (or direct `voicecli nats-serve`); env vars `NATS_URL`, `NATS_NKEY_SEED_PATH`, `LYRA_TTS_ENGINE` set.
4. `supervisorctl reread && supervisorctl update` reloads the two programs. GPU loads; first heartbeat on `lyra.voice.{stt,tts}.heartbeat` arrives within `startsecs` (re-tuned if > 10 s).
5. Hub logs show `worker_id=voicecli-<host>-<pid>` heartbeats (competing-consumer queue group — only one of voicecli / lyra satellite answers each request).
6. Ops runs `make voice-smoke` → passes (voice note → transcript → reply audio round-trip).
7. If any step 4–6 fails, single rollback command flips back to lyra satellites within 15 s TTL.

### Decision — nkey provisioning (lift-and-rename vs fresh)

Two options for M5. Deferred from parent spec to this slice.

| Option | Pros | Cons |
|---|---|---|
| **A. Reuse** `~/.lyra/nkeys/{stt,tts}-adapter.seed` (reassign owner to voicecli user) | Zero new auth.conf changes; ACLs already provisioned by #714 for `{stt,tts}-adapter` role | Role name is misleading (adapter ≠ voicecli); coupling two ownership changes in one step (file owner + process change) |
| **B. Provision fresh** `~/.lyra/nkeys/voice-{stt,tts}.seed` with new `voice-{stt,tts}` role in auth.conf | Clean role name; clean ownership transition; enables S4 to delete `{stt,tts}-adapter` nkey roles cleanly | Requires auth.conf edit + NATS reload before cutover; one extra step in runbook |

**Recommendation: Option B** — cleaner audit trail, aligns role naming with actual subscriber, avoids hidden coupling. Extra step is small (edit + `nats-server --signal reload`). Locked at spec approval unless user overrides.

## Edge Cases

| Case | Handling |
|---|---|
| voicecli `nats-serve` sibling PR not yet merged | Block: PR cannot land until sibling is deployable. CI gate: verify `voicecli>=X` version tag exists before merge. |
| First-heartbeat time > `startsecs=10` | Supervisor marks a process RUNNING after `startsecs` seconds of uninterrupted uptime (time-based, independent of application heartbeats). A process that starts cleanly but takes 20 s to emit its first NATS heartbeat is already RUNNING from supervisor's view. The `startsecs` tuning matters only if the process is crash-looping during GPU warm-up (supervisor treats < `startsecs` exits as start failures and can enter `FATAL`). Measure first-ready-time on first boot; increase `startsecs` only if warm-up exits unexpectedly before it, and document the chosen value in the PR body. |
| Competing-consumer race (lyra_stt answers before voicecli during transition) | Expected & safe — NATS queue group guarantees single-reply per request_id. S3 deliberately leaves lyra_stt / lyra_tts running; S4 deletes them after confidence window. |
| nkey seed file missing or wrong perms | voicecli fails fast at startup; supervisor logs `NATS connect: authorization violation`. Rollback immediately. |
| auth.conf ACL misconfigured (publish allowed, subscribe denied) | voicecli connects but never receives requests. Hub sees no heartbeat from voicecli worker_id → lyra satellite keeps answering. Smoke will pass (via lyra path) but cutover has silently failed. **Detection:** runbook step 2 requires observing `worker_id=voicecli-*` specifically, not just any heartbeat. |
| NATS server not reloaded after auth.conf edit | Same symptom as above. Runbook step: `nats-server --signal reload` or `systemctl reload nats`; verify reload timestamp in logs. |
| Ops on wrong machine | Runbook step 0: `hostname` must equal `roxabituwer`. |
| GPU OOM during parallel stt + tts warm-up | Stagger: `supervisorctl start voicecli_stt`, wait for heartbeat, then `supervisorctl start voicecli_tts`. Document in runbook. |
