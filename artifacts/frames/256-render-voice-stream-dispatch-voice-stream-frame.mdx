## Problem

Lyra can generate TTS audio chunks asynchronously, but there is no path to deliver those chunks to an active Discord voice session. The `DiscordAdapter` lacks `render_voice_stream()`, the `Hub` lacks `dispatch_voice_stream()`, and `PCMQueueSource` is only a stub (landed in #255). Without these, streamed TTS audio never reaches the VoiceClient.

This is Child Issue B of the live audio streaming sub-epic (#185). Issue #255 (VoiceSessionManager + PCMQueueSource stub) must merge first.

## Who

- **Primary:** Lyra bot users in a Discord voice channel — they trigger a TTS response and hear audio streamed in near-real-time.
- **Secondary:** Lyra developers integrating new adapters — `dispatch_voice_stream` establishes the Hub-level dispatch pattern for voice, which future adapters can follow.

## Constraints

- Must follow the existing `Hub.dispatch_audio_stream` pattern for routing through `OutboundDispatcher` with a direct-adapter fallback.
- `PCMQueueSource.read()` must re-frame arbitrary-length input into exactly 3840-byte frames (20 ms @ 48 kHz stereo 16-bit PCM) — discord.py's VoiceClient requires fixed-size reads.
- `is_final=True` chunk triggers EOF + auto-leave in `TRANSIENT` session mode.
- `discord.py[voice]` and `PyNaCl` must be declared in `pyproject.toml`.
- Blocked by #255; both issues should land in the same release cycle to avoid a transient pyright gap on `render_voice_stream`.

## Out of Scope

- Non-Discord adapters (Telegram, etc.) — voice streaming is Discord-only at this stage.
- Multi-guild simultaneous streaming edge cases beyond what `VoiceSessionManager` already handles.
- Session lifecycle management beyond the `TRANSIENT` auto-leave already specified.
- Error recovery / reconnect logic for dropped voice connections.

## Complexity

**Tier: F-lite** — Clear scope, single domain (Discord adapter layer), all interfaces defined by parent spec (`artifacts/specs/185-live-audio-streaming-spec.mdx` Slice B), no architectural unknowns.
