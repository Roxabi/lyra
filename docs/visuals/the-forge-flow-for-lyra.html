Quick Wins

Adopt as-is or with minor changes. No new tooling required — pure prompt/convention upgrades.

📡
Agent Status Protocol
quick win dev-core roxabi-plugins
TFF does
Every sub-agent MUST return one of: DONE, DONE_WITH_CONCERNS, NEEDS_CONTEXT, BLOCKED. The orchestrator branches on these — not on a guess.
We adopt
Add the 4-status protocol to every dev-core sub-agent prompt. Orchestrator workflows (implement, code-review, fix) branch on the returned status explicitly.
DONE — criteria met, tests pass, files changed
DONE_WITH_CONCERNS — done but agent has doubts
NEEDS_CONTEXT — missing info, re-dispatch with more
BLOCKED — fundamental problem, escalate
✅
Self-Review Checklist
quick win dev-core agents
TFF does
Before reporting DONE, every agent mentally runs: Completeness (did I implement everything?), Quality (is this my best work?), Discipline (no overbuilding?), Verification (evidence, not assumptions).
We adopt
Add the 4-axis checklist as a closing block in every dev-core agent definition: backend-dev, fixer, tester, frontend-dev. One short paragraph, big impact on output quality.
## Before reporting DONE, verify:
Completeness: all requirements met, edge cases handled
Quality: clean, clear names, maintainable
Discipline: YAGNI — only what was asked
Verification: evidence only — no "should work"
🧩
Formalize the WHO / HOW Split
quick win roxabi-plugins docs
TFF does
TFF explicitly articulates: Skills = knowledge fragments teaching HOW. Agents = roles defining WHO does the work. These are orthogonal and independently composable.
We adopt
Our plugin system already follows this but we haven't documented it. Add a "Skills vs Agents" section to roxabi-plugins README and marketplace.json. Makes onboarding faster.
# Skills teach HOW
hexagonal-architecture.md
test-driven-development.md

# Agents define WHO
backend-dev → loads hexagonal + tdd skills
tester → loads tdd skill only

Skill Drafter + Detect-Patterns
quick win roxabi-plugins
TFF does
Two dedicated commands: /tff:suggest-skills (AI analyzes completed work, proposes skill fragments) and /tff:detect-patterns (finds recurring code patterns worth extracting).
We adopt
Add a tff-skill-drafter-style agent to dev-core. When a dev session uncovers a reusable pattern, the agent drafts a new skill TOML. Feeds our plugin ecosystem organically.
# After completing a feature:
/suggest-skills
→ agent analyzes commit diff
→ identifies reusable fragments
→ drafts skill .md in roxabi-plugins

Strategic Upgrades

Requires design work and iteration but uses existing infrastructure. High leverage.

〰️
Wave Execution in dev-core:implement
medium effort dev-core:implement dev-core:plan
TFF does
Tasks in a plan are topologically sorted. Dependencies are tracked explicitly. Independent tasks in each wave are dispatched to parallel agents. Checkpointed after every wave.
We adopt
dev-core:plan should output a dependency graph. dev-core:implement should read it, compute waves, and spawn agents in parallel per wave. Currently we spawn sequentially.
# Plan output adds deps:
T01: deps=[] → Wave 0
T02: deps=[] → Wave 0 (parallel)
T03: deps=[T01,T02] → Wave 1
T04: deps=[T01] → Wave 1 (parallel)
🎯
Spec-Reviewer Before Code-Reviewer
medium effort dev-core:code-review dev-core:fix
TFF does
On /tff:ship, spec-reviewer runs first — checks every acceptance criterion against the actual code. Only if this passes does code-reviewer run for quality.
We adopt
dev-core:code-review currently jumps straight to quality. Add a Stage 0 spec-compliance pass: reads acceptance criteria from the issue/spec, checks implementation. Saves wasted quality reviews on incomplete work.
# Updated /code-review flow:
Stage 0: spec-reviewer — criteria met?
↓ if pass:
Stage 1: code-reviewer — quality
Stage 2: security-auditor (always)
💾
Checkpoint / Resume Pattern
medium effort dev-core:implement dev-core:dev
TFF does
After each wave completes, TFF writes CHECKPOINT.md with: completed tasks, current wave, worktree state. /tff:resume reads it and continues exactly where it left off.
We adopt
dev-core:implement and dev-core:dev should write artifacts/checkpoint.md after each significant step. Long sessions (F-full tier) can resume mid-implementation without losing context.
# artifacts/checkpoint.md
session: 2026-03-21T14:22
issue: #42 — auth module
wave: 1 of 3
completed: [T01, T02]
remaining: [T03, T04, T05]
🎭
Orchestrator Pattern — Formalized
medium effort roxabi-plugins dev-core docs
TFF does
Workflows follow strict rules: only 5 step types — Check, Spawn, Handle, Transition, Suggest. No reading codebases, no implementing, no decisions in the orchestrator itself.
We adopt
Our skills already thin-orchestrate but some drift. Add the 5-step model + anti-patterns to CLAUDE.md and to the roxabi-plugins skill authoring guide. Prevents bloated skill files.
# Every workflow step is exactly one of:
Check → read state
Spawn → dispatch Agent
Handle → process result
Transition → update state
Suggest → show next step

Future Investments

Significant tooling or architectural change. Worth tracking — not worth rushing.

🧠
Dual State — Markdown + SQLite Status
heavy investment 2ndBrain lyra knowledge
TFF does
Content lives in markdown (human-editable, git-diffable). Status, IDs, and dependency graph live in Dolt (queryable, structured). Neither source is authoritative for the other's domain.
We adopt
2ndBrain already uses markdown + SQLite. The concept of content in markdown, metadata in DB is exactly what roxabi-vault does. We could formalize the sync rules: vault is authoritative for tags/status, markdown is authoritative for body content.
# 2ndBrain dual state
vault.db ← status, tags, links (db wins)
notes/*.md ← content body (md wins)
sync ← bd-style CLI: vault sync
🫧
Beads / Dolt — Structured Task Tracker
heavy investment dev-core
TFF does
Beads gives atomic task claiming (bd update --claim), bd ready --json for unblocked tasks, dependency-aware scheduling, and Dolt's version-controlled SQL for full history.
We adopt
GitHub Issues + dev-core handles this today, but lacks atomic claiming and dependency-aware "what's ready" queries. Worth evaluating beads as a dev-core backend once wave execution is proven in.
# Beads advantages vs GitHub Issues:
bd ready → only unblocked tasks
bd update --claim → atomic assign + start
dolt diff → versioned task history
offline → no GitHub API needed
