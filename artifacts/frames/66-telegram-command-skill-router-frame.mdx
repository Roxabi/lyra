## Problem

Lyra's Telegram adapter forwards all messages to a single agent as raw text. There is no command system — no `/agenda`, no `/search`, no `/save`. The 2ndBrain Telegram bot has 8 plugins with 19+ commands, and migrating to Lyra means rebuilding that command routing.

Without a command router, Lyra agents cannot leverage the roxabi-plugins skill ecosystem (google-workspace, voice-cli, vault, web-intel, image-prompt-generator) via Telegram. Users must rely on free-text prompts and hope the LLM infers the right tool — unreliable and slow.

The trigger: ADR-010 (Install, Wrap, Declare) defines how external tools integrate. The router is the missing bridge between Telegram messages and those tools.

## Who

- **Primary:** Mickael — daily Telegram user sending commands to Lyra for calendar, tasks, vault, voice
- **Secondary:** Any future Lyra user on any channel (Discord, Signal) — the router should be channel-agnostic

## Constraints

- Must gracefully degrade when CLIs are not installed (e.g., `gws` not on PATH → inform user, don't crash)
- Phase 1 only: skills executed via agent prompt + Bash tool (per ADR-010 Layer 3). No MCP yet.
- Command registry must be config-driven (TOML), not hardcoded
- Router sits in the hub message pipeline, before agent dispatch
- Must not break existing free-text message flow — unknown commands and plain messages fall through to the default agent

## Out of Scope

- MCP-based skill execution (Phase 2 — when Lyra supports MCP client)
- Actual wiring of each skill (gws, voicecli, vault) — separate issues per skill
- Command argument parsing beyond simple positional args (complex CLI-style parsing is a future concern)
- Permission system for commands (all commands available to all users for now)

## Complexity

**Tier: F-lite** — Single domain (backend/hub), clear scope, well-defined input/output. Extends the existing hub pipeline with a command detection + routing layer. No new architectural patterns — config-driven registry + message interception.

Signals:
- ~5-8 files (router module, registry, TOML config, tests, hub integration)
- Single domain: backend (hub message pipeline)
- No unknowns: command routing is a solved pattern
- Dependencies are internal only (hub, agents, adapters already exist)
