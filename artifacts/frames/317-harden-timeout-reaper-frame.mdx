## Problem

The timeout system and reaper process have 6 gaps that reduce operational flexibility, observability, and recovery speed. Key timeouts (`idle_ttl`, `default_timeout`) are hardcoded with no config exposure. The turn-level timeout is disabled, allowing chatty stuck processes to run indefinitely. Idle evictions are silent to the user. Pool eviction leaves orphaned CLI subprocesses alive for up to 20 minutes. The reaper task has no health monitoring. Dead-process entries linger in the pool until the next sweep.

None are critical bugs today, but collectively they create a fragile system where stuck processes go undetected, resources leak, and operators have no tuning knobs without code changes.

## Who

- **Primary:** Lyra operators — need config knobs, monitoring, and clean process lifecycle
- **Secondary:** End users — experience silent context loss on idle eviction, and potential degraded performance from orphaned processes

## Constraints

- No breaking changes to existing config files — new keys are additive
- Must preserve current default behavior (existing timeouts unchanged unless explicitly reconfigured)
- Item 3 (user notification) requires a design choice on delivery mechanism (PoolObserver callback vs dispatch callback)

## Out of Scope

- Rewriting the CLI pool architecture
- Changing the supervisor/process model
- Adding new monitoring dashboards (only exposing data in existing `/health` endpoint)
- Persistent session recovery after eviction
