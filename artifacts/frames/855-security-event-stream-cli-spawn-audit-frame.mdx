## Problem

The `skip_permissions` flag (`cli_protocol.py:78-79`) silently appends
`--dangerously-skip-permissions` to every CLI subprocess spawn when an agent
runs unrestricted. No record links: which agent triggered the unrestricted
turn → what tool allowlist was active → what message ID or trace context drove
it. If an agent deletes files or runs destructive Bash, there is no forensic
trail to reconstruct why and under whose authority.

This was flagged as §P0 #5 in the 2026-04-22 quality audit
(`artifacts/analyses/quality-audit/AUDIT-SUMMARY.md`). The gap is especially
acute given Lyra's multi-agent pool design: multiple agents can be running with
`skip_permissions=True` concurrently with no per-event accountability.

The fix is a structured security event stream. Every CLI spawn emits a
`SecurityEvent` to JetStream subject `lyra.audit.>` — durable, independently
retained from app logs, and queryable by future SIEM/alerting consumers. This
is the observability dual of the trace context already flowing through NATS
(M3 epic #667).

## Who

- **Primary:** Operators running Lyra in production multi-agent deployments —
  need to audit which agents ran unrestricted, when, and why.
- **Secondary:** Security reviewers, compliance teams, future SIEM / alerting
  consumers reading from `lyra.audit.>`.

## Constraints

- `SecurityEvent` dataclass/schema must land in `packages/roxabi-contracts/`
  (existing NATS contract package) — no new packages.
- `core/audit/` module does not yet exist; must be created within the
  `src/lyra/core/` layer.
- Audit emission must be async-safe and non-blocking — must not slow down the
  CLI spawn hot path; failures must not propagate to callers (fire-and-forget
  with structured error logging).
- Must carry `trace_id` so audit events join the OTel trace from M3 (#667).
- Adoption scope: CLI subprocess spawn sites in `cli_protocol.py` and
  `cli_pool_lifecycle.py` (3–4 sites; analysis will confirm exact count).
- JetStream stream provisioning for `lyra.audit.>` is in scope (stream
  creation / retention config).
- `core/auth/guard.py` GuardChain runs at request-time; audit runs at
  spawn-time — these are parallel, not sequential. Guard rejects; audit records.

## Out of Scope

- Layer 2: policy-based per-tool permissions (`Read=allow`, `Bash=ask`, etc.)
  — filed separately when multi-tenant threat model justifies.
- SIEM integration, Langfuse, or alerting consumers — future issues.
- Compliance dashboard or audit query UI.
- Retroactive audit of historical spawns.
- Auditing non-CLI code paths (HTTP, NATS adapters) — out of scope for this
  issue; may follow as Layer 1b.

## Complexity

**Tier: F-full** — cross-package change (`roxabi-contracts` + `lyra/core/`),
new `core/audit/` module, JetStream stream provisioning, 3–4 spawn-site
