## Source

GitHub issue #75: "feat: agent identity / persona system"
Frame: `artifacts/frames/75-agent-persona-frame.mdx` (approved)

## Problem

Two distinct problems exist:

1. **Structural:** Lyra's identity is a raw system prompt string in the agent TOML (`[prompt].system`). It can't be decomposed for reuse across surfaces (voice/TTS, CLAUDE.md templating) and isn't composable.

2. **Injection gap (discovered during analysis):** The system prompt is parsed and stored on the `Agent` dataclass (`agent.system_prompt`) but **never passed to the Claude CLI subprocess**. In `cli_pool.py`, `_build_cmd()` builds the CLI command with `--model`, `--max-turns`, and `--allowedTools` but omits `--system-prompt`. The prompt is effectively dead code — Lyra runs without any custom system prompt.

### Evidence

```python
# cli_pool.py:190-209 — _build_cmd() never references system_prompt
def _build_cmd(self, model_config: ModelConfig, session_id: str | None = None) -> list[str]:
    cmd = ["claude", "--input-format", "stream-json", "--output-format", "stream-json",
           "--verbose", "--model", model_config.model, "--max-turns", str(model_config.max_turns)]
    if model_config.tools:
        cmd.extend(["--allowedTools", ",".join(model_config.tools)])
    # ← No --system-prompt flag

# agent.py:124 — system_prompt IS loaded from TOML
system_prompt = prompt_section.get("system", "")
# agent.py:146 — stored on Agent dataclass
return Agent(system_prompt=system_prompt, ...)
```

The `SimpleAgent.process()` method calls `self._pool.send(pool_id, text, model_cfg)` passing only `ModelConfig` — which doesn't contain the system prompt. The system prompt lives on `Agent`, not `ModelConfig`.

## Outcome

- Lyra has a structured persona config ("soul file") with identity, personality, expertise, and voice fields
- The persona auto-composes into a system prompt that is actually injected into the CLI subprocess
- Persona fields are programmatically accessible for external consumers (voice, templates)
- Backward compatibility: `[prompt].system` override and no-persona-file fallback both work

## Appetite

Unlimited — no time constraint.

## Landscape Research

Surveyed 10 frameworks/standards for agent persona definition. Key findings:

### Approaches to persona structure

| Project | Format | Persona model | Prompt composition |
|---------|--------|---------------|-------------------|
| **OpenClaw / Soul Spec** | Multi-file Markdown (SOUL.md, IDENTITY.md, AGENTS.md) | Soul (philosophy/values) vs. Identity (name/presentation) vs. Agents (capabilities) | All files concatenated into system prompt at session start (20K char cap) |
| **OpenFang** | TOML (`agent.toml`) | Freeform `system_prompt` string inside `[model]` section | No decomposition — raw string |
| **Nanobot** | YAML frontmatter + Markdown body | Metadata in YAML, persona in prose (markdown body = system prompt) | Markdown body injected verbatim |
| **CrewAI** | YAML + Python | `role` + `goal` + `backstory` triad | Three fields composed into system prompt; overridable via `system_template` |
| **AutoGPT** | YAML (`ai_settings.yaml`) | `ai_name` + `ai_role` + `ai_goals[]` | Minimal fields composed into template |
| **Open Souls** | TypeScript + Markdown (`core.md`) | Code-first: `MentalProcesses` state machine + `WorkingMemory` | Programmatic cognitive steps, not config-driven |
| **OpenPersona** | JSON + Markdown (4-layer: Soul/Body/Faculty/Skill) | Full lifecycle: constitution, identity, state, evolution | Multi-file injection with 4-layer separation |
| **Character Card V2** | JSON (in PNG metadata) | `name`, `description`, `personality`, `scenario`, `first_mes`, `extensions` | Template with `{{original}}` placeholder for composition |

### Patterns relevant to Lyra

1. **Soul vs. Identity separation** (OpenClaw/Soul Spec): Philosophy (who you are, values) separate from presentation (name, avatar, theme). Our `[personality]` maps to soul, `[identity]` maps to identity.

2. **Role/Goal/Backstory triad** (CrewAI): Minimal but effective. Consider adding a `goal` field to `[identity]` — a one-sentence objective that guides decision-making.

3. **Extensions / forward-compat** (Character Card V2): An `[extensions]` table in the persona TOML would allow future fields without schema changes.

4. **Markdown-first persona** (Nanobot, OpenClaw): Several frameworks use prose rather than structured fields. Our structured TOML approach is more programmatically useful (voice, templates) but less natural to write. The `compose_system_prompt()` function should produce clean prose, not a mechanical field dump.

5. **Template override** (CrewAI, Character Card V2): CrewAI's `system_template` with `{role}`, `{goal}`, `{backstory}` placeholders is similar to our `[prompt].system` override. Our approach (full override or auto-compose) is simpler and sufficient.

### Design implications

- Our four-section TOML structure (`[identity]`, `[personality]`, `[expertise]`, `[voice]`) is **more granular than most frameworks** (CrewAI uses 3 fields, AutoGPT uses 3, OpenFang uses 1). This is appropriate because Lyra's persona must serve multiple consumers (LLM prompt, voice/TTS, templates).
- The `compose_system_prompt()` function should produce **natural prose** (like OpenClaw's SOUL.md), not a structured dump of fields. Think "You are Lyra, a direct and precise assistant..." not "Name: Lyra. Traits: direct, precise."
- Consider adding an optional `[identity].goal` field (one-sentence objective) — aligns with CrewAI's proven pattern and gives the prompt composer a clear behavioral anchor.

## Shapes

### Shape 1: Separate persona TOML + system prompt injection fix (recommended)

Add a `lyra_default.persona.toml` file with structured `[identity]`, `[personality]`, `[expertise]`, `[voice]` sections. Create `PersonaConfig` frozen dataclass + `load_persona()` loader + `compose_system_prompt()` composer. Fix the injection gap by passing the system prompt to `_build_cmd()`.

**Injection approach:** Pass system prompt via `--system-prompt` CLI flag. This requires `_build_cmd` to accept a `system_prompt: str` parameter (or the full `Agent` config rather than just `ModelConfig`). Two sub-options:

- **A: Expand `ModelConfig` to include `system_prompt`** — keeps the interface narrow but blurs the config boundary (model config ≠ prompt config).
- **B: Pass `Agent` (or a new `AgentRunConfig`) to `CliPool.send()`** — cleaner separation but wider interface change. `_build_cmd` gets access to the full agent context.
- **C: Add `system_prompt` as a separate parameter to `CliPool.send()`** — minimal change, explicit, no dataclass restructuring.

**Trade-offs:**
- Pro: Clean separation of persona from model config. Persona reusable across consumers. Two files is manageable (one per agent).
- Pro: Injection fix is natural — we're already reworking the prompt path.
- Con: Two files to maintain per agent. Hot-reload must watch both files.

**Rough scope:** M (5-8 files: persona TOML, persona dataclass/loader, prompt composer, agent.py, cli_pool.py, simple_agent.py, tests)

### Shape 2: Inline `[persona]` section in agent TOML + injection fix

Same structured fields but inside the existing `lyra_default.toml` as a `[persona]` table. No separate file.

**Trade-offs:**
- Pro: Single file per agent. Simpler hot-reload (one mtime).
- Con: Agent TOML grows larger and mixes concerns (model config + persona + commands).
- Con: Persona can't be shared across multiple agents without duplication.
- Con: Contradicts user preference (user chose separate file in spec interview).

**Rough scope:** S-M (4-6 files: agent TOML, persona dataclass, prompt composer, agent.py, cli_pool.py, tests)

### Shape 3: Minimal injection fix only (defer structured persona)

Just fix the `_build_cmd` gap: pass `--system-prompt` from `Agent.system_prompt`. Keep the raw string format. Defer structured persona to a follow-up.

**Trade-offs:**
- Pro: Smallest change. Fixes the actual runtime bug first.
- Con: Doesn't address the core request (structured, reusable persona).
- Con: Two issues/PRs for what's conceptually one feature.

**Rough scope:** XS (2-3 files: cli_pool.py, simple_agent.py, tests)

## Fit Check

**Shape 1 is the best fit.** The user explicitly chose a separate persona file during the spec interview. The injection fix fits naturally — we're reworking the prompt path anyway. Shape 2 was rejected by the user. Shape 3 is too narrow — it fixes a bug but misses the feature request.

For the injection sub-option, **option C** (add `system_prompt` as a separate parameter to `CliPool.send()`) is recommended: minimal interface change, explicit, doesn't restructure existing dataclasses.

### Hot-reload + persistent process lifecycle

`--system-prompt` is passed at process spawn time. A running Claude CLI process will keep using its original system prompt until killed and respawned. Therefore, when `_maybe_reload` detects a persona file change, it must also trigger `pool.reset(pool_id)` on the next `process()` call so the process respawns with the new prompt. This means `SimpleAgent` needs access to the pool's reset mechanism when a reload is detected.
