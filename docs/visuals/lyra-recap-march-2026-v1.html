Channel layer

Translate platform events ↔ domain Message. Per-platform bounded queues in InboundBus isolate backpressure — a Telegram flood can't starve Discord.

  • telegram.py — 359 lines
  • discord.py — 340 lines
  • InboundBus — 107 lines, feeder tasks
Hub layer

Hub holds the binding registry. Pool (one per conversation scope) owns the asyncio.Task and history. OutboundDispatcher routes replies back per-channel.

  • hub.py — 444 lines
  • pool.py — 162 lines
  • outbound_dispatcher.py — 141 lines
  • command_router.py — 185 lines
Agent + LLM layer

Stateless agents driven by TOML persona configs. Two backends: direct Anthropic SDK and Claude CLI subprocess. Circuit breaker per service.

  • agent.py — 512 lines, ABC
  • anthropic_agent.py — 171 lines
  • cli_pool.py — 410 lines, subprocess
  • circuit_breaker.py — 189 lines

4Decision Log
ADR · scope_id routing (#125)
Pool routing uses (platform, bot_id, scope_id), not user_id
Why: Group chats and threads need conversation-scope isolation — multiple users share one pool per chat. user_id was wrong for groups. scope_id captures chat_id, topic_id, thread_id, or channel_id as appropriate.
Considered: Using session_id — rejected, scope is per-container not per-user-session.
Contract: RoutingKey.to_pool_id() is the single source of truth for the platform:bot_id:scope_id format (ADR-001 §4). Never construct inline.
Design · OutboundDispatcher (#126)
Outbound separated from adapters: one OutboundDispatcher per platform
Why: Inbound and outbound are different concerns. Per-channel outbound queues prevent a slow Discord reply from blocking Telegram sends. The dispatcher drains the queue and calls adapter.send().
Caveat: Two TODO markers remain — reply_message_id is not yet stored in response.metadata after streaming, which matters for edit-in-place across session boundaries.
ADR-013 · temp file lifecycle (STT)
Audio temp file cleanup is the Agent's responsibility, not STTService
Why: STTService only knows the happy path. The Agent has try/finally context across all error branches — if STTService cleaned up, a transcription error would leave orphaned temp files.
Ordering constraint: #139 (typed bus envelope) must ship before #80 (STT) — audio content needs a typed carrier in the bus.
Strategy · Chimera patterns
#134 (smart routing) and #135 (runtime config) promoted from ClawFamily analysis
Why: IronClaw and ScalyClaw analyses identified complexity-based model selection and live !config tuning as high-value, low-risk patterns. Added to Phase 1b tail as independent items — no blockers, fit within existing AgentBase abstraction.
Deferred: Proactive engine, command shield, memory consolidation — Phase 2, needs infra.
Architecture · per-session asyncio.Task (#127)
Each Pool runs one asyncio.Task; Discord threads auto-created on first message
Why: Discord threads map 1:1 to conversation scopes. Creating on first message avoids needing the thread ID upfront. The Task will be cancelled on scope eviction (future).
Considered: Discord Forum channels — deferred, regular threads are simpler for current use case.

6Mental Model Essentials
Key invariants
  • Trust is hardcoded "user" from adapters. Message.from_adapter() always sets trust="user". Only internal Hub code may set trust="system". Never derive trust from inbound channel data.
  • Pool ID = RoutingKey.to_pool_id() only. Format is platform:bot_id:scope_id. ADR-001 §4 forbids constructing this string inline anywhere in the codebase.
  • InboundBus: register before start(). Calling register() after feeders are running raises RuntimeError. A queue registered late would silently receive messages but have no feeder to drain it.
  • Agent is stateless — Pool owns history. One AgentBase singleton serves all Pools. Conversation history lives in Pool.history and Pool.sdk_history. Mutating Agent state corrupts all scopes.
  • Per-platform backpressure isolation. A Telegram flood cannot starve Discord. InboundBus maintains separate bounded queues; feeder tasks drain independently into one staging queue.
Non-obvious coupling
  • scope_id extraction is adapter-specific. Message.extract_scope_id() branches on TelegramContext vs DiscordContext. Adding a new platform requires a new branch here — it raises ValueError at runtime, not at startup.
  • OutboundDispatcher is created by Hub, not adapters. Adapters only implement send(). The dispatcher manages the per-channel outbound queue and calls adapter.send() from a background task.
  • Pool eviction is absent. Pools accumulate indefinitely until process restart. The TODO at pool.py:35 must be resolved before the memory layer (#83) can be designed — the eviction hook is where L0 compaction attaches.
  • cli_pool.py and anthropic_agent.py are parallel, not hierarchical. Both subclass AgentBase but have completely different concurrency models (subprocess vs asyncio). Their fates diverge under #123.
Gotchas
  • content: MessageContent | str — the str branch is legacy. New code should not assume str content. Will be cleaned up after #139.
  • Discord thread_id in DiscordContext can be None. extract_scope_id() always prefers thread over channel — check this logic before adding thread-aware features.
  • Telegram message_id is optional in TelegramContext — service messages (joins, pins) have no message_id. Don't assume it's set.
  • TURN_TIMEOUT_DEFAULT = 60.0s in pool.py. Long-running agent turns (e.g. Whisper transcription) may timeout before STT completes. Review when implementing #80.
7Cognitive Debt Hotspots
high
Pool eviction strategy — undecided, blocks memory layer
pool.py:35 has a TODO with two documented options: deque(maxlen=N) sliding window, or a Pool.append() mutator with compaction callback. This decision is on the critical path to #83 (memory integration), which blocks #99 (command sessions). The longer it stays undecided, the more code accumulates assuming the current unbounded list.
Fix: Write a 1-page ADR. Option B (compaction callback) is likely correct — the memory layer needs hook points for L0→L1 cascade. Decide during #123 LlmProvider design.
high
cli_pool.py — 410 lines, unclear fate under #123
The Claude CLI subprocess pool is the largest module in core. When #123 (LlmProvider protocol) ships with a ClaudeCliDriver, it's unclear whether cli_pool.py becomes that driver (refactored in-place) or gets deleted and replaced. No doc comment explains the tension. If deleted, current CLI test coverage is wasted investment. If promoted, it needs significant reshaping.
Fix: Add a module docstring to cli_pool.py explaining its expected fate. Decide explicitly during the #123 LlmProvider protocol design session.
medium
reply_message_id not stored after streaming — breaks edit-in-place across restarts
Two TODOs — discord.py:271 and telegram.py:285 — note that the placeholder message ID is not stored in response.metadata["reply_message_id"]. Streaming edits work within a session (Pool holds the ID in memory) but break if the process restarts or if session persistence (#67) ever stores and replays responses.
Fix: Complete both TODOs before implementing session persistence (#67). The fix is local to each adapter — should be a 1-hour task.
medium
Plugin handlers have no isolated tests
plugins/echo/handlers.py and plugins/pairing/handlers.py have no dedicated test files. Plugin behavior is only verified end-to-end through the full Hub routing stack. A bug in a handler would surface as a routing failure — hard to pinpoint without unit-level tests.
Fix: Add tests/plugins/test_echo_handler.py and test_pairing_handler.py. Invoke handlers directly, bypass Hub. Small surface area — should take under an hour.
low
content: MessageContent | str — legacy union pollutes the type model
The str branch in Message.content was kept for backwards compatibility during adapter migration. extract_text() handles it but the union creates defensive branching anywhere content is inspected. This will linger until #139 (message normalization) forces a cleanup.
Fix: After #139 ships, make content: MessageContent mandatory and remove the str branch from extract_text(). Tag with a comment now so it's findable.
