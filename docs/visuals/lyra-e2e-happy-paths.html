Architecture Overview

Hub-and-spoke with per-platform inbound/outbound queues, per-scope pools, and stateless singleton agents.

Core Core Message Flows

SCENARIO 01

Basic Text Message

Trigger: User sends a text message in Telegram or Discord
User → Adapter._on_message() → normalize() → _start_typing() → push_to_hub_guarded() → InboundBus.put(platform, msg) → Feeder → Staging Queue → Hub.run() → _is_rate_limited() → resolve_binding() → get_or_create_pool() → pool.submit(msg) → _process_loop() → agent.process(msg, pool) → LlmProvider.complete() → Response → dispatch_response() → OutboundDispatcher.enqueue() → _worker_loop() → adapter.send() → _cancel_typing() → bot.send_message()
SCENARIO 02

Streaming Text Response

Trigger: Agent returns AsyncIterator[str] from process()
Same intake as #1 → agent.process() returns AsyncIterator → dispatch_streaming(msg, chunks) → OutboundDispatcher.enqueue_streaming() → adapter.send_streaming(msg, chunks) → Send placeholder message → _cancel_typing() → Debounce loop (TG: 500ms, DC: 1s) → edit_message_text() on each tick → Final edit with complete text → Overflow chunks as follow-up messages
SCENARIO 03

Markdown Formatting

Trigger: LLM response contains markdown
Response text with markdown → Telegram: _render_text() → telegramify_markdown (markdownify) → MarkdownV2 parse_mode → chunk_text(text, 4096) → Discord: _render_text() → Native markdown (no conversion) → chunk_text(text, 2000)
SCENARIO 04

Typing Indicator

Trigger: Message received by adapter
_on_message() received → _start_typing(chat_id) immediately → TG: _typing_worker() sends chat_action every 3s → DC: _discord_typing_worker() uses channel.typing() ctx mgr → Processing happens (agent, LLM...) → send() or send_streaming() called → _cancel_typing(chat_id) at start of send → Drop safety: on_drop cancels if msg dropped

Commands Commands

SCENARIO 05

Builtin Command

Trigger: /help, /clear, /config, /stop, /circuit, /routing
User sends /help → CommandParser.parse() → CommandContext → Hub pipeline: is_command() = true → CommandRouter.dispatch(msg, pool) → _dispatch_builtin("help", ...) → _help() builds listing of all commands → Response(content) → dispatch_response()
SCENARIO 06

Plugin Command

Trigger: /echo hello, /search query
User sends /echo hello → CommandParser.parse() → CommandContext → CommandRouter.dispatch() → Not builtin, not session cmd → PluginLoader.get_commands(enabled_plugins) → handler = plugin_handlers["/echo"] → asyncio.wait_for(handler(msg, pool, args), timeout) → Response → dispatch_response()
SCENARIO 07

Session Command (/add URL)

Trigger: /add https://example.com
/add URL → CommandRouter.dispatch() → _session_handlers["/add"] found → _dispatch_session("/add", args, msg, pool) → session_driver (isolated LlmProvider, no pool history) → scrape_url(url) via web-intel subprocess → LLM summary of scraped content → vault_add() to knowledge base → Response("Saved: title")
SCENARIO 08

Bare URL Rewrite

Trigger: Plain URL as message text
User sends "https://example.com" (no command prefix) → CommandRouter.prepare(msg) → _BARE_URL_RE.fullmatch(text.strip()) = true → _rewrite_bare_url(msg) → Synthetic CommandContext(prefix="/", name="add", args=url) → Flows into session command path (#7)
SCENARIO 09

Workspace Switch

Trigger: /folder ~/proj or /workspace_name
/folder ~/projects/lyra → CommandRouter.dispatch() → _cmd_folder(msg, args, pool) — admin-only check → Path.expanduser().resolve() → validate is_dir() → pool.switch_workspace(cwd) → sdk_history.clear(), history.clear() → _switch_workspace_fn(cwd) — CliPool kills/respawns subprocess → Response("cwd -> /home/.../lyra")

Audio Audio / Voice

SCENARIO 10

Voice Input (STT)

Trigger: User sends a voice message
Voice message arrives → Adapter._on_voice_message() → _download_audio(file_id) — size check vs LYRA_MAX_AUDIO_BYTES → normalize_audio() → InboundAudio envelope → push_to_hub_guarded(inbound_audio_bus, ...) → Hub audio consumer loop → STTService.transcribe(path) — voicecli faster-whisper → Personal vocab from ~/.voicecli/voicecli.vocab → is_whisper_noise() check → Inject as InboundMessage(text=transcription) → Normal text processing flow (#1)
SCENARIO 11

Voice Output (TTS)

Trigger: Agent response requires TTS output
Agent produces text response → Hub detects TTS route (agent tts config) → TTSService.synthesize(text) → voicecli generate_async() — Qwen TTS engine → WAV output + _wav_duration_ms() → ffmpeg WAV → OGG/Opus (48kHz mono) → _wav_waveform_b64() — 256-byte amplitude array → SynthesisResult(audio_bytes, mime, duration, waveform) → OutboundAudio envelope → OutboundDispatcher.enqueue_audio() → adapter.render_audio() → TG: bot.send_voice() (voice bubble) → DC: IS_VOICE_MESSAGE flag (8192) + OGG
SCENARIO 12

Discord Voice Channel

Trigger: !join in Discord guild
User in voice channel sends !join → _handle_voice_command() → cmd.name == "join" → _handle_join_command() → VoiceSessionManager.join(guild, channel, mode) → VoiceMode.TRANSIENT or PERSISTENT → voice_client.connect() → TTS output routed to voice: → OutboundDispatcher.enqueue_voice_stream() → adapter.render_voice_stream() → VoiceSessionManager.stream(guild_id, chunks) → PCM playback via VoiceClient

Session Session & Memory

SCENARIO 13

Multi-turn Conversation

Trigger: Multiple messages in the same scope
Msg 1 → Pool created, session_id = UUID → pool.append(msg) — user_id, medium, message_count set → agent.process() → LLM sees sdk_history (deque) → pool.extend_sdk_history(new_messages) Msg 2 → Same pool (same scope_id) → pool.submit() → debouncer.collect() → agent.process() — LLM sees accumulated sdk_history → max_sdk_history = 50, popleft() on overflow
SCENARIO 14

Session Reset (/clear)

Trigger: /clear or /new command
/clear → CommandRouter._cmd_clear(pool) → pool.sdk_history.clear() → pool.history.clear() → pool.reset_session() → _session_reset_fn() callback → CliPool.reset() — kills subprocess, new session_id → Response("Conversation history cleared. Starting fresh.")
SCENARIO 15

Pool Eviction (TTL)

Trigger: Pool idle > POOL_TTL (3600s default)
Hub.get_or_create_pool() called → _evict_stale_pools() — throttled: skip if < TTL/10 elapsed → Scan pools: pool.is_idle AND (now - pool.last_active) > TTL → For each stale pool: → pools.pop(pid) → agent.flush_session(pool, "idle") → _summarize_session() via LLM → MemoryManager.upsert_session(snap, summary) → _schedule_extraction() — concept/preference extraction → Pool removed from memory
SCENARIO 16

Cross-session Memory Recall

Trigger: Known user starts a new session
New pool created for known user → agent._ensure_system_prompt(pool) — first turn → MemoryManager.recall(user_id, namespace, first_msg, budget) → DB query: sessions WHERE user_id matches (last 5) → FTS5 concept search in namespace:user_id scope → Freshness filter: _is_stale() with FRESHNESS_TTL_DAYS → _fetch_preferences(user_id) — sorted by staleness → Token-budgeted [MEMORY] + [PREFERENCES] blocks → Injected into system prompt for LLM context
SCENARIO 17

L0 Compaction

Trigger: sdk_history exceeds 80% of COMPACT_THRESHOLD (200k tokens)
After each turn: agent.compact(pool) called → Token count of sdk_history > 80% of 200k → _summarize_session() — LLM generates conversation summary → MemoryManager.upsert_session(snap, summary, status="partial") → L3 partial snapshot written → sdk_history replaced with: [summary_turn] + last 10 turns → Working memory reduced, context preserved

Resilience Routing & Resilience

SCENARIO 18

Smart Routing

Trigger: Every message processed by LlmProvider
Message text arrives at SmartRoutingDecorator → ComplexityClassifier.classify(text) → TRIVIAL: ≤3 words, greeting pattern → SIMPLE: ≤2 words, no keywords → MODERATE: explain/summarize keywords → COMPLEX: analyze/implement/refactor keywords → SmartRoutingConfig maps complexity to model → RoutingDecision logged to history deque → Wrapped provider.complete() called with routed model → Stack: CircuitBreaker -> SmartRouting -> Retry -> Driver
SCENARIO 19

Rate Limiting

Trigger: User sends >20 messages in 60 seconds
Hub.run() receives message from staging queue → _is_rate_limited(msg) → Key: (platform, bot_id, user_id) — per-user → Sliding window: deque of timestamps → Evict timestamps outside current window → len(timestamps) >= RATE_LIMIT (20) → Returns True: message DROPPED → notify user: "Too many messages, please wait" → Inactive-user entries cleaned when deque empties
SCENARIO 20

Bus Backpressure

Trigger: Platform inbound queue full (maxsize=100)
Adapter calls push_to_hub_guarded() → InboundBus.put(platform, msg) — put_nowait() → QueueFull raised → Adapter sends ack to user: "message received, ~Xs wait" → Staging queue (500): feeder drains platform queue → QueueDepthExceeded event emitted when staging > threshold → QueueDepthNormal emitted when depth returns below threshold
SCENARIO 21

Circuit Breaker

Trigger: 5+ consecutive LLM/platform failures
Provider call fails → CircuitBreaker.record_failure() → _failure_count++ (in CLOSED state) → failure_count >= failure_threshold (5) → State: CLOSED -> OPEN, _opened_at = now Subsequent calls: → cb.is_open() = True → fast-fail, DROP After recovery_timeout (60s): → OPEN -> HALF_OPEN → One probe call allowed (_probe_in_flight) → Success: HALF_OPEN -> CLOSED, counters reset → Failure: HALF_OPEN -> OPEN, timer reset

Auth Auth & Multi-Bot

SCENARIO 22

Auth Check

Trigger: Every inbound message, before normalization
Message arrives at adapter._on_message() → user_id = str(msg.from_user.id) → auth.check(user_id, roles=[...]) → 1. BLOCKED check first (even public cmds denied) → 2. Public command bypass (/join) → 3. AuthStore lookup (OWNER/TRUSTED/BLOCKED) → 4. Role map lookup (Discord roles -> TrustLevel) → 5. Default fallback from config → BLOCKED: silently drop, log auth_reject → OWNER/TRUSTED/PUBLIC: normalize() + continue → trust_level attached to InboundMessage
SCENARIO 23

Multi-bot Same Channel

Trigger: 2 bots configured for same platform/channel
Two bots registered: (TELEGRAM, "lyra"), (TELEGRAM, "aryl") → Each has its own adapter, webhook endpoint, auth → Inbound: separate RoutingKey per bot_id → resolve_binding(): (platform, bot_id, scope_id) → Separate pools: telegram:lyra:chat:555 vs telegram:aryl:chat:555 → Outbound: OutboundDispatcher per (platform, bot_id) → _verify_routing() ensures platform+bot_id match → Response never sent by wrong bot → Discord: _owned_threads prevents cross-bot replies
SCENARIO 24

Reply Threading

Trigger: User replies to a bot message
User replies to bot message (reply_to_id set) → ContextResolver.resolve(reply_to_id) → Query conversation_turns WHERE reply_message_id = ? → Returns ResolvedSession(session_id, pool_id) → Hub uses resolved pool_id to route to correct pool → pool.resume_session(session_id) if different session → _session_resume_fn() callback for CLI backend → Conversation continues in correct context → Graceful fallback: if DB absent, uses default routing

Media Media & Observability

SCENARIO 25

Attachments (Image/Video/File)

Trigger: User sends photo, document, video, sticker
Media message arrives → _extract_attachments(msg) in adapter → TG: photo (largest), document, video, animation, sticker → DC: message.attachments (filtered: non-audio) → Attachment(type, url/path/bytes, mime_type, filename) → Attached to InboundMessage.attachments[] → Agent processes with attachment context → Outbound: OutboundAttachment envelope → OutboundDispatcher.enqueue_attachment() → adapter.render_attachment() → TG: send_photo/send_video/send_document → DC: discord.File + messageable.send()
SCENARIO 26

Turn Logging (L1)

Trigger: Every user and assistant turn
Pool._process_one() processes message → pool.append(msg) — user turn → _log_turn_async(role="user", content=msg.text) → Agent produces response → _log_turn_async(role="assistant", content=response.content) → Both fire-and-forget via asyncio.create_task → TurnStore.log_turn() writes to turns.db → Pool_id, session_id, role, platform, user_id → message_id, reply_message_id, timestamp → Query: get_turns(pool_id, user_id, limit)

#21 Circuit Breaker — Full State Sequence

sequenceDiagram
    participant Caller
    participant CB as CircuitBreaker
    participant Provider

    Note over CB: State: CLOSED, failures=0

    loop Failures 1-4
        Caller->>CB: is_open()
        CB-->>Caller: false
        Caller->>Provider: call()
        Provider-->>Caller: ERROR
        Caller->>CB: record_failure()
        Note over CB: failures++
    end

    Caller->>Provider: call() — failure #5
    Provider-->>Caller: ERROR
    Caller->>CB: record_failure()
    Note over CB: failures=5 >= threshold -> OPEN
    Note over CB: _opened_at = now

    Caller->>CB: is_open()
    CB-->>Caller: true (FAST-FAIL)
    Note over Caller: Request dropped immediately

    Note over CB: 60s recovery_timeout elapses...

    Caller->>CB: is_open()
    Note over CB: elapsed >= recovery_timeout -> HALF_OPEN
    CB-->>Caller: false (probe slot granted)
    Caller->>Provider: probe call
    alt Probe succeeds
        Caller->>CB: record_success()
        Note over CB: HALF_OPEN -> CLOSED, counters reset
    else Probe fails
        Caller->>CB: record_failure()
        Note over CB: HALF_OPEN -> OPEN, timer reset
    end

State Machines

Pool Lifecycle

stateDiagram-v2
    [*] --> Created: get_or_create_pool()
    Created --> Active: pool.submit(msg)
    Active --> Active: new message (_touch)
    Active --> Idle: _process_loop exits
    Idle --> Active: pool.submit(msg)
    Idle --> Evicted: TTL expired (3600s)
    Evicted --> [*]: flush_session() -> L3 write
    Active --> Reset: /clear command
    Reset --> Active: next message

Circuit Breaker

stateDiagram-v2
    [*] --> CLOSED
    CLOSED --> CLOSED: success (no-op)
    CLOSED --> CLOSED: failure (count < 5)
    CLOSED --> OPEN: failure count >= 5
    OPEN --> OPEN: all calls fast-fail
    OPEN --> HALF_OPEN: recovery_timeout (60s) elapsed
    HALF_OPEN --> CLOSED: probe success (reset counters)
    HALF_OPEN --> OPEN: probe failure (reset timer)

Session Lifecycle

stateDiagram-v2
    [*] --> New: Pool created, UUID assigned
    New --> Active: First message (user_id, medium set)
    Active --> Active: Messages accumulate in sdk_history
    Active --> Compacted: sdk_history > 80% of 200k tokens
    Compacted --> Active: summary + last 10 turns
    Active --> Flushed: Pool evicted OR /clear
    Compacted --> Flushed: Pool evicted
    Flushed --> [*]: L3 snapshot + extraction tasks
    Flushed --> Recalled: Same user, new session
    Recalled --> Active: [MEMORY] + [PREFERENCES] injected
