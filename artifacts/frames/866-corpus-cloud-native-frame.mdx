## Problem

The corpus (`~/.roxabi/corpus.db`) is only reachable on the machine running `scripts/corpus/`.
No API exists, so nothing outside that machine can query issue data without a direct DB read.
Three related needs are converging:

1. **Live API** over the corpus — consumable by tools, scripts, and a frontend.
2. **Operations cockpit** (`roxabi-dashboard` rewrite) — corpus is the first view; agent health, CI, and dep-graph follow in later phases.
3. **Fast dev loop** — the existing reconciler (`scripts/corpus/`) already handles bulk sync; adding a webhook layer closes the freshness gap and removes the need for high-frequency polling.

The problem is not "go cloud" — M₂ already holds the data. The problem is that the data has no service layer in front of it.

## Who

- **Primary:** Mickael — needs corpus data accessible via API on M₂ (dev) and eventually M₁ (24/7 prod).
- **Secondary:** `roxabi-dashboard` — new rewrite will consume the API as its first data source.
- **Tertiary:** CLI tooling and scripts that today read corpus.db directly on-box.

## Constraints

- **Local-first** — corpus.db lives on M₂ (or M₁); no managed cloud DB required or wanted.
- **Zero inbound ports** — Cloudflare Tunnel handles public exposure; no firewall rules, no port forwarding.
- **Supervisor pattern** — `cloudflared` runs as a separate process under existing project supervisord (`deploy/supervisor/conf.d/`), not embedded in the app.
- **Existing reconciler stays** — `scripts/corpus/` GraphQL sync remains the canonical backfill mechanism; webhook is an incremental layer on top.
- **Path configurable** — `CORPUS_DB_PATH` env var, default `~/.roxabi/corpus.db`.
- **Designed for M₂→M₁ migration** — no hard-coded hostnames; switching machines = rsync db + flip tunnel target.
- **Cross-repo dependency** — the FastAPI app belongs in the new `roxabi-dashboard` repo (corpus.db consumer), not in lyra. This issue tracks the lyra side (DB schema, reconciler wiring, webhook contract); dashboard repo is a dependency.

## Out of Scope

- User OAuth / Cloudflare Access (revisit when a write-capable endpoint lands).
- Multi-org GitHub aggregation (single Roxabi org only in MVP).
- Other data sources: dep-graph of non-lyra repos, vault, CI/CD, agent health.
- Frontend polish, mobile, offline mode.
- `roxabi-dashboard` repo setup / frontend implementation detail (tracked in dashboard repo).

## Open Questions

2. **Webhook registration** — GitHub org webhook must be created manually; one-time step. Document in ADR or runbook?

## Signals

- Multi-component: FastAPI, `cloudflared` daemon, supervisord unit, SQLite upsert, reconciler cron wiring.
- New domain for lyra: outward-facing HTTP service + public tunnel.
- Security surface: HMAC validation, tunnel config, future auth.
- Cross-repo: lyra owns corpus.db + reconciler; `roxabi-dashboard` owns the API + frontend.
- M₂ → M₁ migration path must be trivial by design.

**Tier: F-full** — multi-domain, new architecture (tunnel + service layer), cross-repo dependency.
