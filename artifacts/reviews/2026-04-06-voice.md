**Observation 1 (LOW):** The `gen-nkeys.sh` script generates three nkey users (`hub`, `llm-worker`, `monitor`). No dedicated `stt`/`tts` nkey is created — both adapter processes use the same `.env`-sourced `NATS_NKEY_SEED` as the hub. This means there is no per-process subject isolation at the NATS auth layer: lyra_stt and lyra_tts have the same pub/sub permissions as the hub. Acceptable for a single-machine topology (Machine 1 only, no remote voice processes), but worth noting for future multi-machine expansion where separate nkeys per voice service would scope permissions to `lyra.voice.*` only.

## Hexagonal Adapter Pattern

**Status: PASS — correct structure**

```
Port (interface):    STTProtocol / TtsProtocol   — src/lyra/stt/__init__.py, src/lyra/tts/__init__.py
Hub-side impl:       NatsSttClient / NatsTtsClient — src/lyra/nats/
Adapter-side impl:   STTService / TTSService       — src/lyra/stt/__init__.py (voicecli, adapter process only)
Bootstrap wiring:    init_nats_stt / init_nats_tts  — voice_overlay.py
```

- Protocols are `@runtime_checkable` — correct for `isinstance` checks if needed.
- `STTProtocol.transcribe(path)` and `TtsProtocol.synthesize(text, ...)` match the concrete `NatsSttClient` and `NatsTtsClient` signatures.
- Hub depends only on the protocol. `STTService` and `TTSService` are imported exclusively inside `stt_adapter_standalone.py` and `tts_adapter_standalone.py`.
- `_NatsTtsConfig` in `tts_adapter_standalone.py` is a duck-typed stand-in for `AgentTTSConfig` — avoids importing hub-layer type into adapter process. Uses `type: ignore[arg-type]` with comment. Acceptable trade-off; the field list (`_AGENT_TTS_FIELDS`) is duplicated between `nats_tts_client.py` and `tts_adapter_standalone.py` — a minor sync risk if `AgentTTSConfig` gains new fields.

## Graceful Degradation

**Status: PASS — two distinct error paths**

### STT unavailable

Path: `NatsSttClient.transcribe()` → `STTUnavailableError` → `AudioPipeline._process_audio_item()` catches it → dispatches `stt_unavailable` message → drops audio item.

- Three triggers map to `STTUnavailableError`: `TimeoutError`, `MaxPayloadError`, other network errors.
- `stt_unavailable` message is separate from `stt_unsupported` — correct UX distinction (adapter missing vs STT not configured).
- When `STT_MODEL_SIZE` is not set, `init_nats_stt` returns `None` → hub has no STT → `AudioPipeline` uses existing `stt_unsupported` path.

**Note:** The diff shows `STTUnavailableError` handling in two places:
1. `audio_pipeline.py:938–951` — catches at `_process_audio_item` level (for `AudioPipeline._hub._stt.transcribe`)
2. `middleware_stt.py:135–139` — catches in `SttMiddleware.__call__`

Both paths exist. The middleware handles the primary inline STT flow; the audio pipeline handles the audio bus flow. No gap.

### TTS unavailable

Path: `NatsTtsClient.synthesize()` → `TtsUnavailableError` → `AudioPipeline.synthesize_and_dispatch_audio()` catches it → falls back to `dispatch_response(msg, Response(content=text))`.

- When `LYRA_VOICE_RESPONSES=0` or STT is off, `init_nats_tts` returns `None` → no TTS, text-only responses (existing path).
