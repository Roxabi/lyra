Architecture
Deep technical blueprint — components, queues, domains, data types, dispatch, and refactoring plan.
💡
Do LLM responses flow through InboundBus?
No. The InboundBus is strictly one-way: Adapter → per-platform Queue → staging Queue → Hub.run(). LLM responses take a completely separate return path: Agent → Pool → Hub.dispatch_response() → OutboundDispatcher → Adapter.send(). Each adapter has its own inbound queue AND its own OutboundDispatcher — they never share a bus.

Complete message flow
18 steps from platform event to delivered response — inbound and outbound are completely separate.
Inbound Path (Adapter → Hub → Pool → Agent)
  1. 1 Platform event (Telegram update / Discord on_message)
  2. 2 Adapter normalizes → InboundMessage (with RoutingContext)
  3. 3 push_to_hub_guarded() — circuit breaker check before enqueue
  4. 4 inbound_bus.put(platform, msg) → per-platform bounded queue (100)
  5. 5 _feeder() task drains into staging queue (bounded 500)
  6. 6 Hub.run() awaits inbound_bus.get() — single consumer
  7. 7 MessagePipeline.process(msg): validate → rate limit → resolve binding → check commands
  8. 8 get_or_create_pool() via PoolManager
  9. 9 pool.submit(msg) → debounce → merge
LLM + Outbound (Agent → Hub → Dispatcher → Adapter)
  1. 10 _process_with_cancel() — cancel-in-flight support
  2. 11 _guarded_process_one() — timeout + event emission
  3. 12 agent.process(msg, pool) — Claude SDK or CLI subprocess
  4. 13 Response returned: str or AsyncIterator[str]
  5. 14 hub.dispatch_response() or hub.dispatch_streaming()
  6. 15 OutboundDispatcher — fire-and-forget enqueue
  7. 16 _worker_loop(): verify routing → circuit breaker → retry (1s/2s/4s)
  8. 17 adapter.send() or adapter.send_streaming()
  9. 18 Platform API delivers to user ✓
🎤
Separate Audio Path — InboundAudioBus is a completely separate bus. Voice messages route through inbound_audio_bus → Hub._audio_loop() → STT → re-inject as text. Audio responses: dispatch_audio_stream() → OutboundDispatcher → adapter.render_audio_stream().
Flow as pipeline
TG/DC Event→ normalize()→ InboundMessage→ push_to_hub→ InboundBus→ Hub.run()→ Pipeline→ Pool.submit()→ Agent.process()→ LLM→ dispatch_response()→ OutboundDispatcher→ adapter.send()→ ✓ Delivered
Three queue systems
Lyra uses three independent queuing systems that never share state.
InboundBus
Direction: Adapter → Hub
Structure: One queue per platform → staging
Bounded: 100 items (backpressure)
Consumer: hub.run()
File: core/inbound_bus.py
InboundAudioBus
Direction: Adapter → Hub (audio)
Structure: Separate voice queue
Consumer: hub._audio_loop()
Processing: STT → text → InboundBus
File: core/inbound_audio_bus.py
OutboundDispatcher
Direction: Hub → Adapter
Structure: One per (platform, bot_id)
Features: Retry (1s/2s/4s) + CB
Consumer: _worker_loop()
File: core/outbound_dispatcher.py

Scope model
📌
Scope != User — Pools are per conversation scope.
Pools are keyed by (platform, bot_id, scope_id). Scope is extracted by the adapter:
Telegram DM/group → chat:{chat_id}, forum → chat:{chat_id}:topic:{topic_id}.
Discord → channel:{channel_id} or thread:{thread_id}.
Two users in the same group share one Pool. Two bots in the same channel each have separate Pools.

All variants pass through the same OutboundDispatcher._worker_loop() — retry + circuit breaker is guaranteed for every response type. The dispatcher pattern-matches on the kind field to pick the right adapter method.
Key insights
Two Completely Separate Paths
InboundBus (adapters → Hub) and OutboundDispatcher (Hub → adapters) are independent queuing systems. They share no state, no queue, no bus. LLM responses never touch InboundBus.
Per-Scope, Not Per-User
Pools are keyed by (platform, bot_id, scope_id). In a group chat, all users share one Pool. Two bots in the same channel have separate Pools.
Dispatcher Owns Circuit Breaker
The OutboundDispatcher is the sole owner of the platform circuit breaker. Adapters never check CB state — the dispatcher handles it with exponential backoff.
Stateless Agent, Stateful Pool
Agents are immutable singletons — same instance serves all Pools. All mutable state (history, session, debouncer) lives in the Pool.
Hub Delegates to Focused Modules
Hub has been decomposed: MessagePipeline (validate/route), AudioPipeline (STT/TTS), PoolManager (lifecycle/eviction), PoolProcessor + PoolObserver (extracted from Pool). Hub orchestrates; each module owns its domain.
Audio Has Its Own Everything
Separate InboundAudioBus, separate AudioPipeline (extracted from Hub), separate dispatch methods. Fully isolated audio path from inbound to outbound.
