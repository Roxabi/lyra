## Problem

When a voice or audio message fails to download (network error, Telegram API timeout, Discord attachment read failure), both adapters log the exception and return silently. The user sees no error reply — the message appears received but is never processed. This breaks the basic contract that every user action gets a response.

The gap is asymmetric: the "file too large" error path (`ValueError`) already sends a user-facing reply in both adapters, but the generic `except Exception` fallthrough was added for logging safety and never wired to the reply path. A secondary silent drop exists in `SttMiddleware` when `hub._msg_manager is None` — the pipeline drops the message with `_DROP` and no reply.

## Who

- **Primary:** Any Telegram or Discord user who sends a voice/audio message when the download fails transiently (network blip, API timeout, CDN error).
- **Secondary:** Operators — currently no signal distinguishes "message received + processed" from "message received + silently dropped".

## Constraints

- Adapter download failures (`telegram_inbound.py:186`, `discord_audio.py:158`) occur **before** `push_to_hub_guarded()` — they never enter the middleware pipeline. A pipeline-level `ErrorBoundaryMiddleware` cannot catch them; the fix must live at the adapter boundary using the same pattern already present for `audio_too_large`.
- A new `audio_download_failed` message template is required — reusing `stt_failed` conflates two distinct user-actionable situations (product validation).
- `hub._msg_manager` is currently typed `MessageManager | None`. Making it mandatory (raise at Hub init) is preferred over `NullMessageManager` — a missing manager is a config error, not a runtime condition (devops validation).
- ADR-058 (`typed-error-boundary-and-user-visible-error-contract`) is drafted but is a follow-up — this issue ships the immediate fix; the typed exception hierarchy comes after.

## Out of Scope

- `LyraUserError` typed exception hierarchy — arch follow-up, not a blocker for this fix.
- `ErrorBoundaryMiddleware` as pipeline-level safety net — valid follow-up; does not cover the adapter paths fixed here.
- NATS and CLI adapters — no audio download path, unaffected.
- `hub_dispatch.py:129` silent swallow of `dispatch_response()` failures — secondary gap, acceptable as out of scope for this issue.
- Retry logic for transient download failures.

## Complexity

**Tier: F-lite** — Clear scope, single domain (adapters + middleware), 5–7 files, zero architectural unknowns. All design decisions resolved in pre-implementation validation session.

Signals:
- 2 adapter files + 1 middleware file = 3 primary callsites
- 1 new exception types or inline raise pattern
- 1 new message template key (`audio_download_failed`)
- 1 hub init change (`_msg_manager` mandatory)
- No new architectural patterns introduced
