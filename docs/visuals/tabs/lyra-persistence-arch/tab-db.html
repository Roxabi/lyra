turns.db L1 Memory
Managed by: TurnStore

Append-only turn ledger. Every message exchanged — both user inputs and assistant replies. The source of truth for get_last_session(), context recall, and audit. Called "L1 Memory" in code because it's the raw layer before semantic extraction.

idx(session_id, timestamp) idx(pool_id, timestamp)

Key query: get_last_session(pool_id) → SELECT session_id WHERE role='assistant' ORDER BY timestamp DESC LIMIT 1

Note: A "Lyra session" is NOT a first-class entity here. Sessions are implicit — inferred from the session_id column on turns. No sessions table exists. (→ Architecture finding #7)

No managing store — file exists, no tables

Legacy file. Agent configs were migrated to auth.db → agents. This file is never connected by any store. It wastes an open file descriptor on every startup.

Fix: Delete the file. Remove any connection attempts in bootstrap. (→ critical finding #1)
~/.claude/projects/*/*.jsonl EXTERNAL
Managed by: Claude CLI (external)

NOT a Lyra DB. Written by the Claude CLI subprocess. One file per Claude session — filename = session UUID. Contains the full conversation JSONL that Claude uses for --resume.

Path pattern~/.claude/projects/-home-…/<session_uuid>.jsonl
Written whenWhen the subprocess exits (stream-json mode flushes on exit, not during streaming)
Read by Lyra_session_file_exists() → glob */{session_id}.jsonl
Resume argclaude --resume <session_uuid> → CLI loads the JSONL
Timing bug: In --input-format stream-json mode, the .jsonl is NOT written while the subprocess is alive. So _session_file_exists() always returns False for the current active session, causing spurious resume failures even when context is intact. (→ finding #5)
~/.roxabi-vault/ (AsyncMemoryDB) SEMANTIC
Managed by: MemoryManager via roxabi-vault SDK

Structured semantic long-term memory. NOT a raw turn log. Extracted by the agent after sessions end — facts about users, preferences, concepts, contacts. Freshness-aware with per-type TTL thresholds.

Key flows:
Read: recall(user_id, namespace, first_msg, token_budget=1000) → 5 recent sessions + concepts → injected into prompt
Write: upsert_session/concept/preference(SessionSnapshot, data) → fires async AFTER session ends
Gap: No FK to session_id in turns.db → can't trace which memory came from which session (→ finding #8)
_resume_session_ids IN-MEMORY
Lives in: CliPoolWorkerMixin

An in-memory Python dict: pool_id → session_id. This is the single source of truth for what --resume arg the next Claude CLI spawn uses. Populated by _try_auto_resume() after each turn and by the reaper before killing idle processes.

Typedict[str, str] in Python process memory
Written by_try_auto_resume() post-turn, reaper pre-kill
Read by_spawn() → dict.pop(pool_id)
Survives restart?❌ Lost — recovery via Path 3 + file check (fragile)
Finding: This is the weakest durability link in the resume chain. If Lyra crashes between turns, the next spawn starts fresh even if the .jsonl exists. (→ finding #4)
How the stores relate

auth.db is the config+state monolith. It is NOT normalized across concerns — one file owns grants, agent configs, tokens, Discord thread pointers, and user prefs. This is the primary architectural debt.

turns.db feeds Path 3 resume (get_last_session) and is the audit trail. It is purely append-only — reads are always O(log n) via index scan.

message_index.db is the most targeted: one purpose, one table, O(1) PK lookup. The cleanest store in the codebase.

pairing.db carries a deprecated table in its DDL — schema confusion without functional impact.

roxabi-vault is a fully separate system. Lyra treats it as a black-box SDK. No FK linkage to Lyra's other DBs.
