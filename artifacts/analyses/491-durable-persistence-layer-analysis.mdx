## Problem

Lyra has a systemic fire-and-forget problem: 10 call sites launch SQLite writes as background tasks with no retry, no tracking, and in 3 cases no error logging. When a write fails silently, data is lost forever — thread claims vanish, turns disappear from history, sessions lose their routing.

### Incident (2026-04-02)

Thread `1488865367508910080` was working (8 messages exchanged April 1st). `persist_thread_claim` ran as a raw `asyncio.create_task()` with no error handling. The claim was never persisted to `discord.db`. After a restart triggered by `OperationalError: database is locked` on `auth.db`, the thread was treated as unowned — all messages silently dropped. The user saw no error, no response, nothing.

The monitoring system (6 health checks, 5-minute interval) reported "All checks passed" throughout — the dead CLI backend returned in 0ms, which looked like a healthy queue drain from the outside.

### Root cause: Port contract violations

In hexagonal architecture terms, the Domain Core (`Pool`, `PoolProcessor`, `Hub`) calls Driven Adapter methods (`TurnStore.log_turn()`, `ThreadStore.claim()`) via `asyncio.create_task()` without awaiting the result. The Port says "I will persist this" — fire-and-forget means the Core never confirms it did.

This was a premature optimization. The codebase already awaits DB reads in the same hot path (reply-to resume, last-session lookup in `message_pipeline.py`), and LLM calls dominate latency at 100–5000ms. Fire-and-forget saves <10ms per write — invisible relative to the LLM bottleneck.

### Severity breakdown

- **3 CRITICAL** — `ThreadStorePort.claim()`: untracked `create_task`, no error logging, thread ownership lost forever
- **2 HIGH** — `MemoryPort.flush_session()`: session memories permanently lost on eviction failure; `pool_manager.py:75` has no error logging at all
- **5 MEDIUM** — `TurnStorePort` and `MessageIndexPort` via `PoolObserver._fire_and_forget()`: errors are logged but no retry, data lost

## Hexagonal architecture alignment

The existing persistence diagram (`lyra-persistence-arch`, Architecture tab) already defines the target:

```
DRIVING ADAPTERS  →  PORTS (interfaces)  →  DOMAIN CORE  →  DRIVEN ADAPTERS
Discord/Telegram     TurnStorePort          Hub/Pool        SqliteTurnStore
CLI/Admin            SessionStorePort       Pipeline        SqliteMessageIndex
                     MessageIndexPort       Agent           SqliteAuthStore
                     ThreadStorePort        Guard           VaultMemory
                     MemoryPort             Dispatcher      CliSubprocessAgent
```

The fix follows the **Repository pattern from DDD**: each Port is a Repository interface. The Domain Core `await`s Port methods. The Driven Adapter implementation decides how to fulfill the contract.

### Why a monolithic `PersistenceAdapter` is wrong

Each Port has its own contract, its own data shape, its own failure modes. A single adapter would couple unrelated concerns. Per-Port decorators keep the hexagonal structure clean:

- `WalTurnStore(TurnStorePort)` decorates `SqliteTurnStore` or `PostgresTurnStore`
- `WalThreadStore(ThreadStorePort)` decorates `SqliteThreadStore` or `PostgresThreadStore`
- `WalMessageIndex(MessageIndexPort)` decorates `SqliteMessageIndex` or `PostgresMessageIndex`

The Domain Core is unaware — it just `await`s the Port.

## Architecture: two-phase approach

### Phase 1 — Enforce Port contracts (now)

Replace all 10 fire-and-forget calls with regular `await`. `PoolObserver._fire_and_forget()` disappears entirely — the observer `await`s Port methods directly.

```
Domain Core  ──await──>  TurnStorePort.log_turn()  ──>  SqliteTurnStore (WAL mode, <5ms)
                                                              ↓
                                                         turns.db ✓ durable
```

**Why this is safe:**
- SQLite WAL mode + 30s `busy_timeout` already configured (`sqlite_base.py:45`)
- LLM calls are 100–1000x slower (100–5000ms vs <5ms for SQLite write)
- DB reads are already awaited in the same hot path (`message_pipeline.py:301, 317, 355`)
- Zero new infrastructure — remove code, don't add it

### Phase 2 — WAL Repository decorators (multi-machine, future)

When a centralized external DB is needed, each Port gets a **WAL-buffered decorator** — same Protocol interface, transparent to the Core:

```
Domain Core  ──await──>  TurnStorePort.log_turn()  ──>  WalTurnStore (decorator)
                                                              ↓
                                                    ┌─────────────────┐
                                                    │ Local WAL queue  │ ← sync insert (~1-2ms)
                                                    │ (SQLite table)   │
                                                    └────────┬────────┘
                                                             ↓ background flush + retry
                                                    ┌─────────────────┐
                                                    │ PostgresTurnStore│ ← remote, pluggable
                                                    └─────────────────┘
```

**Multi-machine topology:**

```
Machine 1 (hub)                     Machine 2 (adapter)
┌──────────────────┐                ┌──────────────────┐
│ Hub → WalTurnStore│               │ Adapter → WalThread│
│       ↓ local WAL │               │           ↓ local  │
└───────┬──────────┘                └───────┬──────────┘
        ↓ flush                             ↓ flush
   ┌─────────┐                         ┌─────────┐
   │ Postgres │◄────────────────────────│ Postgres │
   └─────────┘                         └─────────┘
```

Each machine writes to its local WAL first (fast, always available). A background flusher pushes to the remote store with retry + exponential backoff. On crash, startup replay flushes unflushed entries.

## Design constraints

- **Phase 1: backward-compatible** — no schema changes, no new dependencies, no new abstractions. Just `await` the existing methods.
- **Phase 2: per-Port decorators** — each Port gets its own WAL decorator. No monolithic adapter. The decorator implements the same Protocol as the direct store.
- **Repository pattern** — Domain Core depends only on Port Protocols. Driven Adapter selection is a bootstrap concern (config/DI).
- **No NATS dependency** — persistence is local-first. NATS remains transport-only.
- **WAL is opt-in** — single-machine deployments use direct `SqliteXxxStore`. WAL decorators activate when `LYRA_PERSISTENCE_MODE=wal` or when a remote backend is configured.

## Monitoring gap (also fixed)

The `dead_backend_hits` counter has been implemented locally (not yet deployed):

- `cli_pool.py` — `_dead_backend_hits` counter, incremented by `pool_processor.py` when response completes in <100ms
- `health.py` — exposed as `dead_backend_hits` in `/health/detail` JSON
- `checks.py` — new `check_dead_backend()` fails if hits > 0
- `pool.py` — `record_dead_backend_hit()` added to `PoolContext` protocol
- `hub_outbound.py` — Hub implements it, delegates to `cli_pool`

This would have caught today's incident: 14 suspiciously-fast responses would have triggered a Telegram alert within 5 minutes.
