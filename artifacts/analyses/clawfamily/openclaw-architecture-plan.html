Storage Backend

SQLite with Vector + FTS

  • FTS table — full-text search for keyword matching (always available)
  • Vector table (chunks_vec) — semantic search via embeddings
  • Embedding cache — prevents redundant API calls across providers
  • File-based per agent at ~/.openclaw/agents/<agentId>/
Embedding Providers

6 Pluggable Providers

  • OpenAI — batch API support for async processing
  • Gemini — Google's embedding models
  • Voyage AI — specialized embeddings
  • Mistral — LLM-based embeddings
  • Ollama — local model support
  • Local QMD — built-in fallback (no API required)

Lyra Relevance
The ContextEngine interface is a powerful pattern: pluggable strategies for assembling context within token budgets, with lifecycle hooks for compaction and subagent spawning. The hybrid RRF search (FTS + vector) with graceful degradation when no embeddings are available is worth adopting. The per-agent workspace isolation prevents cross-agent memory leakage.

The binding hierarchy is critical for multi-channel agents: messages resolve to agents via specific-to-general tiers. System 03

Event System & Concurrency

Multiple decoupled event streams with lane-based command queuing for concurrency control. No single bus bottleneck.

Design Decision

Why multiple streams?

Instead of a single event bus, OpenClaw separates concerns into 4+ independent streams. Each has its own sequence counter and listener set.

Single Bus (traditional)
  1. All events through one channel
  2. Single bottleneck under load
  3. Listeners filter irrelevant events
  4. Coupled consumers
Multi-Stream (OpenClaw)
  1. Typed streams per concern
  2. No cross-stream contention
  3. Listeners opt into relevant streams
  4. Independent scaling
Event Streams

4 Independent Streams

Agent Events

Lifecycle, tool, assistant, error. Per-run monotonic seq counters. Fed to WebSocket gateway.

Diagnostic Events

14 event types: token usage, webhook lifecycle, session state, stuck detection, tool loops. OTEL-ready.

Heartbeat Events

Periodic status: sent/ok-empty/ok-token/skipped/failed. Simple polling endpoint.

System Events

Infrastructure: config reload, gateway events, presence, update availability.

Lane-Based Command Queue
src/process/command-queue.ts
// Named lanes with independent concurrency control
enqueueCommandInLane<T>(lane: string, task: () => Promise<T>, opts?)
  → Promise<T>

// Built-in lanes:
//   "main"           — default, general commands
//   "cron"           — scheduled tasks
//   "auth-probe:*"   — per-provider auth checks

// Per-lane features:
//   - Configurable concurrency (setCommandLaneConcurrency)
//   - Generation tracking (stale tasks rejected after lane reset)
//   - Graceful drain mode (markGatewayDraining)
//   - Custom error types: CommandLaneClearedError, GatewayDrainingError
Lyra Relevance
The lane-based queue prevents head-of-line blocking — cron tasks don't block user messages. Generation tracking invalidates stale tasks after resets. The diagnostic event stream with stuck-session detection and tool-loop detection provides production observability. These are patterns Lyra should adopt for its asyncio-based architecture.

Security

Plugin Guardrails

  • HTTP route overlap detection prevents conflicting endpoints
  • Auth validation on all registered routes
  • Prompt injection constraints on hook payloads
  • Per-plugin API factory with scoped access control
  • Versioned registry with cache-key tracking for identity
Lyra Relevance
OpenClaw's plugin system is more mature than Lyra's directory-based MVP. Key patterns to adopt: typed extension points (tools, hooks, channels, providers as separate registration paths), per-plugin API factory with access control, and hook priority ordering for lifecycle interception. The HTTP route overlap detection prevents plugin conflicts.
System 05

LLM Provider Abstraction

Auth profiles for multiple providers with live model discovery, token rotation, and fallback chains.

Auth Profiles

Multi-Provider

Each provider has its own auth profile: API keys, OAuth tokens, session management. Profiles are independently rotated and validated.

Anthropic OpenAI Gemini Ollama Custom
Model Discovery

Live + Auto-Update

Models are discovered at runtime from provider APIs. Catalog auto-updates ensure new models are available without code changes. Health probing validates availability.

Reliability

Fallback Chains

Multi-provider setups define fallback order. Token rotation handles rate limits. Cooldown tracking prevents hammering failed providers.

Tool Schema Guardrails
OpenClaw enforces strict rules: no Type.Union in input schemas, no anyOf/oneOf, use stringEnum for string lists. This prevents LLM confusion from ambiguous schemas.

Study Further
Multi-stream events — evaluate if Lyra's scale needs decoupled streams vs a single bus.
Typed plugin SDK — 8 extension points may be overengineered for Lyra's current stage.
Hot-reload config — useful but complex; evaluate priority vs simpler restart.
Diagnostic events — OTEL integration is valuable for production observability.
Key Architectural Warning
OpenClaw's TypeScript codebase is 39MB / 2,850 files. This scale is a product of the npm ecosystem's module fragmentation. Lyra should aim for the same capability with Python's more concise idioms — target 10-20% of the file count for equivalent functionality. Prioritize the interfaces and patterns, not the implementation scale.
