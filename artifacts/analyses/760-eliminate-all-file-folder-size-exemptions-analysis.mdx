## Source

> "Remove all 10 file exemptions (≤300 lines) and 5 folder exemptions (≤12 files) to achieve zero exemptions and clean CI."
> — Issue #760 body

## Problem

10 files exceed the 300-line limit and 5 folders exceed the 12-file limit, bypassing pre-commit hooks. This creates technical debt, reduces code navigability, and masks architectural issues where modules have grown beyond their intended scope.

## Outcome

All exemptions removed, `make lint` passes clean, CI enforces uniformly.
- Test coverage ≥ current baseline (no regressions)
- Public API unchanged (internal refactors only)
- Module boundaries align with domain (discoverability improved)

## Appetite

~3 weeks (~20 days) — fixed time, variable scope.

## Constraints

- **No breaking changes:** All refactors must preserve public APIs
- **Test coverage:** Must maintain or improve coverage; baseline required before extraction
- **Sequential dependencies:** Some files depend on others being split first
- **Architecture compliance:** All moves respect Hexagonal layer boundaries

## Context

Issue #753 (dispatch extraction) completed significant refactoring, creating momentum for this cleanup. Commit `5ded6c5` (April 16, 2026) extracted IdentityResolver + OutboundRouter from Hub, reducing hub.py from 791 → 541 lines. The exemption files are stale and must be updated.

The codebase follows Hexagonal/Ports & Adapters architecture:
- **Domain Core** (core/) — no external dependencies, pure logic
- **Adapters** (adapters/) — inbound/outbound platform implementations
- **Infrastructure** (infrastructure/) — DB, external services
- **Bootstrap** (bootstrap/) — wiring and lifecycle

## Questions Explored

## Shapes

### Shape 1: In-place decomposition

- **Description:** Split each large file in-place, creating sibling modules within the same folder. Folders: create subfolders under each package. Minimal structural change, surgical extraction.
- **Trade-offs:**
  - Pro: Lower risk, incremental PRs possible, easier review
  - Con: Less architectural improvement, may need future reorganization
- **Rough scope:** M-L (10 file splits, 5 folder reorgs, ~25 new files)

### Shape 2: Domain-driven reorganization

- **Description:** Reorganize packages by domain (core/ → core/agent/, core/memory/, core/cli/). Move files to new homes, split where needed. Higher initial cost, cleaner long-term architecture.
- **Trade-offs:**
  - Pro: Cleaner module boundaries, improved discoverability, aligns with Hexagonal architecture domains
  - Con: Higher risk, more import updates, larger PR, longer timeline
- **Rough scope:** L-XL (15+ file moves + 20+ splits + 5 folder reorgs, ~35 new files)

### Shape 3: Extraction-first

- **Description:** Extract shared utilities first (streaming state, session stores), then reorganize. May over-extract but maximizes reuse.
- **Trade-offs:**
  - Pro: Maximizes code reuse, creates shared libraries
  - Con: May over-fragment, risk of premature abstraction
- **Rough scope:** XL (25+ new files, complex import graph)

## Fit Check

| Requirement | Shape 1 | Shape 2 | Shape 3 |

| Appetite (~20 days) | ✅ | ⚠️ (may exceed) | ❌ |

**Selected shape:** In-place decomposition (Shape 1) — lower risk, fits appetite, still achieves architectural improvement via subfolder organization. Domain-driven reorganization deferred to future initiative.
