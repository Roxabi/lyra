Architecture
Hub-and-spoke asyncio engine. Stateless agents over stateful pools.
Message Flow
User → Adapter → InboundBus → Hub → Pool → Agent → LLM Backend → OutboundDispatcher → Adapter → User
Voice msg → Adapter → STT (Whisper) → transcript → normal flow → TTS (Chatterbox) → audio reply
Architecture Map
💡
Do LLM responses flow through InboundBus? No. The InboundBus is strictly one-way: Adapter → per-platform Queue → staging Queue → Hub.run(). LLM responses take a completely separate return path: Agent → Pool → Hub.dispatch_response() → OutboundDispatcher → Adapter.send().
Core Modules
core
src/lyra/core/

Hub, Pool, InboundMessage / OutboundMessage types, routing logic, auth middleware, AgentStore (SQLite WAL + write-through cache).

adapters
src/lyra/adapters/

Telegram (python-telegram-bot, long-poll + webhook) and Discord (discord.py, gateway WebSocket, watch channels) adapter implementations.

commands
src/lyra/commands/

CommandRouter, builtin commands (/help, /clear, /stop, /new), session commands (/add, /explain, /summarize — isolated LLM), CommandLoader extension system.

backends
src/lyra/backends/

LLM backend drivers: claude-cli (subprocess, streaming #321), ollama (local HTTP API), openai-compat (any OpenAI-compatible endpoint).

agents
src/lyra/agents/

Bundled system agent TOML files. Seed sources only — imported into ~/.lyra/auth.db via lyra agent init. Runtime reads DB exclusively.

Queue Topology
Inbound Queue
Type: InboundMessage envelope
Scope: Per-channel, bounded capacity
Owner: InboundBus
Adapters push normalized events here. Hub consumes and dispatches to the correct Pool based on scope_id resolution.
Audio Queue
Types: InboundAudio / OutboundAudio
Pipeline: STT (Whisper) → text → normal flow
TTS: Chatterbox, async encode
Decoupled from the main message queue. STT/TTS processing happens off the main event loop.
Outbound Queue
Owner: OutboundDispatcher
Mode: Fire-and-forget with retry
Streaming: Token-level chunks (#321)
Receives completed responses (or token chunks) and delivers to the originating adapter with back-pressure and retry logic.
Inbound vs Outbound Swim Lanes
Inbound Path
  • 1Platform event arrives at adapter (Telegram long-poll or Discord gateway)
  • 2Adapter normalizes to InboundMessage(platform, bot_id, scope_id, user_id, text)
  • 3AuthMiddleware checks user access level against config.toml
  • 4Message pushed to InboundBus (per-channel bounded queue)
  • 5Hub consumes queue, resolves scope_id → Pool (create if absent)
  • 6CommandRouter checks prefix — dispatches to builtin / session / LLM pool
  • 7Pool.run_turn() invokes Agent with history + circuit breaker check
Outbound Path
  • 1Agent produces response (full text or token stream if streaming=true)
  • 2Response pushed to OutboundDispatcher queue (fire-and-forget)
  • 3Streaming: dispatcher debounces tokens (80ms window) before sending edit
  • 4Dispatcher calls adapter.send_message() or adapter.edit_message()
  • 5Platform-specific formatting: telegramify_markdown for TG, native MD for Discord
  • 6Retry on transient platform errors (rate limit, network timeout)
  • 7Message delivered to user. Pool retains history for next turn.
Key Architectural Decisions
🧩
Stateless agents / stateful pools. Agent objects hold no mutable state. All conversation history, circuit state, and session index live in the Pool. A Pool is created per scope_id on first message and evicted after inactivity.
⚡
Sequential per-user, parallel cross-user. Each Pool processes one message at a time (ordered guarantees, no interleaved responses). Different Pools run concurrently via asyncio — no global lock.
📏
300-line gate. All source files are kept to ≤300 LOC. Larger modules are decomposed (#395–#410). Enforced by CI. Keeps every file reviewable in a single pass.
💾
DB-first agent config (ADR-029). Runtime reads exclusively from ~/.lyra/auth.db. TOML files are seed sources only. Hot-reload via updated_at polling — no restart required for config changes.
Deployment Topology

Lyra runs on any Linux or WSL2 machine with Python 3.12+. A single machine is enough — add more to split workloads.

Hub machine
Always-on · 24/7

Runs the Lyra server, adapters, and memory. Managed by supervisord. Any Linux box, Raspberry Pi, or cloud VM works.

GPU machine (optional)
On-demand · LLM / Voice

Offload local LLM inference or voice (STT/TTS) to a GPU-equipped machine. Connected via NATS or HTTP. Only needed for local models.

Dev machine (optional)
Development · testing

Run Lyra locally for development. Deploy to the hub via make deploy (git pull → pytest → restart). Any OS with Python 3.12+.

💡
Single-machine is the default. Most setups run everything on one box. The multi-machine split is optional — useful when you want 24/7 uptime on a low-power hub while keeping GPU workloads on a separate machine.
