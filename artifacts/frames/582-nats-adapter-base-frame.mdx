## Problem

Lyra's NATS adapters (TTS, STT) duplicate a boilerplate lifecycle ‚Äî connect, wait for hub readiness, subscribe to a subject, run a handler loop, shut down ‚Äî without sharing any code. Two concrete problems fall out of this:

1. **nkey bypass** (`tts_adapter_standalone.py:73`, `stt_adapter_standalone.py:44`) call `nats.connect()` directly instead of `nats_connect()`. When nkey auth is enforced on the NATS server, both processes fail to authenticate or connect as anonymous clients ‚Äî silent security gap (#563).
2. **No schema-version guard** ‚Äî adapters don't call `check_schema_version()` before processing payloads. The first time the hub sends a v2 envelope, adapters silently misinterpret it (#566).
3. **Queue group constants** ‚Äî `"tts-workers"` and `"stt-workers"` are bare string literals in the adapter files, not exported from `queue_groups.py`. Future adapters will repeat the same pattern.

The root cause is structural: there is no shared base class or template for NATS request-reply adapters. Every new adapter re-invents the same lifecycle, re-introduces the same risks.

## Who

- **Primary:** Lyra backend ‚Äî the adapter processes (`lyra_tts`, `lyra_stt`) running under supervisord on Machine 1.
- **Secondary:** Future adapters (e.g. a new voice codec adapter, WhatsApp voice path) that will inherit the base.

## Constraints

- `NatsAdapterBase` must live in `src/lyra/nats/adapter_base.py` (new file in the existing NATS module).
- `_connect()` wraps `nats_connect()` ‚Äî bare `nats.connect()` calls in subclasses must be impossible by design (no exposure in the base API).
- `_register_subject()` must only accept constants typed as `str` but enforced by runtime validation against the pattern in `_validate.py` ‚Äî bare arbitrary strings passed directly should raise at startup, not at handler time.
- `_validate_envelope()` uses the existing `check_schema_version()` from `_version_check.py`.
- `_shutdown()` order is: drain ‚Üí unsubscribe ‚Üí close (mirrors the pattern in existing three-process adapter code).
- TTS adapter config (`_NatsTtsConfig`, `_AGENT_TTS_FIELDS`) stays in `tts_adapter_standalone.py` ‚Äî the base carries no domain logic.
- `LYRA_STT_TIMEOUT` env var is absorbed as a base-level config parameter (#567: timeout configurable per-adapter via a default or env var).

## Out of Scope

- NATS Jetstream / durable consumers ‚Äî adapters remain ephemeral queue-group subscribers.
- Channel adapters (`NatsChannelProxy`, `nats_outbound_listener.py`) ‚Äî different pattern (publish/subscribe proxy, not request-reply service); separate issue (#529).
- Health endpoint over HTTP ‚Äî `health()` returns a dict consumed by the NATS health reply; no HTTP server.
- New platform adapters (WhatsApp, Slack) ‚Äî migration targets are TTS and STT only.
- `_version_check.py` rate-limit internals ‚Äî consumed as-is.

## Complexity

**Tier: F-lite** ‚Äî Clear scope, single domain (NATS adapters), 4‚Äì5 files touched. New base class + two migrations. No architectural unknowns; all building blocks (`nats_connect`, `wait_for_hub`, `check_schema_version`, `queue_groups.py`, `_validate.py`) already exist.
