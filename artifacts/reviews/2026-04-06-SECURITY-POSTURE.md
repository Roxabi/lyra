# Lyra — Security Posture
_Synthesized from the 2026-04-06 48-hour security review. Update when security decisions change._

---

## Threat Model

### Trust Zones

| Zone | Trust Level | Notes |
|------|------------|-------|
| Hub process | Fully trusted | All business logic; originates all outbound messages |
| Adapter processes (Telegram, Discord) | Semi-trusted | First-party code; communicates over network bus |
| NATS bus | Semi-trusted | Internal network boundary; platform_meta is sanitized at inbound |
| Platform APIs (Telegram, Discord) | Untrusted | User-controlled content; normalized through adapter before entering domain |
| Voice adapter processes (STT, TTS) | Semi-trusted | Same nkey as hub (current); no per-service permission scoping |

### Trust Boundaries

```
[Platform users]
  → [Telegram/Discord APIs]
  → [Adapter process] ─── normalize(), verify sender identity
  → [NATS bus] ──── TLS 1.3 + nkey (when activated)
  → [NatsBus.handler()] ──── sanitize_platform_meta() ← PRIMARY TRUST BOUNDARY
  → [MiddlewarePipeline] ──── scope validation (TurnStore)
  → [Hub / Agent / LLM]
```

### NATS Bus Trust Level

NATS is a **semi-trusted internal boundary**. Key facts:
- Adapters are first-party processes, but communicate over a network bus — they can be compromised.
- The sanitizer enforces that `platform_meta` only carries what the hub expects (9-key allowlist).
- Session scope validation (`thread_session_id` validated against `pool_id` ownership) prevents cross-pool hijacking.
- **Production currently runs without server-side nkey enforcement** — `nats.conf` is ready but not yet activated (MEDIUM risk, see below).

## Authentication

### nkey Status

| Component | nkey Wired | Notes |
|-----------|-----------|-------|
| Hub (`nats_connect()`) | Yes (`src/lyra/nats/connect.py`) | Reads seed from `NATS_NKEY_SEED_PATH`; fails closed if missing |
| Telegram adapter (`nats_connect()`) | Yes | Same helper |
| Discord adapter (`nats_connect()`) | Yes | Same helper |
| TTS adapter standalone | **NO** | Calls `nats.connect()` directly, bypassing `nats_connect()` — `src/lyra/bootstrap/tts_adapter_standalone.py:73` |
| STT adapter standalone | **NO** | Calls `nats.connect()` directly, bypassing `nats_connect()` — `src/lyra/bootstrap/stt_adapter_standalone.py:44` |
| NATS server enforcement | **NOT ACTIVATED IN PRODUCTION** | `nats.conf` prepared; production still runs `nats-local.conf` (no auth) |

**MEDIUM — TTS and STT adapters bypass nkey auth:**
Root cause: `tts_adapter_standalone.py` and `stt_adapter_standalone.py` predate the `nats_connect()` helper and were not updated when nkey support was added (commit `343ff69`). When `NATS_NKEY_SEED_PATH` is set and the NATS server enforces auth, these processes will either fail to connect or connect as unauthenticated clients.

Fix: replace `import nats` + `await nats.connect(nats_url)` with `from lyra.nats import nats_connect` + `await nats_connect(nats_url)` in both files. Add dedicated seeds for these processes to `gen-nkeys.sh`.

### Server-Side Enforcement Gap

**MEDIUM (deployment, not code):** Production NATS runs `nats-local.conf` (no auth). `nats.conf` (TLS 1.3 + nkey) is ready to activate but not yet deployed. Until `nats.conf` is activated, any process that can reach `localhost:4222` can publish to any NATS subject without credentials.

Fix: switch from `nats-local.conf` to `nats.conf` in the supervisor/systemd launch config.

### Key Storage

- Seeds: `/etc/nats/nkeys/{hub,llm-worker,monitor}.seed` — outside repo, mode `0600`, root-owned
- Generated by `sudo deploy/nats/gen-nkeys.sh` (idempotent; skip-if-exists)
- No seeds or `auth.conf` are committed to git
- Only three nkeys generated: `hub`, `llm-worker`, `monitor` — no dedicated STT/TTS keys

### Key Rotation

No rotation tooling or expiry mechanism exists. To rotate: delete `/etc/nats/nkeys/`, re-run `gen-nkeys.sh`, restart all processes. No documented runbook.

### Fail-Closed Behavior

- `NATS_NKEY_SEED_PATH` unset → connects without authentication (explicit dev mode, documented)
- Seed file missing/unreadable/empty → `sys.exit()` with clear error (correct fail-closed)
- **No file permission check** — `_read_nkey_seed()` does not validate `st_mode`; a world-readable seed file is silently accepted

## Input Sanitization

### platform_meta Allowlist

`PLATFORM_META_ALLOWLIST` in `src/lyra/nats/_sanitize.py:13-23` — 9 keys:

| Key | Used for |
|----|---------|
| `guild_id` | Discord server routing |
| `channel_id` | Channel routing |
| `message_id` | Reply reference (Telegram/Discord API integer) |
| `thread_id` | Thread routing |
| `channel_type` | Channel type discrimination |
| `chat_id` | Telegram chat routing |
| `topic_id` | Topic routing |
| `is_group` | Group vs DM discrimination |
| `thread_session_id` | Session resume lookup |

All other keys (including `_`-prefixed internal keys like `_session_update_fn`) are stripped. Sanitization applied in `NatsBus.handler()` immediately after deserialization, before the item reaches the staging queue.

### Prompt Injection Risk

**None.** `platform_meta` values are used for routing logic only — they do not appear in LLM prompt templates or agent context.

- `thread_session_id` → session DB lookup only (parameterized query, SQLi impossible)
- `message_id` → used as `reply_to` integer in platform API calls
- `guild_id`, `channel_id`, `chat_id`, etc. → routing metadata only

### Remaining Input Validation Gaps

**LOW — No length or type validation on allowlisted values:**
Values for allowlisted keys pass through without type coercion or length capping. A crafted NATS message with `thread_session_id` set to an arbitrarily long string passes the allowlist filter unchanged and reaches `TurnStore.get_session_pool_id()`. The parameterized query prevents SQL injection, but the unbounded string traverses the full pipeline.

Fix: add value-shape validation in `sanitize_platform_meta()` — coerce known string fields to `str`, cap at 256 chars, coerce `is_group` to `bool`.

## Known Bug: scrub_nats_url() Not Called in Error Exit Paths

**HIGH — Credentials in error output on NATS connection failure**

`scrub_nats_url()` is correctly applied to `log.info()` (Fix 1 of commit `8c19400`), but the failure-path `sys.exit()` calls still use the raw URL:

- `src/lyra/bootstrap/adapter_standalone.py:46` — `sys.exit(f"Failed to connect to NATS at {nats_url!r}: {exc}")`
- `src/lyra/bootstrap/hub_standalone.py:154` — same pattern

If `NATS_URL` contains embedded credentials (`nats://user:secret@host:4222`), the raw URL is printed to stderr and captured by supervisord logs — defeating Fix 1.

Fix (one line each, `scrub_nats_url` is already imported):
```python
# adapter_standalone.py:46 and hub_standalone.py:154 — replace:
sys.exit(f"Failed to connect to NATS at {nats_url!r}: {exc}")
# with:
sys.exit(f"Failed to connect to NATS at {scrub_nats_url(nats_url)!r}: {exc}")
```

## Cache and DoS Mitigations

All 5 caches are now bounded (commit `4f5c884`):

| Cache | Limit | TTL/Timeout | Notes |
|-------|-------|------------|-------|
| `NatsOutboundListener._cache` | `_MAX_CACHE_SIZE = 500` entries | `_CACHE_TTL_SECONDS = 120s` (reaper) | Hard eviction on overflow with WARNING log |
| `NatsOutboundListener._stream_queues` | `_MAX_QUEUE_SIZE = 256` chunks/stream | 120s `wait_for` timeout | `put_nowait` + catch `QueueFull`; drops chunk, logs WARNING |
| `NatsOutboundListener._stream_outbound` + `_stream_tasks` | `_MAX_STREAMS = 100` concurrent streams | 120s task timeout | New stream_ids rejected when at limit |
| `NatsBus._staging` | 500 (default, configurable via `[inbound_bus]`) | N/A | Was hardcoded; now operator-tunable |
| Discord `thread_sessions` | 500 entries | N/A | Insertion-order eviction |

**Reaper:** `_reap_stale()` task fires every `_REAPER_INTERVAL_SECONDS = 30s`. Uses snapshot iteration (no concurrent modification). Evicts `_cache`, `_cache_ts`, `_stream_outbound`, `_stream_tasks`, `_stream_queues` in one pass. Cancellation via `stop()` with `contextlib.suppress(CancelledError)`.

**Gap — `_terminated_streams` not cleaned by reaper (LOW):**
`NatsOutboundListener._terminated_streams` is a `set[str]` with no eviction by the reaper. Stream IDs are removed in `_drain_stream`'s `finally` block, so completed streams are cleaned. But if a stream is tombstoned via `_handle_stream_error` and never starts a drain task, the ID lingers indefinitely. At high churn this leaks memory.

**Gap — Non-deterministic tombstone eviction (LOW):**
`remember_terminated` in `nats_stream_decoder.py:91` uses `set.pop()` to evict one entry when `_MAX_TERMINATED_STREAMS = 500` is reached. `set.pop()` removes an **arbitrary** element — an active tombstone protecting a live stream_id could be evicted while stale entries remain.

Fix: replace `set` with `collections.OrderedDict` for FIFO eviction.
