## Problem

When the Anthropic API fails (rate limit, 503, timeout), Lyra has no recovery mechanism. Errors cascade silently to all users, the hub keeps firing requests at a broken backend, and there is no user-facing feedback about degraded service. This creates a poor user experience and risks hammering a rate-limited API, making recovery slower.

The missing layer is API resilience at the LLM call site: a circuit breaker that detects consecutive failures, opens to reject requests fast (with a user-friendly message), then probes for recovery. This is the third and innermost layer of the 3-layer rate-limiting stack (hub throttle → pairing brute-force → LLM circuit).

## Who

- **Primary:** All Lyra users — they receive cascading errors or silent failures when the API degrades.
- **Secondary:** Operators/admins — need visibility into circuit state to diagnose outages and confirm recovery.

## Constraints

- **Async-native:** Implementation must be `asyncio`-compatible; no blocking calls. Wraps coroutines, not functions.
- **Anthropic SDK streaming:** `messages.stream()` returns an async context manager — the circuit breaker must handle this correctly.
- **Config-driven:** Failure threshold and recovery timeout must be configurable via TOML or agent config (no hardcoding).
- **Minimal blast radius:** New module `core/llm_circuit.py`; modifications limited to `AnthropicAgent` and `core/hub.py`.
- **Part of Phase 0 epic #101:** Must not block or be blocked by #102 (pairing rate limiter).
- **Tests required:** All state transitions (CLOSED→OPEN→HALF_OPEN→CLOSED/OPEN) must be unit-tested.

## Out of Scope

- Machine 2 failover / cloud fallback (#23) — different concern, different scope.
- Per-user circuit breaking — one shared circuit for the Anthropic API is sufficient.
- Retry logic with exponential backoff — the circuit breaker rejects fast; retries are the caller's responsibility (user re-sends after `retry_after`).
- Frontend/UI changes — user feedback is delivered via the existing adapter message mechanism.

## Status Visibility

Both surfaces targeted:
- **Admin command** (Telegram/Discord `!circuit` or `/circuit`) — returns current state, failure count, `retry_after` if open.
- **HTTP endpoint** `GET /status` — exposes circuit state for monitoring/health checks.

## Complexity

**Tier: F-full** — New cross-cutting async module with state machine logic, integration into streaming SDK calls, dual status surfaces, and full state-transition test coverage.

Signals observed:
- New architectural pattern (circuit breaker state machine) not previously present in codebase
- Wraps async streaming context manager — non-trivial integration
- Touches hub, agent, and API surface (3 integration points)
- Config-driven with TOML — requires config schema changes
- Full test suite for 4 state transitions required
