## Problem

When strict pyright was enabled (#395 / PR #410), seven `reportUnknown*` and
`reportMissing*` rules had to be globally suppressed because config-loading
sites (`bootstrap/config.py`, `config.py`, `bootstrap/agent_factory.py`) return
bare `dict` from TOML parsing. These untyped dicts cascade through the entire
codebase, generating ~3,000 false positives across src (1,273) and tests (1,986).

The suppressions mask real type errors and undermine the value of strict mode.
Until they are removed, new code can silently introduce type-unsafe patterns
that pyright cannot catch.

## Who

- **Primary:** Developers working on Lyra — every contributor benefits from
  stricter type checking catching real bugs before runtime.
- **Secondary:** CI pipeline — once suppressions are removed, the type gate
  becomes meaningful for these rule families.

## Constraints

- **Approach:** Pydantic v2 models for **all config structures** (~25 config
  dataclasses across 12 files). Domain objects (~50 dataclasses: messages,
  events, render events, LLM results, commands) stay as stdlib dataclasses.
- **Config migration scope:** Every dataclass that represents configuration
  loaded from TOML/env/DB becomes a Pydantic `BaseModel`. This includes:
  - `config.py` (4 classes), `adapters/telegram.py`, `adapters/discord_config.py`
  - `monitoring/config.py`, `core/runtime_config.py` (2 classes)
  - `core/agent_config.py` (11 classes), `tts/__init__.py`, `stt/__init__.py`
  - `core/tool_display_config.py`, `core/stores/pairing_config.py`
- **Incremental:** Re-enable pyright rules one at a time, confirming 0 errors
  per rule before moving to the next. Each rule can be its own commit/PR slice.
- **Src + tests together:** Each rule must be clean in both `src/` and `tests/`
  before re-enabling — no split between src-first and tests-later.
- **No blockers:** No in-flight PRs conflict with this work.

## Out of Scope

- Migrating domain dataclasses to Pydantic (messages, events, render events,
  LLM results, circuit breakers, commands — ~50 classes).
- Refactoring config loading logic beyond what's needed for type safety.
- Runtime validation improvements (nice side-effect of Pydantic, but not the goal).

## Complexity

**Tier: F-full** — Touches multiple config-loading domains (bootstrap, adapters,
LLM, agent factory), requires designing a Pydantic model hierarchy, and involves
7 incremental rule re-enablement passes across the entire codebase.

Signals observed:
- 17 files with `dict[str, Any]` in src alone
- ~25 config dataclasses across 12 files to migrate to Pydantic
- 7 suppressed pyright rules to re-enable incrementally
- Multiple domains affected: bootstrap, adapters, LLM drivers, agent config
- New dependency introduction (pydantic, pydantic-settings)
- ~3,000 cascading type errors to resolve
