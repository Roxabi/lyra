# Wire Format Review — 2026-04-06

## Verdict: WARN

Two issues found, neither breaks production today, both require action before
Slice 2 of #534 ships:

1. `OutboundMessage.schema_version` has no check on the receive side (adapter).
2. `SttMiddleware` line 91 has an unconditional `assert hub._msg_manager is not None`
   that crashes with `AssertionError` instead of a user-visible reply when the Hub
   is instantiated without a `MessageManager` (valid in tests and custom bootstrap).

## InboundMessage Unification (520c33b)

### Audio field preservation

All `InboundAudio` fields are carried into `InboundMessage` via `AudioPayload`:

| `InboundAudio` field | `AudioPayload` field | Status |
|----------------------|----------------------|--------|
| `audio_bytes` | `audio_bytes` | Preserved |
| `mime_type` | `mime_type` | Preserved |
| `duration_ms` | `duration_ms` | Preserved |
| `file_id` | `file_id` | Preserved |
| — (absent) | `waveform_b64` | New field, `None` in compat path |

Platform identity fields (`platform`, `bot_id`, `scope_id`, `user_id`,
`user_name`, `is_mention`, `timestamp`, `trust_level`, `trust`,
`platform_meta`, `routing`) are copied 1:1 in `_convert_legacy()`.

`text` / `text_raw` are intentionally empty strings — transcription has not
run at conversion time.

### Type discriminator

`modality="voice"` is the discriminator. It was already present on
`InboundMessage` and is set explicitly in `_convert_legacy()`. The STT
middleware keyed on `msg.modality != "voice"` for the passthrough guard —
correct.  Re-entrance guard (`msg.text != ""`) correctly skips messages that
entered as pre-transcribed voice (pipeline-transcribed path, H-8).

### Consumers

| Consumer | Pre-534 path | Post-534 path | Status |
|----------|-------------|---------------|--------|
| `AudioPipeline.run()` | Consumed `InboundAudio` from `inbound_audio_bus` | Deleted | Replaced by `SttMiddleware` |
| `SttMiddleware` | (new) | Consumes `InboundMessage(modality="voice", audio=AudioPayload)` | Correct |
| `AnthropicAgent` | Had `_audio is not None and self._stt is None` branch | Removed | STT-None is now filtered upstream by `SttMiddleware`; agents never see `audio != None` without STT |
| `SimpleAgent` | Had `if self._stt is None` guard | Replaced with `assert self._stt is not None` | Correct — assert is accurate given upstream guarantee |
| `NatsOutboundListener.cache_inbound()` | Accepted `InboundMessage | InboundAudio` | Unchanged | Signature retained, no break |

The legacy `lyra.inbound.audio.*` NATS subject is still subscribed via
`InboundAudioLegacyHandler` (Slice 1 compat shim). The shim converts and
injects into the unified bus via `Bus.inject()`. This is correctly wired in
`hub_standalone.py` and `unified.py`.

### Potential issue — `SttMiddleware` hard assert on `_msg_manager` (MEDIUM)

`middleware_stt.py:91` — `assert hub._msg_manager is not None`

`hub.py` declares `msg_manager: MessageManager | None = None` — it is
optional. Production bootstrap always sets it (`hub_standalone.py:247`), but
any custom bootstrap or test that creates `Hub()` without a `MessageManager`
and sends a voice message will hit `AssertionError` instead of receiving the
`stt_unsupported` reply template.

`simple_agent.py` uses `assert self._stt is not None` in the same pattern —
but that assertion is architecturally sound because the STT guarantee is
enforced by the middleware ordering. The `_msg_manager` one is not: the
ordering of middleware stages does not guarantee `_msg_manager` is set.

**Fix:** replace the assert with a graceful fallback, mirroring the pattern
used throughout the rest of the hub:
```python
# middleware_stt.py line 91 — replace:
assert hub._msg_manager is not None
# with:
if hub._msg_manager is None:
    log.error("SttMiddleware: _msg_manager not configured — dropping voice message")
    return _DROP
```

## schema_version (d970d3a + 747211e)

### Coverage across envelopes

| Envelope | `schema_version` field | `SCHEMA_VERSION_*` constant | `check_schema_version` on receive |
|----------|------------------------|-----------------------------|------------------------------------|
| `InboundMessage` | `int = 1` | `SCHEMA_VERSION_INBOUND_MESSAGE = 1` | Yes — `NatsBus._make_handler()` |
| `InboundAudio` (legacy) | `int = 1` | `SCHEMA_VERSION_INBOUND_AUDIO = 1` | Yes — `NatsBus._make_handler()` + `InboundAudioLegacyHandler._handle()` |
| `OutboundMessage` | `int = 1` | `SCHEMA_VERSION_OUTBOUND_MESSAGE = 1` | **No — missing on adapter receive side** |
| `TextRenderEvent` | `int = 1` | `SCHEMA_VERSION_TEXT_RENDER_EVENT = 1` | Yes — `NatsRenderEventCodec.decode()` |
| `ToolSummaryRenderEvent` | `int = 1` | `SCHEMA_VERSION_TOOL_SUMMARY_RENDER_EVENT = 1` | Yes — `NatsRenderEventCodec.decode()` |

### Missing check — `OutboundMessage` on adapter receive side (LOW, pre-Slice-2 action)

`nats_outbound_listener.py` — `_handle_send()` (line 144), `_handle_stream_start()`
(line 190), `_handle_attachment()` (line 165) — all call `_deserialize_dict(outbound_data,
OutboundMessage)` without a `check_schema_version` gate.

`OutboundMessage` carries `schema_version: int = 1` and
`SCHEMA_VERSION_OUTBOUND_MESSAGE = 1` exists in `message.py`, but the constant
is never imported or used anywhere. A future hub version bumping
`OutboundMessage`'s schema would silently misinterpret the payload on old
adapters — exactly the failure mode the versioning system was designed to prevent.

This is low severity today (version is 1 everywhere, no bump is pending), but
the invariant is architecturally broken and should be resolved before the first
version bump.

**Fix:** add `check_schema_version` to the outer envelope handler in
`NatsOutboundListener._handle_message()` before dispatching to `_handle_send` /
`_handle_stream_start` / `_handle_attachment`. The outer dict already has
`msg_type`; `outbound_data` is the inner payload dict where `schema_version`
lives. The check belongs on `outbound_data`, not the outer envelope.

### Mismatch handling correctness

`check_schema_version` rules:
- Missing field → version 1 (legacy compat). Correct.
- `bool` value → drop (explicitly excluded despite `bool` being `int` subclass in Python). Correct.
- `int <= 0` → drop. Correct.
- `int > expected` → drop. Correct.
- `int in [1, expected]` → accept (forward-compat for older senders). Correct.

### Rate-limit robustness (747211e)

Module-level `_last_log_ts: dict[str, float]` is process-wide, intentionally
shared across all `NatsBus` and `NatsOutboundListener` instances. This is the
correct scope: log volume is a process concern, counter isolation is a
per-instance concern. Both are independently correct.

`_reset_log_state()` test helper + `autouse` fixture in `conftest.py` prevents
test-ordering flakiness. Pattern is sound.

The 60-second `_LOG_INTERVAL_S` constant is reasonable for a deployment-caused
flood. No concern.

## Type Safety

### `decode_stream_events` return type fix (747211e)

`AsyncIterator["RenderEvent"]` → `AsyncGenerator["RenderEvent", None]`

Correct. `async def` with `yield` is a generator, not just an iterator.
`AsyncIterator` was technically accepted by mypy in some configurations but
`AsyncGenerator` is the precise annotation. No behavioral change.

### `NatsOutboundListener.version_mismatch_count()` and `NatsBus.version_mismatch_count()`

Both are public, identically named, and return from separate per-instance dicts.
Counter isolation is correct: `NatsBus` instance A and instance B have independent
counts. The naming symmetry between bus and listener is useful for monitoring.

### `_ENVELOPE_VERSIONS` dispatch table in `nats_bus.py`

```python
_ENVELOPE_VERSIONS: dict[type, tuple[str, int]] = {
    InboundMessage: ("InboundMessage", SCHEMA_VERSION_INBOUND_MESSAGE),
    InboundAudio: ("InboundAudio", SCHEMA_VERSION_INBOUND_AUDIO),
}
```

The table is looked up with `self._item_type`. If a `NatsBus` is instantiated
with a type not in the table (e.g., a future third inbound envelope), the
`_make_handler()` line `envelope_name, expected = _ENVELOPE_VERSIONS[self._item_type]`
raises `KeyError` at subscription time — loud, fast failure rather than silent
mismatch. This is acceptable, but a `KeyError` message at runtime won't
immediately point to the fix. Low concern.
