## Goal

Expose corpus.db via a live HTTP API reachable at a public custom domain. GitHub org webhook keeps data fresh in near-real-time; existing reconciler heals gaps when M₂ is off.

## Users

- **Primary:** Mickael — queries corpus API from any machine; uses the dep-graph frontend tab.
- **Secondary:** `roxabi-dashboard` frontend — consumes `GET /api/*` endpoints.
- **Tertiary:** CLI scripts that today read corpus.db directly (can migrate to API at their own pace).

## Constraints

- `CORPUS_DB_PATH` env var configures DB path; default `~/.roxabi/corpus.db`.
- No inbound ports opened; `cloudflared` handles all public traffic.
- `cloudflared` runs under supervisord — ¬embedded in FastAPI process.
- Reconciler (`scripts/corpus/`) unchanged; wiring additions only (startup trigger + hourly schedule).
- Auth MVP: public read, webhook HMAC only. ¬OAuth, ¬Cloudflare Access.

## Endpoint Contracts

### `GET /api/issues`

Returns all issues. Supports optional query params.

| Param | Type | Description |
|-------|------|-------------|
| `repo` | string | Filter by repo slug (e.g. `Roxabi/lyra`) |
| `state` | `open` \| `closed` \| `all` | Default: `all` |
| `label` | string | Filter by label name (exact match) |

**Response 200:**
```json
{
  "issues": [
    {
      "key": "Roxabi/lyra#864",
      "repo": "Roxabi/lyra",
      "number": 864,
      "title": "...",
      "state": "open",
      "url": "https://github.com/Roxabi/lyra/issues/864",
      "labels": ["feat", "size:L"],
      "milestone": null,
      "is_stub": false,
      "created_at": "2026-04-01T10:00:00Z",
      "updated_at": "2026-04-22T08:00:00Z",
      "closed_at": null
    }
  ],
  "total": 1
}
```

### `GET /api/issues/:key`

`:key` = URL-encoded `{repo}#{number}` (e.g. `Roxabi%2Flyra%23864`).

**Response 200:** single issue object (same shape as above) + `blocking` + `blocked_by` arrays.

```json
{
  "key": "Roxabi/lyra#864",
  "...": "...",
  "blocking": [{"key": "Roxabi/lyra#865", "number": 865, "repo": "Roxabi/lyra"}],
  "blocked_by": []
}
```

**Response 404:** `{"detail": "Issue not found"}`

### `GET /api/graph`

Edge list for the dep-graph renderer.

**Response 200:**
```json
{
  "nodes": [
    {"key": "Roxabi/lyra#864", "repo": "Roxabi/lyra", "number": 864,
     "title": "...", "state": "open", "labels": []}
  ],
  "edges": [
    {"src": "Roxabi/lyra#861", "dst": "Roxabi/lyra#864", "kind": "blocks"},
    {"src": "Roxabi/lyra#800", "dst": "Roxabi/lyra#864", "kind": "parent"}
  ]
}
```

`kind: blocks` — `src` blocks `dst`. `kind: parent` — `src` is parent of `dst` (child). Matches `edges` table direction.

> **Schema note:** `edges` needs a `kind` column (`blocks` | `parent`) to distinguish the two edge types. If the current `scripts/corpus/` schema lacks it, add a migration as part of this spec's implementation.

### `GET /api/repos`

Repos present in corpus.db + their sync state.

**Response 200:**
```json
{
  "repos": [
    {"repo": "Roxabi/lyra", "last_synced_at": "2026-04-22T07:00:00Z", "last_cursor": "..."}
  ]
}
```

### `POST /webhook/github`

GitHub org webhook receiver.

**Request headers required:**
- `X-Hub-Signature-256`: `sha256=<hmac>` (HMAC-SHA256 of raw body with `GITHUB_WEBHOOK_SECRET`)
- `X-GitHub-Event`: event type

**Validation:** compute `hmac.new(secret, body, sha256).hexdigest()`, compare via `hmac.compare_digest`. Return 401 if mismatch.

**Handled events → table mapping:**

| `X-GitHub-Event` | Action(s) handled | Target table |
|---|---|---|
| `issues` | `opened`, `closed`, `reopened`, `edited`, `labeled`, `unlabeled`, `milestoned`, `demilestoned` | `issues`, `labels` |
| `issues` (any) | full upsert on `issues`; replace all `labels` rows for that key | `issues`, `labels` |
| `issue_dependencies` | `blocked_by_added`, `blocked_by_removed`, `blocking_added`, `blocking_removed` | `edges` (blocks) |
| `sub_issues` | `parent_issue_added`, `parent_issue_removed`, `sub_issue_added`, `sub_issue_removed` | `edges` (parent/child) |

**`issue_dependencies`** — emitted when a blocking relationship is added/removed. `blocked_by_added` and `blocking_added` describe the same edge from opposite sides; handle only one direction (e.g. act on `blocked_by_*`, ignore `blocking_*` as duplicates) to avoid double-writes. Upsert/delete `edges(kind='blocks', src_key=<blocker>, dst_key=<blocked>)`.

**`sub_issues`** — emitted when parent/child links change. Same duplicate-direction caveat: act on `sub_issue_added`/`sub_issue_removed`, ignore `parent_issue_*`. Upsert/delete `edges(kind='parent', src_key=<parent>, dst_key=<child>)`.

> Payload shapes are not fully documented by GitHub; infer keys from the first real delivery captured in webhook logs and adjust the handler. The reconciler is authoritative for edge state — webhooks are an optimisation, not a contract.

**Unhandled events:** return 200 (ack, no-op).

**Response 200:** `{"ok": true}`
**Response 401:** `{"detail": "Invalid signature"}`

**Mismatch healing:** after each webhook upsert, if `sync_state.last_synced_at` for that repo is older than 1 hour, trigger a reconciler run in background (non-blocking).

### `GET /health`

**Response 200:**
```json
{
  "status": "ok",
  "db": "~/.roxabi/corpus.db",
  "db_reachable": true,
  "issue_count": 312
}
```

`db_reachable: false` if corpus.db cannot be opened (returns 200 still — caller decides on action).

## `cloudflared` Supervisord Unit

File: `deploy/supervisor/conf.d/cloudflared.conf`

```ini
[program:cloudflared]
command=cloudflared tunnel run %(ENV_CLOUDFLARED_TUNNEL_NAME)s
environment=TUNNEL_TOKEN="%(ENV_CLOUDFLARED_TUNNEL_TOKEN)s"
autostart=false
autorestart=false
stdout_logfile=%(ENV_HOME)s/.roxabi/logs/cloudflared.log
stderr_logfile=%(ENV_HOME)s/.roxabi/logs/cloudflared.err
```

**One-time tunnel provisioning (documented steps):**
1. `cloudflared login` (authenticates to CF account).
2. `cloudflared tunnel create dashboard-roxabi` → saves credentials JSON under `~/.cloudflared/`.
3. In CF dashboard: add DNS CNAME `dashboard.roxabi.dev` → `<tunnel-id>.cfargotunnel.com`.
4. Set `CLOUDFLARED_TUNNEL_NAME=dashboard-roxabi` in supervisord env or `.env`.
5. `make cloudflared start`.
6. Verify: `curl https://dashboard.roxabi.dev/health`.

**M₂ → M₁ migration steps:**
1. `rsync ~/.roxabi/corpus.db roxabituwer:~/.roxabi/`.
2. Copy `~/.cloudflared/` credentials to M₁ (or re-login).
3. Start supervisord stack on M₁.
4. In CF dashboard: edit ingress to point at M₁ address (or create new tunnel entry).
5. Verify `/health` through tunnel from external machine.
6. Stop `cloudflared` + FastAPI on M₂.

## Success Criteria

- [ ] `scripts/corpus/` reconciler runs on FastAPI startup (non-blocking background task).
- [ ] Reconciler runs on a configurable interval (default 3600s); interval set via `CORPUS_SYNC_INTERVAL_SECONDS`.
- [ ] `deploy/supervisor/conf.d/cloudflared.conf` exists; `make cloudflared start` launches the daemon.
- [ ] `GET /health` returns `{"status": "ok", "db_reachable": true}` with corpus.db reachable.
- [ ] `GET /api/issues` returns all issues; `?repo=` and `?state=` filters work.
- [ ] `GET /api/issues/:key` returns a single issue with `blocking` + `blocked_by`; 404 on unknown key.
- [ ] `GET /api/graph` returns nodes and edges matching `issues` and `edges` tables.
- [ ] `GET /api/repos` returns repos from `sync_state` table.
- [ ] `POST /webhook/github` with valid `X-Hub-Signature-256` upserts issue to `issues` + `labels` tables.
- [ ] `POST /webhook/github` with `action: blocked_by_added` on `issue_dependencies` event upserts `edges(kind='blocks', ...)`; `blocked_by_removed` deletes it.
- [ ] `POST /webhook/github` with `action: sub_issue_added` on `sub_issues` event upserts `edges(kind='parent', ...)`; `sub_issue_removed` deletes it.
- [ ] Opposite-direction actions (`blocking_*`, `parent_issue_*`) are acknowledged with 200 but produce no DB write (dedup).
- [ ] `POST /webhook/github` with invalid signature returns HTTP 401.
- [ ] `POST /webhook/github` with unrecognised event type returns HTTP 200 (no-op).
- [ ] `CORPUS_DB_PATH` env var overrides default `~/.roxabi/corpus.db`; app does not hard-code path.
- [ ] Tunnel is reachable at `dashboard.roxabi.dev`; `GET /health` succeeds through tunnel.
- [ ] Frontend dep-graph tab renders graph from `GET /api/graph` (static data replaced with live API call).
- [ ] Repo dropdown populated from `GET /api/repos`; selecting a repo filters the graph.
- [ ] M₂ → M₁ migration steps are documented and verified (rsync + tunnel flip).
- [ ] No OAuth or Cloudflare Access required to read `GET /api/*` in MVP.
