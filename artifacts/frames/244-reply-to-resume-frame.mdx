## Problem

When a user replies to an old Telegram message, Lyra currently ignores the reply-to context and either starts a new session or continues the current active one. This means users lose the thread of a previous conversation — there is no way to "pick up where we left off" by replying to the message that triggered the original exchange.

The 2ndBrain project solved this with a `SessionArchive` + `context_resolver.py` approach: each outgoing message is indexed by its `message_id` in a `conversation_turns` table. When a reply arrives, the `message_id` is looked up to retrieve the original `session_id`, which is then restored in the pool.

Lyra already has the pool and session model in place (issue #83). The missing pieces are: (1) the `message_id → session_id` index (provided by #67), and (2) the resolver that fires the lookup and resets the pool.

## Who

- **Primary:** Telegram users of Lyra who carry on multi-thread conversations and want to resume old context by replying to a specific message.
- **Secondary:** The pool/session layer — needs a clean API for session handoff without breaking in-flight state.

## Constraints

- **Blocked by #67** — the `conversation_turns` table (with `message_id` and `session_id` columns) must exist before this feature can query it.
- Telegram-only for this iteration; Discord deferred.
- Session file must exist on disk (`~/.claude/projects/<project>/<session_id>.jsonl`) — if pruned, fall back gracefully.
- Must survive pool resets (`/new` command) — the message index lives in the DB, not the pool.

## Out of Scope

- Discord support (deferred to follow-up issue).
- Session archiving / compression (separate concern).
- Backfilling `message_id` for turns logged before #67 lands.

## Complexity

**Tier: F-lite** — Clear scope, single backend domain (session/pool layer + Telegram adapter). The logic mirrors 2ndBrain's proven pattern, reducing unknowns. Main risk is the DB query integration and graceful fallback path.
