## Problem

With Hub now running as a standalone `lyra_hub` process (#457), the adapters still hold a direct reference to a `Hub` instance via their constructor. This coupling breaks the distributed model: an adapter process can't instantiate a Hub from a remote process, and the current bootstrap produces two isolated Hub instances when adapters run separately.

The fix is to sever that reference entirely. Adapters become pure I/O translators: platform events → NATS inbound subject, NATS outbound subject → platform API calls. The infrastructure for both directions is already in place — `NatsBus` (C2) and `NatsChannelProxy` (C4) — but the adapters have not been rewired to use them yet.

The deferred gate ("until hub-per-adapter becomes a bottleneck") is lifted: Hub extraction is stable, NatsBus + NatsChannelProxy are implemented, and starting work now avoids accumulating coupling debt.

## Who

- **Primary:** The Lyra runtime — adapters running as independent processes (`lyra_telegram`, `lyra_discord`) communicating with `lyra_hub` over NATS.
- **Secondary:** Developers running `--adapter all` in embedded dev mode (must continue to work with `LocalBus`, no NATS required).

## Constraints

- `NatsBus` (inbound, `src/lyra/nats/nats_bus.py`) and `NatsChannelProxy` (outbound, `src/lyra/nats/nats_channel_proxy.py`) are already implemented — do not rewrite them.
- Dev mode (`--adapter all`) must keep working with `LocalBus` — NATS must remain opt-in.
- Streaming chunk protocol (#454) must work over NATS — `NatsChannelProxy` already handles `OutboundAudioChunk`/`RenderEvent` pub; adapters must subscribe and process.
- Key files: `adapters/telegram.py`, `adapters/telegram_inbound.py`, `adapters/discord.py`, `adapters/discord_inbound.py`, `adapters/_shared.py`.
- C6 ops work (supervisor priority, deploy rolling restart, ADR-021 amendment) is a separate slice — do not include here.

## Out of Scope

- NatsBus implementation (already done).
- NatsChannelProxy implementation (already done).
- Hub-side changes — Hub already uses `NatsChannelProxy` to dispatch outbound; no Hub changes needed for this slice.
- Hub-per-adapter architecture (ADR-021 Option B) — still deferred.
- NATS TLS / nkey hardening (Slice D).
- Ops, supervisor ordering, deploy scripts (C6).

## Complexity

**Tier: F-lite** — Two adapters, single domain (adapter I/O wiring), clear scope. Infrastructure exists; work is connecting it and threading the dev/prod bus selection through bootstrap.

Signals:
- Affects 4–5 files, all in `adapters/` + `nats/`.
- No new abstractions needed — `NatsBus` and `NatsChannelProxy` cover both directions.
- Mode selection (LocalBus vs NatsBus) is a bootstrap-level concern, not a new pattern.
- Outbound subscription loop on the adapter side is the only genuinely new logic.
