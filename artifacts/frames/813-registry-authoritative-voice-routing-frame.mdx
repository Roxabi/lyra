## Problem

The hub-side voice clients (`NatsSttClient` / `NatsTtsClient`) currently run two independent load balancers in series on every request. The `VoiceWorkerRegistry` picks the least-loaded alive worker from its own heartbeat view (per-worker subject, 15 s TTL); on failure, `_request_with_fallback` re-dispatches to the queue-group subject where the NATS server picks any still-subscribed worker.

These two views of liveness disagree in exactly the failure modes the fallback is supposed to catch: a paused, network-partitioned, kernel-hung, or deadlocked worker still appears connected to NATS but cannot respond. In that state, the queue-group fallback can re-hand the request to the same unresponsive worker, hang for the full `_CLIENT_TIMEOUT_S`, and surface as `STTUnavailableError` / `TTSUnavailableError` while a healthy peer sits idle. PR #811's CI run reproduced this deterministically — local runs were merely lucky.

A second, symmetric gap: when a worker exits cleanly, NATS removes the subscription, the per-worker request raises `NoRespondersError`, and `_request_with_fallback` treats that as terminal (only `TimeoutError` triggers fallback). Clean death bypasses fallback entirely.

## Who

- **Primary:** The hub — every voice request path (`NatsSttClient.transcribe`, `NatsTtsClient.synthesize`) goes through this code. Failure = user-visible "voice unavailable" despite healthy workers.
- **Secondary:** #733 integration tests (blocked — assume the new semantics). Voicecli worker-side queue-group subscriptions (retired by a trailing cleanup ticket, not this one).

## Constraints

- Keep the per-worker request subject `lyra.voice.{stt,tts}.request.{worker_id}` — it is correct and already load-aware.
- Keep the `SUBJECTS.stt_request` / `SUBJECTS.tts_request` constants for backwards compatibility — worker-side subscriptions stay in voicecli until a later cleanup PR retires them.
- Registry eviction on failure must be idempotent across concurrent requests — multiple in-flight calls may observe the same bad worker simultaneously.
- No changes to heartbeat TTL (15 s) or to the voicecli worker side in this PR.
- Errors other than `TimeoutError` / `NoRespondersError` remain terminal for the current request (no silent swallowing).
- **Regression risk to accept:** if heartbeats stop flowing while workers remain TCP-connected (e.g., NATS subject permission change, broken heartbeat publisher), the current queue-group fallback would still route; the new design fails closed with `STTUnavailableError`. This is intentional — a silent routing path masking a broken heartbeat channel is worse than a loud failure — but must be documented in the ADR / inline comment so future debugging checks heartbeat flow first.

## Out of Scope

- Retiring worker-side queue-group subscriptions in voicecli (follow-up ticket).
- Changes to the heartbeat protocol, TTL, or `VoiceWorkerRegistry` schema.
- #733 integration tests themselves — this ticket unblocks them but does not deliver them.
- Any change to the request/response payload envelope (ADR-049 `ContractEnvelope` stays put).
