OpenClaw-RL

Conversation-to-Training Pipeline Architecture

A fully asynchronous RL framework that intercepts live multi-turn conversations via an OpenAI-compatible API, scores each turn using a Process Reward Model, and continuously optimizes the policy in the background — all without interrupting user interaction.

Component 1
OpenClaw Agent
User conversations via OpenAI-compatible /v1/chat/completions. Sends X-Session-Id and X-Turn-Type headers.
Component 2
API Proxy Server
FastAPI proxy. Classifies main vs side turns. Tokenizes, collects log-probs, triggers PRM. File: openclaw_api_server.py
Component 3
PRM Judge
Process Reward Model on separate GPUs. Majority voting (m=3). Scores: +1 / -1 / 0. Or extracts hindsight hints (OPD).
Component 4
SLIME Trainer
Megatron + SGLang. Receives samples via output_queue. GRPO loss or OPD distillation. Async weight updates.
API Proxy Server — Detail
Request Handling Pipeline
Main Turns (trainable)
  • Forward to SGLang with logprobs=True
  • Tokenize prompt + response via chat template
  • Extract per-token log-probabilities
  • Buffer as turn_data with session tracking
  • When next turn arrives, previous turn gets its "next state"
  • Fire async PRM scoring task
Side Turns (non-trainable)
  • Forward to SGLang normally
  • No tokenization, no log-probs collected
  • No training data produced
  • Used for tool calls, system prompts, side tasks
Session lifecycle: Each session tracked by session_id. Turn counter increments per main turn. When session_done=true, last turn is submitted (with loss_mask=0 unless at-least-one guarantee kicks in), then session state is cleaned up. Submission pauses during weight updates (HTTP 503) and resumes automatically.
PRM Judge — Scoring Mechanism
Binary RL Scoring
Prompt: "You are a PRM evaluating an AI assistant..."
  • Input: assistant response + next state (user reply or tool result)
  • \boxed{1} good — task progressed, user moves on, tool succeeds
  • \boxed{-1} bad — correction, redo, error, rephrase
  • \boxed{0} neutral — ambiguous, unrelated follow-up
  • m=3 independent votes, majority wins. Ties = 0.
OPD Hint Extraction
Prompt: "You are a PRM for hindsight hint extraction..."
  • Input: same (response + next state)
  • \boxed{1} + [HINT_START]...[HINT_END]
  • \boxed{-1} no useful hindsight, sample dropped
  • m=3 votes; select longest non-trivial hint among positive votes
  • Hint must be >10 chars, concrete, actionable
  • If no valid hint: sample is discarded entirely
Two Learning Paradigms

Path A Binary RL (GRPO)

1. Serve
Policy generates response + rollout log-probs
↓
2. Next State
Next user/env message arrives as natural feedback
↓
3. PRM Score
m=3 majority vote → scalar +1 / -1 / 0
↓
4. Advantage
GRPO: At = r for all response tokens (uniform broadcast)
↓
5. Loss
PPO-clip + KL
L = Lpg(clip ε=0.2, εhigh=0.28) + 0.02 · LKL

Path B On-Policy Distillation (OPD)

1. Serve
Policy generates response + rollout log-probs
↓
2. Next State
Next message arrives; judge extracts hindsight hint
↓
3. Enhanced Teacher
Hint appended to original prompt. Teacher = same model with hint-augmented context.
↓
4. Token-Level Advantage
At = log πteacher(at | s+hint) − log πθ(at | s)
↓
5. Loss
PPO-clip + KL (or Top-K reverse KL distillation)
Default: token-level PPO with teacher gap
Alt: DKL(πθK+1 || πteacherK+1) over top-K + tail bin
Key difference: Binary RL uses a scalar reward broadcast uniformly to all tokens. OPD uses a per-token directional signal derived from the teacher's log-probability gap. OPD is richer but requires a valid hint — samples without useful hints are dropped entirely, while Binary RL's at-least-one guarantee ensures every session contributes.
SLIME Trainer
Training Loop
File: train_async.py
  • rollout_manager.generate() drains output_queue
  • Waits for rollout_batch_size samples (default: 16)
  • Sorts by sample.index for deterministic ordering
  • actor_model.async_train() computes loss & updates
  • Periodic update_weights() syncs to SGLang
  • Checkpoints every save_interval steps
Sample Structure
  • tokens = prompt_ids + response_ids
  • response_length — token count of response
  • rollout_log_probs — from policy at serve time
  • loss_mask — [1]*n or [0]*n (exclude from loss)
  • reward — {"score": float} for Binary RL
  • teacher_log_probs — [T] tensor for OPD
  • teacher_topk_* — [T,K] tensors for Top-K OPD

Actor (Megatron)
Training with tensor-parallel=4, recompute-all, CPU optimizer offload. Holds the trainable policy weights.
Rollout (SGLang)
Serves the policy as OpenAI API. TP=2 per engine. Handles inference requests from the proxy during training.
PRM (SGLang)
Separate SGLang instance for the judge model. Runs m=3 parallel evaluations. Also computes teacher log-probs for OPD.
Async Loop — Weight Update Cycle
Continuous Training Cycle
PHASE 1
Serve & Collect
SGLang serves requests.
Proxy collects samples.
PRM scores async.
→
PHASE 2
Train Batch
Drain rollout_batch_size samples.
Compute loss, backprop.
Serving continues.
→
PHASE 3
Weight Sync
Pause submission (503).
Actor → SGLang sync.
Resume & purge records.
↺
Key invariant: The user never sees downtime. During Phase 2 (training), SGLang continues serving with the old weights. Only Phase 3 (weight sync) briefly pauses new sample submission — ongoing requests complete normally. The submission_enabled threading.Event controls this gate. Record files are purged on pause to prevent stale data.
