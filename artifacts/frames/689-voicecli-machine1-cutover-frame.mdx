## Problem

Slice S3 of #658 (voicecli NATS decouple). Lyra currently runs its own `lyra_stt` / `lyra_tts` satellite workers on Machine 1 alongside `lyra_hub`. The parent refactor moves voice workload into voicecli's own `nats-serve {stt,tts}` process so the Lyra hub only consumes voice via NATS topics. This slice is the production cutover: swap the supervisor programs, prove the round-trip, and keep a rollback path hot.

Observable impact: user voice notes must continue to transcribe and reply; any outage during swap is directly user-facing (Telegram/Discord voice loop).

## Who

- **Primary:** Mickael (ops) — executing the cutover, on the hook for rollback if smoke fails
- **Secondary:** voice-note users of Lyra (Telegram + Discord) — cannot tolerate extended voice-path outage

## Constraints

- **Blocked by** voiceCLI sibling: `voicecli nats-serve {stt,tts}` must be merged and deployable before cutover can run
- Must preserve existing NATS subject contract (`lyra.voice.{stt,tts}.*`) — hub code unchanged
- nkey provisioning decision pending: reuse `~/.lyra/nkeys/{stt,tts}-adapter.seed` OR provision fresh `~/.lyra/nkeys/voicecli-{stt,tts}.seed` (resolve during impl / PR review)
- GPU cold-start may exceed default `startsecs=10` — must re-tune if measured boot > 10s
- Rollback must be a single `supervisorctl` sequence — no code revert required

## Out of Scope

- voicecli-side implementation of `nats-serve` (sibling repo, separate PR)
- Deleting `lyra_stt` / `lyra_tts` satellite code (#658 S4 — blocked until this cutover lands in prod)
- Changing NATS subject schema or auth.conf beyond nkey provisioning
- Discord/Telegram adapter changes
- Production NATS server migration

**Tier: F-lite** — clear scope, single domain (supervisor/infra), ≤5 files, no new architecture. Parent spec (`artifacts/specs/658-voicecli-nats-decouple-spec.mdx`) covers the "why" and subject contract; this slice is execution + runbook.
