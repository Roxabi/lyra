Memory — File-Based, No Database

Philosophy: Pure filesystem storage at workspace/memory/ — no SQLite, no vector store, no embedding search. Grep-searchable by design.
Short-Term

Session Messages

  • Up to ~500 messages in context window
  • Append-only for LLM cache efficiency
  • Session tracks last_consolidated index
  • In-memory only, no persistence until consolidation
Long-Term

MEMORY.md + HISTORY.md

  • MEMORY.md — Markdown facts extracted by LLM during consolidation
  • HISTORY.md — Grep-searchable log with timestamps
  • Consolidation triggers when unconsolidated messages exceed threshold (default 100)
  • LLM-driven via save_memory() tool call

Bus — Minimal asyncio.Queue

MessageBus

Two Queue Architecture

  • Two asyncio.Queue instances: inbound + outbound
  • Single-process — no need for complex pub/sub
  • Channels don't wait for responses (non-blocking)
  • Per-session task tracking for cancellation

Built-in Tools

Core Capabilities

  • ReadFile, WriteFile, EditFile, ListDir
  • Exec — shell execution
  • WebSearch (Brave) + WebFetch
  • Message — send messages
  • Spawn — launch subagents

MCP Integration

External Tool Protocol

  • Supports stdio or HTTP/SSE transports
  • Auto-discovery of available tools
  • 30s timeout per tool call

LLM Providers — Registry-Driven

Metadata-Driven Provider Registry

Base class LLMProvider with abstract chat() method. Registry auto-detects the correct provider from API key prefixes, api_base URL matching, and environment variables. Zero manual configuration needed.

Other Notable Features

Subagent Manager

Isolated Background Tasks

  • Spawns background tasks with own event loop
  • Each subagent gets restricted ToolRegistry
  • No message/spawn/cron tools to prevent recursion
  • Isolated from parent agent scope
Heartbeat System

Periodic Health Checks

  • 30-minute interval execution
  • Reads HEARTBEAT.md checklist
  • LLM decides actions via virtual tool call
  • Notifies user only if findings exist
Session Manager

State Persistence

  • Key format: channel:chat_id
  • JSONL persistence format
  • History alignment to user turns
  • WeakRef dict for locks — auto-cleanup when sessions GC'd
Cron

Scheduled Tasks

  • croniter-based scheduling
  • Jobs stored in workspace/cron/
  • Pickled Job objects for persistence
Virtual Tool Calls

Structured Decisions

  • LLM makes structured decisions via tool call interface
  • Used for heartbeat actions and memory consolidation
  • Not free-text — enforces schema compliance
08

Lyra Comparison — Key Takeaways

Verdict: Nanobot proves that a personal AI agent can be remarkably effective with minimal infrastructure. These patterns are worth evaluating for Lyra's architecture.
ADOPT Recommended Patterns
  • Simple asyncio.Queue bus — Appropriate for single-process architectures. Two queues (inbound + outbound) is enough.
  • WeakRef locks for sessions — Auto-cleanup when sessions are garbage collected, no manual lifecycle management.
  • Virtual tool calls — Structured decisions via tool schema instead of free-text parsing. More reliable and auditable.
  • Provider registry with metadata detection — Zero-config LLM provider selection via API key prefix / URL matching.
STUDY Worth Investigating
  • File-based memory — Simpler than a database but limited in scale. Grep-searchable is elegant for small datasets.
  • Heartbeat with LLM-driven decisions — Novel approach: the LLM reads a checklist and decides what to act on, not hardcoded rules.
  • Subagent isolation — Restricted ToolRegistry per subagent prevents recursive spawning. Smart safety boundary.
