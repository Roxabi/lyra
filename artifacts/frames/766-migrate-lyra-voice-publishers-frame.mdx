## Problem

With #763 merged, `roxabi_contracts.voice` is now the canonical source of truth for the Tts/Stt wire contract — models (`TtsRequest`, `TtsResponse`, `SttRequest`, `SttResponse`), `SUBJECTS` namespace, and fixtures live in one typed, tested package. But on the lyra side, `src/lyra/nats/nats_tts_client.py` and `src/lyra/nats/nats_stt_client.py` still carry **inline dataclass definitions** and **hard-coded subject string literals** that were reconciled against voiceCLI during the port. Until these two files switch to importing from the package, the drift loop ADR-049 targets is still open: any future edit to the contracts package can silently diverge from the lyra runtime, and CI has no way to catch it.

This issue closes the lyra half of the loop (Cohort B). It is strictly a port: no wire changes, no redesign — replace local defs with imports, add the workspace source, regenerate the lockfile. The voiceCLI side (Cohort A, `tts_adapter.py` / stt equivalent) ports in a parallel but separate issue under the same epic.

## Who

- **Primary:** lyra hub maintainer wiring the voice clients to the new shared schema — responsible for the migration and for catching any last-mile drift between the inline defs and `roxabi_contracts.voice`.
- **Secondary:** reviewers auditing that no behavioral change leaks through (same subjects, same serialized shape, pyright clean); future contracts-domain ports (image, memory, llm) that use this migration as the template.

## Constraints

- Import-only migration: the wire contract MUST NOT change. Any field/subject mismatch surfaced during the port is a drift bug — document and reconcile explicitly (matches what #763 did on the contracts side).
- Workspace source wiring: root `pyproject.toml` `[tool.uv.sources]` must declare `roxabi-contracts = { workspace = true, extras = ["testing"] }`. Pinning via git tag is out of scope for this issue — workspace is the local-dev contract until the package is first tagged and published.
- `uv.lock` must be regenerated in the same PR so CI is reproducible.
- No inline `Tts*`/`Stt*` dataclasses or subject f-strings may remain anywhere under `src/lyra/nats/`. Grep-clean is an acceptance gate.
- `uv run pyright` must pass with the new imports; existing lyra tests must continue to pass without modification.
- Blockers #763 (voice port) and #765 (CONTRACT_VERSION in contracts.envelope + compat shim) are both closed.

## Out of Scope

- voiceCLI Cohort A migration (`tts_adapter.py` and the stt equivalent) — tracked separately under Epic #761.
- Behavioral changes to the NATS clients: connection handling, retry logic, circuit breaker, readiness — untouched.
- Tagging / publishing `roxabi-contracts` v0.x — happens after all four domains (voice, image, memory, llm) land.
- Adding new fields or new subjects to the voice contract.
- Refactoring `src/lyra/nats/` module layout beyond what the import swap requires.

## Complexity

**Tier: F-lite** — Clear scope, single domain (lyra NATS client layer), no new architecture, 4 files touched (`nats_tts_client.py`, `nats_stt_client.py`, root `pyproject.toml`, `uv.lock`). ADR-049 already decided the shape; this is execution.

Signals observed:
- Single domain: `src/lyra/nats/` + root project config — no cross-package runtime change on the lyra side.
- Port-only: contracts source exists and is tested (#763 closed).
- No unknowns: lines to delete are enumerable; the import-replacement is mechanical.
- One judgment call: any residual drift between lyra's inline defs and `roxabi_contracts.voice` must be called out in the PR so resolution is auditable.
- Dependencies resolved: #763, #765 both closed.
