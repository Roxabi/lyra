🎤 Full Voice Loop
Send a voice message → Whisper transcribes → Claude responds → TTS replies with audio. Full voice conversations on Telegram and Discord voice channels.
🔗 Paste a Link, Save Knowledge
Just paste a URL. Lyra auto-saves it to your personal knowledge vault with title and tags. Search later with /search. Build your second brain via chat.

Troubleshooting
🐍 Python version too old
python --version
# Need 3.12+. Install via pyenv or deadsnakes PPA.
Lyra uses modern Python features (StrEnum, tomllib). Older versions will fail at import.
🤖 Claude CLI not authenticated
claude --version
claude   # starts interactive — if it works, you're authenticated
Claude CLI needs an active subscription. Visit claude.ai/download to install and authenticate.
🔌 Port 8443 already in use
# Change the health port:
LYRA_HEALTH_PORT=9443 lyra start
# Or in supervisor config: environment=LYRA_HEALTH_PORT="9443"
Default health port is 8443. If another service uses it, override via env var.
🔑 Bot token invalid or missing
lyra bot list              # check stored credentials
lyra bot add --platform telegram --bot-id lyra  # re-add
Tokens are encrypted in auth.db. If the bot fails to start, re-add the token.
📄 config.toml errors
lyra config validate   # schema check
lyra config show       # dump resolved config
Common issues: missing owner_users, wrong TOML syntax. Validate before starting.
📦 uv sync fails
pip install --upgrade uv
uv sync --refresh
Ensure uv is up to date. Use --refresh to clear the cache if dependencies fail to resolve.

🚀 lyra-stack — recommended for production

✦ lyra-stack

A single supervisord instance manages all Roxabi daemons (lyra, voicecli_tts, voicecli_stt…). Each project repo owns its supervisor configs; lyra-stack/conf.d/ contains only symlinks. Recommended for Machine 1 (24/7 hub).
bash
# Clone alongside lyra
git clone https://github.com/Roxabi/lyra-stack ~/projects/lyra-stack

# One-time setup: register lyra symlinks + start supervisord
cd ~/projects/lyra
make register      # registers lyra conf symlinks into lyra-stack

cd ~/projects/lyra-stack
make start         # start supervisord (idempotent)
make ps            # status of all programs

# Lyra-specific shortcuts
make lyra          # start lyra
make lyra reload   # restart
make lyra logs     # tail stdout
make lyra errors   # tail stderr
make lyra status   # status

# Or from inside lyra repo:
cd ~/projects/lyra
make lyra          # delegates to lyra-stack
make deploy        # git pull → pytest → supervisorctl restart on Machine 1
  • Adapters run as independent processes — telegram can restart without touching discord
  • Machine 1 remote control: make remote status, make remote logs, make remote reload
  • make deploy runs git pull + pytest + restart on Machine 1 in one command

Under the hood — message flow
flow
1. Adapter receives message → normalizes to InboundMessage(platform, bot_id, scope_id, text, …)
2. Enqueued on per-platform asyncio Queue (bounded=100, immediate ACK on full)
3. InboundBus staging queue → Hub.run_loop()
4. Hub: resolve_binding(platform, bot_id, scope_id) → agent="lyra_default"
5. Pool created/reused for this scope — one Pool = one isolated conversation
6a. text starts with "/" → CommandRouter.dispatch() → handler → response  (zero tokens!)
6b. normal text → agent.process(msg, pool) → LLM (claude subprocess)
7. OutboundDispatcher: verify routing → deliver reply to originating channel only
Scope isolation
RoutingKey format
# (platform, bot_id, scope_id) → (agent, pool_id)
telegram · lyra · chat:555   → pool telegram:lyra:chat:555
discord  · lyra · thread:888 → pool discord:lyra:thread:888
Two different bots talking to the same user in the same channel have completely isolated history.
Scope ID extraction
TG DM/groupchat:{chat_id}
TG forum topicchat:{id}:topic:{id}
Discord threadthread:{thread_id}
Discord channelchannel:{channel_id}
Agents
Create, edit, and manage agents. Agents live in SQLite (~/.lyra/auth.db) — TOML files are seed sources only.
💾
AgentStore (v4): Agent config is now stored in SQLite (~/.lyra/auth.db). TOML files are seed sources — edit them, then run lyra agent init to import into DB. Runtime reads from DB only.
Create / update an agent
1
Scaffold a new agent TOML
lyra agent create my_agent   # interactive wizard → writes TOML
📁
User overrides: Place a file at ~/.lyra/agents/my_agent.toml to override the system default without modifying the repo. User-level files take precedence during seed.
2
Edit the TOML
tomlsrc/lyra/agents/my_agent.toml
[agent]
name              = "my_agent"
memory_namespace  = "my_agent"      # isolates memory from other agents
persona           = "lyra_default"  # system prompt from ~/.roxabi-vault
show_intermediate = true            # true → show ⏳ intermediate turns to user
permissions       = []

[model]
backend   = "claude-cli"            # "claude-cli" | "anthropic-sdk"
model     = "claude-sonnet-4-6"
max_turns = 10
tools     = ["Read", "Grep", "Glob", "WebFetch", "WebSearch"]
# Bash + Write omitted — prevents prompt injection → RCE
skip_permissions = true            # skip Claude CLI permission prompts

[agent.smart_routing]               # requires backend = "anthropic-sdk"
enabled = false

[agent.smart_routing.models]
trivial  = "claude-haiku-4-5-20251001"
simple   = "claude-haiku-4-5-20251001"
moderate = "claude-sonnet-4-6"
complex  = "claude-opus-4-6"

[plugins]
enabled = ["echo", "search"]        # search enables /search + /add + /explain + /summarize

[workspaces]                         # each key → / slash command
lyra     = "~/projects/lyra"
projects = "~/projects"

# Voice config (all optional — omit to inherit voicecli globals)
[tts]
engine   = "qwen-fast"   # qwen | qwen-fast | chatterbox | chatterbox-turbo
voice    = "Ono_Anna"
language = "French"      # overridden by user pref if "detected"
chunked  = true

[stt]
language_detection_threshold = 0.95
language_fallback            = "en"
3

lyra agent init          # seed DB from TOML files (skip if exists)
lyra agent init --force  # overwrite existing DB rows with TOML

Voice

STT — Speech to text (optional)
✅
STT is optional. Omit STT_MODEL_SIZE from .env → STT is disabled and Lyra starts normally. Set it to enable STT. The .env.example ships small for low-VRAM setups; use large-v3-turbo for best accuracy.
bash.env
# Omit entirely to disable STT
STT_MODEL_SIZE=small  # .env.example ships small; use large-v3-turbo for best accuracy
STT_DEVICE=auto               # cpu | cuda | auto
STT_COMPUTE_TYPE=auto         # float16 | int8 | auto
📚
Personal vocab: Drop voicecli.vocab at ~/.voicecli/voicecli.vocab (one word/phrase per line). Loaded automatically — shared with the voicecli dictate daemon.

tiny~0.2GBFastest, low accuracy
small~0.5GBFast, decent accuracy
medium~1.5GBGood balance
large-v3-turbo~3.0GB⭐ Recommended
large-v3~3.0GBMost accurate, slowest

STT transcription echo
Transcription feedback
When STT is enabled and a user sends a voice message, Lyra echoes the transcription back to the user before processing it through the agent. This gives immediate feedback that the voice was understood correctly.
chat
You:  🎤 [voice message]
Bot:  📝 "What time is the meeting tomorrow?"   ← STT echo
  ↓ agent processes transcribed text
Bot:  🔊 "The meeting is at 3pm..."                ← TTS reply
STT noise filtering
Whisper noise tokens (auto-discarded)
Transcriptions matching these tokens (or empty strings) are treated as noise. Lyra replies with stt_noise and does not forward to the agent.
pythonstt/__init__.py
WHISPER_NOISE_TOKENS = {"[music]", "[applause]", "[laughter]", "[silence]", "[noise]"}

discordguild messages only (no DMs)

Architecture
Deep technical blueprint — components, queues, domains, data types, dispatch, and refactoring plan.
💡
Do LLM responses flow through InboundBus?
No. The InboundBus is strictly one-way: Adapter → per-platform Queue → staging Queue → Hub.run(). LLM responses take a completely separate return path: Agent → Pool → Hub.dispatch_response() → OutboundDispatcher → Adapter.send(). Each adapter has its own inbound queue AND its own OutboundDispatcher — they never share a bus.
Text message lifecycle
InboundMessage → RoutingKey → Binding → Pool → Agent.process() → Response | AsyncIterator → OutboundMessage

Complete message flow
18 steps from platform event to delivered response — inbound and outbound are completely separate.
Inbound Path (Adapter → Hub → Pool → Agent)
  1. 1 Platform event (Telegram update / Discord on_message)
  2. 2 Adapter normalizes → InboundMessage (with RoutingContext)
  3. 3 push_to_hub_guarded() — circuit breaker check before enqueue
  4. 4 inbound_bus.put(platform, msg) → per-platform bounded queue (100)
  5. 5 _feeder() task drains into staging queue (bounded 500)
  6. 6 Hub.run() awaits inbound_bus.get() — single consumer
  7. 7 MessagePipeline.process(msg): validate → rate limit → resolve binding → check commands
  8. 8 get_or_create_pool() via PoolManager
  9. 9 pool.submit(msg) → debounce → merge
LLM + Outbound (Agent → Hub → Dispatcher → Adapter)
  1. 10 _process_with_cancel() — cancel-in-flight support
  2. 11 _guarded_process_one() — timeout + event emission
  3. 12 agent.process(msg, pool) — Claude SDK or CLI subprocess
  4. 13 Response returned: str or AsyncIterator[str]
  5. 14 hub.dispatch_response() or hub.dispatch_streaming()
  6. 15 OutboundDispatcher — fire-and-forget enqueue
  7. 16 _worker_loop(): verify routing → circuit breaker → retry (1s/2s/4s)
  8. 17 adapter.send() or adapter.send_streaming()
  9. 18 Platform API delivers to user ✓
🎤
Separate Audio Path — InboundAudioBus is a completely separate bus. Voice messages route through inbound_audio_bus → Hub._audio_loop() → STT → re-inject as text. Audio responses: dispatch_audio_stream() → OutboundDispatcher → adapter.render_audio_stream().
Flow as pipeline
TG/DC Event→ normalize()→ InboundMessage→ push_to_hub→ InboundBus→ Hub.run()→ Pipeline→ Pool.submit()→ Agent.process()→ LLM→ dispatch_response()→ OutboundDispatcher→ adapter.send()→ ✓ Delivered
Three queue systems
Lyra uses three independent queuing systems that never share state.
InboundBus
Direction: Adapter → Hub
Structure: One queue per platform → staging
Bounded: 100 items (backpressure)
Consumer: hub.run()
File: core/inbound_bus.py
InboundAudioBus
Direction: Adapter → Hub (audio)
Structure: Separate voice queue
Consumer: hub._audio_loop()
Processing: STT → text → InboundBus
File: core/inbound_audio_bus.py
OutboundDispatcher
Direction: Hub → Adapter
Structure: One per (platform, bot_id)
Features: Retry (1s/2s/4s) + CB
Consumer: _worker_loop()
File: core/outbound_dispatcher.py

Scope model
📌
Scope != User — Pools are per conversation scope.
Pools are keyed by (platform, bot_id, scope_id). Scope is extracted by the adapter:
Telegram DM/group → chat:{chat_id}, forum → chat:{chat_id}:topic:{topic_id}.
Discord → channel:{channel_id} or thread:{thread_id}.
Two users in the same group share one Pool. Two bots in the same channel each have separate Pools.

Key insights
Two Completely Separate Paths
InboundBus (adapters → Hub) and OutboundDispatcher (Hub → adapters) are independent queuing systems. They share no state, no queue, no bus. LLM responses never touch InboundBus.
Per-Scope, Not Per-User
Pools are keyed by (platform, bot_id, scope_id). In a group chat, all users share one Pool. Two bots in the same channel have separate Pools.
Dispatcher Owns Circuit Breaker
The OutboundDispatcher is the sole owner of the platform circuit breaker. Adapters never check CB state — the dispatcher handles it with exponential backoff.
Stateless Agent, Stateful Pool
Agents are immutable singletons — same instance serves all Pools. All mutable state (history, session, debouncer) lives in the Pool.
Hub Delegates to Focused Modules
Hub has been decomposed: MessagePipeline (validate/route), AudioPipeline (STT/TTS), PoolManager (lifecycle/eviction), PoolProcessor + PoolObserver (extracted from Pool). Hub orchestrates; each module owns its domain.
Audio Has Its Own Everything
Separate InboundAudioBus, separate AudioPipeline (extracted from Hub), separate dispatch methods. Fully isolated audio path from inbound to outbound.

Command Routing
How messages dispatch from raw platform event through 6 ordered layers to handler, LLM, or error — and why ! behaves differently from /.
The two prefixes
/
Strict mode
Hard command — / prefix
Known command → dispatched to registered handler
Unknown command → returns "Unknown command: /foo" error immediately
LLM never sees an unrecognised / command
!
Soft mode
Fallthrough allowed — ! prefix
Known command → dispatched identically to / — same registry, same handler
Unknown command → router returns None → hub falls through to LLM pool
Only error behavior differs — same command set, same dispatch order
