## Source

> Four folders in `src/lyra/` exceed the 12-file cap defined in `.claude/stack.yml` quality gates:
> - `src/lyra/core/` — 53 files
> - `src/lyra/adapters/` — 30 files
> - `src/lyra/bootstrap/` — 25 files
> - `src/lyra/core/hub/` — 28 files
>
> V4 was deferred due to higher blast radius (~150+ import-path updates across `src/` and `tests/`).

— Issue #773, follow-up to #760

## Problem

Four folders exceed the 12-file quality gate cap, making navigation cognitively expensive and violating architectural hygiene. The exemptions are tracked in `tools/folder_exemptions.txt`.

**Current state (verified):**

| Folder | Files | Subdirs | Status |
|--------|-------|---------|--------|
| `src/lyra/core/` | 53 | `commands/`, `hub/`, `pool/`, `processors/`, `stores/` | ❌ flat files exceed cap |
| `src/lyra/adapters/` | 30 | none | ❌ no decomposition |
| `src/lyra/bootstrap/` | 25 | none | ❌ no decomposition |
| `src/lyra/core/hub/` | 28 | none | ❌ exceeds cap |

**Precedent:** #395 successfully decomposed `core/` into `stores/`, `hub/`, `pool/`, `commands/` using a scripted approach with AST-aware import updates. That work reduced flat files from 68 → 41. #773 continues this pattern for the remaining oversized folders.

## Outcome

**Target:** `core/` (flat), `adapters/`, `bootstrap/` contain ≤12 files per folder after decomposition. `tools/folder_exemptions.txt` reduced to `core/hub/` only (deferred).

**Deferred:** `core/hub/` (28 files) requires deeper analysis — the 14 hub-core files form a cohesive unit around Hub coordination. Revisit after other folders land.

## Appetite

**3 days, 4–5 commits** (one per major folder). Each commit = one decomposition slice with all import updates, verified green on pyright + pytest + lint-imports.

## Shapes

### Shape 1: Per-folder scripted decomposition (frame's proposed approach)

Split each oversized folder into cohesive subfolders as specified. Use AST-based import rewriting tool. One folder per commit.

**Decomposition mapping (draft — requires refinement):**

| Folder | Target subfolders | Files per subfolder | Notes |
|--------|-------------------|---------------------|-------|
| `core/` (52 flat) | `agent/` (11), `cli/` (9), `memory/` (5), `messaging/` (5), `auth/` (3) | ≤11 each | **19 orphan files** (see below) |
| `adapters/` (30) | `discord/` (10), `telegram/` (6), `nats/` (3), `shared/` (10) | ≤10 each | Clean split |
| `bootstrap/` (25) | `standalone/` (5), `wiring/` (2), `lifecycle/` (3), `stores/` (2), `infra/` (4), `factory/` (8) | ≤8 each | Clean split |
| `core/hub/` (28) | `middleware/` (4), `outbound/` (6), `pipeline/` (3) | ≤6 each | **14 hub-core files remain** (see below) |

**Core/ orphan files (19):** audio_payload, builtin_commands, circuit_breaker, debouncer, error_utils, guard, persona, processor_registry, render_events, runtime_config, scope, session_lifecycle, smart_routing_protocol, stream_processor, tool_display_config, tool_recap_format, trace, tts_dispatch, workspace_commands

Resolution options:
1. Refine mapping in spec phase (add shared/, audio/, processors/ additions)
2. Move some orphans to existing subdirs (guard → auth/, persona → agent/, processor_registry → processors/)
3. Accept limited flat files remain (≤19, still over cap)

**Core/hub/ remaining (14 files):** hub.py, hub_dispatch.py, hub_protocol.py, hub_registration.py, hub_shutdown.py, hub_rate_limit.py, hub_circuit_breaker.py, hub_pool_delegation.py, identity_resolver.py, path_validation.py, pool_manager.py, event_bus.py, audit_consumer.py, _dispatch.py

**Trade-offs:**
- Pro: Incremental, low risk per commit, easy to roll back
- Pro: Follows precedent from #395
- Con: `core/hub/` decomposition leaves 14 files in hub-core (still over cap)
- Con: Requires AST-based import rewriter (may need to build/borrow tool)

**Rough scope:** L (3 folders × ~30-50 import sites each = ~120 total, hub-core deferred)

### Shape 2: Merge-then-decompose (atomic PR)

Create all subfolders on a feature branch. Do all decompositions in one atomic PR.

**Trade-offs:**
- Pro: Single review cycle
- Con: High blast radius — one bad import breaks everything
- Con: Harder to bisect if regressions appear later
- Con: No incremental validation

**Rough scope:** XL

### Shape 3: Hybrid incremental with tooling validation

Start with `core/` (largest, most complex). Validate the AST tooling works. Then apply to `adapters/`, `bootstrap/`, `core/hub/` in subsequent slices.

**Trade-offs:**
- Pro: Validates tooling on highest-value target first
- Pro: Can adjust approach based on learnings
- Con: Slightly longer overall (tooling refinement between slices)
- Con: Still has hub-core 14-file problem

**Rough scope:** L

## Fit Check

**Shape 1 is recommended** — it matches the frame's constraints and follows the successful precedent from #395.

**Shape 2 eliminated** — the frame explicitly requires "one-subfolder-at-a-time to minimize blast radius per commit". An atomic PR violates this constraint.

**Shape 3 not needed** — Shape 1 already validates incrementally. The AST tooling approach is well-documented in #395 plan.

**Two open issues require spec-phase resolution:**

1. **core/ orphan files (19):** The proposed subfolders (agent/, cli/, memory/, messaging/, auth/) don't cover all 52 flat files. Resolution options:
   - Refine mapping: add shared/, audio/, move some to existing processors/
   - Accept limited flat remainders (but this violates cap)
   - Recommendation: resolve in spec phase with full file → subdir mapping

2. **core/hub/ deferred:** Decomposition leaves 14 hub-core files (exceeds 12-file cap). Recommendation: defer hub decomposition, land other 3 folders first, create follow-up issue for hub/.

## Risks

1. **Circular imports during intermediate state** — Mitigated by moving one subfolder at a time, verifying pyright after each.
2. **Relative import depth errors** — AST-based rewriter catches these; manual review still required.
3. **Layer constraint violations** — `.importlinter` config may need updates for new subfolder paths.
4. **Test drift** — Tests must be updated in lockstep with source moves.
