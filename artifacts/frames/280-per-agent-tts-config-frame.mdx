## Problem

`TTSService` is instantiated **once globally** at startup, using only the config of the first agent (alphabetically) that has a `[tts]` section. Every other agent's TTS configuration is silently ignored — a warning is logged, but the values are discarded.

The practical result: all agents share one voice, one engine, one set of parameters. Per-agent voice identity is impossible despite the underlying data model (DB column, dataclass, TOML parsing, overlay helper) already being fully in place.

The gap is purely at the call site: nothing wires the per-agent config into `TTSService.generate()` at request time, and the startup logic actively discards all but one config.

## Who

- **Primary:** Mickael (developer/operator) — configuring distinct voice identities for Lyra agents (e.g. Vivian in French vs. a different agent in English).
- **Secondary:** End users interacting with Lyra via Telegram/Discord — they experience the voice directly.

## Constraints

- `TTSService` must remain backward-compatible: a global default config still applies when no per-agent override is present.
- `AgentTTSConfig` already exists in the DB — no schema migration needed beyond adding `exaggeration` and `cfg_weight` fields.
- `lyra agent edit` interactive editing must work within the existing CLI structure.
- No new TTS engines or external services are introduced — this is purely config plumbing.

## Out of Scope

- Hot-reload of TTS config at runtime (without restart).
- Platform-specific TTS routing (Telegram vs. Discord get the same per-agent config).
- New TTS engine integrations beyond what voiceCLI already supports.
- Web or GUI interface for editing agent TTS config.
- Bulk migration / validation tooling for existing agent TOML files.

## Complexity

**Tier: F-lite** — Clear scope, single TTS domain, no new architectural patterns. All 5 tasks are mechanical wiring of already-existing infrastructure. Risk is low; main care is not breaking the global fallback path.
