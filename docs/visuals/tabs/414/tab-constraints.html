Key Constraints
DB = Single Source of Truth
Every personality and voice field is loaded from the DB. No fallback chain (TOML → DB → default). DB has it or it doesn't.
One-Shot Migration
Existing tts_json data is split once. No dual-read compatibility layer, no gradual rollout.
TOML + DB Restructured
Both agent TOML files and DB schema updated to reflect the taxonomy. Seeder and loader updated accordingly.
Spec Decision: The Taxonomy
Classifying each field as core, LLM-only, or voice-only is the key design decision. DB schema follows.
Industry Takeaways for Lyra
What We Learn From External Analysis
  • Lyra's "core" category is novel. No major platform shares personality fields between LLM and TTS — they all require manual duplication. This is a genuine innovation.
  • OpenAI's dual-instructions model exists for a reason. What shapes good text (formality, vocabulary) and good speech (pacing, filler words) don't always overlap 1:1. The spec should validate each "core" field for both consumers.
  • Industry consensus: personality = text, voice = technical. Every platform treats personality as narrative text in system prompts, and voice as technical sliders/IDs. Lyra's structured personality fields (French descriptors) are more expressive than most.
  • LiveKit's Agent/AgentSession split is the closest architectural parallel. Same agent personality reusable across different voice configs — similar to what Lyra achieves at the data layer.
  • Risk: "speed" and "accent" are borderline. OpenAI and ElevenLabs treat pacing/accent as voice delivery params, not personality. The spec should decide if these are truly "core" or voice-only fields that also hint to the LLM.
  • No platform does fallback chains for personality. Validates the "DB is sole source" constraint — personality should be explicit, not derived from cascading defaults.
