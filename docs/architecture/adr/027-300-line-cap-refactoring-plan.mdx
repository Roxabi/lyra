## Status

Accepted

## Context

Epic #293 mandates that no source file in `src/lyra/` exceeds 300 lines. After the
first pass (issues #294–#318), 12 files remain over the cap. Some were outside the
original scope; others closed as done but regrew. This ADR provides the authoritative
decomposition plan so implementers can work in parallel without stepping on each other.

Constraint: extracted modules must also be ≤300 lines each. `__init__.py` re-exports
must be preserved. Zero behavioural changes are permitted.

## Risk Assessment

### High risk

**Batch D (hub dispatch extraction):** `Hub` is the central dependency. All adapters, all agents, all pools, and all tests that call `dispatch_*` go through `Hub`. Delegating wrappers preserve the interface, but any mistake in the wrapper signature (missed `await`, wrong argument order) produces a runtime error that may only appear under load. Mitigation: run the full test suite after each extracted method, not after the whole batch.

**Batch A (bootstrap):** `_resolve_bot_agent_map` is called during startup. An import error kills the process before any logs are emitted. Mitigation: add a smoke-test import assertion in the test suite.

### Medium risk

**Batch F (adapters shared split):** Several adapter files import from `_shared.py`. If the re-export approach is not used, every importing file must be updated simultaneously. A partial update produces `ImportError` at runtime. Mitigation: use the re-export approach — keep all symbols importable from `_shared.py` via `from ._shared_audio import *` style re-exports.

**Batch E (cli_pool worker):** The `_spawn` function holds the subprocess creation logic. Any argument mislabelling breaks the CLI backend silently (pool returns `CliResult(error=...)` rather than raising). Mitigation: ensure `tests/` cover the spawn path (mocked subprocess).

### Low risk

**Batches B, C, G:** Small extractions, narrow import surfaces, no protocol changes.

**Batch H (trimming):** Pure cosmetic, zero behavioural risk. The only failure mode is an off-by-one in line counting.

### Cross-cutting risks

- **`__init__.py` re-export preservation:** `core/__init__.py` imports `Action`, `MessagePipeline`, `PipelineResult`, `RoutingKey` via `hub.py`. These must remain importable from `lyra.core` after extraction. Since `hub.py` already re-exports them via `# noqa: F401`, the pattern is established — new extracted modules follow the same pattern.
- **Circular imports:** `hub_dispatch.py` must import `Hub` for type annotations but `hub.py` would import `hub_dispatch`. Use `TYPE_CHECKING` guard to break the cycle: runtime imports only the dispatch functions, type annotations are string-form or behind `TYPE_CHECKING`.
- **`cli_pool_worker.py` circular reference:** same solution — `CliPool` is only needed for type annotations in worker functions.

## Options Considered

### Option A: Extract new modules aggressively (this plan)

Split by concern, creating new modules for each extracted group. Minimises per-file line count, maximises single-responsibility.

- **Pros:** Definitively solves the cap. Each module has a clear, named responsibility. Easier to test in isolation.
- **Cons:** More files to navigate; import graph grows. Each extraction carries refactoring risk.

### Option B: Formatting cleanup only

Rely solely on removing blank lines, collapsing log strings, and inlining short lambdas.

- **Pros:** Zero semantic risk. No new files.
- **Cons:** Not viable for files that are 30+ lines over (hub.py at 232 over, agent_factory.py at 74 over, etc.). Only viable for the trimming candidates.

### Option C: Raise the limit for select files

Exempt hub.py and agent_factory.py from the cap as architectural exceptions.

- **Pros:** No refactoring work.
- **Cons:** Undermines the epic's stated goal. These files are also the ones most likely to continue growing. Rejected.

## Decision

Option A for files more than 20 lines over the cap. Option B (trimming) attempted first for files 25 lines or fewer over the cap, with Option A as fallback if trimming is insufficient.

## Consequences

### Positive

- All 12 files brought within the 300-line cap.
- Each new extracted module is named by its single responsibility (`hub_dispatch`, `memory_upserts`, `pairing_config`, etc.), improving discoverability.
- Parallel execution (Batches B–H) allows the work to be distributed across developers with no coordination overhead beyond Batch A.

### Negative

- The `lyra.core` and `lyra.adapters` namespaces gain 6–8 new internal modules.
- Developers must learn the new module boundaries when adding features.

### Neutral

- All `__init__.py` re-exports are preserved. External callers (`from lyra.core import Hub`, `from lyra.core.memory import MemoryManager`) continue to work without change.
- The `# noqa: F401` re-export pattern already in use in `hub.py` is extended to new extraction sites.
