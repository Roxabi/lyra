Architectural Assessment

What to fix, clarify, drop, and create for a clean hexagonal architecture

CRITICAL — delete/remove, no value HIGH — causes real bugs or durability loss MEDIUM — architectural smell, future pain LOW — nice to have, no immediate impact FIXED — already resolved
#1 — agents.db: dead file, wastes file handles
CRITICAL
~/.lyra/agents.db exists on disk but has zero tables. Agent configs were migrated to auth.db → agents. The file is opened by bootstrap but never used. Every startup wastes a file descriptor and creates misleading file presence in the DB directory.
Delete the file. Audit bootstrap for any agents.db connection attempt and remove it. No migration needed — no data lives there.
#2 — auth.db carries 5 distinct domains
HIGH
A single SQLite file owns: auth grants (AuthStore), agent configs (AgentStore), bot tokens (CredentialStore), Discord thread session pointers (ThreadStore), and user TTS preferences (PrefsStore). This violates hexagonal architecture — the auth domain has nothing to do with agent configuration or platform-specific state. It creates implicit coupling: any schema migration must consider all 5 stores touching the same file.
Split into: auth.db (grants only) · config.db (agents + bot_agent_map + bot_secrets + agent_runtime_state) · platform.db (discord_threads + user_prefs). Requires store connection rewiring and a one-time migration script.
#3 — paired_sessions: deprecated table still in DDL
HIGH
pairing.db still creates the paired_sessions table on every connect() even though it was deprecated in issue #245. Grants now go directly to auth.db → grants. The table is always empty but its presence in the DDL misleads anyone reading pairing.py about how the system works.
Remove paired_sessions from DDL in pairing.py. Add a one-time DROP TABLE migration to clean existing files. Update any code that still references it.
#4 — _resume_session_ids: in-memory, lost on restart
HIGH
The dict _resume_session_ids[pool_id] in CliPoolWorkerMixin is the only thing that gives _spawn() its --resume argument. If Lyra restarts between turns (crash, redeploy, supervisord reload), this dict is empty. Recovery falls to Path 3 (TurnStore) + _session_file_exists() — but Path 3 is blocked by finding #5 (file not on disk while process was alive). Result: restart after any in-progress conversation causes silent context loss.
Add a pool_state(pool_id TEXT PK, session_id TEXT, updated_at TEXT) table to turns.db. Write session_id there at the same time as _resume_session_ids. On startup, repopulate _resume_session_ids from pool_state rows that have matching .jsonl files.
#5 — _session_file_exists() timing: stream-json mode doesn't flush .jsonl while alive
HIGH
Claude CLI in --input-format stream-json mode writes the session .jsonl file only when the subprocess exits. So _session_file_exists() — which globs ~/.claude/projects/*/{session_id}.jsonl — always returns False for the current active session (while it's alive). This causes Path 2 (ThreadStore) and Path 3 (TurnStore) to wrongly conclude the session is unavailable, even when the subprocess is running and context is intact. Root cause of the session resume failures documented in issue #415.
In resume_and_reset(): before calling _session_file_exists(), check if the pool's entry is alive for that session_id. If alive, skip the file check entirely — the subprocess has the context in memory. Only check the file for dead/old sessions where the subprocess already exited.
#6 — Path 3 guard: returned FRESH instead of SKIPPED (Bug #415)
✅ FIXED
When last_sid == pool.session_id in Path 3, the code fell through to return FRESH if path2_attempted else SKIPPED. Since Path 2 had been attempted (and failed due to finding #5), this returned FRESH → sent a spurious ⚠️ "Couldn't resume" warning to the user, even though the pool was already on the correct session. The actual spawn still used the right session via _resume_session_ids.
Added explicit return ResumeStatus.SKIPPED after the guard's debug log in message_pipeline.py line ~364.
#7 — Sessions are implicit — no sessions table
MEDIUM
A "Lyra session" is not a first-class entity. It's a session_id UUID on turn rows in conversation_turns. To know the current session for a pool, you must query get_last_session(pool_id) — an index scan. To know how many sessions a pool has had, you SELECT DISTINCT session_id. This makes resume logic more complex and harder to reason about. It also makes the accidental session-sharing bug (#415) invisible in the data — you need domain knowledge to notice that a pool is using a session_id that belongs to a different user's pool.
Add pool_sessions(pool_id TEXT, session_id TEXT, started_at TEXT, last_active_at TEXT, is_current BOOL) to turns.db. Write a row when a new session starts. Update is_current on session change. This makes session history explicit, queryable, and debuggable.
#8 — ThreadStore (discord_threads) inside auth.db: platform state in the auth domain
MEDIUM
discord_threads is Discord-specific platform state. In hexagonal architecture, platform adapters own their own state — the core should not have platform-specific tables. Moving this to auth.db made sense for convenience (one DB file), but it couples the core persistence layer to a specific adapter's schema. If a new adapter needs similar thread tracking, it would add more platform tables to auth.db, compounding the problem.
As part of finding #2 split: move discord_threads to a discord.db owned by the Discord adapter. The adapter passes session_id to the Hub via platform_meta, not by having the Hub query its private tables directly.
#9 — Memory vault has no FK to session_id in turns.db
LOW
When MemoryManager.upsert_session(SessionSnapshot) writes a session summary to the vault, it's keyed by namespace + user_id only — not session_id. There's no way to trace which vault memory came from which Claude session, or to correlate vault entries with their corresponding rows in turns.db. Makes debugging memory correctness hard.
Include session_id in memory upserts as metadata (e.g. source_session_id field). No schema change needed if vault entries support arbitrary metadata. Enables cross-store debugging.
#10 — message_index.db: no automatic pruning on startup
LOW
MessageIndex.cleanup_older_than(days) exists but is never called automatically. The message_index will grow unboundedly. Platform message IDs from months ago serve no resume use (sessions expire from Claude's side well before that).
Call cleanup_older_than(90) during Hub.startup() — or add a periodic maintenance task. Make retention configurable via config.toml.
