## Problem

`roxabi_nats._serialize.register_type_checking_import(module_path, type_name)` maintains `_TYPE_CHECKING_IMPORTS`, a process-global mutable registry consulted by `_get_hints()` at deserialization time (via `importlib.import_module`). Surfaced during PR #724 review (roxabi-nats SDK extraction); flagged independently by the architect and security-auditor agents as the one architectural smell worth fixing before external plugins start consuming the SDK.

**Why now:** PR #724 is already shipping the short-term mitigation (demote to `_register_type_checking_import`, drop from `__all__` + `__init__.py`). That makes the helper hub-internal by convention but leaves the design debt in place. External consumers (voiceCLI, roxabi-vault, imageCLI) will start pinning `roxabi-nats` within weeks — the breaking change in the constructor is cheap now, expensive later.

**Observable impact:**
- Any module that imports `roxabi_nats` can mutate the registry. Order-dependent, hard to reason about across adapter instances.
- Parallel tests that need different type-hint sets interfere through the shared registry.
- `importlib.import_module` on a caller-controlled string is a code-execution sink. Internally trusted today; blast radius grows with every external consumer.
- Initialization via side effect — nothing in `NatsAdapterBase.__init__` tells you the adapter needs type hints registered somewhere else.

## Who

- **Primary:** `roxabi-nats` SDK consumers — current lyra hub + Telegram/Discord adapters that register `CommandContext` for deserialization. Future consumers (voiceCLI, roxabi-vault, imageCLI) adopting the SDK at `v0.2.0`.
- **Secondary:** SDK maintainers reasoning about adapter instantiation in isolation (tests, new transports).

## Constraints

- **Breaking change in `NatsAdapterBase.__init__`** — lands in `roxabi-nats/v0.2.0`. External consumers pin to `v0.1.x` until migrated.
- **Fail fast at init, not at deserialize time** — non-existent module paths must raise `ValueError` when the adapter is constructed, not on first message.
- **ADR-045 Consequences section** must be updated to remove the "known design debt" note once the global is gone.
- **Deprecation path** — `_register_type_checking_import` stays for one release emitting `DeprecationWarning`; `_TYPE_CHECKING_IMPORTS` dict removed.
- **All lyra call sites migrated in the same PR** — hub + Telegram + Discord adapters pass `type_registry` at construction. No mixed state.
- **Short-term mitigation in PR #724 must have landed first** — this refactor depends on the `_register_type_checking_import` rename being in place.

## Out of Scope

- **No other SDK surface changes** — `serialize`/`deserialize` gain an optional `resolver` param, nothing else moves.
- **No new transport adapters** — this is not the place to add extra `NatsAdapterBase` subclasses.
- **No ADR-045 rewrite** — only the Consequences section gets edited.
- **No changes to external consumers** (voiceCLI, roxabi-vault, imageCLI) — they pin `v0.1.x` and migrate in their own repos at their own pace.
- **No CONTRACT_VERSION bump coordination** — unrelated concern tracked elsewhere.

## Complexity

**Tier: F-lite** — clear scope, single package (`packages/roxabi-nats/` + lyra call sites), no unknowns. Confirmed by `size:F-lite` label on #729.

Signals observed:
- File count: ~5–8 (1–2 in `packages/roxabi-nats/src/roxabi_nats/`, 3 lyra call sites for hub + Telegram + Discord adapters, 1 ADR, 1–2 test files).
- Single domain (backend / SDK plumbing).
- Design is already decided in the issue proposal — no `/analyze` step needed.
- The breaking-change signature means spec + plan gates earn their keep: wrong constructor shape or missing fail-fast check is cheap to catch on paper, expensive in a released SDK.
