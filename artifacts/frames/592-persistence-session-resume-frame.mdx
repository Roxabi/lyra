## Problem

Conversation persistence is broken across three related scenarios, all rooted in incomplete or missing session-persistence wiring in the adapter and pool layers.

**Bug #1 â€” Restart does not resume session.** After a Lyra daemon restart, Discord DMs and all Telegram chats start a fresh session instead of rehydrating from the TurnStore. Discord threads work correctly because `discord_inbound.py` injects `thread_session_id` and wires a `_session_update_fn` callback â€” this mechanism is absent for Discord DMs and entirely missing from the Telegram adapter.

**Bug #2 â€” `/clear` does not rotate session ID.** `pool.reset_session()` clears history buffers but never replaces `pool.session_id` with a new UUID. Pre- and post-clear turns accumulate under the same session ID in TurnStore, and reply-to routing is permanently confused about session boundaries.

**Bug #3 â€” Reply-to loads the wrong session (race condition).** When `MessageIndex.resolve()` finds the target session but `pool.is_idle == False`, the resume is silently dropped and the current active session is used instead. The reply is routed to the wrong context.

**Bug #4 â€” Guild channels and Telegram groups use per-user pools instead of per-channel shared pools.** `discord_normalize.py` applies `user_scoped()` to guild channels (non-thread), and `telegram_normalize.py` applies `user_scoped()` to all group chats, giving each user their own isolated pool in shared spaces. This is inconsistent with how Discord threads work (one pool per thread, shared across users). Guild channels and Telegram groups should share one pool per channel/group regardless of sender.

## Who

- **Primary:** Any Lyra user on Discord DMs or Telegram â€” restart wipes their conversation context.
- **Secondary:** Users who rely on `/clear` to start a clean session, and users who use reply-to threading to switch context.

## Constraints

- Fix must not change the public interface of `TurnStore`, `MessageIndex`, or `ThreadStore`.
- Session wiring added to Telegram adapter must follow the same pattern already established in `discord_inbound.py` (threads path) for consistency.
- The reply-to race fix must queue the resume (not drop it) â€” dropping is observable user-facing data loss.
- No schema migrations required (stores already exist and are wired at bootstrap).

## Out of Scope

- Backfilling session history for messages already lost due to restarts.
- Adding session-resume to Discord threads (already works correctly).
- Changing the TurnStore query interface or adding new DB columns.
- UI/command changes beyond the existing `/clear` behaviour contract.
