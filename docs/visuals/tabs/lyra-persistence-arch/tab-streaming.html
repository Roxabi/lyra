Streaming & Output Types

Does the output type change how stores are consumed? Short answer: no — writes always happen at dispatch time, not during streaming.

🔒 Invariant: Stores are READ-ONLY during streaming

  • All store writes happen at exactly two checkpoints: before pool submit (user turn) and at _on_dispatched (after platform delivery)
  • The streaming window — from first text chunk to final result event — is a no-write zone for ALL stores
  • Tool calls, intermediate thinking, and multi-turn sub-processing do NOT trigger intermediate store writes
  • Output type affects what the user sees during streaming, not when or what Lyra writes to its stores
A — Text Streaming

Standard response. Claude emits text chunks in sequence. Lyra's _capture() tee wraps the iterator: chunks flow to the OutboundDispatcher AND are collected in a buffer for turn logging.

Before: TurnStore + MessageIndex log user turn
During: text chunks → OutboundDispatcher → Adapter.send_streaming() → Platform (chunked edits)
End of stream: result event carries final session_id → stored in pool.session_id
_on_dispatched: TurnStore + MessageIndex + ThreadStore + _resume_session_ids ALL updated
reply_message_id: available only after platform ACKs → that's why assistant turn is logged AFTER dispatch
B — Tool Use

Claude emits tool_use events mid-stream. Lyra renders tool progress to the user (show_tool_recap flag). Tool results are sent back to Claude via stdin. From Lyra's store perspective: completely transparent.

Claude → stdout: tool_use {name, input}
Lyra renders: tool progress badge (if show_tool_recap=true) — outbound only, no store write
Tool executes externally (MCP/plugin) → returns result
Lyra → stdin: tool_result {content}
Store writes: same as text — only at _on_dispatched, after ALL tools complete and full response is sent
Key: A turn with 5 tool calls still produces exactly ONE TurnStore entry for the assistant. All tool intermediates are inside the Claude CLI's own session — invisible to Lyra's stores.
C — Thinking / Intermediate

If show_intermediate=True on the agent, Lyra streams thinking text to the user as it arrives. This is display-only — no store interaction.

Claude → stdout: thinking content events
IF show_intermediate: rendered to user via OutboundDispatcher (streaming update)
Store impact: zero — thinking is not persisted to TurnStore, not indexed, not logged
Only the final text content ends up in TurnStore.content (collected by _capture())
D — Multi-turn (within one pool turn)

Claude CLI's --max-turns N allows the agent to complete multiple internal LLM turns before returning. From Lyra's perspective this is still ONE pool turn: one user message in, one final response out.

Claude CLI internally: turn 1 → tool → turn 2 → tool → turn N → final text
Lyra sees: a stream of events — tool badges + final text chunks
One _on_dispatched: fires once, after all internal turns complete and response is sent
TurnStore: one row — full final content. Internal sub-turns are not separately logged.
Clean: Store consistency is guaranteed regardless of how many internal turns Claude runs. The session_id in the final result event is authoritative.
