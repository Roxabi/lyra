## Context

Promoted from [frame](../frames/458-adapters-nats-rewire-frame.mdx). Part of #445 (Distributed NATS epic), Slice C5. Blocker #457 (Hub extraction) is closed. `NatsBus` and `NatsChannelProxy` are implemented and partially aligned with ADR-036. ADR-037 resolves the placement and streaming protocol questions.

Background ADRs:
- ADR-035: NATS subject naming convention
- ADR-036: `NatsChunkEnvelope` streaming protocol (single persistent outbound subject, stream_id multiplexing)
- ADR-037: `NatsOutboundListener` placement + `adapter_standalone.py` bootstrap contract

## Goal

Remove the `Hub` reference from `TelegramAdapter` and `DiscordAdapter`, rewiring them as pure I/O translators: platform events → NATS inbound subject, NATS outbound subject → platform API calls. Dev mode (`--adapter all` with `LocalBus`) continues to work without changes.

## Users

- **The Lyra runtime** — `lyra_telegram` and `lyra_discord` run as standalone processes communicating with `lyra_hub` via NATS.
- **Developers** — `--adapter all` keeps using `LocalBus`; NATS remains opt-in.

## Expected Behavior

### Inbound — text message

**Current:** `telegram_inbound._push_to_hub()` passes `inbound_bus=adapter._hub.inbound_bus` to `push_to_hub_guarded()`. Same for `discord_inbound`. The adapter must hold a live `Hub` reference.

**Target:** Both `telegram_inbound._push_to_hub()` and `discord_inbound._push_to_hub()` pass `inbound_bus=adapter._inbound_bus` — a `Bus[InboundMessage]` injected at construction. In NATS mode this is a `NatsBus`; in dev mode it is the `LocalBus` that Hub also reads from.

### Inbound — audio message

The audio path (`handle_voice_message` in `telegram_inbound.py`) currently passes `inbound_bus=adapter._hub.inbound_audio_bus`. This is also updated to `inbound_bus=adapter._inbound_audio_bus` in S1. The actual value is `LocalBus[InboundAudio]` (a no-op sink) in both dev and NATS modes — audio over NATS is deferred.

### Outbound — single-shot text reply

**Current:** Hub's `OutboundDispatcher` calls `adapter.send(original_msg, outbound)` directly (same process).

**Target:**
1. Hub's `OutboundDispatcher` calls `NatsChannelProxy.send(original_msg, outbound)`.
2. `NatsChannelProxy` publishes to `lyra.outbound.{platform}.{bot_id}` with a `NatsSendEnvelope` carrying `stream_id=original_msg.id` and serialized `outbound`.
3. Adapter's `NatsOutboundListener` receives the envelope, looks up `original_msg` from its local cache by `stream_id`, and calls `adapter.send(original_msg, outbound)`.

**`stream_id` correlation:** `stream_id = original_msg.id`. No new field on `InboundMessage` — the adapter's inbound path caches `{msg.id → InboundMessage}` before publishing to NATS; the outbound listener resolves `original_msg` from this cache.

### Outbound — streaming reply

**Per ADR-036:** All streaming chunks are published to `lyra.outbound.{platform}.{bot_id}` (same persistent subject as single-shot), with `NatsChunkEnvelope` envelopes (stream_id, seq, event_type, payload, done). `stream_id = original_msg.id`.

**Target for `NatsChannelProxy.send_streaming()`:** Update to publish `NatsChunkEnvelope` per ADR-036 on the single outbound subject instead of the current `lyra.outbound.stream.*` ephemeral subjects (which deviate from ADR-036 and are not implemented here).

`NatsOutboundListener` receives chunks on the shared subscription, routes them by `stream_id` into per-stream `asyncio.Queue`s, and calls `adapter.send_streaming(original_msg, events)` for each completed stream — or immediately for the first chunk, with the queue providing the async iterator.

### Outbound — attachment

`NatsChannelProxy.render_attachment()` already publishes to `lyra.outbound.{platform}.{bot_id}` with `type=attachment`. `NatsOutboundListener` handles this envelope type, looks up `original_msg` from cache, and calls `adapter.render_attachment(attachment_msg, original_msg)`.

### Dev mode (LocalBus — no change to behavior)

`--adapter all` starts Hub + adapters in the same process. Bootstrap passes `hub.inbound_bus` (a `LocalBus`) directly to adapter constructors. Adapters do **not** start a `NatsOutboundListener`; Hub's `OutboundDispatcher` already holds a direct reference to the registered adapter and calls it without NATS. No behavioral change.

### Audio (deferred)

`inbound_audio_bus` is a `LocalBus[InboundAudio]` no-op sink in both modes. Audio-over-NATS is deferred. `NatsChannelProxy` audio methods remain stubs.

### Wiring

- `TelegramAdapter._inbound_bus` ← injected via constructor (replaces `_hub.inbound_bus`)
- `TelegramAdapter._inbound_audio_bus` ← injected via constructor (replaces `_hub.inbound_audio_bus`)
- `telegram_inbound._push_to_hub()` ← passes `inbound_bus=adapter._inbound_bus` to `push_to_hub_guarded()`
- `telegram_inbound.handle_voice_message()` ← passes `inbound_bus=adapter._inbound_audio_bus`
- `telegram_inbound._push_to_hub()` ← calls `adapter._outbound_listener.cache_inbound(msg)` before publishing (if listener present)
- `NatsChannelProxy.send_streaming()` ← updated to publish `NatsChunkEnvelope` on `lyra.outbound.{platform}.{bot_id}` (not separate stream subjects)
- `NatsOutboundListener` in `src/lyra/adapters/nats_outbound_listener.py` (per ADR-037)
- Adapter starts listener in `astart()` (Telegram) / `start()` (Discord); stops in `close()`

- [ ] `DiscordAdapter.__init__` same — no `hub: Hub` parameter
- [ ] `telegram_inbound._push_to_hub()` calls `push_to_hub_guarded(inbound_bus=adapter._inbound_bus, ...)` (not `adapter._hub.inbound_bus`)
- [ ] `discord_inbound._push_to_hub()` calls `push_to_hub_guarded(inbound_bus=adapter._inbound_bus, ...)`
- [ ] `telegram_inbound.handle_voice_message()` passes `inbound_bus=adapter._inbound_audio_bus` (not `adapter._hub.inbound_audio_bus`)
- [ ] Adapters do not import `Hub` from `lyra.core.hub` — including `TYPE_CHECKING` guards
- [ ] `NatsChannelProxy.send_streaming()` publishes `NatsChunkEnvelope` per ADR-036 on `lyra.outbound.{platform}.{bot_id}` (not on `lyra.outbound.stream.*` subjects)
- [ ] `NatsOutboundListener` in `src/lyra/adapters/nats_outbound_listener.py` subscribes to `lyra.outbound.{platform}.{bot_id}`, handles `type=send`, `type=attachment`, and `NatsChunkEnvelope` (streaming)
- [ ] `NatsOutboundListener` uses adapter-side cache `{msg_id → InboundMessage}` for outbound correlation; never embeds `original_msg` in NATS envelopes
- [ ] `NatsOutboundListener.start()` / `stop()` wired into adapter lifecycle (listener running after `astart()`/`start()`, stopped in `close()`)
- [ ] `--adapter all` dev mode works end-to-end with `LocalBus` — no `NatsOutboundListener` started in dev mode
- [ ] `bootstrap/adapter_standalone.py` wires NATS adapter processes per ADR-037: NATS connection → NatsBus → adapter construction → `resolve_identity()` → NatsOutboundListener → uvicorn (Telegram) / Gateway (Discord)

## Edge Cases & Risks

| Scenario | Handling |
|----------|---------|
| NATS connection dropped mid-stream | `NatsOutboundListener` stream times out per ADR-036 gap policy; user sees partial/error response |
| Chunk gap detected (`seq` skip) | Treat as `STREAM_ABORTED` per ADR-036 — surface error to user, close stream |
| Adapter starts before Hub (race) | NatsBus publish succeeds (NATS queues message); Hub processes when it connects |
| `InboundMessage` cache eviction | Cache entry removed after listener receives `done=true` on the outbound stream; TTL fallback for abandoned turns |
| Dev mode `Hub` parameter removal | Existing tests passing `hub=...` to adapter constructors must be updated in the same PR |
| `TYPE_CHECKING` Hub import | Remove entirely — no static typing needs it after the constructor change |
