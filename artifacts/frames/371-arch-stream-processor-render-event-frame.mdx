## Problem

`LlmProvider.complete()` buffers the entire LLM response before returning — `tool_use` events are processed internally by drivers and never surfaced to the domain core. The channel adapter layer (`send_streaming()`) receives only `AsyncIterator[str]`, making real-time tool call visibility impossible in Telegram and Discord.

The root gap is architectural: Lyra has no event model between the LLM layer and the outbound adapters. There is no `LlmEvent` hierarchy (`TextLlmEvent`, `ToolUseLlmEvent`, `ResultLlmEvent`), no `StreamProcessor` to aggregate and throttle them into display-ready events, and no `RenderEvent` types for channel-agnostic output. Six concrete gaps (G1–G6) block the path from current to hexagonal target.

This was surfaced by a UX reflection on rendering Claude `tool_use` blocks in real time, and generalised during a full architecture gap analysis. Three target files (`llm/events.py`, `core/stream_processor.py`, `core/render_events.py`) do not yet exist.

## Who

- **Primary:** Lyra engine internals — the LLM → StreamProcessor → ChannelAdapter pipeline that must be wired end-to-end
- **Secondary:** Telegram and Discord users who currently see no tool call progress during streaming responses

## Constraints

- `LlmProvider.complete()` must remain fully intact — zero regression on the current non-streaming path
- Domain Core (`core/`) must import nothing from `aiogram`, `discord.py`, or `anthropic` — hexagonal boundary enforced
- `StreamProcessor` must be config-driven: thresholds (`names_threshold`, `group_threshold`, `bash_max_len`, `throttle_ms`) and per-tool `show` flags in `config.toml`
- Each of the 5 implementation phases must be independently mergeable to `staging`
- `StreamProcessor` requires 100% unit test coverage with no network dependencies
- Phase 3 (StreamProcessor) is blocked by Phases 1 + 2; Phase 4 is blocked by Phase 3; Phase 5 is independent

## Out of Scope

- Any changes to the `complete()` path or existing non-streaming behaviour
- New agent capabilities, conversation logic, or tool additions
- Voice / audio / token-level streaming (separate issues: #144, #185, #320)
- Promotion to production (handled independently via `/promote`)

## Complexity

**Tier: F-full** — 6 architectural gaps spanning 3 distinct layers (LLM drivers, domain core, channel adapters), 5 sequentially-dependent implementation phases, new event and render type hierarchies, and config-driven aggregation/throttle behaviour.

Signals observed:
- Multiple domains: `llm/`, `core/`, `adapters/`
- New architectural patterns: Ports & Adapters hexagonal boundary, event-driven streaming pipeline
- Multiple new files: `llm/events.py`, `core/stream_processor.py`, `core/render_events.py`
- Phase dependency chain: P1 + P2 (parallel) → P3 → P4; P5 independent
- Significant test surface: ~20 unit test cases across StreamProcessor, LLM drivers, and channel adapters
