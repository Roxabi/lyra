## Problem

PR #361 introduced the `lyra.integrations` layer with `ScrapeProvider` and `VaultProvider` protocols, establishing a clean pattern for wrapping external tool calls. However, two genuine gaps remain: `ffmpeg` (called directly in `tts/__init__.py` for WAV-to-OGG conversion) and `supervisorctl.sh` (called directly in `commands/svc/handlers.py` for service management). These direct calls bypass the integration layer, making the affected code harder to test and inconsistent with the established pattern.

## Who

- **Primary:** Developers maintaining Lyra — consistent integration layer makes external tool usage predictable and testable.
- **Secondary:** Contributors — clear protocol boundaries lower the bar for understanding how Lyra interacts with external tools.

## Constraints

- Follow the existing provider protocol pattern established by PR #361 (`lyra.integrations.base`)
- `AudioConverter` protocol: `async convert_wav_to_ogg(wav_path, ogg_path) -> None`
- `ServiceManager` protocol: `async control(action, service) -> str`
- Implementations: `FfmpegConverter` in `lyra/integrations/audio.py`, `SupervisorctlManager` in `lyra/integrations/supervisor.py`
- Inject providers via the same mechanism used for vault/scraper providers

## Out of Scope

- `claude` CLI in `core/cli_pool_worker.py` — core infrastructure, intentionally bare
- `systemctl` in `monitoring/checks.py` — one-liner health check, no business logic
- Any behavioral changes to ffmpeg conversion or supervisorctl operations themselves

## Complexity

**Tier: F-lite** — Clear scope, single domain (integrations layer), follows an established pattern with 2 well-defined providers.

- 2 new protocol + implementation files
- 2 existing files updated to use injected providers
- No architectural decisions needed — pattern already proven by PR #361
