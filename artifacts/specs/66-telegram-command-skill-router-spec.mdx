## Context

Lyra's hub forwards all messages to `agent.process()` as raw text. There is no command system — `/agenda`, `/search`, `/help` are treated as free-text. The command router adds a detection + dispatch layer inside the hub run loop, routing `/command` messages to skill handlers while passing everything else to the default agent.

Builds on ADR-010 (Install, Wrap, Declare): the router is the Lyra-side bridge that connects agent TOML config (Layer 3) to the CLI tools on PATH (Layer 1).

## Goal

Messages starting with `/command` are intercepted by the hub, routed to a configured skill handler, and the result is sent back via the originating adapter — without touching the default agent.

## Users

- **Primary:** Mickael — sends `/agenda`, `/tasks`, `/save`, `/voice` via Telegram daily
- **Secondary:** Any Lyra user on any channel — the router is channel-agnostic

## Expected Behavior

1. User sends `/agenda` via Telegram
2. Telegram adapter normalizes message, pushes to bus (existing flow)
3. Hub pulls message, resolves binding (existing flow)
4. **New:** Hub resolves the agent from binding, asks agent's `command_router` if message is a command
5. **New:** Router detects `/agenda` prefix → looks up command registry
6. **New:** Registry maps `/agenda` → skill handler config `{skill: "google-workspace", action: "calendar-today"}`
7. **New:** Skill handler checks CLI availability via `shutil.which()`, executes command via `asyncio.create_subprocess_exec`, formats result
8. Hub dispatches response via adapter (existing flow)

**Fallback paths:**
- Unknown `/command` → respond with "Unknown command. Type `/help` for available commands."
- CLI not on PATH → respond with "Command unavailable: `gws` is not installed."
- Subprocess timeout (>30s) → respond with "Command timed out. Please try again."
- No `/` prefix → normal agent flow (unchanged)

**Execution model note:** Commands bypass `agent.process()` entirely — the router calls CLI subprocesses directly via `asyncio.create_subprocess_exec`. This is a deliberate divergence from the frame's "Bash tool via agent" description. Rationale: commands are user-initiated, deterministic, and don't need LLM reasoning. Routing through the agent would add latency and token cost for a simple subprocess call. The `[commands]` section in agent TOML acts as a second permission boundary (distinct from the `tools` allowlist), which is appropriate since commands have a different threat model (no prompt injection, no LLM discretion).

**Args passthrough:** Command arguments are passed positionally as-is. `/search my query` → args = `["my", "query"]`. No flag parsing, no quoting — simple split on whitespace. Complex argument parsing is out of scope (deferred).


## Technical Notes

### Router ownership: per-agent, not per-hub

`CommandRouter` is owned by the agent, not the hub. Each agent has its own command registry from its TOML `[commands]` section. The hub resolves the agent first (via binding), then asks `agent.command_router`.

This avoids ambiguity when multiple agents are registered — each agent declares its own commands. Phase 1 constraint: single-agent usage (Lyra has one active agent). Multi-agent command conflict resolution is deferred.

```python
# In hub.run(), after resolving agent:
if agent.command_router and agent.command_router.is_command(msg):
    try:
        response = await agent.command_router.dispatch(msg)
    except Exception as exc:
        log.exception("command dispatch failed for %s: %s", key, exc)
        response = Response(content=GENERIC_ERROR_REPLY)
    try:
        await self.dispatch_response(msg, response)
    except Exception as exc:
        log.exception("dispatch_response() failed for %s: %s", key, exc)
    continue
# else: existing pool.lock → agent.process() flow
```

### TOML command config

Commands are declared in the agent TOML file under `[commands]`:

```toml
[commands."/help"]
builtin = true
description = "List available commands"

[commands."/agenda"]
skill = "google-workspace"
action = "calendar-today"
cli = "gws"
description = "Show today's calendar"

[commands."/echo"]
skill = "echo"
action = "echo"
cli = "echo"
description = "Echo back the message (test command)"
```

### Skill registry (Phase 1)

Hardcoded dict mapping `(skill, action)` → CLI command template:

```python
SKILL_REGISTRY: dict[tuple[str, str], list[str]] = {
    ("echo", "echo"): ["echo"],
    ("google-workspace", "calendar-today"): ["gws", "calendar", "list", "--today", "--json"],
    # ... added per skill integration issue
}
```

Args from the user message are appended to the template. Phase 2: move to config file.

### Permission model

`[commands]` is a second permission boundary, distinct from the `tools` allowlist in `[model]`:
- `tools` governs what the LLM can call autonomously (agent-driven)
- `commands` governs what the user can trigger directly (user-driven, deterministic)

Different threat model: commands don't involve prompt injection or LLM discretion.
