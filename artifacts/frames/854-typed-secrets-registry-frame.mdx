## Problem

`bootstrap/infra/health.py` exposes `_read_secret(name: str)` — a stringly-typed
filename lookup that reads from `$LYRA_VAULT_DIR/secrets/{name}`. The `name`
parameter is an arbitrary string passed directly into a path join: a caller
supplying `"../../etc/passwd"` would read any file readable by the lyra user.
Today only one hardcoded caller exists (`"health_secret"`), so the surface is
not exploitable, but the function signature accepts any string and provides no
defence-in-depth.

The audit (§P0 #4, `artifacts/analyses/quality-audit/AUDIT-SUMMARY.md`) flagged
this as a latent generic-surface risk — the function is safe only by convention,
not by construction.

## Who

- **Primary:** bootstrap module maintainers — adding/rotating secrets requires
  passing a new string literal; there is no compile-time or type-system
  enforcement that the name is valid.
- **Secondary:** security auditors — the open string surface makes it impossible
  to enumerate expected secret names without grepping callsites.

## Constraints

- Scope is strictly the bootstrap layer (`src/lyra/bootstrap/`); no adapter or
  hub code changes.
- The `Secrets` class must preserve the current `LYRA_VAULT_DIR` resolution
  and silent-fallback-to-empty-string behaviour so existing deployments keep
  working without config changes.
- Lane H container work (#693) will migrate secrets to Podman secrets /
  systemd-creds; the registry must make that one-line swap trivial
  (replace `_read` internals, keep properties intact).
- No new external dependencies; `functools.cached_property` is sufficient.

## Out of Scope

- Migrating secrets to Podman / systemd-creds (Lane H / #693).
- Adding validation or non-empty enforcement on individual secret values.
- Rotating or changing secret filenames on disk.
- Any change to adapter config loading or `config.toml` handling.

no new architecture.
