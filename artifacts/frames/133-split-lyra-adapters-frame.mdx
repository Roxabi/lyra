## Problem

`python -m lyra` runs as a **single process** — Telegram adapter, Discord adapter, Hub, and agent
pools all share one asyncio event loop. Restarting any component requires taking down the entire
process, interrupting all active conversations across all channels.

As Lyra grows (voice input, voice output, new channels), each service must be independently
restartable. A flaky Telegram connection or a Discord library update should not require a full
daemon restart.

Additionally, production (Machine 1) is missing the voice daemons (`voicecli_tts`, `voicecli_stt`)
that are already running locally. Machine 1 handles Telegram voice messages and will gain a voice
interface in Phase 2 — the voice services must be running there too.

## Who

- **Primary:** Mickael (operator) — needs to restart individual adapters (e.g., on config change,
  library update, or transient failure) without interrupting unrelated conversations.
- **Secondary:** Lyra users — reduced downtime during adapter restarts; only the affected channel
  goes offline briefly instead of all channels simultaneously.

## Constraints

- **IPC design: Option A (hub embedded per adapter)** — each adapter process (`lyra_telegram`,
  `lyra_discord`) embeds its own Hub + agent pool. No inter-process communication protocol needed.
  Hub-as-a-service (Option B) is the named Phase 2 upgrade path when cross-platform bridging or
  N > 3 adapters are required.
- **Python-first paradigm** — all new entry points are Python packages with clean `__main__`
  / CLI entrypoints, consistent with the existing pattern.
- **Supervisor pattern** — the lyra-stack supervisord manages all daemons. Each new process gets
  a `*.conf` file in `lyra/supervisor/conf.d/` (production-side) and symlinked to
  `~/projects/lyra-stack/conf.d/` (local-side).
- **Makefile targets** — `make telegram reload`, `make discord reload`, `make tts reload`,
  `make stt reload` must work on both local and production.
- **Both environments** — local (`~/supervisor/conf.d/`) and production (`lyra/supervisor/conf.d/`)
  must be updated together.
- **SQLite is shared** — WAL mode handles concurrent readers/writers; both adapter processes share
  the same memory DB. Agent long-term memory remains unified.
- **Phase 1b tail still in progress** — #83 (agent integration) and #99 (hub command sessions)
  are unmerged. This issue must not conflict with them; it targets supervisor/infra only.
- **Independent of #112** — the hub refactor (#112 epic) is complete. This issue builds on the
  stable Phase 1b hub.

## Out of Scope

- **Option B (hub as a service / Unix socket IPC)** — intentionally deferred. Revisit when
  cross-platform bridging or a central cognitive routing layer (Phase 2) is needed.
- **Option C (Redis message broker)** — rejected; overkill for personal scale.
- **New adapter logic** — this issue is infrastructure only. No changes to adapter behavior,
  hub routing, or agent logic.
- **Machine 2** — voice daemons are specific to Machine 1 (always-on hub). Machine 2 is
  on-demand and does not run Lyra services.
- **WebSocket / HTTP adapter protocol** — out of scope for Option A; each adapter is
  self-contained.

## Complexity

**Tier: F-full** — Multi-domain infra change spanning Python entry points, supervisor configs
(two environments), Makefile targets, and production deployment of `voicecli`.
