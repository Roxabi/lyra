ADR-029 — DB-first agent config. The SQLite agents table is the single runtime source of truth. TOMLs are seed-only. lyra agent create writes to the DB directly — no TOML is created. Hot-reload uses updated_at polling. New in v10: persona_json (inline persona), voice_json (merged TTS+STT), patterns_json (per-agent URL rewriting), fallback_language.

What is an agent?

An agent is a named configuration that binds a system prompt, a model, a set of tools, optional persona_json, and voice_json to a conversational identity. When a bot receives a message, it looks up its assigned agent by name from the DB and passes the message to a Pool running that agent.

Agents are stateless singletons — their configuration is shared across all concurrent conversations. Conversation state (history, circuit) lives in the Pool, not the agent.

TOML files vs DB — what goes where

SourceRoleRead at runtime?
src/lyra/agents/*.tomlSeed files — versioned defaultsNo — only via lyra agent init
~/.lyra/agents/*.tomlUser-level overrides (higher precedence)No — only via lyra agent init
~/.lyra/auth.db (agents table)Runtime source of truthYes — on every message via AgentStore cache
~/.roxabi-vault/personas/*.persona.tomlPersona file source (loaded into persona_json on migration)Only at migration / force-init time
DB takes precedence. Editing a TOML file after lyra agent init has no effect until you run lyra agent init --force. The running daemon reads from the DB only. persona_json, voice_json, and patterns_json use COALESCE on upsert — passing null preserves the existing DB value.

CLI reference

Creating agents v10 — DB-first, no TOML

bashDB-first — writes directly to auth.db
lyra agent create my_agent \
  --model claude-sonnet-4-6 \
  --system-prompt "You are a helpful assistant." \
  --tools Read,Bash,Skill
No TOML, no init step. lyra agent create calls AgentStore.upsert() directly. The agent is immediately available to the daemon on the next message.

Seeding from TOML (first-time setup)

bash
lyra agent init              # seed DB from TOML files (skips existing)
lyra agent init --force      # overwrite existing DB rows from TOMLs

Use init once on a new machine, or after updating seed TOMLs. Note: on first connect, AgentStore._populate_343() automatically migrates persona → persona_json and tts_json+stt_json → voice_json for existing rows.

Viewing and editing agents

bash
lyra agent list              # all agents in DB
lyra agent show my_agent     # full config for one agent
lyra agent edit my_agent     # interactive editor, writes to DB (bumps updated_at)
lyra agent validate my_agent # check config for errors

Assignment, unassign, and deletion

bash
lyra agent assign my_agent --platform telegram --bot <bot_id>
lyra agent unassign --platform telegram --bot <bot_id>
lyra agent delete my_agent     # refuses if a bot is still assigned

Hot-reload mechanics v10 split

On every inbound message, AgentBase runs two reload checks:

  1. _maybe_reload_config()Fetches AgentStore.get(name) — O(1) dict lookup. Compares row.updated_at to local cache. If changed, calls agent_row_to_config(row) to deserialize all DB columns (including persona_json, voice_json, patterns_json) into the Agent dataclass. Invokes compose_system_prompt_from_json() for inline personas.
  2. _maybe_reload_plugins()Separately checks whether extension commands (CommandLoader) need reloading — runs CommandReloadManager if the agent's commands config has changed. Split from config reload so plugin reloads don't block on config equality.
Hot-reload is lazy and per-message — there is no background polling timer. At personal-scale message volumes, two O(1) dict lookups per message is negligible.
Persona file changes don't auto-reload. If you edit a .persona.toml file in the vault, run lyra agent edit <name> and save any field to bump updated_at and trigger the reload path. Alternatively use lyra agent init --force <name>.

Persona — inline JSON vs file reference

As of v10 (ADR-029), persona data is stored inline in the DB as persona_json. File-based personas are migrated automatically. Two paths exist:

Inline persona (persona_json) v10

Stored directly in the DB. Edited via lyra agent edit. Composed into system prompt by compose_system_prompt_from_json() at runtime.

jsonpersona_json structure
{
  "identity": {
    "display_name": "Lyra",
    "tagline": "personal AI assistant",
    "creator": "Roxabi",
    "role": "...",
    "goal": "..."
  },
  "personality": {
    "traits": ["curious","concise"],
    "style": "direct",
    "tone": "warm",
    "humor": "dry"
  },
  "expertise": {
    "areas": ["Python","asyncio"],
    "instructions": ["Be concise","No filler"]
  }
}
File-based persona (legacy)

TOML files at ~/.roxabi-vault/personas/<name>.persona.toml. Loaded by load_persona() and migrated into persona_json on first DB connect. The persona column retains the name as a reference but runtime uses persona_json.
