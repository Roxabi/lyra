## Problem

The Hub currently depends directly on `asyncio.Queue` for internal message passing. This tight coupling blocks the introduction of distributed messaging (NATS) in Phase 2, because swapping the transport requires touching every call site that creates or interacts with queues.

Extracting a `Bus` ABC with `publish()`/`subscribe()` semantics allows the Hub to depend on an interface, not an implementation. The first concrete implementation (`LocalBus`) wraps the existing `asyncio.Queue` with identical behavior — zero regression, pure refactor.

This is a prerequisite for #50 (NatsBus) and part of Phase 2 NATS introduction (#60).

## Who

- **Primary:** Hub core — the message-routing layer that currently owns `asyncio.Queue` instances
- **Secondary:** Adapter authors and future NatsBus consumers — they benefit from a stable `Bus` contract

## Constraints

- `LocalBus` must preserve existing backpressure behavior exactly
- All existing tests must pass without modification (zero regression)
- Config toggle (`bus: local | nats`) must be wired but only `local` is functional in this issue
- No NATS dependency introduced — NatsBus implementation is #50

## Out of Scope

- NatsBus implementation (#50)
- NATS server setup or connection management
- Message serialization changes (current `bytes` contract stays)
- Multi-bus topologies or routing

## Complexity

**Tier: F-full** — New abstraction layer (ABC), migration of Hub internals from concrete queue to interface, config wiring, and careful preservation of backpressure semantics across all existing paths.

Signals:
- New architectural pattern (Bus protocol) touching Hub core
- Multiple domains: core abstraction, Hub integration, config system
- Must preserve subtle runtime behavior (backpressure)
- Dependency chain: blocks #50, part of Phase 2 (#60)
