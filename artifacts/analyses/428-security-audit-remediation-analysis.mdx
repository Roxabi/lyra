## Source

GitHub issue #428 — "Security audit: 1 Critical, 4 Medium, 4 Low findings + prompt injection analysis". Full report: `artifacts/security-audit-2026-03-26.md`.

## Problem

The 2026-03-26 security audit identified 9 actionable findings across error handling, prompt injection, config trust, system prompt exposure, and health endpoint architecture. Three additional backlog items improve defense-in-depth. All are fixable within the Lyra codebase — no external dependencies or upstream blockers (M1's "blocked" status was stale; `--system-prompt-file` exists).

### Current state of each finding

| ID | Finding | File(s) | Current code | Gap |
|----|---------|---------|-------------|-----|
| **C1** | `/svc` leaks raw exceptions to chat | `commands/svc/handlers.py:79,82` | `f"Error: {exc}"` sent to chat | Should use `GENERIC_ERROR_REPLY` pattern already used everywhere else |
| **M1** | System prompt in `/proc/cmdline` | `core/cli_pool_worker.py:123-127` | `--system-prompt <value>` as CLI arg | `--system-prompt-file` exists in Claude CLI — write tmpfile (`mode=0o600`), pass path, delete in `finally` after subprocess opens |
| **M2** | `LYRA_CONFIG` accepts arbitrary paths | `bootstrap/config.py:116` | `os.environ.get("LYRA_CONFIG", "config.toml")` no validation | Validate against trusted base. Config base = `Path.home()` (covers `~/projects/lyra/`, `~/.lyra/`, and any home-relative path). Production configs in `/etc/lyra/` would need the base extended — acceptable for single-tenant. |
| **M3** | Health endpoint exposes operational state | `bootstrap/health.py:22-86` | All detail behind auth, but single endpoint | Split into `/health` (liveness: `{"ok": true}`, no auth) and `/health/detail` (operational data, returns 401 on missing/bad auth — unlike current `/health` which silently degrades) |
| **M4** | Scraped content not XML-escaped before LLM | `core/processors/_scraping.py:136` | Raw `{scraped}` inside `<webpage>` tags | `add_vault.py:88-92` already escapes `&`, `<`, `>` — reuse pattern |
| **L1** | `LYRA_MESSAGES_CONFIG` no path validation | `bootstrap/config.py:234` | `os.environ.get("LYRA_MESSAGES_CONFIG")` | Same fix as M2 — validate against trusted base |
| **L2** | `/svc` stdout (PIDs, paths) returned verbatim | `commands/svc/handlers.py:72` | `output or "(no output)"` | Strip PIDs and absolute paths from output |
| **L3** | `/config` reveals `effective_model`/`effective_max_steps` | `bootstrap/health.py:114-117` | Returned to any authenticated caller | Remove `effective_*` computed fields — callers can derive from raw values |
| **L4** | HTTP scheme allowed in scraping URLs | `core/processors/_scraping.py:76` | `startswith(("http://", "https://"))` | Warn on `http://` (log), or reject and require `https://` |
| **BL1** | File-based secrets (Claw pattern) | `bootstrap/health.py:24`, env vars | `LYRA_HEALTH_SECRET` in env → `/proc/environ` | Read from file (`/run/secrets/` or `~/.lyra/secrets/`) |
| **BL2** | Centralized error sanitization | `commands/svc/handlers.py` + future | Per-handler error handling | Shared `safe_error_response()` utility |
| **BL3** | Plain text not tagged as user input | `agents/simple_agent.py:236`, `anthropic_agent.py:176` | `text = msg.text` (raw) | Wrap in `<user_message>` tags like voice uses `<voice_transcript>` |

### Existing patterns to reuse

1. **`GENERIC_ERROR_REPLY`** (`core/message.py:14`) — already used in `pool_processor.py`, `middleware_stages.py`, `message_pipeline.py`, both outbound adapters. C1 is the only exception handler that doesn't use it.
2. **XML escaping** (`processors/add_vault.py:88-92`) — `content.replace("&", "&amp;").replace("<", "&lt;").replace(">", "&gt;")` + "treat as untrusted" label.
3. **`_TRUSTED_BASE` validation** (`integrations/web_intel.py:27,33`) — `Path.resolve().is_relative_to(_TRUSTED_BASE)`. Exact pattern needed for M2/L1. Note: `integrations/supervisor.py` also uses this pattern but only exists on `feat/362-wrap-ffmpeg-supervisorctl` (not yet merged to staging). Reference `web_intel.py` as the canonical source.
4. **`hmac.compare_digest`** (`bootstrap/health.py:26`) — constant-time auth already correct.

## Appetite

No time constraint. Quality over speed.

## Shapes

### Shape 1: Single-pass surgical

Fix all 12 items in one branch, one PR. Each finding gets a focused commit. No new abstractions — reuse existing patterns inline.

**Trade-offs:**
- Pro: Fastest to complete, all fixes visible in one diff, easy to track against audit
- Pro: Reuses existing patterns (GENERIC_ERROR_REPLY, XML escape, _TRUSTED_BASE) — no new abstractions
- Con: Large PR (12 commits, 10+ files) — harder to review
- Con: BL2 (centralized error sanitization) doesn't fit the "no new abstractions" approach

**Rough scope:** M — ~10 files, all using known patterns

### Shape 2: Themed batches (3 PRs)

Group findings by security domain. Each PR is self-contained and independently reviewable:
1. **Error handling** (C1, L2, BL2) — sanitize `/svc` output, build shared `safe_error_response()`
2. **Prompt injection** (M4, BL3) — escape scraped content, tag plain text as `<user_message>`
3. **Config & infra** (M1, M2, M3, L1, L3, L4, BL1) — config path validation, system prompt file, health split, secrets from files

**Trade-offs:**
- Pro: Each PR is focused, easy to review, can be merged independently
- Pro: If one batch has issues, others aren't blocked
- Con: More overhead (3 branches, 3 reviews, 3 merges)
- Con: Overkill for fixes that are mostly 5-15 min each

**Rough scope:** M — same total work, split into 3 chunks

### Shape 3: Infra-first, then apply

Build shared infrastructure first (centralized error sanitizer, input escaping helper, config path validator, file-based secret loader), then apply across all findings:
1. PR1: Shared security utilities (`lyra.core.security` module)
2. PR2: Apply utilities to all 12 findings

**Trade-offs:**
- Pro: Creates reusable security patterns for future development
- Pro: Consistent approach across all findings
- Con: Over-engineered — most patterns already exist (GENERIC_ERROR_REPLY, add_vault escape, _TRUSTED_BASE)
- Con: New module creates coupling for one-time fixes

**Rough scope:** L — adds abstraction layer on top of existing patterns

## Fit Check

**Shape 1 (single-pass surgical) is the best fit.**

- The audit provides precise file locations and 5-30 min fix estimates per finding. These are targeted changes, not architectural refactoring.
- Existing patterns (`GENERIC_ERROR_REPLY`, XML escaping, `_TRUSTED_BASE`) cover 9 of 12 findings with zero new code. Only M1 (tmpfile for system prompt), M3 (health split), and BL1 (file-based secrets) need new logic.
- Shape 2 adds review overhead for no benefit — findings don't conflict, so a single pass with per-finding commits is just as reviewable.
- Shape 3 is eliminated — creating a new `security` module for patterns that already exist elsewhere is premature abstraction.

For BL2 (centralized error sanitization), a lightweight `safe_error_response()` helper is needed. `core/message.py` is a pure data module (dataclasses, enums) — adding a function with logging side-effects is a layering oddity. Better home: a small `core/error_utils.py` that imports `GENERIC_ERROR_REPLY` from `message.py` and adds the `log.exception()` + generic-reply pattern.
