Routing
Command dispatch, prefix handling, 6-layer routing pipeline, and circuit breaker.
The Two Prefixes
/
Strict mode
Hard command — / prefix
Known command → dispatched to registered handler immediately
Unknown command → returns "Unknown command: /foo" error — LLM never invoked
No fallthrough. Strict contract — unknown / commands always error.
!
Soft mode
Fallthrough allowed — ! prefix
Known command → dispatched identically to / — same registry, same handler
Unknown command → router returns None → falls through to LLM pool
Only error behavior differs. Same command set, same dispatch order.
💬
Plain text bypasses the command router entirely and goes directly to the LLM pool. No prefix = no command check.
6-Layer Dispatch Pipeline
1 Native platform commands — handled at adapter level before entering Hub
2 Builtin commands — /help, /clear, /stop, /new always registered
3 Workspace commands — /workspace <key> sets cwd for pool
4 Session commands — /vault-add, /explain, /summarize isolated LLM call
5 CommandLoader extensions — extension commands commands/ directory
6 LLM pool fallthrough — agent generates response final destination
ℹ
Layer 1 (native platform commands) is new in v12. Discord voice commands like !join and !leave are handled directly in the adapter — they never reach the Hub router.
Scope ID Resolution

The scope_id is the unique conversation identity. It determines which Pool handles a message.

PlatformContextscope_id formatNotes
Telegram Group / channel chat:<chat_id> All users in the chat share one Pool
Telegram DM dm:<user_id> Private, one Pool per user
Telegram Forum topic chat:<chat_id>:topic:<topic_id> Each topic is an isolated scope
Discord Guild channel channel:<channel_id> Per-channel shared Pool
Discord Thread thread:<thread_id> Thread ID = scope for session continuity
Discord DM dm:<user_id> Private, one Pool per user
★
Discord thread support. When auto_thread: true, each new watch-channel conversation creates a thread. The thread ID is used as scope_id, giving full session isolation per-conversation even within the same channel.
Circuit Breaker

Per-user rate limiting with three states. Prevents cascade failures from a single user's repeated errors.

CLOSED

Normal operation. All messages processed. Failure counter increments on backend errors. Circuit opens when threshold is reached.

OPEN

Too many failures. Messages rejected immediately — user notified with a clear message. After cooldown period, transitions to HALF‑OPEN.

HALF-OPEN

Probe mode. One message allowed through. Success → back to CLOSED. Failure → back to OPEN. Auto-recovery without restart.

CommandRouter patterns_json

Each agent can configure URL auto-rewrite rules and custom command patterns via patterns_json in the DB.

jsonagents.patterns_json
{
  "bare_url": true,            // auto-rewrite bare URLs → /vault-add <url>
  "patterns": [
    {
      "match": "^https://github\\.com",
      "rewrite": "/gh {url}"         // custom rewrite for GitHub URLs
    }
  ]
}
🔗
bare_url opt-in is per-agent. When true, any message containing only a URL (no other text) is automatically rewritten to /vault-add <url> before routing. Useful for watch channels that process links.
