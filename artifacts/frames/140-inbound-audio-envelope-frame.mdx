## Problem

Audio messages arrive from Telegram (voice notes, audio files) and Discord (audio attachments) as raw bytes + platform-specific metadata. There is no typed, platform-agnostic envelope for inbound audio — adapters currently have no `normalize_audio()` contract, so any downstream consumer (e.g. an STT pipeline) would need to handle platform-specific structures directly.

This mirrors the gap that `InboundMessage` (#137) solved for text: the hub and agents should never read platform-specific fields above the adapter layer. The same principle must apply to audio.

## Who

- **Primary:** Hub run loop and future STT/transcription agents — will receive a typed `InboundAudio` envelope instead of raw bytes + ad-hoc metadata.
- **Secondary:** Future adapter authors — `normalize_audio()` on `ChannelAdapter` gives them a clear contract to implement.

## Constraints

- `InboundAudio` must live in `lyra/bus/messages.py` alongside `InboundMessage`.
- Field set is defined by the issue: `platform`, `bot_id`, `scope_id`, `user_id`, `audio_bytes`, `mime_type`, `duration_ms`, `file_id`, `timestamp`.
- Both Telegram and Discord adapters must implement `normalize_audio(raw) → InboundAudio`.
- Telegram: handle voice messages (ogg/opus) and audio files.
- Discord: handle audio attachments.
- Must include unit tests for both adapters.

## Out of Scope

- STT / transcription pipeline (this issue only defines the envelope and normalization; processing is downstream).
- Audio playback / outbound audio.
- Web gateway or other adapter audio support.
- Persisting audio bytes or structured logging of audio metadata.
- `InboundAudio` being pushed onto a bus queue (the envelope is defined here; queue integration is a follow-on).

## Complexity
