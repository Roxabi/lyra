## Problem

`adapters/telegram.py` is 1,143 lines — a monolithic adapter mixing gateway handling, text formatting, audio processing, and command handling. This mirrors the same pattern already addressed in Discord (#293). The file's size makes it hard to navigate, review, and test individual concerns in isolation. Changes to formatting logic risk breaking audio pipelines and vice versa.

Parent issue #293 tracks the broader adapter decomposition; this is the Telegram extraction step.

## Who

- **Primary:** Maintainers working on Telegram adapter features or bug fixes — currently every change requires navigating 1,100+ lines of mixed concerns.
- **Secondary:** Contributors touching audio or formatting logic across adapters — extracted modules enable cross-adapter patterns.

## Constraints

- Must follow the same extraction pattern used for Discord adapter decomposition
- `uv run pytest` must pass after extraction
- `uv run ruff check .` must be clean
- `adapters/telegram.py` must shrink to ≤300 lines (core adapter logic only)
- No behavioral changes — pure mechanical refactor

## Out of Scope

- Refactoring the extracted code itself (formatting logic, audio pipeline)
- Shared base classes across Telegram/Discord adapters
- New features or API changes
- Helper methods (`_reply_safe`, `_start_typing`, etc.) — these stay in the adapter for now

## Complexity

**Tier: F-lite** — Single domain (adapters), clear extraction boundaries defined in the issue, no architectural unknowns.

Signals:
- Mechanical refactor with known file splits
- LOC targets already specified (formatting ~100, audio ~120, adapter ≤300)
- Same pattern already executed for Discord adapter
- No new abstractions or cross-domain impact
