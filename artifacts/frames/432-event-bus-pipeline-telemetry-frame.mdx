## Problem

After #431 introduced a composable middleware pipeline, each stage has well-defined entry/exit points — but no way to observe what happens at those seams. Debugging "why was this message dropped?" requires grepping raw logs with no structured trace. Pipeline metrics (per-stage latency, drop rates, throughput) don't exist. Plugins have no hook points to react to pipeline decisions without coupling to internal middleware code.

An EventBus existed before (PR #260, issue #44) and was removed in the simplification audit as a "dead abstraction" — it had no real consumers beyond logging. This time, the bus is justified by **concrete consumers**: audit logger, pipeline metrics, plugin hooks, real-time dashboard feed, and trace ID propagation.

## Who

- **Primary:** Developers debugging message routing — need structured audit trail to retrace pipeline decisions without grepping logs.
- **Secondary:** Plugin authors who want to react to pipeline events (`ROUTING_RESOLVED`, `LLM_SUBMITTED`) without touching middleware internals. Future consumers: real-time dashboard (#337), observability stack (#274), trace propagation (#270).

## Constraints

- **Fire-and-forget only** — the bus is for side-effects. Consumers cannot block, short-circuit, or modify messages. The middleware pipeline (#431) stays sequential and drives all control flow.
- **No global singleton** — bus is injected via constructor (lesson learned from ADR-022's removed EventBus).
- **Backpressure via drop** — per-subscriber queues with `put_nowait`; slow consumers lose telemetry events, never block the pipeline.
- **Dependency: #431 closed** — emission points live in the middleware runner, which is now in place.
- **First consumer ships with the bus** — audit logger must be in the same PR to avoid recreating a dead abstraction.

## Out of Scope

- **Control-flow event bus** — no migration of routing decisions to events. The exploration doc (`artifacts/explorations/event-bus-refactoring.md`) concluded this is not recommended.
- **Persistent event storage** — events are in-memory, fire-and-forget. Durable audit log is a separate concern.
- **Plugin registration API** — this issue provides the subscribe mechanism; a plugin SDK or discovery system is future work.
- **Metrics export** — Prometheus/StatsD export is tracked under #274, not here.

## Complexity

**Tier: F-full** — Multi-domain feature touching hub core (event definitions, bus), middleware runner (emission points), bootstrap wiring, and a first consumer. While individual components are small (~30-40 LOC each), the integration surface spans pipeline internals, bootstrap lifecycle, and establishes a new extension pattern for the project.

Signals observed:
- New architectural pattern (event bus as extension point)
- Multiple new files (events, bus, consumer, tests)
- Cross-cutting: hub core + middleware + bootstrap wiring
- Historical context requires careful scoping (prior removal of same pattern)
- Establishes convention for future consumers (plugins, dashboard, metrics)
