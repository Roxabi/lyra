## Problem

When the hub crashes (or hits an unhandled exception) after publishing `stream_start` and one or more chunks but before `stream_end`, the adapter's `_drain_stream` coroutine blocks on `q.get()` indefinitely. Although a 120-second timeout exists on the queue read (line 235 of `nats_outbound_listener.py`), the adapter has no way to distinguish "hub is slow" from "hub is dead" — it always waits the full timeout before giving up.

Meanwhile, the stream's cache entry (`_cache`), queue (`_stream_queues`), task (`_stream_tasks`), and outbound metadata (`_stream_outbound`) remain allocated with no cleanup path until the timeout fires. If the hub restarts and the user retries, duplicate or stale state can accumulate.

**Observable impact:**
- End users see a frozen response for up to 2 minutes before the timeout kicks in.
- Adapter memory leaks one cache + queue + task set per orphaned stream.
- No error feedback reaches the user — the stream just silently dies.

## Who

- **Primary:** Adapter processes (Telegram, Discord) — they hold the blocked coroutines and leaked resources.
- **Secondary:** End users — they experience a hung response with no feedback.

## Constraints

- Must introduce a new `stream_error` envelope type without breaking existing `stream_start` / chunk / `stream_end` flow.
- Hub must track active streams so it can publish `stream_error` on graceful shutdown (SIGTERM/SIGINT).
- Adapter must handle `stream_error` arriving at any point during `_drain_stream` — before first chunk, mid-stream, or after all chunks but before `stream_end`.
- The 120s timeout in `_drain_stream` remains as defense-in-depth for ungraceful crashes (kill -9, network partition).
- Must not break #529 (outbound listener extraction) or #530 (message versioning) — these are independent.

## Out of Scope

- JetStream / persistent streams — this is a fire-and-forget recovery signal, not durable messaging.
- Automatic retry of failed streams from the hub side — that's a separate concern.
- Changes to the adapter's `send_streaming()` contract — the error handling stays internal to `_drain_stream`.

## Complexity

**Tier: F-lite** — Single domain (NATS), clear scope, ~4 files touched.
