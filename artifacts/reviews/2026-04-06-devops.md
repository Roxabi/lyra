## Deploy SHA Cache Fix (817fd9e)

### What was the bug

Timer fires every ~70s. Sequence before fix:

```
fetch → LYRA_LOCAL != LYRA_REMOTE → pull → uv sync → pytest FAIL
→ git reset --hard LYRA_LOCAL → uv sync → exit 1
→ timer fires again → LYRA_LOCAL still old, LYRA_REMOTE still same bad SHA
→ repeat × 1390+
```

The script never remembered it had already tried and failed that remote SHA.

### Is the fix correct

Yes. Logic is correct:

```
LYRA_REMOTE computed from origin/staging before pull
→ on pytest failure: echo LYRA_REMOTE >> FAIL_FILE
→ on next run: grep -Fxq LYRA_REMOTE FAIL_FILE → skip silently
→ when staging moves forward: new LYRA_REMOTE != cached SHA → skip not triggered → normal deploy
```

The `grep -Fxq` uses `-x` (whole-line match) and `-F` (literal, not regex) — correct for 40-char hex SHAs.

### Directory creation

`mkdir -p "$(dirname "$LOG_FILE")"` creates `~/.local/state/lyra/logs/` which creates `~/.local/state/lyra/` as a side effect. `FAIL_FILE` appends to `~/.local/state/lyra/deploy_failed_shas.txt` — its parent directory is guaranteed to exist before first write. Safe.

### Risk: stale SHA cached incorrectly

**Not possible.** The SHA written is `LYRA_REMOTE` — the remote SHA at the time of the failed test. The script only skips when `LYRA_REMOTE` (new fetch) matches a cached entry. If the remote SHA changes (new commit pushed to staging), `grep -Fxq` won't match and deploy runs normally.

### Risk: FAIL_FILE growth

**WARN**: FAIL_FILE is append-only with no truncation or pruning logic. In a project with frequent broken staging pushes, this file grows without bound. Currently low-risk (each SHA is 41 bytes including newline), but no operator documentation exists for when to prune it besides "delete the file to force a retry."

The comment says "Cleared automatically when staging moves forward" — this is **inaccurate**. The file is NOT cleared when staging advances; old bad SHAs simply stop matching. The comment should say "entries for old SHAs are silently skipped as staging moves forward."

## Startup Ordering + Readiness Probe (453b8fa)

### Supervisor priority ordering

| Program | priority | startsecs |
|---------|----------|-----------|
| `lyra_hub` | 100 | 10s |
| `lyra_telegram` | 200 | 5s |
| `lyra_discord` | 200 | 5s |

Supervisor `priority` controls start order within a `supervisorctl start all` call — lower priority starts first. This is correct: hub at 100 starts before adapters at 200.

**Important caveat**: supervisor `priority` does NOT enforce a hard dependency. It controls ordering within a batch start, not blocking. Adapters can begin their startup before hub reaches RUNNING state. The NATS readiness probe (`wait_for_hub`) is the actual guard that prevents message loss.

### Readiness probe correctness

Hub wiring order in `hub_standalone.py` (lines 430–437):
1. `await hub.inbound_bus.start()` — subscribes to `lyra.inbound.*`
2. `await legacy_audio_handler.start()` — subscribes to audio subjects
3. `for d in dispatchers: await d.start()` — outbound dispatchers ready
4. `readiness_sub = await start_readiness_responder(nc, [hub.inbound_bus])` — probe responder installed

The probe responder is installed **after** all subscriptions and dispatchers are live. This is the correct ordering — adapters that receive a readiness reply know the hub is fully wired.

In `adapter_standalone.py`:
- Telegram: `wait_for_hub(nc)` called at line 131 — **after** all `astart()` calls, before `poll_tasks` are created. Correct.
- Discord: `wait_for_hub(nc)` called at line 267 — same pattern. Correct.

### Timeout behavior

`PROBE_TIMEOUT_S = 30.0`, `PROBE_INTERVAL_S = 0.5` (readiness.py lines 24–25).

On timeout: adapter logs WARNING and **proceeds** (graceful degradation, line 103–108). This is intentional — adapters never hard-block on hub unreachability.

**INFO**: Hub `startsecs=10` means supervisor considers hub "started" after 10s of continuous uptime. The readiness probe timeout is 30s. Gap: adapters can start probing before hub even reaches `startsecs`. The 30s window covers the 10s supervisor warm-up plus ~20s of hub wiring. Adequate for current startup times, but if hub wiring ever slows (large agent DB, slow NATS connect), the 30s timeout could expire before the hub is ready, and adapters would start without hub confirmation. Not currently a problem.

### `depends_on` directives

Supervisor does not support `depends_on`. The priority mechanism combined with the NATS readiness probe is the correct substitute for this deployment.

### Documentation for ops

The commit message documents the two-layer mechanism clearly. No runbook update was made to `docs/ARCHITECTURE.md` or any ops runbook. **WARN**: operators restarting services manually (e.g., `supervisorctl restart lyra_telegram`) will not trigger the readiness probe — the probe only runs at adapter bootstrap. This is acceptable since manual restart implies hub is already running.

## Embedded NATS Auto-Start (746b4af)

### Is NATS started by the hub process

Only in **unified mode** (`lyra start` / `_bootstrap_unified`). The `EmbeddedNats` class is instantiated inside `ensure_nats()` in `embedded_nats.py`, which is called from `unified.py`. The standalone hub (`_bootstrap_hub_standalone`) does NOT use `ensure_nats()` — it reads `NATS_URL` directly and exits if unset (hub_standalone.py line 146–151).

**Production three-process mode is unaffected** — hub and adapters require an external NATS_URL and will exit if it is missing. The embedded NATS is development/zero-config only.

### Process model impact

Three-process mode (production):
```
lyra.service → supervisord → lyra_hub (requires NATS_URL)
                           → lyra_telegram (requires NATS_URL)
                           → lyra_discord (requires NATS_URL)
```
NATS is provided externally. `lyra.service` declares `Requires=nats.service` (deploy/lyra.service line 5) — correct.

Unified mode (development):
```
lyra start → _bootstrap_unified → ensure_nats()
                                   → EmbeddedNats.start() if no NATS_URL
                                   → single process runs hub + adapters
```

### NATS shutdown in unified mode

`EmbeddedNats.stop()` is called in `ensure_nats()`'s exception path (line 181). In unified mode, `nc.close()` is in a `finally` block. The `EmbeddedNats` stop is registered via `atexit._kill_sync` as orphan protection. On clean exit: `stop()` is called explicitly. On crash/kill: atexit handler runs `_kill_sync()` (synchronous `process.kill()`). This is correct.

### Port conflict risk

`EmbeddedNats` binds to `127.0.0.1:4222` with `--no_auth`. Risk: if a system NATS is already running on 4222, `wait_ready()` will succeed on the existing server (TCP connect succeeds), but the process started by `asyncio.create_subprocess_exec` will have already exited with a "port in use" error. The `returncode` check at the top of the `wait_ready` poll loop (line 77) catches this: it reads stderr and raises `RuntimeError`. The error is surfaced via `sys.exit()` in `ensure_nats()` (line 166). Handled correctly.

The binary is pinned to `127.0.0.1` (`-a 127.0.0.1`) — no external exposure.

### `nats-server` binary availability

If `nats-server` is not on PATH, `FileNotFoundError` is raised with a helpful install message (`make nats-install`). This only affects `lyra start` (unified mode). Production is unaffected.

## Monitoring and Health Checks

### /health endpoint

- `GET /health` — unauthenticated, returns `{"ok": true}`. Used by deploy script's FATAL/BACKOFF check indirectly (via supervisorctl status, not HTTP).
- `GET /health/detail` — bearer token auth via `~/.lyra/secrets/health_secret`. Now includes `buses` field (commit 453b8fa) surfacing `inbound_bus.subscription_count`.

### Deploy script health gate

```bash
# deploy.sh lines 110-118
for i in $(seq 1 12); do
    sleep 5
    FAILED=$("$SCTL" status 2>&1 | grep -c "FATAL\|BACKOFF" || true)
    if [ "$FAILED" -eq 0 ]; then HEALTHY=true; break; fi
done
```

Waits up to 60s (12 × 5s) for all programs to exit FATAL/BACKOFF. This is a supervisor-level check, not an HTTP health check. It does not verify that the readiness probe has completed or that the hub is accepting messages — only that supervisor considers processes "running."

**Gap**: After deploy restart, supervisor restarts hub first (priority 100), then adapters. Adapters have `startsecs=5`. The deploy health gate starts checking after the restart commands return — but `supervisorctl restart` is async for the start phase. The 60s gate is generous enough to cover both `startsecs` windows (10s hub + 5s adapters) plus the 30s readiness probe worst case. Adequate.

### lyra-monitor

`deploy/lyra-monitor.service` and `deploy/lyra-monitor.timer` exist. No changes to these files in the reviewed commits. Monitor is unaffected.

## Rollback Safety

### Manual rollback

`git reset --hard <prev_sha>` + `uv sync` + `supervisorctl restart lyra_hub lyra_telegram lyra_discord` — unchanged behavior.

### Automated rollback (deploy.sh)

On test failure:
1. `LYRA_REMOTE` written to `FAIL_FILE`
2. `git reset --hard LYRA_LOCAL` — reverts to pre-pull state
3. `uv sync --all-extras --frozen` — restores deps for old SHA
4. `exit 1` — deploy script exits without restarting services

**Correct**: services are never restarted on a failed deploy, so production stays on the last good SHA throughout.

**Risk**: if `uv sync` after rollback fails (e.g., network timeout), services continue running with potentially mismatched deps for the brief window between `git pull` and `git reset --hard`. This is a pre-existing risk, not introduced by these commits.

### Embedded NATS rollback

If `lyra start` (unified mode) is rolled back from a version that had embedded NATS to one that did not, the `nats-server` binary is still present on PATH — no cleanup needed. No rollback concern.
