Voice overview

Lyra integrates with voiceCLI — a separate project that runs TTS (text-to-speech) and STT (speech-to-text) daemons. When voice is enabled, Lyra can receive voice messages from Telegram and respond with audio.

Voice requires voiceCLI daemons to be running. See ~/projects/voiceCLI and the supervisor pattern in lyra-stack/.

Per-agent voice config (voice_json) v10

As of v10, TTS and STT configuration are merged into a single voice_json column. The old tts_json and stt_json columns are retained for migration compatibility but the runtime uses voice_json.

jsonvoice_json structure
{
  "tts": {
    "voice": "en_US-lessac-medium",
    "speed": 1.0
  },
  "stt": {
    "model": "whisper-base",
    "language": "en"
  }
}

The fallback_language column is wired through to both TTS synthesize() and STT language_fallback. Edit via lyra agent edit <name>.

Setup

  1. Start voiceCLI daemons
    bash
    make tts    # start voicecli_tts
make stt    # start voicecli_stt
  2. Enable voice in config.toml
    toml
    [voice]
enabled = true
tts_url = "http://localhost:8020"
stt_url = "http://localhost:8021"
  3. Configure voice_json on the agent

    Use lyra agent edit <name> to set the voice_json field and optionally fallback_language.

  4. Reload Lyra
    bash
    make lyra reload

Voice commands in chat

Command / ActionEffect
/voice onEnable voice responses for this session
/voice offDisable voice responses (text only)
Send voice messageLyra transcribes with STT, processes, and responds with TTS audio

Architecture

Telegram voice msg → STT daemon → text transcript → AgentBase → text response → TTS daemon → audio file
Known issue: audio_loop contains blocking I/O calls in an async context. This can cause latency spikes under concurrent load. A fix is tracked as a high-priority item.
