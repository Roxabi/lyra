## Problem

`nats_channel_proxy.py` (hub) and `nats_outbound_listener.py` (adapter) are an implicit semantic pair ‚Äî they must agree on event types and wire encoding, but have no shared source of truth.

The current hub-side serializer uses an inline `isinstance` chain that only handles `TextRenderEvent` and `tool_summary`. Adding any new `RenderEvent` subclass compiles fine but silently drops events on the adapter side. This was the root cause of tool-use embeds and final messages being dropped on Discord in NATS mode.

**Why now:** Discovered 2026-04-01 during Discord bug investigation. Part of #561 (NATS hardening).

## Who

- **Primary:** Hub ‚Üî adapter streaming pipeline ‚Äî `NatsChannelProxy` and `NatsOutboundListener`
- **Secondary:** Discord and Telegram users in NATS mode (affected by dropped events)

## Constraints

- 3 files maximum ‚Äî low risk refactor
- Must not change `StreamProcessor`, `RenderEvent` types, or adapter `send_streaming()`
- Must not affect non-NATS direct path
- Existing hotfix in `nats_outbound_listener.py` is correct ‚Äî make it permanent and safe

## Out of Scope

- New `RenderEvent` types
- Changes to `StreamProcessor`
- Changes to adapter `send_streaming()`
- Non-NATS direct path modifications
- Wire protocol changes
