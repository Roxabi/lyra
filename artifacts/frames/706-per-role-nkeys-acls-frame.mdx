## Problem

The NATS server enforces nkey authentication (#523) and each Lyra process ships with its own seed (#563), but `auth.conf` only authenticates them — there are no subject-level permissions. Any authenticated identity can publish or subscribe to any subject. A buggy or malicious process (or a plugin that inherits `nats_connect()` via the upcoming `roxabi-nats` SDK, ADR-045) can impersonate the hub by publishing to `lyra.inbound.*`, or snoop adapter traffic by subscribing to `lyra.outbound.>`. Security review flagged this as Broken Access Control (OWASP 5), Medium severity.

Now is the right time: ADR-045 (SDK extraction) is blocked on this — once `roxabi-nats` is consumable by plugins, plugin code will hold an nkey, and without ACLs that nkey is equivalent to "full bus access." We need the authorization block in place before plugin consumers land.

Observable impact today: zero in production (no malicious plugin exists yet). Observable impact the day a plugin lands without this: any plugin process can forge adapter inbound traffic to the hub.

## Who

- **Primary:** operator running Lyra on Machine 1 (`roxabituwer`) — must regenerate `auth.conf` and reload nats-server without dropping connections.
- **Secondary:**
  - Lyra processes (`lyra_hub`, `lyra_telegram`, `lyra_discord`, `lyra_tts`, `lyra_stt`, `lyra_llm_worker`, `lyra_monitor`) — each identity gains a subject allowlist; any missed subject breaks the process silently.
  - Future `roxabi-nats` plugin consumers (ADR-045) — inherit the plugin ACL shape documented here when they land.

## Constraints

- **Identity alignment.** Existing nkey identities (`hub`, `llm-worker`, `monitor`, `tts-adapter`, `stt-adapter`) and quadlet-expected-but-not-yet-generated identities (`telegram-adapter`, `discord-adapter`) must all be covered. Issue body's `tts-worker`/`stt-worker` map to `tts-adapter`/`stt-adapter`.
- **`gen-nkeys.sh` extension.** Script currently generates 5 seeds; must extend to 7 (add `telegram-adapter`, `discord-adapter`) and emit per-user `permissions { publish, subscribe }` blocks in `auth.conf`.
- **Zero-downtime reload.** `nats-server --signal reload` must pick up the new ACLs without dropping active client connections. Falling back to `systemctl restart` drops all connections and is not acceptable for production.
- **Fail-loudly on ACL miss.** NATS silently drops denied publishes and returns nothing from denied subscriptions. We need an explicit subject audit (grep all `publish()` / `subscribe()` call sites across `lyra`, `voicecli`, `roxabi-intel`) mapped to roles in the spec before rollout — treat the subject → role matrix as a deliverable.
- **Local dev unaffected.** `nats-local.conf` (no auth) stays in place for local dev; ACLs only apply when `auth.conf` is loaded (production path).
- **Existing nkey helper is the only client entry point.** `nats_connect()` already routes every Python client; no change needed on the client side — this is a server-config change + seed-generation change only.

## Out of Scope

- **Plugin role ACL implementation.** Plugin namespace (`lyra.plugin.<name>.>` shape TBD) is deferred until `roxabi-nats` SDK extraction (ADR-045) lands. Frame/spec will document the expected shape and where to wire it in, but no plugin nkey is generated in this issue.
- **JWT / account-based authz.** We stay on nkey + inline `authorization { users }` ACLs; NATS accounts/JWT are a bigger architectural shift, not in scope here.
- **Subject schema refactor.** Whatever subject patterns exist today become the allowlist. If the audit surfaces overlap/ambiguity (e.g. `lyra.voice.*.request` shared by hub and workers), we document it, not refactor it.
- **TLS / mTLS.** Separate track; not bundled.
- **Rotation tooling.** Key rotation stays on `gen-nkeys.sh` with its existing "delete and regenerate" workflow; no incremental key-add support added here.

## Complexity

**Tier: F-lite** — single-domain change (NATS server config + `gen-nkeys.sh` extension + one deploy-docs update). Scope is focused (≤10 files), but security-sensitive with silent-failure risk → frame + spec + plan warranted.

Signals observed:
- Single domain: NATS deploy + auth config.
- Clear scope: 6 identities (7 counting future plugin) × 2 ACL sets (publish, subscribe) = bounded matrix.
- Prior art exists (#523 client wiring, #563 seed generation) — same patterns extended, no new architecture.
- Risk: silent denial → adapters go dark on unmapped subjects. Mitigated by pre-rollout subject audit (spec deliverable) and `nats-server` reload logs as the verify step.
