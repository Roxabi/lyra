## Problem

Supervisor starts `lyra_hub`, `lyra_telegram`, and `lyra_discord` concurrently with no ordering guarantees. Core NATS uses at-most-once delivery — messages published before the other side subscribes are silently dropped.

Two race windows exist:

1. **Adapter starts before hub** — user messages arrive on `lyra.inbound.*` before the hub's `NatsBus` has called `start()` and created subscriptions. Inbound messages are lost.
2. **Hub starts before adapter** — hub publishes responses on `lyra.outbound.*` before the adapter's `NatsOutboundListener` has subscribed. Outbound replies are lost.

Today there is zero readiness signaling between processes. The health endpoint (`/health`) is a static `{"ok": true}` — it does not reflect subscription readiness. Supervisor configs have no `priority` directives.

The impact is worst during restarts (deploys, crashes) when users are actively chatting — messages vanish with no error or retry.

## Who

- **Primary:** End users — messages sent during the startup window disappear silently.
- **Secondary:** Operators — silent message loss is hard to detect and debug; no metric or log signals the gap.

## Constraints

- Core NATS only (at-most-once). JetStream migration is tracked separately in #460.
- Three-process supervisor model must be preserved (hub + telegram + discord as separate programs).
- Unified mode (`lyra start`) is less affected — sequential in-process wiring — but should still benefit from readiness checks.
- Solution must not add hard startup dependencies that prevent individual process restarts.
- Embedded NATS auto-start (#520) may interact — the embedded server must also be ready before any process subscribes.

## Out of Scope

- JetStream / durable subscriptions (#460) — that's a larger architectural change.
- Changing the three-process supervisor model.
- Adapter-to-adapter direct communication.
- Message replay or guaranteed delivery beyond the startup window.
