## Problem

NATS subscriptions in `NatsBus` (hub-side inbound) and `NatsOutboundListener` (adapter-side outbound) do not use queue groups. During a rolling deploy or supervisor restart, two instances of the same process briefly coexist — both receive every message on the shared subject, causing duplicate delivery (double sends to users).

The TTS and STT standalone adapters already use `queue=QUEUE_GROUP` correctly. The core hub and channel adapters do not.

## Who

- **Primary:** End users (Telegram/Discord) — receive duplicate bot messages during deploys
- **Secondary:** Operators — must be aware of the deploy-time duplicate window; currently no mitigation

## Constraints

- NATS queue groups are the idiomatic solution — one subscriber per group receives each message
- Group name must be deterministic and consistent across restarts (not random)
- Must not break single-instance mode (queue group of 1 behaves identically to no group)
- Existing dedup cache in `NatsOutboundListener` is a safety net, not a fix — it only catches exact duplicates within the cache window

## Out of Scope

- Exactly-once delivery guarantees (JetStream) — queue groups provide at-most-once-per-group, sufficient here
- Multi-machine horizontal scaling — this is about rolling restarts on a single machine
- Changes to TTS/STT adapters — they already have queue groups

## Complexity

**Tier: F-lite** — Clear scope, ~4 files touched (2 subscription sites + tests), well-understood NATS pattern already used elsewhere in the codebase.

Signals:
- Pattern already established in `stt_adapter_standalone.py` and `tts_adapter_standalone.py`
- Two primary subscription sites to modify
- No architectural changes — adding a parameter to existing `.subscribe()` calls
- Needs integration testing with supervisor restart scenario
