## Problem

User-facing strings are scattered across four source files (`telegram.py`, `discord.py`, `message.py`, `command_router.py`) with no abstraction layer. The same strings are duplicated between adapters, and there is no mechanism for internationalization or per-adapter message formatting. Changing any message requires a code edit, a review cycle, and a redeploy — with no guarantee that all copies stay in sync.

The issue is structural: Lyra has no centralized message registry. Each adapter owns its own strings, leading to drift as the adapter count grows. A 2ndBrain reference (`messages.yaml`) already documents the full vocabulary of required messages, confirming the scope is well-understood and bounded.

The immediate trigger is Phase 0 (Bot core parity, #101): before adding new capabilities, the existing hardcoded strings must be extracted into a maintainable system that supports at least two languages (EN, FR) and per-adapter formatting variants.

## Who

- **Primary:** Developers maintaining Lyra — they bear the cost of duplicated strings, missed sync, and manual translation work.
- **Secondary:** End users of Lyra bots (Telegram, Discord) who receive messages; multilingual users currently always see English regardless of preferred language.

## Constraints

- Must use TOML (matches Lyra's existing config philosophy — `pyproject.toml`, agent TOML files).
- Language selection must be configurable per agent via TOML (`[i18n] default_language = "fr"`), not per user.
- `MessageManager` must be injectable into adapters and the command router at Hub startup time.
- Python standard library `tomllib` (≥3.11) is the target parser — no extra dependencies.
- Part of Epic #101 (Phase 0) — must not introduce breaking changes to the adapter interface.

## Out of Scope

- Languages beyond English and French (EN + FR only for this issue).
- Per-user language preference (language is config-level, not session-level).
- Dynamic language switching at runtime (language is set at Hub startup; restart required to change).
- Pluralization rules or ICU message format — simple `str.format_map` / `.format(**kwargs)` substitution only.
- A UI or admin interface for managing messages.

## Complexity

**Tier: F-full** — Introduces a new configuration abstraction layer (TOML message registry + `MessageManager` class) that cuts across the adapter boundary, plus a new injection pattern at Hub startup. The design is clear, but the new patterns warrant deeper analysis of how `MessageManager` integrates with the existing Hub/adapter wiring and what risks arise from the config loading path.

Signals observed:
- New pattern: `MessageManager` injection at Hub startup (no existing precedent in codebase).
- New config layer: `messages.toml` alongside existing TOML agent configs — load ordering and error handling need analysis.
- Cross-cutting: touches Telegram adapter, Discord adapter, command router, and Hub init.
- Single domain: message templating (¬multi-domain).
- Scope fully specified: acceptance criteria + TOML schema + class API already defined in issue.
