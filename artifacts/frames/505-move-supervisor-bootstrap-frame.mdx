## Problem

The supervisord bootstrap (config, start/ctl scripts, conf.d symlinks) currently lives in `lyra-stack/`, a separate orchestration repo. Since Lyra is the primary service and the hub of the supervisor tree, the process manager config should be co-located with it. This simplifies deployment — one repo owns its own daemon lifecycle — and removes the indirection of a separate stack repo for what is fundamentally Lyra's concern.

## Who

- **Primary:** Mickael — deploying and managing Lyra services on local and production machines
- **Secondary:** Any future contributor who needs to understand where supervisor config lives

## Constraints

- Must remain compatible with the existing systemd user unit (`lyra-stack.service` → renamed or re-pointed)
- All current `make` targets (`make ps`, `make lyra`, `make tts`, `make stt`, etc.) must keep working — from both `lyra/` and `lyra-stack/`
- Production (`roxabituwer`) runs 24/7 — migration must be atomic (no partial state where supervisor can't find configs)
- Other projects (`voiceCLI`, `2ndBrain`) currently register their conf.d files via symlinks into `lyra-stack/conf.d/` — their registration path changes

## Out of Scope

- Changing supervisor program definitions (what gets supervised stays the same)
- Migrating away from supervisord to another process manager
- Changing the systemd linger/user-unit pattern
- Production deployment (this PR targets local; production deploy is a follow-up)

## Complexity

**Tier: F-lite** — single domain (infra/deploy), well-defined file moves, no new architecture or unknowns.

- ~8-10 files touched across lyra + lyra-stack
- Known patterns (supervisord, systemd, Makefile)
- Single domain (deployment/infra)
- 0 unknowns — all paths and configs are documented
- Risk: medium (breaking daemon startup if paths are wrong) but easily testable
