## Problem

Lyra has no user authentication beyond platform trust. Anyone who can message the bot on Telegram or Discord gets full access — there is no way to invite specific users, revoke access, or distinguish admin from regular user. This blocks multi-user support entirely.

The 2ndBrain codebase solved this with a production-grade pairing system (8-char codes, SHA-256 hashing, rate limiting, session expiry). Lyra needs the same security properties, adapted to its hub-and-spoke architecture — specifically its TOML config pattern and SQLite-backed storage via `roxabi-memory`.

## Who

- **Primary:** Mickael (admin) — needs to control who can interact with Lyra across channels, generate invite codes, revoke access.
- **Secondary:** Invited users — need a simple `/join <CODE>` flow to pair with the bot on any supported channel.

## Constraints

- **Storage:** Must use `roxabi-memory` (SQLite + namespace scoping), not JSON files. Pairing codes and sessions stored as memory entries with appropriate category/type.
- **Config:** TOML-based (`[pairing]` section in `lyra.toml`), following the existing `[admin]` and `[circuit_breaker]` pattern.
- **Admin identity:** Reuse the existing `admin_user_ids` set from `lyra.toml` `[admin]` section — no new admin concept.
- **Identity format:** Unified `{platform}:user:{id}` keys (e.g., `telegram:user:123456`, `discord:user:789012`).
- **Security:** SHA-256 hashed codes (never plaintext), rate limiting on `/join`, 1h TTL on codes, 30-day session expiry.
- **Async:** All operations must be async-compatible (Lyra is fully asyncio).
- **Parent epic:** Part of #101 (Phase 0 — Bot core parity).

## Out of Scope

- Role-based permissions beyond admin/user (no granular roles)
- OAuth or external identity providers
- Cross-hub / multi-hub pairing (single hub only)
- Persistent user profiles (pairing = access control, not profile management)

## Complexity

**Tier: F-lite** — Single domain (backend auth), clear design from 2ndBrain reference, well-defined interface (`PairingManager`), no new architectural patterns beyond adapting existing storage/config conventions. ~4-5 files: core pairing module, config integration, command handlers, tests.

Signals observed:
- Single domain: backend only, no frontend/UI changes
- Clear scope: 2ndBrain reference provides proven design
- Contained integration: hooks into existing `CommandRouter` and `admin_user_ids`
- No unknowns: storage and config patterns well-established in codebase
