## Problem

The Lyra hub has a working asyncio bus but no live adapters connected to it. Two channels need to
be wired in parallel: Telegram (existing 2ndBrain adapter, must not regress) and Discord (new).

The core challenge is not simply connecting two libraries — it is defining a **unified routing
model** that correctly handles a `channel × bot_id` matrix. Multiple bot identities can send and
receive on multiple channels simultaneously. The message model must capture enough context to
route correctly and to attribute every message to its sender + receiving bot + platform context.

Platform context types differ significantly:
- **Telegram**: conversations (private DMs, group chats, topics in supergroups)
- **Discord**: channels, threads, forum posts, voice text channels — each with distinct IDs and
  parent relationships

Without a unified `InboundMessage` / `OutboundMessage` contract, every skill and handler will
need platform-specific branches, making the system brittle as new adapters are added.

## Who

- **Primary:** Mickael (operator) — wires bots, observes routing, debugs message flow
- **Secondary:** End users of the bots — Telegram users, Discord server members. They experience
  the bot's response latency and correctness, not the internals.

## Constraints

- **Channel × bot_id matrix**: a single hub instance may manage multiple bot tokens, each
  registered on different channels/servers. Routing must key on `(bot_id, channel_id, platform)`.
- **Attribution required**: every message must carry sender identity (user), receiving bot identity,
  and platform context (thread, channel, topic). Needed for logging, skill dispatch, and future
  per-bot personas.
- **Platform context heterogeneity**: Telegram contexts (conversation, supergroup topic) and
  Discord contexts (channel, thread, forum) must map to a common abstraction without losing
  platform-specific metadata.
- **asyncio only**: hub is pure asyncio. Adapters must be fully non-blocking. No threads, no
  blocking I/O in the event loop.
- **Single process**: both adapters run in the same Python process as the hub. No microservices,
  no inter-process queues.
- **Zero regression on Telegram**: existing 2ndBrain Telegram adapter is live. The migration must
  be seamless — same bot behavior, no message loss.

## Security

The hub and message handler are an attack surface that must be hardened before real traffic:

- **Inbound origin verification**: Telegram webhooks must validate the `X-Telegram-Bot-Api-Secret-Token`
  header. Discord uses a gateway (persistent WebSocket) — bot token must never be logged or exposed.
- **Sender allowlist / denylist**: each bot should carry a configurable policy — who is allowed to
  trigger skills? At minimum a `user_id` allowlist per bot, checked before dispatch.
- **Message sanitization**: raw text from external users must be treated as untrusted input before
  being passed to skills or LLM prompts. No direct string interpolation.
- **Secret management**: bot tokens (`TELEGRAM_TOKEN`, `DISCORD_TOKEN`) sourced exclusively from
  environment variables or a secrets store — never hardcoded or logged.
- **Rate limiting**: per `(bot_id, user_id)` token bucket to prevent abuse and runaway LLM costs.
- **Hub isolation**: the internal asyncio bus must not be reachable from the network. Only adapter
  layer touches external protocols; hub is internal-only.

## Out of Scope

- Skill migration: existing Telegram skills stay as-is for this issue. Adapter wiring only.
- Auth / permission system: no role-based access control on Discord messages in this issue.
- Discord slash commands / interactions: message routing only for now.
- Multi-guild Discord: single server scope for this issue.
