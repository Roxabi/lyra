## nkey Authentication

### Key storage and gitignore

- Seeds live at `/etc/nats/nkeys/{hub,llm-worker,monitor}.seed` — outside the repo,
  owned `root:root 0600`, generated by `sudo deploy/nats/gen-nkeys.sh`.
- Only the generation script (`deploy/nats/gen-nkeys.sh`) and server config template
  (`deploy/nats/nats.conf`) are tracked in git. No seeds or `auth.conf` are committed.
- `.gitignore` has no explicit pattern for `*.seed` — but the files are generated
  into `/etc/nats/nkeys/`, which is never inside the repo tree, so gitignore coverage
  is moot in practice. Low residual risk.

### Server-side enforcement

- Production `nats.conf` includes `/etc/nats/nkeys/auth.conf` which lists the three
  user public keys. Any client without a matching nkey is rejected at the NATS
  handshake layer — enforcement is server-side.
- `max_connections: 20` limits blast radius from unexpected clients.
- TLS 1.3-only cipher suites are configured; mutual TLS (`ca_file`) is present,
  meaning the server also verifies client certificates. The combination of TLS + nkey
  provides two independent auth layers.

### Rotation story

- No key rotation tooling exists. `gen-nkeys.sh` is idempotent/skip-if-exists.
  To rotate: delete `/etc/nats/nkeys/`, re-run the script, redeploy.
- No documented rotation runbook or expiry policy. Acceptable for a personal project;
  worth noting for production hardening.

### Failure modes

- If `NATS_NKEY_SEED_PATH` is unset, `nats_connect()` connects **without authentication**.
  This is the explicit dev mode. Documented in `.env.example` and `DEPLOYMENT.md`.
  The local config (`nats-local.conf`) also has no auth. Risk is scoped to dev/local.
- If the seed file is missing, unreadable, or empty, `sys.exit()` is called with a
  clear error — the process refuses to start rather than falling back to no-auth.
  This is the correct fail-closed behavior.
- `tts_adapter_standalone.py` and `stt_adapter_standalone.py` still call `nats.connect()`
  directly (bypassing `nats_connect()`). These processes connect without nkey auth even
  when `NATS_NKEY_SEED_PATH` is set. See Issue 1 below.

## platform_meta Sanitization

### Fields covered

Allowlist (`PLATFORM_META_ALLOWLIST`) contains 9 keys:
`guild_id`, `channel_id`, `message_id`, `thread_id`, `channel_type`,
`chat_id`, `topic_id`, `is_group`, `thread_session_id`.

All other keys (including `_`-prefixed keys like `_session_update_fn`) are stripped.
Stripping is logged at DEBUG level (key names only, not values — correct).

### Prompt injection risk

`platform_meta` values do **not** reach LLM prompt construction. Traced the full
pipeline:

- `thread_session_id` → session scope validation → `resume_session()` (DB lookup) — never
  interpolated into text.
- `message_id` → used as `reply_to` integer in Telegram/Discord API calls — not rendered
  as text.
- `guild_id`, `channel_id`, `chat_id`, etc. → routing metadata only — not interpolated
  into agent prompts.
- `_session_update_fn` is a callable injected by adapters _before_ NATS publish and
  stripped by `sanitize_platform_meta` on inbound — correct design.

No prompt injection vector exists through platform_meta.

### Boundary placement

Sanitization is applied in `NatsBus.handler()` immediately after deserialization, before
the item reaches the staging queue. This is the correct boundary — the NATS trust boundary.
Any message arriving over NATS (including from a compromised adapter process) has its
platform_meta filtered before entering the hub.

The `_session_update_fn` callable is adapter-injected locally (same process) and therefore
never survives the NATS hop — the sanitizer's stripping of `_`-prefixed keys is a correct
defense-in-depth measure even though the callable could not arrive over NATS in practice.

### Schema validation

Fields are allowlist-filtered but **not** type-validated. Values for allowlisted keys
arrive as raw JSON deserialized types (`str`, `int`, `bool`, `None`). For example,
`thread_session_id` is a string read directly from NATS JSON — no length or format
check. See Issue 2 below.

## Trust Model

```
[Telegram/Discord platform] → adapter process → NATS (TLS 1.3 + nkey)
                                                        ↓
                                               NatsBus.handler()
                                               sanitize_platform_meta()   ← trust boundary
                                                        ↓
                                               hub pipeline
                                               scope validation (TurnStore)
                                                        ↓
                                               pool.resume_session()
```

Adapters are semi-trusted: they are first-party processes but communicate over a
network bus. The sanitizer enforces that platform_meta only carries what the hub
expects. The scope validation (commit 1f0c911) ensures `thread_session_id` cannot
be used to hijack a session belonging to a different pool.

### MEDIUM: TTS and STT adapters bypass nkey auth

`src/lyra/bootstrap/tts_adapter_standalone.py:73`
`src/lyra/bootstrap/stt_adapter_standalone.py:44`

Category: Broken Auth (OWASP A2)
Confidence: 88%

Exploit scenario: When `NATS_NKEY_SEED_PATH` is set on the hub and platform adapters,
the NATS server requires nkey authentication. The TTS and STT adapter processes call
`nats.connect(url)` directly, bypassing `nats_connect()`. If the NATS server's
`auth.conf` does not include entries for these processes (the generated `auth.conf`
lists only `hub`, `llm-worker`, and `monitor`), the TTS/STT adapters will fail to
connect or connect without auth. Conversely, if the server has no `authorization`
block for them, they connect as unauthenticated clients, violating the intended
authentication perimeter. An attacker who can inject a NATS client (e.g. via a
compromised LAN node) gains the same access as the TTS/STT processes.

Root cause: `tts_adapter_standalone.py` and `stt_adapter_standalone.py` import
`nats` directly instead of `lyra.nats.nats_connect`. These files predate the
`nats_connect()` helper and were not updated in commit 343ff69.

Remediation:
  1. Replace `import nats` + `await nats.connect(nats_url)` with
     `from lyra.nats import nats_connect` + `await nats_connect(nats_url)` in both
     files. Add `llm-worker` or dedicated seeds for these processes to `gen-nkeys.sh`.
     (recommended)
  2. Add a `NATS_NKEY_SEED_PATH` env var wired via supervisor conf.d for these
     programs and update their seed generation as a separate step.

### LOW: thread_session_id value not length- or format-validated

`src/lyra/nats/_sanitize.py:26–44`
`src/lyra/core/hub/message_pipeline.py:323`
`src/lyra/core/hub/middleware_submit.py:164`

Category: Injection / Input Validation (OWASP A3 / A8)
Confidence: 65%

Exploit scenario: A crafted NATS message with `thread_session_id` set to an
arbitrarily long string (e.g. 100 KB) passes the allowlist filter unchanged and is
passed directly to `TurnStore.get_session_pool_id()` as a SQL parameter. The
parameterized query (`SELECT ... WHERE session_id = ?`) prevents SQL injection, but
the unbounded string is stored in memory and included in log output. An attacker
with access to the NATS bus could cause amplified memory usage or log bloat across
multiple concurrent requests.

Root cause: `sanitize_platform_meta` is an allowlist filter, not a schema validator.
Values are passed through without type coercion or length capping.

Remediation:
  1. Add a value-shape check in `sanitize_platform_meta`: coerce known string fields
     to `str`, cap length at a reasonable bound (e.g. 256 chars), and strip or reject
     values that don't match. (recommended)
  2. Add a length check specifically in the `thread_session_id` consumer before the
     TurnStore lookup.
