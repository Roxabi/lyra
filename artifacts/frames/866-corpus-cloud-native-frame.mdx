## Problem

The corpus sync currently targets a local SQLite file (`~/.roxabi/corpus.db`) with scheduled polling. This creates:
- **Local-only access** — tools must run on same machine
- **No API** — direct DB access only
- **No web UI** — no frontend can consume the data
- **Manual sync** — cron/timer required

A cloud-native approach would enable:
- Real-time webhook updates (zero API quota)
- REST API for any consumer (CLI, web, other tools)
- Optional frontend for browsing/searching issues
- Access from any machine

## Who

- **Primary:** Mickael — wants the corpus accessible beyond `roxabituwer`
- **Secondary:** `roxabi-dashboard` (future) — would consume the API
- **Tertiary:** CI/CD pipelines — could query corpus for issue metadata

## Constraints

- **Free tier only** — no paid infrastructure
- **Simple auth** — leverage GitHub OAuth if possible
- **SQLite compatibility** — D1 is SQLite, should match existing schema
- **Low latency** — edge deployment for webhook responsiveness
- **No lock-in** — must be able to export/sync back to local if needed

## Out of Scope

- Migrating existing `dep-graph` tool immediately (can be gradual)
- Full `roxabi-dashboard` implementation (separate effort)
- Mobile app

## Open Questions

1. **Frontend location** — Cloudflare Pages vs local-only vs skip?

3. **Local sync strategy** — keep local corpus.db or go cloud-only?
4. **Write permissions** — webhook only, or allow user edits via API?
5. **Existing tools migration** — when/how do CLI tools switch to API?
