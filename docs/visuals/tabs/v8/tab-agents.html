ADR-029 — DB-first agent config. As of v8 the SQLite agents table is the single runtime source of truth. TOMLs are seed-only. lyra agent create writes to the DB directly — no TOML is created. Hot-reload uses updated_at polling instead of file mtime.

What is an agent?

An agent is a named configuration that binds a system prompt, a model, a set of tools, and optional persona to a conversational identity. When a bot receives a message, it looks up its assigned agent by name from the DB and passes the message to a Pool running that agent.

Agents are stateless singletons — their configuration is shared across all concurrent conversations. Conversation state (history, circuit) lives in the Pool, not the agent.

Agent config fields

FieldRequiredDescription
namereqUnique agent identifier (snake_case). Used in config.toml bot assignments.
modelreqLLM model string e.g. claude-sonnet-4-6
system_promptreqStatic system prompt injected into every conversation
toolsoptList of tool names to enable e.g. ["Bash", "Read", "Skill"]
personaoptName of a .persona.toml file to load (voice, style overrides)
max_turnsoptMax agentic turns before auto-stopping. Default: 10
workspacesoptNamed directories exposed as /workspace <key> commands (override cwd)
platformoptRestrict agent to one platform: telegram | discord
provideroptBackend: anthropic (default) | claude-cli

TOML files vs DB — what goes where

SourceRoleRead at runtime?
src/lyra/agents/*.tomlSeed files — versioned defaultsNo — only via lyra agent init
~/.lyra/auth.db (agents table)Runtime source of truthYes — on every message
*.persona.tomlVoice / style overlaysYes — loaded when agent reloads
DB takes precedence. Editing a TOML file after lyra agent init has no effect until you run lyra agent init --force. The running daemon reads from the DB only.

CLI reference

Creating agents v8 updated

bashDB-first (v8 — no TOML created)
lyra agent create my_agent \
  --model claude-sonnet-4-6 \
  --system-prompt "You are a helpful assistant." \
  --tools Read,Bash,Skill
No TOML, no init step. lyra agent create calls AgentStore.upsert() directly. The agent is immediately available to the daemon on the next message. You do not need to run lyra agent init afterward.

Seeding from TOML (first-time setup)

bash
lyra agent init              # seed DB from src/lyra/agents/*.toml (skips existing)
lyra agent init --force      # overwrite existing DB rows from TOMLs

Use init once on a new machine, or after updating seed TOMLs that you want to push to the DB.

Viewing agents

bash
lyra agent list              # all agents in DB
lyra agent show my_agent     # full config for one agent

Editing agents hot-reload

bash
lyra agent edit my_agent     # interactive editor, writes to DB

After saving, upsert() bumps the updated_at timestamp. The running daemon picks up the change on the next inbound message — no restart needed.

Assignment and deletion

bash
lyra agent assign my_agent --platform telegram --bot <bot_id>
lyra agent validate my_agent   # check config for errors
lyra agent delete my_agent     # refuses if a bot is still assigned

Hot-reload mechanics

On every inbound message, AgentBase._maybe_reload() runs:

  1. Fetch cached rowCalls AgentStore.get(name) — an O(1) dict lookup. No DB query; no file stat.
  2. Compare updated_atCompares row.updated_at (ISO-8601 string) to the locally cached value. If equal, nothing happens.
  3. Rebuild configIf changed, calls agent_row_to_config(row) to deserialize the DB row into an Agent dataclass. This is the only deserialization path at runtime.
  4. Reload personaIf the row has a persona field, load_persona(row.persona) reads the .persona.toml file and composes it into the new config.
Hot-reload is lazy and per-message — there is no background polling timer. At personal-scale message volumes, one O(1) dict lookup per message is negligible.
Persona-only changes don't trigger reload. If you edit a .persona.toml file on disk, the daemon won't pick it up automatically. You must also run lyra agent edit <name> (save any field) or lyra agent init --force <name> to bump updated_at and trigger the reload path.

Persona files

A persona file (*.persona.toml) overlays voice/style settings on top of the agent config. It is still file-based and is loaded when the agent reloads from the DB row.

tomlexample.persona.toml
[persona]
name = "Lyra"
voice = "warm, concise, British English"
avoid = ["em dash", "filler phrases"]
signature = ""   # optional sign-off

Persona files remain under src/lyra/personas/ and are not stored in the DB. They are referenced by name in the agent's persona field.
