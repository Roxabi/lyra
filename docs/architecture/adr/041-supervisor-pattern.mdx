## Context

We manage multiple daemon processes across several projects (lyra, voiceCLI, imageCLI, forge, idna, litellm). These need to run on two machines with different operational profiles:

| Machine | Role | Uptime | Supervisor Behavior |
|---------|------|--------|---------------------|
| ROXABITOWER (local) | Dev + AI workloads | On-demand | Lazy startup via `make` targets |
| roxabituwer (prod) | Always-on hub | 24/7 | Auto-start via systemd on boot |

Previously, we had two supervisor setups in lyra:
- `lyra/supervisor/` вЂ” configs + scripts (used via symlinks)
- `lyra/deploy/supervisor/` вЂ” standalone supervisord for prod

This caused confusion and duplicate supervisord processes running simultaneously.

---

### Key Principles

1. **Configs live in project repos** вЂ” Each project owns its `supervisor/conf.d/*.conf` files, version-controlled
2. **Hub aggregates via symlinks** вЂ” Local `~/projects/conf.d/` symlinks to project configs
3. **Prod runs standalone** вЂ” Production uses `lyra/deploy/supervisor/` directly, managed by systemd
4. **Cross-project services** вЂ” Services like litellm live in `~/.<service>/supervisor/`, symlinked to hub

---

## Consequences

### Positive

- **Single supervisord process** locally (no duplicates)
- **Clear ownership** вЂ” each project owns its configs
- **Lazy startup** вЂ” `make lyra` auto-starts supervisor if needed
- **Systemd integration** вЂ” prod auto-starts on boot via linger

### Negative

- **2ndBrain runs separate supervisor** вЂ” knowledge_bot has `autostart=true` in its own supervisord (acceptable for isolation)
- **litellm config not versioned** вЂ” lives in `~/.litellm/`, not in a repo (acceptable for infrastructure-only service)

### Mitigations

- 2ndBrain isolation is intentional вЂ” knowledge_bot has different lifecycle requirements
- litellm can be moved to a repo later if needed
