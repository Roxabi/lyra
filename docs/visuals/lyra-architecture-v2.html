Lyra — Architecture Deep Dive

Hub-and-spoke message flow, domain structure, data types & #293 refactoring target

Do LLM responses flow through InboundBus?

No. The InboundBus is strictly one-way: Adapter → per-platform Queue → staging Queue → Hub.run(). LLM responses take a completely separate return path: Agent → Pool → Hub.dispatch_response() → OutboundDispatcher → Adapter.send(). Each adapter has its own inbound queue AND its own OutboundDispatcher — they never share a bus. dispatch_response() is not a bus — it's a direct enqueue into the per-adapter outbound queue with circuit breaker + retry.

Inbound Path (Adapter → Hub → Pool → Agent)
  1. 1 Platform event (Telegram update / Discord on_message)
  2. 2 Adapter normalizes → InboundMessage (with RoutingContext)
  3. 3 inbound_bus.put(platform, msg) → per-platform bounded queue
  4. 4 Feeder task drains into staging queue (bounded 500)
  5. 5 Hub.run() awaits inbound_bus.get() — single consumer
  6. 6 MessagePipeline.process(msg): validate → rate limit → resolve binding → check commands
  7. 7 pool.submit(msg) → debounce → agent.process(msg, pool)
  8. 8 Agent calls LLM (Anthropic SDK or Claude CLI subprocess)
Outbound Path (Agent → Hub → Dispatcher → Adapter)
  1. 1 Agent yields Response or AsyncIterator[str]
  2. 2 Pool calls ctx.dispatch_response(msg, response) or ctx.dispatch_streaming(msg, chunks)
  3. 3 Hub wraps as OutboundMessage, attaches RoutingContext
  4. 4 Hub looks up OutboundDispatcher for (platform, bot_id)
  5. 5 dispatcher.enqueue(msg, outbound) — fire-and-forget into per-adapter queue
  6. 6 Worker loop: verify routing → check circuit breaker → retry with backoff (1s/2s/4s)
  7. 7 adapter.send(original_msg, outbound) or adapter.send_streaming(original_msg, chunks)
  8. 8 Platform API delivers to user
Separate Audio Path

InboundAudioBus is a completely separate bus from InboundBus. Audio clips (voice messages) are routed through inbound_audio_bus → Hub._audio_loop() → STT transcription → then re-injected as text. Audio responses go via dispatch_audio_stream() → OutboundDispatcher.enqueue_audio_stream() → adapter.render_audio_stream().

Hub
Central orchestrator. Owns the adapter registry, agent registry, binding table, pool map, and all dispatch methods. Single consumer of InboundBus.
core/hub.py (1,366 LOC)
InboundBus
Per-platform bounded queues + staging queue. Feeder tasks drain platform queues into staging. Hub.run() is the sole staging consumer.
core/inbound_bus.py (129 LOC)
MessagePipeline
Routing logic: validate platform → rate limit → resolve binding (RoutingKey→Binding) → check commands → submit to pool. Currently nested in hub.py.
core/hub.py (nested, ~270 LOC) → target: core/message_pipeline.py
Pool
One per conversation scope (per chat/thread/channel — NOT per user). Holds history, session ID, debouncer, cancel-in-flight logic, processing engine.
core/pool.py (493 LOC)
AgentBase
Stateless singleton. Immutable config (persona, plugins, memory namespace). All mutable state in Pool. Subclasses: AnthropicAgent (SDK), CliAgent (subprocess via CliPool).
core/agent.py (1,218 LOC)
OutboundDispatcher
Per-adapter outbound queue. Sole owner of circuit breaker check. Retry with exponential backoff (1s/2s/4s). Routing verification. User error notification.
core/outbound_dispatcher.py (414 LOC)
CommandRouter
Routes commands: built-ins (/help, /config, /clear), session commands (/add, /summarize — isolated LLM), plugin commands, passthrough to pool.
core/command_router.py (500 LOC)
MemoryManager
L0 working (context window) + L1 TurnStore (raw turns to SQLite) + L3 semantic (FTS5 + embeddings). Recall, concept/preference upsert, session flush.

Protocol interface. Telegram (aiogram v3) and Discord (discord.py v2). Normalizes inbound → InboundMessage, renders outbound via send()/send_streaming()/render_audio().

§3 Key Data Types in Flight

Text message lifecycle
InboundMessage → RoutingKey → Binding → Pool → Agent.process() → Response | AsyncIterator[str] → OutboundMessage + RoutingContext
Routing types
RoutingKey(platform, bot_id, scope_id) — lookup key in binding table
Binding(agent_name, pool_id) — resolution result
RoutingContext(platform, bot_id, scope_id, thread_id, platform_meta) — carried on every outbound for dispatcher verification
Outbound envelope types
OutboundMessage — text + optional buttons + routing context
OutboundAudio — pre-synthesized voice note (bytes + mime)
OutboundAudioChunk — streaming TTS chunk
OutboundAttachment — file/image/video delivery
Scope != User — Pools are per conversation scope

Pools are keyed by (platform, bot_id, scope_id). Scope is extracted by the adapter: Telegram DM/group → chat:{chat_id}, forum → chat:{chat_id}:topic:{topic_id}. Discord → channel:{channel_id} or thread:{thread_id}. Two different users in the same group chat share one Pool. Two bots in the same channel each have separate Pools (different bot_id).

§5 Key Architectural Insights

Two Completely Separate Paths
InboundBus (adapters → Hub) and OutboundDispatcher (Hub → adapters) are independent queuing systems. They share no state, no queue, no bus. LLM responses never touch InboundBus.
Per-Scope, Not Per-User
Pools are keyed by (platform, bot_id, scope_id). In a group chat, all users share one Pool. Two bots in the same channel have separate Pools (different bot_id in the key).
Dispatcher Owns Circuit Breaker
The OutboundDispatcher is the sole owner of the platform circuit breaker. Adapters never check or record CB state — the dispatcher worker loop handles it, with exponential backoff (1s → 2s → 4s).
Stateless Agent, Stateful Pool
Agents are immutable singletons — same instance serves all Pools. All mutable state (history, session, debouncer) lives in the Pool. Zero race conditions by design.
Hub Is a God Object (for now)
At 1,366 lines, Hub mixes orchestration, audio pipeline, rate limiting, pool lifecycle, and dispatch. #294 splits it into Hub + MessagePipeline + AudioPipeline + PoolManager. Complementary to #282 (process-level split).
Audio Has Its Own Everything
Separate InboundAudioBus, separate Hub._audio_loop(), separate dispatch methods (dispatch_audio, dispatch_audio_stream, dispatch_voice_stream). Clean extraction target for #294.
