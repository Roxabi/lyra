## Status

Accepted (2026-04-24)

Supersedes: ADR-053 Decisions 4 and 5 (UID pinning + secrets file location).
ADR-053 Decisions 1, 2, 3 (tier strategy, naming normalization, image digest pinning) remain in force unchanged.

## Context

ADR-053 was accepted on 2026-04-24 and defined Quadlet topology assuming bot tokens came from scoped env files at `~/.lyra/env/{hub,telegram,discord}.env`. Issue #417 (March 2026, "auth.db split") changed the credential model: tokens are now Fernet-encrypted in `~/.lyra/config.db` (table `bot_secrets`) and decrypted at runtime via `~/.lyra/keyring.key` using `LyraKeyring`.

The running adapter entry point (`src/lyra/bootstrap/standalone/adapter_standalone.py`, lines 68ŌĆō82 and 171ŌĆō184) opens `config.db` + `LyraKeyring` directly ŌĆö it does not read `TELEGRAM_TOKEN` or `DISCORD_TOKEN` from the environment. The `SystemExit` on missing env var in `src/lyra/adapters/{telegram,discord}/*_config.py` is a legacy (non-standalone) code path not exercised by the production daemons.

Two bugs existed in the ADR-053 Quadlet units:

**Bug 1 ŌĆö named volume instead of bind-mount.** `lyra-data.volume` had no `Device=` directive. Podman created an empty named volume at first use. Host `~/.lyra/config.db` and `~/.lyra/keyring.key` were never exposed inside the container. Credential lookup failed at startup: `sys.exit("No Telegram adapters started ŌĆö check credentials")`.

**Bug 2 ŌĆö UID 1500 cannot read host files owned by UID 1000.** ADR-053 Decision 4 pinned `User=lyra` (UID 1500, set in `Dockerfile` line 21). Host `~/.lyra/` is owned by `mickael` (UID 1000). In rootless Podman, the user namespace mapping does not bridge this gap without `UserNS=keep-id`. Container processes received `EACCES` on all `~/.lyra/*` reads.

These two bugs are coupled: fixing only the volume without fixing the UID leaves the credentials still inaccessible; fixing only the UID without fixing the volume leaves the files still unmounted.

## Decisions

### 1. Data volume = bind-mount of `%h/.lyra`

`lyra-data.volume` is rewritten as a bind-mount:

```ini
[Volume]
Type=none
Device=%h/.lyra
Options=bind
```

`%h` expands to `$HOME` in Quadlet unit files (`/home/mickael` on MŌéü). This exposes `config.db`, `keyring.key`, `auth.db`, all agent directories, and every other runtime file in `~/.lyra/` to the containers without any one-time copy or migration step. Existing prod state is preserved exactly as-is.

Rationale: every credential and store that Lyra needs at runtime already lives in `~/.lyra/`. A named volume would require a provisioning step (`podman volume import` or a first-run copy) that adds operational complexity and failure modes. The bind-mount is the correct primitive for a single-machine deployment where the host directory is the source of truth.

### 2. Drop `User=lyra`; add `UserNS=keep-id` to all three `.container` files

`User=lyra` (UID 1500) is removed from `lyra-hub.container`, `lyra-telegram.container`, and `lyra-discord.container`. `UserNS=keep-id` is added in its place in the `[Container]` security block.

`UserNS=keep-id` maps the container's effective UID to the host user's UID (1000, `mickael`). The container process sees itself as UID 1000 inside the container namespace and owns all files in `~/.lyra/` without permission errors.

**What is retained:** `NoNewPrivileges=true`, `ReadOnly=true`, `DropCapability=all` ŌĆö the defense-in-depth posture from ADR-053 is unchanged. Rootless Podman confinement is intact.

**What is changed:** the "explicit non-root user inside container" layer (UID 1500) is superseded by the host-UID-matching approach. The Dockerfile `USER lyra` instruction (line 29) and `useradd -u 1500` (line 21) are left in place ŌĆö the unit-level `UserNS=keep-id` overrides the image-declared user at container startup.

**Security posture:** running as host UID 1000 inside a rootless container is the standard rootless Podman pattern. The container still cannot escalate privileges (`NoNewPrivileges=true`), cannot write outside its bind-mounts (`ReadOnly=true`), and has no Linux capabilities (`DropCapability=all`). The only capability lost relative to ADR-053 is the container-internal UID != host UID isolation, which was causing the actual production failure.

### 3. Adapter data mount changed to `rw` (`:z` without `:ro`)

`lyra-telegram.container` and `lyra-discord.container` previously mounted `lyra-data.volume:/home/lyra/.lyra:ro,z`. This is changed to `:z` (default read-write).

Adapters write to files inside `~/.lyra/`: Discord adapter writes `discord.db` (thread mapping), both adapters write to `message_index.db`, and auth pairings are written to `auth.db`. Read-only mount caused write failures at runtime. Hub mount was already writable and is unchanged.

### 4. Env files stripped of bot tokens; `telegram.env.example` and `discord.env.example` deleted

Bot tokens are no longer sourced from env files. Since #417, they come from `config.db` via `LyraKeyring`.

**`telegram.env.example` and `discord.env.example` are deleted.** The corresponding `EnvironmentFile=` lines are removed from `lyra-telegram.container` and `lyra-discord.container`. An empty or comment-only env file is a documentation smell ŌĆö it suggests vars that are no longer read by the process. Deleting them removes the false signal and eliminates the `~/.lyra/env/telegram.env` / `discord.env` provisioning step from the deploy checklist.

**`hub.env.example` is retained and simplified.** It now contains only `LYRA_HEALTH_SECRET` and `LYRA_HEALTH_PORT`. Hub still needs `LYRA_HEALTH_SECRET` at runtime (bearer token for the `/health` endpoint), and there is no other mechanism to inject it. The `EnvironmentFile=` line in `lyra-hub.container` is preserved.

Optional vars (`LYRA_LOG_DIR`, `ROXABI_VAULT_DIR`) are removed from `hub.env.example` ŌĆö they were already set inline in the `.container` file; duplicating them in the env file created confusion about the authoritative source.

### 5. Keyring file exposure

`~/.lyra/keyring.key` is now accessible inside all three containers via the bind-mount (Decision 1). No special handling is required beyond ensuring the file exists on MŌéü before starting units.

**Rotation procedure:** rotating `~/.lyra/keyring.key` requires restarting all three containers (`lyra-hub`, `lyra-telegram`, `lyra-discord`) to re-initialize `LyraKeyring` with the new key. Containers hold the decrypted key in memory; a restart is the only way to force re-read.

## Alternatives Considered

### A. Named volume + `podman unshare chown` on first deploy

Keep the named volume from ADR-053 and add a provisioning step: `podman unshare chown -R 1500:1500 ~/.lyra/` to make host files owned by container UID 1500, then `podman volume import` to seed the named volume.

Rejected. This requires a destructive `chown` on `~/.lyra/` which changes ownership of files that other host tools (supervisord-managed lyra processes, CLI tools) also read as UID 1000. Rollback requires a second `chown`. The bind-mount (Decision 1) avoids all of this with no host-side mutation.

### B. `UserNS=auto:uidmapping=1000:1500`

Explicitly map container UID 1500 to host UID 1000 using `UserNS=auto` with a custom UID mapping.

Rejected. `UserNS=auto` with custom mapping requires entries in `/etc/subuid` / `/etc/subgid` and the exact Podman syntax varies by version. `UserNS=keep-id` is the standard rootless Podman idiom for this exact case (container runs as caller's UID) and is well-documented in `podman-run(1)`. Simpler, less error-prone.

### C. Dropping Quadlet and returning to supervisord

Supervisord processes run as the host user (UID 1000) by default and have native access to `~/.lyra/*`. No volume or UID issue.

Rejected. ADR-053 Decision 1 (Quadlet as strategic ecosystem target) stands. The two bugs are fixable at the unit level without abandoning the container strategy.

### D. Keep `telegram.env.example` / `discord.env.example` as empty files with comments

Retain the example files with a comment explaining tokens come from `config.db`.

Rejected. An env file with only a comment and no `KEY=value` pairs has no operational value ŌĆö it cannot be diffed against a live file to verify configuration. Keeping the `EnvironmentFile=` line in the container unit while pointing to a file that contains no variables is also a source of confusion ("why does this exist?"). Deleting the files and removing the `EnvironmentFile=` lines is cleaner and removes one provisioning step.

## Consequences

### Positive

- Containers can now read `config.db` and `keyring.key` ŌĆö credential lookup works.
- Adapter write operations (`discord.db`, `message_index.db`, `auth.db`) succeed.
- No provisioning step for `telegram.env` / `discord.env` on new deployments.
- `hub.env.example` is minimal and accurately reflects what the running hub actually reads from the environment.

### Negative

- Containers run as host UID 1000 rather than container-internal UID 1500. The "container user != host user" isolation layer is removed. Accepted trade-off: rootless Podman + `NoNewPrivileges` + `DropCapability=all` provides confinement regardless of UID matching.
- All three containers must be restarted after `~/.lyra/keyring.key` rotation.
- MŌéü operators who created `~/.lyra/env/telegram.env` and `~/.lyra/env/discord.env` can delete those files ŌĆö the units no longer reference them. (Not harmful if left in place; just dead files.)
