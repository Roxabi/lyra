Composants

Channel Adapters
TelegramAdapter · DiscordAdapter · CLIAdapter · HTTPAdapter
  • inbound_queue: Queue[Message]
  • outbound_queue: Queue[Response]
  • normalize(raw_event) → Message
  • format(Response) → channel output

Chaque adapter est indépendant. Normalise les événements entrants en Message standard et formate les Response sortants en format canal (Markdown, clavier inline, voix, fichier).

Bus Central
asyncio.Queue — orchestration du routage
  • inbound: Queue[Message]
  • outbound: Queue[Response]
  • router: Router

Le Bus n'est pas un simple relay — il orchestre le routage via le Router. Maintient la correspondance session → pool pour la continuité de contexte.

Router
Binding table · Session continuity
  • bindings: (channel, user_id) → (agent_id, pool_id)
  • default_binding(channel) → pool
  • session_lookup(session_id) → pool
  • register(channel, user_id, agent_id)

Même session_id → même agent pool. Garantit la continuité de contexte entre les messages d'une même conversation.

Agent Identity
AgentIdentity — profil complet de l'agent
  • agent_id: str
  • name: str
  • personality: str
  • system_prompt: str
  • tools: list[Tool]
  • memory_scope: MemoryScope
  • llm_config: LLMConfig
  • skills: list[str]

asyncio.Lock par pool_id — empêche les réponses concurrentes dans une même session. Plusieurs pools tournent en parallèle pour des users/sessions différents. Sub-agents possibles via pattern ReAct.

LLM Backends
AnthropicWrapper · OllamaWrapper · OpenAIWrapper
  • provider: anthropic | ollama | openai
  • model: str
  • endpoint: str | None
  • temperature: float
  • max_tokens: int
  • streaming: bool

Anthropic API = défaut (cloud). Ollama = Machine 2 FastAPI /llm (fallback / offline). OpenAI = futur.

CLI Wrappers
VoiceCLIWrapper · EmbeddingCLIWrapper
  • subprocess + CWD management
  • env vars injection
  • stdout/stderr parsing
  • voicecli --chunked

⚠ CWD critique : voicecli cherche voicecli.toml dans le répertoire courant. Doit toujours être invoqué depuis ~/projects/voiceCLI/.

🔄 Agent → ToolForwardMessage  →  Bus outbound (type: tool_forward)  →  Adapter format (keyboard / prompt)  →  User répond  →  Message(tool_result: ToolResult)  →  Bus → Agent (même session_id)  →  Agent reprend l'exécution
Invariant clé : ce n'est PAS une nouvelle session — même session_id, même pool, asyncio.Lock maintenu pendant l'attente. L'agent est suspendu, pas terminé.

  B->>A: outbound Queue
  A->>U: final answer — Lock released

Système de mémoire — 5 niveaux

L1 1
Working Memory
Python dict · in-process
Context window actuel, tool calls en cours.
TTL: single request
L2 2
Session Memory
asyncio-safe in-memory store · keyed by session_id
Messages récents, état session, préférences user pour la session.
TTL: session lifetime
L3 3
Episodic Memory
Markdown daté · ~/.lyra/memory/episodic/YYYY-MM-DD/
Événements clés, décisions, résumés de sessions passées. Écrit en fin de session.
TTL: permanent (per user)
L4 4
Semantic Memory
SQLite + BM25 + sqlite-vec · ~/.lyra/memory/semantic.db
Faits, connaissances apprises, recherche hybride cross-session. Requête à chaque tour agent.
TTL: permanent (global)
L5 5
Procedural Memory
YAML/TOML skill definitions + Python · skills/
Graphes de skills, patterns ReAct, tool chains. Capacités de l'agent.
TTL: permanent (codebase)
